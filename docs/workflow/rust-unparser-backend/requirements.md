# Rust Unparser Backend

## Original request

> "fltk generates both parsers and unparsers. The parsers recently gained the ability to generate a Rust backend with PyO3 wrappers. Unparsers are still Python-only. I'd like to add a Rust backend option for unparsers as well. The pattern is the same: We generate a core Rust layer that does not have *any* python dependencies. It can only work with the Rust-based CST. Then we generate an optional PyO3 wrapper that lets you call the generated unparser from Python -- but critically, only with the Python-wrapped Rust CST, not with the pure-python CST. So you have to pair your rust unparser with a rust parser."

## What the user is asking for

FLTK is a parser-generator toolkit. Given a grammar file (`.fltkg`), it can generate two things: a **parser** (turns source text into a Concrete Syntax Tree) and an **unparser** (turns a CST back into formatted source text). Both generators currently emit Python code. The parser generator recently gained a second backend that emits Rust code with optional PyO3 wrappers so the Rust parser can be called from Python. The user wants the same treatment for the unparser generator: a new Rust backend that emits a Rust unparser for a given grammar, following the same two-layer architecture the parser backend uses.

The Rust unparser must faithfully reproduce the Python unparser's formatting behavior. The Python unparser does not produce strings directly -- it produces a `Doc` combinator tree, a structured intermediate representation of formatted output. That tree then passes through a spacing-resolution pass and a Wadler-Lindig pretty-printing renderer to become a final string. This formatting pipeline is the core value of the unparser: it provides width-aware line breaking, indentation nesting, grouping, joining, trivia preservation, and configurable spacing control. The Rust unparser must reproduce all of this formatting behavior -- producing strings directly without the formatting pipeline is not acceptable. The Python implementation's `Doc` combinator tree, spacing resolution, and Wadler-Lindig renderer are the reference for what the Rust side must achieve; the exact internal representation is a design decision.

### The two-layer architecture

The Rust parser backend established a pattern that the Rust unparser backend should follow:

1. **Core Rust layer.** A pure-Rust module with zero Python dependencies. It operates entirely on the Rust CST types generated by the Rust CST generator (`gsm2tree_rs.py`). This layer is compiled unconditionally and can be used in pure-Rust applications with no Python runtime.

2. **PyO3 wrapper layer.** An optional layer, gated behind a `python` Cargo feature. It wraps the core Rust unparser so it can be called from Python. Critically, the PyO3 wrapper accepts only the Python-wrapped Rust CST handles (the `Py{ClassName}` types), not the pure-Python CST classes. This means a consumer who wants a Rust unparser callable from Python must also use the Rust parser backend for parsing -- the Rust unparser and pure-Python parser are not mix-and-matchable.

### What a generated Rust unparser does

The Python unparser generator (`gsm2unparser.py`) produces a Python class called `Unparser` with one public method per grammar rule (`unparse_{rule_name}(node)`). Each method walks the CST node's children list, pattern-matching on labels and child types, and builds up a tree of formatting combinators (the `Doc` type hierarchy defined in `fltk/unparse/combinators.py`). The `Doc` tree includes text content, line breaks, indentation nesting, grouping (for Wadler-Lindig pretty-printing), and spacing control nodes. A separate resolution pass (`resolve_spacing_specs`) and then a renderer turn the `Doc` tree into a final string.

The Rust unparser must be the Rust analog of this full pipeline. It has two parts:

- **Generated code** (per-grammar): Rust code that walks the Rust CST and produces formatting output from it. The core tree-walking and pattern-matching logic -- visiting each node's children, checking labels, dispatching to sub-rules -- maps naturally to Rust.

- **Runtime library**: Grammar-independent Rust code covering the formatting pipeline -- combinator types (or equivalent), accumulator/builder, spacing resolution, and rendering. This is the shared runtime that all generated Rust unparsers link against, analogous to how `fltk-parser-core` (under `crates/fltk-parser-core/`) provides grammar-independent parser runtime types. The exact crate name, boundaries, and internal decomposition are design decisions.

## What matters in the codebase

### The Rust parser backend (the pattern to follow)

The file `fltk/fegen/gsm2parser_rs.py` contains `RustParserGenerator`, which is the direct analog for what the Rust unparser generator needs to be. It is a Python class whose `generate()` method returns a complete `.rs` file as a string. The generated Rust file has two sections:

- A pure-Rust section containing `pub struct Parser` and per-rule parsing methods like `parse_{rule_name}()`, `parse_{rule_name}__alt{N}__item{M}()`, etc.
- A `#[cfg(feature = "python")]` gated section containing `PyParser` with PyO3 `#[pymethods]` that unwrap Python handles and delegate to the core parser.

The parser backend depends on a shared runtime crate, `fltk-parser-core` (under `crates/fltk-parser-core/`), which provides grammar-independent types like `PackratState`, `ErrorTracker`, `ApplyResult`, and `Cache`. The analogous `fltk-unparser-core` crate would live under `crates/` and provide the grammar-independent `Doc` combinator types, the accumulator/builder, the spacing-resolution logic, and the renderer.

The CST backend lives in `fltk/fegen/gsm2tree_rs.py` (`RustCstGenerator`). It generates Rust structs where each grammar rule gets a struct with `children: Vec<(Option<{ClassName}Label>, {ClassName}Child)>`. The child value is an enum whose variants are `Span(Span)` for terminals and `{ChildClass}(Shared<{ChildClass}>)` for rule references. `Shared<T>` is `Arc<RwLock<T>>` from `fltk-cst-core`. The Rust unparser will traverse these structures rather than the Python CST's `(label, value)` tuples.

### The Python unparser generator (the logic to port)

`fltk/unparse/gsm2unparser.py` contains `UnparserGenerator`, which produces a Python `Unparser` class. The generated class has:

- A `terminals: str` constructor parameter (the original source text, used to extract text from spans).
- One public method per grammar rule: `unparse_{rule_name}(self, node: {ClassName}) -> Optional[UnparseResult]`.
- Internal helper methods for alternatives, items, and quantified loops: `unparse_{rule_name}__alt{N}(...)`, `unparse_{rule_name}__alt{N}__item{M}(...)`, etc.
- Utility methods like `_has_preservable_trivia()`, `_count_newlines()`, `_count_newlines_in_trivia()`.

The core unparsing algorithm walks `node.children` positionally (using an integer `pos` index), checks labels against the label enum, dispatches on child type (`isinstance` for rule nodes, `is_span` for terminals), extracts terminal text via `extract_span_text(span, terminals)`, and accumulates output through the `DocAccumulator`.

A key difference between the Python and Rust CST access patterns: In Python, the unparser uses `extract_span_text(span, terminals)` which slices the `terminals` string at `span.start:span.end`. In Rust, spans carry their own source text via `Arc<SourceTextInner>` and expose `span.text() -> Option<String>`. The Rust unparser does not need a `terminals` parameter.

### The formatting pipeline (Doc combinators, resolution, rendering)

The Python unparser's formatting pipeline has three stages, all of which need Rust equivalents:

1. **Unparsing** (the generated `Unparser` class): CST -> `Doc` tree (via `DocAccumulator`)
2. **Spacing resolution** (`resolve_spacing_specs` in `fltk/unparse/resolve_specs.py`): Resolves `AfterSpec`, `BeforeSpec`, and `SeparatorSpec` control nodes into concrete spacing, handling trivia preservation, spec merging, and collapsing redundant line breaks.
3. **Rendering** (`Renderer` in `fltk/unparse/renderer.py`): Implements the Wadler-Lindig pretty-printing algorithm to turn the `Doc` tree into a final string, making flat-vs-break decisions for `Group` nodes based on line width.

The `Doc` hierarchy has roughly 15 types: `Text`, `Comment`, `Line`, `Nbsp`, `SoftLine`, `HardLine`, `Group`, `Nest`, `Concat`, `Join`, `Nil`, `AfterSpec`, `BeforeSpec`, `SeparatorSpec`. The `DocAccumulator` is an immutable linked-list builder with push/pop operations for nesting Group, Nest, and Join contexts.

None of these types currently exist in Rust. The Rust unparser runtime will need equivalent functionality -- the generated Rust unparser code will build formatting structures, and the runtime will provide the spacing resolution and rendering that produce the final string.

### Build system integration points

The CLI entry point for Rust code generation is `fltk/fegen/genparser.py`, which has subcommands `gen-rust-cst`, `gen-rust-parser`, and `gen-rust-lib`. The `Makefile` has corresponding targets and a master `gencode` target that runs them all. The Rust unparser will need analogous CLI and Makefile integration.

The lib.rs generator (`fltk/fegen/gsm2lib_rs.py`) uses a `LibSpec` with a `Submodule` list that currently covers `cst` and `parser`. The unparser will need to be wired into this mechanism so that downstream consumer crates can include it in their generated `lib.rs`.

The Cargo workspace (`Cargo.toml`) currently has members `"."`, `"crates/fltk-cst-core"`, `"crates/fltk-parser-core"`. The unparser runtime will need to fit into this workspace structure.

### Test infrastructure

Existing unparser tests (`fltk/unparse/test_*.py`) exercise the Python unparser with Python-backend CSTs only. The Rust parser backend has parity tests (`tests/test_rust_parser_parity_fixture.py`) that run both backends on the same corpus and compare results. An analogous parity-testing strategy applies to the Rust unparser: given the same input text, parse it with both backends, unparse with both backends, and confirm the formatted output matches.

There is a Rust parser test fixture at `tests/rust_parser_fixture/` that is an independent Cargo workspace with its own `Cargo.toml`. The Rust unparser will need a similar testing setup.

## correctness-1

- **File:line:** `fltk/unparse/gsm2unparser_rs.py:341` (Text) and `:343-344` (Concat), in `_doc_to_rust_expr`.
- **What's wrong:** The helper emits `Doc::text("...")` and `Doc::concat(vec![...])`. In the runtime crate `fltk-unparser-core`, `text` and `concat` are *free module functions* (`crates/fltk-unparser-core/src/doc.rs:134`, `:200`, re-exported in `lib.rs:22-24`); there is **no** inherent `impl Doc` — the only `impl` on `Doc` is `impl Drop` (`doc.rs:67`). Rust does not allow calling a free function via `Type::func` syntax, so `Doc::text(...)` / `Doc::concat(...)` resolve to nothing (E0599: "no function or associated item named `text`/`concat` found for enum `Doc`"). The enum variants are `Text(String)` / `Concat(Vec<Rc<Doc>>)` — capitalized and with different arg types — so the `Doc::`-qualified lowercase names match neither a variant nor an associated fn. (The crate's own code/tests construct these via the free functions `text(...)`/`concat(...)`, e.g. `accumulator.rs:251,270`, `render.rs:288,298`.)
- **Why:** The faithful Python analog `_doc_to_combinator_expr` (`gsm2unparser.py:419,423`) emits *module*-qualified `combinators.text(...)` / `combinators.concat(...)`, i.e. free functions on the combinators module — not class-qualified. The Rust port mistranslated "module-qualified free function" into "type-associated function" (`Doc::text`) instead of a plain free-function call. The singleton/struct-variant cases on the surrounding lines are correct because `Doc::Nil`/`Doc::Nbsp`/`Doc::Line`/`Doc::SoftLine`/`Doc::HardLine { .. }` really are enum variants; only the two helper-backed cases (`text`, `concat`) are wrong.
- **Consequence:** Any grammar + `FormatterConfig` that routes a `Text` or `Concat` spacing `Doc` through `_doc_to_rust_expr` produces uncompilable Rust. This is directly reachable: a rule-level (or, once item-level anchors land, item-level) `JOIN_BEGIN` separator such as `join ... by ","` yields a `Text(",")` separator that flows JOIN_BEGIN → `push_join(self._doc_to_rust_expr(op.separator))` (`gsm2unparser_rs.py:203`), emitting `acc.push_join(Doc::text(","));`. The generated `.rs` then fails to build. The fixture/parity tests use the default config (no join separator) so they will not surface it, but design §2.6's planned join-from-to fixture will. Cross-backend behavioral parity is also violated: the Python backend generates working code for the same config, the Rust backend generates code that cannot compile.
- **Suggested fix:** Emit the free functions instead of `Doc::`-qualified names — `text("...")` and `concat(vec![...])` — and add `text`, `concat` (they are already `pub use`-exported from `fltk_unparser_core`) to the generated header `use` list in `_gen_header` (currently imports only `Doc`, so even unqualified `text(...)` would not resolve without the added import). Tests assert these output strings literally (`test_doc_to_rust_expr_text_is_escaped`, `test_doc_to_rust_expr_concat_recurses`, `test_doc_to_rust_expr_nested_concat`) and will need updating; consider a parity/compile check on a join-separator config so string-only assertions stop masking this.

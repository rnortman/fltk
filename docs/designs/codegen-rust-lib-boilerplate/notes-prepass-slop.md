## slop-1

**File:** `fltk/fegen/gsm2lib_rs.py:208–210`

**Quote:**
```python
# Rust identifier pattern: start with letter or underscore, followed by
# alphanumerics, underscores, or a leading underscore-only prefix (like _native).
# Allows identifiers starting with underscore (e.g. _native).
```

**What's wrong:** The comment restates what the regex `^[A-Za-z_][A-Za-z0-9_]*$` already says, and includes a self-contradictory sentence ("followed by … or a leading underscore-only prefix"). The leading-underscore case is already covered by `[A-Za-z_]` at position one and needs no separate callout. Reads like a draft note left in.

**Consequence:** An LLM talking to itself mid-write. Signals the comment wasn't reviewed before commit.

**Suggested fix:** Replace with one line: `# Standard Rust identifier: letter or underscore, then alphanumerics or underscores.`

## slop-2

**File:** `fltk/fegen/gsm2lib_rs.py:322`

**Quote:**
```python
lines.append("// Generated node code no longer reads crate::UNKNOWN_SPAN directly; each generated")
lines.append("// extension caches the sentinel via its own PyOnceLock (UNKNOWN_SPAN_CACHE) by importing")
lines.append("// `fltk._native.UnknownSpan` at runtime.  The static is retained here so that any")
lines.append("// external code that might hold a reference to the crate-internal value still works;")
lines.append("// it is not dead in the sense of the module being broken, but no generated code reads it.")
```

**What's wrong:** This is a five-line explanation of *why the static is retained despite being unused* — the kind of note an author writes to themselves mid-refactor. In a *generator*, these lines are emitted verbatim into every generated lib.rs for fltk._native. The justification is only true for the current state of the codebase; if the static is ever removed the generator will still emit this misleading rationale.

**Consequence:** Design-rationale prose embedded in generated output, visible to any downstream consumer who reads their own generated file. Looks unfinished.

**Suggested fix:** Extract the rationale into a comment in `gsm2lib_rs.py` above the `if spec.unknown_span_static:` block, not into the emitted code. The emitted comment (if any) can be a one-liner: `// UNKNOWN_SPAN is set at module init and exposed as UnknownSpan.`

## slop-4

**File:** `src/lib.rs` (fltk repo)

**Quote (removed lines):**
```rust
-    // PoC grammar classes in a submodule (avoids clobbering canonical Span/SourceText
-    // if a PoC rule name collides, and keeps top-level clean).
     register_submodule(m, "poc_cst", cst_generated::register_classes)?;
-
-    // Fegen grammar classes in a submodule (both grammars produce Identifier, Items, Trivia).
     register_submodule(m, "fegen_cst", cst_fegen::register_classes)?;
-
     Ok(())
```

**What's wrong:** Not a slop issue with the new code — the removal of explanatory comments is fine when moving to generated output. However the `src/lib.rs` in the diff is still a hand-maintained file (not yet a generated artifact): the diff shows it being edited by hand to reorder mod declarations (`cst_fegen` / `cst_generated` swap) and strip comments. If `src/lib.rs` is now supposed to be generated by `gen-rust-native-lib`, the file should carry a `// @generated` header. Without it there is no indication the file must not be hand-edited.

**Consequence:** Future maintainers will hand-edit the file not knowing it is a generated artifact, then the next `make gencode` will overwrite their changes silently.

**Suggested fix:** Add `// @generated — do not edit by hand. Regenerate with: make gencode` as the first line of `src/lib.rs`, or ensure `RustLibGenerator.generate()` emits it when producing the native lib.

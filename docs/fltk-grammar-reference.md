# FLTK Grammar Language (`.fltkg`) Reference

This document is the canonical description of the FLTK grammar language: its syntax,
semantics, and constraints. It is written for grammar authors and maintainers who need
ground truth. Every statement here is grounded in the FLTK source; citations are given as
`file:line` so that any claim can be checked against the code.

A `.fltkg` grammar is compiled into two kinds of artifact:

- **CST node classes** (the public-API data types your application reads), generated by
  `fltk/fegen/gsm2tree.py` (Python) and `fltk/fegen/gsm2tree_rs.py` (Rust).
- A **parser** (a packrat / memoized recursive-descent parser that supports left
  recursion), generated by `fltk/fegen/gsm2parser.py` (Python) and
  `fltk/fegen/gsm2parser_rs.py` (Rust).

Both backends are intended to be behaviorally equivalent so that the Rust backend is a
near-drop-in replacement for the Python backend. The places where they diverge are called
out explicitly (recursion-depth limits, the regex engine boundary, negative-position
handling).

The compilation pipeline is: `.fltkg` text → CST (parsed by the generated FLTK grammar
parser, `fltk/fegen/fltk_parser.py`) → Grammar Semantic Model (GSM) via `Cst2Gsm`
(`fltk/fegen/fltk2gsm.py`) → validation/classification transforms (`fltk/fegen/gsm.py`) →
CST node classes and parser (the two generators above).

---

## 1. The self-describing grammar

The authoritative syntax specification is `fltk/fegen/fegen.fltkg`: a `.fltkg` grammar that
describes `.fltkg` grammars. It is simultaneously the spec and a worked example of every
construct. Reproduced verbatim (`fegen.fltkg:1-22`):

```
// FLTK grammar for FLTK grammars
grammar := , rule+ ;
rule := name:identifier , ":=" , alternatives , ";" , ;
alternatives := items , ( "|" , items , )* ;
items :=
  ( no_ws:"." | ws_allowed:"," | ws_required:":" )? ,
  item ,
  ( ( no_ws:"." | ws_allowed:"," | ws_required:":" ) , item , )* ,
  ( no_ws:"." | ws_allowed:"," | ws_required:":" )? ,
  ;
item := ( label:identifier . ":" )? . disposition? . term . quantifier? , ;
term :=
  identifier | literal | "/" . regex:raw_string . "/" | "(" , alternatives , ")" ;
disposition := suppress:"%" | include:"$" | inline:"!" ;
quantifier := optional:"?" | one_or_more:"+" | zero_or_more:"*" ;
identifier := name:/[_a-z][_a-z0-9]*/ ;
raw_string := value:/([^\/\n\\]|\\.)+/ ;
literal := value:/("([^"\n\\]|\\.)+"|'([^'\n\\]|\\.)+')/ ;
_trivia := ( line_comment | line_comment? : | block_comment )+ ;
line_comment := prefix:"//" . content:/[^\n]*/ . "\n" ;
block_comment := start:"/*" . content:/(?:[^*]|\*+[^\/\*])*/ . end:/\*+\// ;
```

`fegen.fltkg` is ground truth for what the toolchain currently parses. Two other grammar
files exist in the tree and should **not** be treated as the spec:

- `fltk/fegen/bootstrap.fltkg` is a slightly older, near-identical variant (no
  `ws_required:":"` third separator; a plainer `_trivia`; literal rather than regex
  `block_comment` end). It is used to bootstrap the toolchain.
- `fltk/fegen/fltk.fltkg` is an extended, explicitly-incomplete variant whose own header
  declares it "actually broken and was never completed" (`fltk.fltkg:2`). It sketches
  aspirational constructs (`invocation`, `expression`/`add`, `var`) that correspond to GSM
  types (`gsm.py:267-288`) which are **not wired into the live pipeline** (see §10). Do not
  rely on anything that appears only in `fltk.fltkg`.

## 2. Lexical structure

A grammar is plain UTF-8 text. There are no statement keywords; structure comes entirely
from operators and glyphs:

- `:=` separates a rule name from its body.
- `;` terminates a rule.
- `|` separates alternatives.
- `. , :` are the three item separators (§5).
- `% $ !` are the three dispositions (§6.3).
- `? + *` are the three quantifiers (§6.4).
- `( )` group a sub-expression.
- `" "` / `' '` delimit literals.
- `/ /` delimit regexes.

**Span offsets are codepoint-indexed, not byte-indexed**, in both backends. The Python
backend operates on `str` directly (`terminalsrc.py`); the Rust backend builds a
codepoint-to-byte table at construction and converts match offsets back to codepoint indices
(`terminalsrc.rs:43-44, 60-68, 156-165`). Multibyte literals and regexes are fully supported
(`rust_parser_fixture.fltkg:39-45`, e.g. `arrow := %"→"` and `latin_word := /[À-ÿ]+/`).

### 2.1 Identifiers (rule names and labels)

`identifier := name:/[_a-z][_a-z0-9]*/ ;` (`fegen.fltkg:16`). An identifier is lowercase
snake_case: it must start with `_` or `a-z`, followed by any of `_`, `a-z`, `0-9`. No
uppercase letters, no leading digit. A leading underscore is legal (`_trivia` relies on
this). This constrains the rule and label names you write **in `.fltkg` itself**. A *consumer
grammar* is free to define its own `identifier` rule with any regex it likes (e.g.
`unparsefmt.fltkg:87` uses `/[a-zA-Z_][a-zA-Z0-9_]*/`, allowing uppercase); that is a property
of the consumer's language, not of the `.fltkg` meta-grammar.

### 2.2 Literals (string terminals)

`literal := value:/("([^"\n\\]|\\.)+"|'([^'\n\\]|\\.)+')/ ;` (`fegen.fltkg:18`). A literal
is a double- or single-quoted string. It may not span a newline and may not be empty (the
regex requires at least one character). Backslash escapes are allowed inside. The captured
text is decoded with `ast.literal_eval` (`fltk2gsm.py:166`), so Python string-escape rules
apply (`\n`, `\t`, `\"`, etc.). A literal matches its exact decoded text. Examples: `":="`,
`"|"`, `"//"`, `'group'`.

### 2.3 Regexes

A regex term is written between `/` delimiters; the inner text is a `raw_string`
(`raw_string := value:/([^\/\n\\]|\\.)+/ ;`, `fegen.fltkg:13,17`). The inner text is any
sequence of characters other than an unescaped `/`, a newline, or `\`, with `\.` escapes
permitted. The raw text is stored verbatim as `gsm.Regex.value` (`fltk2gsm.py:168-170`).
Examples: `/[0-9]+/`, `/[_a-z][_a-z0-9]*/`, `/\s+/`. The regex-engine boundary between the
two backends is a hard constraint and is covered in §8.

### 2.4 Comments and trivia

Comments are not built-in lexer syntax; they are defined **as grammar rules** within the
trivia machinery (`fegen.fltkg:19-21`):

- `line_comment := prefix:"//" . content:/[^\n]*/ . "\n" ;` — `//` through end of line.
- `block_comment := start:"/*" . content:/(?:[^*]|\*+[^\/\*])*/ . end:/\*+\// ;` —
  `/* ... */`.

How the `_trivia` rule governs whitespace and comment handling across the whole grammar is
covered in §7.

## 3. Rule definitions and grammar structure

`grammar := , rule+ ;` (`fegen.fltkg:2`): a grammar is one or more rules. The leading `,`
(WS_ALLOWED separator) tolerates leading whitespace and comments.

`rule := name:identifier , ":=" , alternatives , ";" , ;` (`fegen.fltkg:3`): a rule is
`name := <alternatives> ;`. In the GSM this is `gsm.Rule(name, alternatives)`
(`gsm.py:27-40`), where `alternatives` is a `Sequence[Items]` — one `Items` per
`|`-separated alternative. `Grammar.identifiers` is a `name → Rule` map
(`gsm.py:22-24`).

### 3.1 What each rule generates

- **One CST node class**, named `snake_to_upper_camel(rule_name)` (`gsm2tree.py:46-47`).
  E.g. `line_comment` → `LineComment`.
- **One `NodeKind` enum member**, the uppercased class name (`gsm2tree.py:95-97, 134-143`).
  E.g. `LineComment` → `NodeKind.LINECOMMENT`.
- **One memoized parser method** (§9).

### 3.2 Name derivation

`snake_to_upper_camel` (`naming.py:7-22`): lowercase the whole string, split on `_`,
`capitalize()` each segment, and join. Consecutive, leading, and trailing underscores
collapse (`a__b` → `AB`, `_foo_bar` → `FooBar`, `foo_` → `Foo`).

A name consisting only of underscores derives to the empty string and is **rejected**
(`validate_no_underscore_only_names`, `gsm.py:323-345`) for both rule names and labels.
Names like `_foo` are fine, which is why `_trivia` is legal.

The generated node-class names and accessor names are **public API for out-of-tree
consumers** (per `CLAUDE.md`). Renaming a rule or a label is a breaking change for downstream
code. There is no `Node` suffix; the class name is exactly the camel-cased rule name
(`class_name_for_rule_node`, `gsm2tree.py:46-47`).

## 4. Alternation and sub-expressions

`alternatives := items , ( "|" , items , )* ;` (`fegen.fltkg:4`): one or more `items`
sequences separated by `|`. Each alternative is a separate `gsm.Items`
(`fltk2gsm.py:47-48`). A rule body, and the body of a parenthesized sub-expression, are both
lists of alternatives.

**Sub-expressions** are a `term` form: `"(" , alternatives , ")"` (`fegen.fltkg:13`). A
parenthesized group is a nested `alternatives`, i.e. a `Sequence[Items]` term
(`gsm.py:175-181`, `fltk2gsm.py:131-132`). Sub-expressions:

- May contain their own `|` alternation and their own internal separators
  (`rust_parser_fixture.fltkg:59-61`, e.g.
  `grouped := %"(" , (left:num | left:name) , %")"`).
- Carry an **implicit INCLUDE disposition** when no disposition glyph is written
  (`fltk2gsm.py:117-122`: a sub-expression term defaults to INCLUDE).
- Splice their children into the parent node automatically (`inline_to_parent=True`,
  `gsm2parser.py:355-372`; merged via `extend_children`, `gsm2parser.py:803-806`). This
  automatic splicing is unrelated to the `!` INLINE disposition (§6.3), which is **not**
  implemented.

### 4.1 Match resolution: PEG ordered first-match

Alternation is **PEG-style ordered choice**: alternatives are tried in source order and the
**first one that succeeds wins** (`gsm2parser.py:718-732`, a chain of
`if self.<alt_i>(pos): return Success(...)` falling through to `Failure`).

There is **no longest-match arbitration**. A shorter or earlier alternative that succeeds
shadows a longer later one even if the later alternative would have consumed more input.
Order your alternatives most-specific-first.

There is **no cut / commit operator** and **no lookahead predicates** (no PEG `&` / `!`).
The only term kinds are the four in `fegen.fltkg:13`; no syntactic predicate or cut exists in
the GSM or generators (§9.3). Authors cannot explicitly prune the search.

Backtracking is local: once a rule's alternative succeeds at a position, the result is
memoized and is not re-derived there (§9). Backtracking happens **between** alternatives of a
rule, not as a global search across committed rule boundaries.

### 4.2 No precedence or associativity operators

FLTK has **no precedence or associativity declarations**. There is no `%left`, `%right`, or
precedence-level construct anywhere in the GSM or generators. Precedence is encoded
structurally, by stratifying rules (e.g. `expr` / `term` / `factor` layering). Associativity
is encoded by recursion direction: direct left recursion is left-associative (§9.1);
right-associativity is expressed with right recursion (ordinary recursive descent, bounded by
the depth guard in §9.4).

### 4.3 Nil (empty-match) semantics

These definitions are used by the validators:

- A rule is nil if **any** of its alternatives is nil (`gsm.py:59`).
- An `Items` is nil iff its initial separator is nil and every item and trailing separator is
  nil (`gsm.py:96-117`).
- A parenthesized term is nil if **any** of its alternatives is nil (`gsm.py:188-190`).

## 5. Items and separators: whitespace semantics

`items` (`fegen.fltkg:5-10`) is the core sequencing construct: an optional **leading**
separator, a sequence of `item`s each preceded (after the first) by a separator, then an
optional **trailing** separator. At each slot the three separators are mutually exclusive
choices: `no_ws:"."`, `ws_allowed:","`, `ws_required:":"`.

GSM representation: `gsm.Items(items, sep_after, initial_sep)` (`gsm.py:82-93`).
`initial_sep` is the leading separator (default `Separator.NO_WS`, `gsm.py:93`);
`sep_after[i]` is the separator following `items[i]`. A missing trailing separator defaults
to `NO_WS` (`fltk2gsm.py:93-102`). The `Separator` enum is `NO_WS`, `WS_REQUIRED`,
`WS_ALLOWED` (`gsm.py:66-69`).

Separators control whether **trivia** (whitespace and comments, per the `_trivia` rule of §7)
is consumed between adjacent items. The operative semantics, from the parser generator
`_gen_separator_handling` (`gsm2parser.py:620-697`):

| Glyph | `Separator`   | Trivia between the items                                                                 |
|-------|---------------|------------------------------------------------------------------------------------------|
| `.`   | `NO_WS`       | **Not consumed.** Items must be adjacent. The generator emits no code (`gsm2parser.py:642-643`). |
| `,`   | `WS_ALLOWED`  | **Optional.** Trivia is consumed if present; its absence is not a failure.                |
| `:`   | `WS_REQUIRED` | **Required.** Trivia must be present; otherwise the parse fails at that point (`gsm2parser.py:684, 696-697`). |

Separator nil-ness (`gsm.py:71-79`): `NO_WS` and `WS_ALLOWED` are nilable; `WS_REQUIRED` is
**never** nil, because it forces a trivia match and the trivia rule itself cannot be nil
(§7). A leading or trailing `:` therefore makes an `Items` non-nilable (`gsm.py:102-103`).

Common patterns: the trailing `,` seen on most rules in `fegen.fltkg` allows whitespace and
comments before the next construct. Real grammars use all three separators, e.g.
`stmt := lhs:atom : "=" : rhs:atom ;` (`rust_parser_fixture.fltkg:19`, WS_REQUIRED around the
operator) and `from_spec := "from" : ( after:"after" : )? . from_anchor:anchor ;`
(`unparsefmt.fltkg:40`, mixing `:` and `.`).

A rule that uses any whitespace separator (`,` or `:`), anywhere including inside its
sub-expressions, is flagged as trivia-bearing (`rule_has_whitespace_separators`,
`gsm2tree.py:49-67`) and participates in trivia handling.

## 6. Items: labels, dispositions, quantifiers

`item := ( label:identifier . ":" )? . disposition? . term . quantifier? , ;`
(`fegen.fltkg:11`). An item is: an optional `label:` prefix, an optional disposition glyph,
the term, and an optional quantifier. GSM: `gsm.Item(label, disposition, term, quantifier)`
(`gsm.py:120-125`), built by `Cst2Gsm.visit_item` (`fltk2gsm.py:108-128`), which is where the
defaulting rules below are applied.

### 6.1 Terms

`term := identifier | literal | "/" . regex:raw_string . "/" | "(" , alternatives , ")" ;`
(`fegen.fltkg:13`). Four kinds (`fltk2gsm.py:130-140`; GSM `Term` union, `gsm.py:175-181`):

1. **identifier** → `gsm.Identifier`, a reference to another rule. The referenced rule must
   exist; an unknown reference is a hard error both at CST-model build time
   (`gsm2tree.py:614-616`) and at parser build time (`gsm2parser.py:310-311`).
2. **literal** → `gsm.Literal` (decoded exact string).
3. **regex** → `gsm.Regex` (raw pattern, §8).
4. **sub-expression** → `Sequence[Items]` (§4).

Each term's contribution to the CST child model (`gsm2tree.py:612-623`):

- An identifier contributes a child of the referenced rule's node type.
- A literal or regex contributes a `Span` child (§7.1).
- A sub-expression contributes the union of its alternatives' models.

### 6.2 Labels and labeled CST access

A label is an `identifier` followed by `:` before the term (`fegen.fltkg:11`). It names the
child so generated code can fetch it by name.

**Label defaulting** (`fltk2gsm.py:113-116`): if no label is written **and the term is a bare
rule reference**, the label defaults to the rule's own name. So `expr := term , ...`
(`toy.fltkg:1`) auto-labels the `term` child `term`, and `atom := num:num | name:name` could
equally be written `atom := num | name`. Literals, regexes, and sub-expressions are **not**
auto-labeled.

Underscore-only labels are rejected (`gsm.py:305-320`), for the same reason as rule names
(§3.2).

Each label produces, on the node class, a nested `Label` enum member and a **quintet of
accessor methods** (`_emit_label_quintet`, `gsm2tree.py:820-867`):

- `append_<label>(child)` and `extend_<label>(children)` — mutators.
- `children_<label>() -> Iterator[T]` — every child carrying that label.
- `child_<label>() -> T` — exactly one; raises unless exactly one exists
  (`gsm2tree.py:345-357`).
- `maybe_<label>() -> Optional[T]` — zero or one; raises if more than one
  (`gsm2tree.py:358-370`).

Cardinality is enforced **at call time**, not at generation time: the generator does not
restrict which accessor you may call for a given quantifier. Pick the accessor whose
cardinality matches what you expect (§6.4).

A single label may map to multiple types (a "union label"), which widens the accessor's
return annotation to a `typing.Union` and inserts a `typing.cast` in the iterator body
(`gsm2tree.py:91-93, 338, 640-642`). This is pinned by `rust_parser_fixture.fltkg:53`,
`val := item:num | item:name | item:/[!@#$]+/`.

Generic, label-agnostic access also exists on every node:
`children: list[tuple[Optional[Label], child]]` (the raw ordered child list), `child()`
returning the sole `(label, child)` pair (`gsm2tree.py:307-321`), and the generic mutators
`append` / `extend` / `extend_children` / `insert` / `remove_at` / `replace_at` / `clear`
(§7.5).

### 6.3 Dispositions: `%` suppress, `$` include, `!` inline

`disposition := suppress:"%" | include:"$" | inline:"!" ;` (`fegen.fltkg:14`). GSM enum
`Disposition.{SUPPRESS, INCLUDE, INLINE}` (`gsm.py:194-197`). Disposition controls whether
and how a matched term appears in the parent's CST children list.

**Defaulting** when no disposition glyph is written (`fltk2gsm.py:117-122`):

- If the item has a label **or** the term is a sub-expression → **INCLUDE**.
- Otherwise → **SUPPRESS**.

So by default, labeled items and parenthesized groups are kept, while bare unlabeled
literals, regexes, and rule references are dropped. This is why
`rule := name:identifier , ":=" , alternatives , ";" , ;` keeps `name` and `alternatives`
(labeled) but drops the `":="` and `";"` literals.

Effects:

- **`%` SUPPRESS** — the term is parsed (advancing position) but contributes **no child**
  (`gsm2parser.py:802`, the append is gated on `disposition != SUPPRESS`). Used to consume
  required punctuation, e.g. `paren_expr := %"(" , inner:atom , %")"`
  (`rust_parser_fixture.fltkg:16`). Suppression of a sub-expression is **forbidden**: the
  CST generator asserts the suppressed term is not a `Sequence` (`gsm2tree.py:628-630`).
  Suppressed terms create gaps in child span coverage (§7.3).
- **`$` INCLUDE** — the term contributes a child: the referenced rule's node for an
  identifier, or a `Span` for a literal or regex. An unlabeled `$`-literal appears as an
  **unlabeled `Span` child** (`rust_parser_fixture.fltkg:48`,
  `tagged := $"tag" . value:/[a-z]+/`).
- **`!` INLINE — accepted syntactically, but NOT implemented.** The `disposition` rule
  defines `inline:"!"` so the glyph parses, and the CST/tree generator models it
  (`gsm2tree.py:631-636`, splicing the referenced rule's model into the parent with cycle
  detection at `gsm2tree.py:1008-1012`). However, **both parser generators reject it**:

  - Python: `raise NotImplementedError("Inline items not yet supported: {item}")` for any
    `item.disposition == INLINE` (`gsm2parser.py:782-784`).
  - Rust:
    `raise NotImplementedError("INLINE disposition is not supported in Rust parser generation")`
    (`gsm2parser_rs.py:824-826` and `1010-1012`).

  Consequently **`!rule` cannot appear in any grammar that is compiled to a parser.** The
  only uses of `!` in the tree are in `fltk.fltkg` (`:11`, `:34`), the explicitly-broken
  aspirational grammar (§1). The live grammars define the glyph but never apply it to an
  item. To splice a group's children into the parent node, use a sub-expression (§4), which
  is a separate, fully-working mechanism and must not be confused with the `!` disposition.

### 6.4 Quantifiers

`quantifier := optional:"?" | one_or_more:"+" | zero_or_more:"*" ;` (`fegen.fltkg:15`). An
absent quantifier means required (exactly one). GSM `Quantifier` exposes `min()` / `max()`
over an `Arity` enum (`gsm.py:200-264`):

| Glyph  | GSM constant   | min..max  | Meaning                                  |
|--------|----------------|-----------|------------------------------------------|
| (none) | `REQUIRED`     | 1..1      | exactly one (`fltk2gsm.py:124-126` default) |
| `?`    | `NOT_REQUIRED` | 0..1      | optional                                 |
| `+`    | `ONE_OR_MORE`  | 1..many   | one or more                              |
| `*`    | `ZERO_OR_MORE` | 0..many   | zero or more                             |

`is_optional()` is `min == ZERO`; `is_multiple()` is `max == MULTIPLE` (`gsm.py:213-220`). An
item can be nil if its quantifier is optional or its term is nil (`gsm.py:127-129`).

Quantifier matching is **greedy with no give-back**. The `+`/`*` loop consumes as many
repetitions as it can (`gsm2parser.py:505-618`); it never returns matched repetitions to
satisfy a following item. Two guards apply:

- A **per-iteration progress guard** breaks the loop if a match did not advance position
  (`gsm2parser.py:570-577`; the comparison uses `<=`, so a position regression also breaks).
- The **`+` minimum guard** fails the item after the loop if position never advanced
  (`gsm2parser.py:595-599`).

A `+`/`*` (multiple) label naturally yields several children under one label; use
`children_<label>()` for all of them, or assert cardinality with `child_<label>()` /
`maybe_<label>()` (§6.2). Fixtures: `items := item:atom+`, `opt_item := item:atom?`,
`zero_items := item:atom*` (`rust_parser_fixture.fltkg:22-28`).

**Constraint — no repeated nilable item.** A `+` or `*` item whose term can match empty is
rejected (`validate_no_repeated_nil_items`, `gsm.py:418-453`), because it would loop forever.
This is checked recursively into sub-expressions. The emptiness test is an
under-approximation for context-sensitive regexes (e.g. `\ba*`, `(?=x)` test as non-empty at
build time but can match zero-width at runtime), so both backends additionally keep the
runtime progress guard above as defense-in-depth (`gsm.py:438-444`).

## 7. The trivia rule and global whitespace handling

`_trivia` is a **reserved rule name** (`TRIVIA_RULE_NAME = "_trivia"`, `gsm.py:18`). It
defines what counts as inter-token whitespace and comments. In `fegen.fltkg:19` it is:
`_trivia := ( line_comment | line_comment? : | block_comment )+ ;`.

How it governs the grammar:

- **Auto-injection.** A grammar that defines no `_trivia` gets a built-in one matching
  `[\s]+` (`add_trivia_rule_to_grammar`, `gsm.py:477-504`).
- **Reachability classification.** Rules reachable from `_trivia` are flagged
  `is_trivia_rule=True` (`classify_trivia_rules`, `gsm.py:348-379`; `_mark_trivia_reachable`,
  `gsm.py:382-403`). So `line_comment` and `block_comment` become trivia rules transitively.
- **Where it runs.** Trivia is consumed **only at separator positions**. At every `,`
  (optional) or `:` (required) separator, the parser invokes the memoized trivia parser
  (`gsm2parser.py:674-694`). `.` (NO_WS) positions skip trivia entirely. Separators appear at
  three slots per `Items`: the initial separator (`gsm2parser.py:767-776`) and after each
  item (`gsm2parser.py:820-829`).
- **Recursion avoidance.** Inside a trivia rule, the generator does not recurse into
  `_trivia`; it falls back to a raw `\s+` regex (`gsm2parser.py:655-665`), since a trivia
  rule consuming trivia would loop.

### 7.1 Trivia invariants (hard constraints)

- **Non-nil.** `_trivia` must not match empty (`validate_trivia_rule_not_nil`,
  `gsm.py:406-415`). This is what makes `WS_REQUIRED` (`:`) meaningful — a nilable trivia
  rule would make `:` vacuously satisfiable, and `Separator.WS_REQUIRED.can_be_nil()` returns
  `False` precisely because of this (`gsm.py:77-78`).
- **Separation.** A non-trivia rule may **not** reference a trivia rule by name
  (`validate_trivia_separation`, `gsm.py:456-474`). Trivia is reachable only via separators,
  never named as a term. Since trivia-ness propagates by reachability, ordinary rules cannot
  name `line_comment` or `block_comment` either.

### 7.2 Trivia capture (optional, off by default)

When `context.capture_trivia` is set, consumed trivia is appended as an **unlabeled `Span`
child** of the enclosing node (`gsm2parser.py:666-673, 688-694`); otherwise it is discarded.
A rule with any whitespace separator gains a trivia member in its CST model
(`gsm2tree.py:1019-1023`), but children only materialize when capture is enabled.

## 8. Regex terms: the cross-backend engine boundary

A `/.../` term stores its inner text verbatim as `gsm.Regex.value` (`fltk2gsm.py:168-170`).
Build-time emptiness analysis (for the no-repeated-nil check, §6.4) uses Python `re`:
`re.compile(value).match("") is not None`; an invalid regex conservatively returns `False`
(`gsm.py:165-172`).

### 8.1 Python backend matching

`TerminalSource.consume_regex` compiles the pattern with Python `re` and anchors it at the
current position via `re.compile(regex).match(self.terminals, pos=pos)`
(`terminalsrc.py:177-181`). `re.match` with `pos=` anchors the match start but still sees the
full string before `pos`, so `\b`, `\B`, and lookbehind resolve against the preceding
context. (`gsm2parser.py:340-341` carries a `TODO pre-compile regexes`; the Python runtime
recompiles on each call today.)

### 8.2 Rust backend matching

The Rust backend uses `regex_automata::meta::Regex` (`terminalsrc.rs:8`), re-exported from
`fltk-parser-core` so generated code and runtime share one version
(`crates/fltk-parser-core/src/lib.rs:20-23`). Patterns are compiled once, lazily, into
`OnceLock` cells (`parser.rs:25-33`). `consume_regex` performs an anchored search over the
full haystack with the search span starting at the byte offset for `pos`
(`terminalsrc.rs:141-166`), specifically to preserve the same context-before-`pos` behavior
for `\b`/`\B` as Python's `re.match(..., pos=)` (pinned by `consume_regex_context_before_pos`
and `consume_regex_word_boundary_reject_mid_word_via_context`, `terminalsrc.rs:368-389`).

### 8.3 The regex subset restriction (hard limitation)

Grammar regexes must use the **common subset of Python `re` and `regex-syntax`**.
Specifically, **lookahead `(?=...)`/`(?!...)`, lookbehind `(?<=...)`/`(?<!...)`, and
backreferences `\1` are not supported by the Rust engine** — `regex_automata::meta::Regex`
rejects them at compile time (`gsm2parser_rs.py:6-15`). This is a permanent default,
documented in ADR `docs/adr/2026/06/10-rust-parser-codegen/README.md`.

Enforcement: every generated Rust parser emits a `#[test] fn all_regex_patterns_compile`
test; an unsupported pattern fails `cargo test` with a message naming the offending pattern
(`gsm2parser_rs.py:990`, `parser.rs:1346-1347`).

Author consequence: a grammar regex using lookahead, lookbehind, or backreferences will
compile and run on the **Python** backend but **fail to build on the Rust** backend. For
cross-backend (drop-in) compatibility, restrict regexes to the common subset.

### 8.4 Codepoint indexing and position edge cases

Span offsets are codepoint-indexed in both backends (§2). Both backends accept `pos == len`
(an empty match at end-of-input): the Rust path guards `pos > self.len()` and documents the
`pos == len` case (`terminalsrc.rs:139, 141-144`), and the Python path admits it too —
`consume_literal` only fails when `pos + literal_len > terminals_len`, and `consume_regex`
calls `re.match(..., pos=pos)`, both of which succeed at `pos == len` for an empty
literal/match (`terminalsrc.py:168-181`). The Rust backend additionally rejects `pos < 0`
(returning `None`); this is a deliberate divergence — negative positions are unreachable from
generated code (`terminalsrc.rs:33-37, 110-113, 141-144`).

## 9. The parser: packrat memoization, backtracking, left recursion

Each grammar rule generates one memoized parser method. The parser is a packrat / memoized
recursive-descent parser. The memoization engine and the left-recursion algorithm live in
the runtime (`fltk/fegen/pyrt/memo.py` for Python; `crates/fltk-parser-core/src/memo.rs` for
Rust), not in generated code, so they apply to every rule of every grammar automatically.

### 9.1 Left recursion is supported

**FLTK supports left recursion, both direct and indirect/mutual.** The mechanism is a
**packrat memoizer with seed-growing**, a simplified variant of the Warth et al. algorithm.
There is no opt-in; it applies to all rules.

The mechanism, from `Packrat.apply` (`memo.py:82-156`, authoritative):

1. Recall the memo entry for `(rule, pos)`.
2. On a cache miss, write a **poison** entry (`Poison(recursion_info=None)`) at this
   position, push the rule on the invocation stack, and run the rule. The poison is the
   in-progress / left-recursion sentinel.
3. If the rule re-enters itself at the same position while running, the recall finds the
   poison, records the recursion (the cycle head and the involved rules), and **returns
   `None` at the recursion point** so that a *different* alternative can produce a
   non-recursive **seed parse** (`memo.py:96-106`).
4. After the initial call returns, if recursion was detected and a seed parse succeeded,
   `_grow_seed` runs (`memo.py:143-156`).
5. `_grow_seed` (`memo.py:228-257`) repeatedly re-runs the head rule at the start position
   with a cache bypass for the involved rules, keeping each longer result, until the parse
   stops advancing. The longest result wins.

Recursions are keyed by **position** (`memo.py:80`), so independent recursions at different
offsets do not interfere.

**Associativity.** Direct left recursion produces **left-associative** nesting. For
`expr := expr "+" | "x"` on input `x+`, the result is `expr(left=expr("x"), "+")` — each
growth iteration wraps the previous result as the left child, not a flattened node. This
follows from `_grow_seed` (`memo.py:228-257`): each cycle re-runs the head rule, which reads
the just-stored shorter result as its left child and writes the longer wrapped result back
(`memo.py:251-252`), so nesting accretes leftward. (`test_regression_recursive_inlining.py:35-178`
pins the related property that a recursive result is nested rather than spuriously inlined;
it parses a single-`+` input, so it does not by itself distinguish left from right
associativity.)

### 9.2 A base case is mandatory for any left-recursive rule

Seed growing needs a non-recursive seed. When the recursion point returns `None` and no
alternative yields a seed, the overall result is `None` ("Did not find a seed parse",
`memo.py:147-149`). **A rule that is left-recursive in every alternative simply fails to
parse.** Always provide at least one non-left-recursive alternative as the base case, and —
because alternation is ordered first-match — remember the seed comes from the first
alternative that succeeds without recursing.

Exercised forms (all pinned in `rust_parser_fixture.fltkg`):

- Direct: `expr := lhs:expr . "+" . rhs:atom | atom:atom` (`:30-32`).
- Indirect / mutual: `lval := inner:rval . "!" | base:name ;`,
  `rval := inner:lval . "?" | base:num ;` (`:34-37`).
- Through a sub-expression: `rec_via_sub := (inner:rec_via_sub . "+" | inner:atom) . suffix:name ;`
  (`:63-66`).
- Growth that descends into nesting: `nest_sum := lhs:nest_sum . "+" . rhs:nest | first:nest ;`
  (`:68-73`).

The toolchain self-hosts on a left-recursive grammar: `fegen.fltkg`'s own
`alternatives`/`items` rules go through the same memoizer.

### 9.3 Memoization, backtracking, and cut

- **Memoization granularity is per top-level rule** (plus the `_trivia` rule). Each rule gets
  a per-rule cache `_cache__parse_<rule>: dict[pos, MemoEntry]` and a memoized entry method
  that delegates to `Packrat.apply` (`gsm2parser.py:441-472`). Sub-rules created for
  alternatives, items, and sub-expressions are plain helper methods and are **not**
  independently memoized (`gsm2parser.py:380-405`), so packrat guarantees hold at rule
  granularity. A `MemoEntry.result` is tri-state: poison (in-progress), a value (success), or
  `None` (cached failure) (`memo.py:59-62, 107-109`). Cached failures are reused, so a rule
  that failed at a position is not retried there.
- **Backtracking is local and position-based.** Positions are plain integers passed by value;
  a failed alternative or optional item simply does not advance `pos`, and the next
  alternative starts from the original `pos`. There is no global backtracking stack. Optional
  (`?`) and `*` items that fail are non-fatal; only a **required** item's failure returns
  `Failure` and aborts the current alternative (`gsm2parser.py:795-818`).
- **There is no cut / commit operator and no lookahead predicates** (§4.1). The only implicit
  commitment is memoization: once a rule succeeds at a position, it is not re-derived there.

### 9.4 Recursion-depth limits: backend divergence

This is a real cross-backend behavioral difference for pathological (typically deep or
right-recursive) input:

- **Rust** has a configurable depth guard. `DEFAULT_MAX_DEPTH = 1000` (`memo.rs:74`); `apply`
  checks the depth, sets a **sticky** `depth_exceeded` flag, and returns `None`
  (`memo.rs:190-201`). Callers must check `depth_exceeded()` after parsing and discard the
  result if set (`memo.rs:104-106, 158-161`); the Python bindings raise `RecursionError`. The
  default is sized for an ~8 MiB stack; smaller thread stacks require lowering it
  (`parser.rs:1-11`).
- **Python** has no depth guard at all (`Packrat` has no `max_depth` or `depth_exceeded`,
  `memo.py:77-80`). Deep or right-recursive input is bounded only by CPython's own recursion
  limit / native stack, and raises `RecursionError` (or crashes) rather than returning a
  clean failure.

### 9.5 A deliberate hard-error corner case

`_recall` contains one untested corner case that both backends turn into a hard abort rather
than risk a misparse: Python raises `NotImplementedError("Untested corner case...")`
(`memo.py:181-187`) and the Rust port `panic!`s in the same spot (`memo.rs:225-228`). A
grammar that somehow reaches this path aborts rather than producing incorrect output.

## 10. CST construction: nodes, spans, gaps, identity

### 10.1 Node shape

Each rule generates one `@dataclass` (`gsm2tree.py:237`) with three fields
(`gsm2tree.py:262-277`):

- `kind: Literal[NodeKind.X] = NodeKind.X` — the rule discriminant, defaulting to its own
  `NodeKind` member.
- `span: terminalsrc.Span | fltk._native.Span = UnknownSpan` — a codepoint-indexed source
  range; the type is the union of the Python-backend span and the Rust native span.
- `children: list[tuple[Optional[Label], child]] = field(default_factory=list)` — ordered
  children, each tagged with its `Label` enum member or `None`.

A node references only its children — there are no parent or sibling pointers. Rule-reference
children appear as the referenced rule's node; `$`/included literals and regexes appear as
`Span` children.

### 10.2 Spans and merging

- Spans are built with `Span.with_source(start, end, source)` (`terminalsrc.py:130-149`). An
  alternative/item parser initializes its result node with a sentinel end of `-1` and
  overwrites the span at the end with `(start, final_pos)` (`gsm2parser.py:745-763,
  832-838`). The span start is captured **before** any mutation so it is never read back off
  the result node (the Rust backend cannot read `result.span.start`,
  `gsm2parser.py:535-536, 743-744`).
- `Span.merge` is the smallest covering span; `Span.intersect` is the overlap, or
  `UnknownSpan` (`Span(-1, -1)`) if the spans are disjoint (`terminalsrc.py:109-128, 152`).
- `Span` equality **ignores `_source` and `kind`** (`compare=False`, `terminalsrc.py:54-55`),
  so source-bearing and sourceless spans, and spans from either backend, compare by
  `(start, end)` only.

### 10.3 Gaps from suppression

A `%`-suppressed term advances position but appends no child (§6.3). The parent node's span
still covers the suppressed text, but no child entry exists for it, so the union of child
spans can be a strict subset of the parent span. **Suppression creates gaps in child
coverage.** Whitespace consumed at separators is likewise gap-producing unless
`capture_trivia` is enabled (§7.2).

### 10.4 Mutators and strictness

Beyond the per-label quintet (§6.2), every node has generic mutators. The newer named
mutators are strict and enforce cross-backend-parity ordering:

- `insert` / `replace_at` validate **child type, then label type, then index**, matching the
  Rust order (`gsm2tree.py:531-603`). Unknown child classes are rejected with `TypeError`
  (`gsm2tree.py:447-493`); the native `Span` is admitted lazily so the module stays
  pure-Python-importable.
- A label-free node rejects any non-`None` label; a labeled node requires `None` or its own
  `Label` enum (`gsm2tree.py:515-529`).
- `insert` clamps out-of-range indices (matching `list.insert` and Rust);
  `remove_at` / `replace_at` raise `IndexError` out of range (`gsm2tree.py:558-603`).
- `append` / `extend` are grandfathered as un-strict (no type check) for backward
  compatibility; only the newer named mutators are strict (`gsm2tree.py:283-297, 395-396`).

### 10.5 Node identity and cross-backend equality

The `NodeKind` enum and per-node `Label` enums carry a cross-backend `__eq__`/`__hash__`
keyed on a plain string `_fltk_canonical_name` (e.g. `"NodeKind.LINECOMMENT"`,
`"Items.Label.NO_WS"`), set post-class (`gsm2tree.py:99-156, 158-169`). Same-type comparisons
use the member name; cross-type comparisons compare canonical names; foreign operands return
`NotImplemented`. As a result, Python-backend and Rust-backend enum members **compare equal
across backends** — the load-bearing piece of the drop-in-replacement contract.
`SpanKind.SPAN` has the same bridge (`terminalsrc.py:24-45`).

A parallel `*_cst_protocol.py` module (`gsm2tree.py:721-1006`) mirrors the concrete CST
surface with `typing.Protocol` classes, for structural typing across backends without
importing a concrete backend.

### 10.6 At least one included member per node

A rule whose every item is suppressed and which has no sub-expression or labeled member
produces an empty model and is **rejected**: `"Model class ... would have no members"`
(`gsm2tree.py:251-256`). Every node type must have at least one child member.

## 11. Error reporting

`ErrorTracker` (`errors.py:24-49`) records the furthest position reached and the set of
expected literals and regexes at that position, attributed to the current rule (top of the
invocation stack). On failure at a literal or regex, the terminal source calls
`fail_literal` / `fail_regex` (`gsm2parser.py:171-180, 220-229`). `format_error_message`
(`errors.py:126-152`) renders `Syntax error at line L col C`, echoes the offending line with
a caret, and lists the expected tokens grouped by rule. Control, bidi, and zero-width
characters in the echoed line are escaped (`escape_control_chars`, `errors.py:96-123`),
byte-identically to the Rust port (`crates/fltk-cst-core/src/escape.rs`).

## 12. Vestigial / unsupported constructs

The GSM defines `Invocation`, `Expression`, `Add`, and `Var` term types (`gsm.py:267-288`),
which are part of the `Term` union but are **dead** — vestiges of the aspirational
`fltk.fltkg` (§1):

- The live grammar never produces them (`fltk2gsm.visit_term` only yields
  identifier/literal/regex/sub-expression, `fltk2gsm.py:130-140`).
- The Python parser generator has no branch for them; `_gen_consume_term_expr` raises
  `NotImplementedError` on any unrecognized term (`gsm2parser.py:374-375`).
- The Rust parser generator explicitly rejects `Invocation`
  (`gsm2parser_rs.py:768-770`).

These constructs, the `let X : Stack[String]` `var` syntax, and the `!` INLINE disposition
(§6.3) are not part of the usable grammar language. Author grammars using only the constructs
described in §1–§10.

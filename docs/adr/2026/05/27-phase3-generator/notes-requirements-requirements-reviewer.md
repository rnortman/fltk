## requirements-1

**Section:** "Public API" (lines 116–121)

**What's wrong:** The requirements specify a concrete Python class name (`RustCstGenerator`) and method name (`gen_rs_module`), which are design decisions, not requirements.

**Why:** The review protocol says: "flag any tip into module structure, function names, file paths, internal types, implementation steps." The observable behavior is: "given a `gsm.Grammar`, produce a compilable Rust source string." The class/method names are implementation detail.

**Consequence:** Needlessly constrains the designer. If a better API shape emerges (e.g., a free function, or subclassing `CstGenerator`), the designer must either deviate from the requirements or file a change request.

**Suggested fix:** Specify the observable contract only: the module must expose a callable that accepts a `gsm.Grammar` and returns a `str` of compilable Rust source. Leave naming to design.

## requirements-2

**Section:** "Processing" step 1 (line 54), "Public API" (lines 116–121)

**What's wrong:** The requirements simultaneously say the constructor "applies `add_trivia_rule_to_grammar` and `classify_trivia_rules` internally" (Public API section) and say the input is "A `gsm.Grammar` object (after `add_trivia_rule_to_grammar` and `classify_trivia_rules` have been applied)" (System behavior / Input, line 46). These are contradictory — either the caller preprocesses the grammar or the generator does, not both.

**Why:** OQ-trivia-preprocessing acknowledges this but the "System behavior / Input" section already states the input is preprocessed, while the Public API section says the constructor does the preprocessing.

**Consequence:** Implementer gets conflicting instructions. Ambiguous who owns the trivia step; if both caller and generator do it, trivia rules could be double-added.

**Suggested fix:** Pick one. The recommendation in OQ-trivia-preprocessing (constructor does preprocessing) is fine — but then the "Input" section must say the input is a raw grammar.

## requirements-3

**Section:** "Protocols / Method signatures" (lines 173–189)

**What's wrong:** Specifying exact Rust method signatures, parameter types, PyO3 attribute strings, and function bodies is design, not requirements. The observable requirement is behavioral: the generated classes must have the same Python-visible API as Phase 2's hand-written classes.

**Why:** The review protocol says to flag design-as-requirements. These signatures are internal Rust details — the Python caller never sees `&Bound<'_, PyAny>` or `PyResult<Py<PyList>>`. The Phase 2 tests already define the behavioral contract.

**Consequence:** The designer is locked into specific Rust patterns even where alternatives would be cleaner (e.g., different PyO3 API choices). Makes the requirements doc fragile to PyO3 version changes.

**Suggested fix:** Remove the Rust method signatures section. The behavioral contract is already captured by AC-6 (Phase 2 tests pass) and AC-11 (error message format). Keep only the Python-visible API constraints.

## requirements-4

**Section:** "Generated Rust file structure (fixed ordering)" (lines 137–156)

**What's wrong:** Specifying internal file structure (ordering of enum definitions, struct definitions, method blocks) is design. The only observable that matters is: the generated code compiles and `register_classes` works.

**Why:** The review protocol: "flag any tip into... file paths, internal types." The ordering of Rust items within a file is not externally observable.

**Consequence:** Constrains the designer to one file layout when the compiler doesn't care. Also specifies a comment format ("Auto-generated by gsm2tree_rs.py — do not edit.").

**Suggested fix:** Remove the file structure specification. Keep AC-10 (preamble includes required `use` declarations) if desired, as that is a correctness constraint.

## requirements-5

**Section:** AC-11 (line 101)

**What's wrong:** The error message for `maybe_{label}` is specified as `"Expected at most one {label} child but have at least 2"` (fixed string), while `child_{label}` uses `"Expected one {label} child but have {count}"` (dynamic count). The Phase 2 design document (lines 260–264) shows `maybe_name` using `format!("Expected at most one name child but have {count}")` — a dynamic count, not "at least 2". These are different behaviors: "at least 2" implies early termination on finding count > 1; `{count}` implies scanning all children.

**Why:** The exploration (section 3, line 168) describes the committed hand-written code as using "at least 2", while the Phase 2 design shows `{count}`. The requirements doc follows the exploration, but says the generator must "match Phase 2 pattern." These sources disagree.

**Consequence:** Ambiguity on whether `maybe_{label}` should break early (and say "at least 2") or scan completely (and report the actual count). If the existing tests assert a specific string, only one implementation will pass — but the requirements doc doesn't say which.

**Suggested fix:** Determine what the committed Phase 2 code actually does (and what the test asserts), then align the requirements to that. The tests are the ground truth per AC-1.

**Suggested fix:** Remove the `lib.rs` code block. AC-4 and AC-5 already capture the observable requirements.

## requirements-7

**Section:** "Constraints" — "Deterministic output" (line 197)

**What's wrong:** Requiring "byte-identical output across invocations" is an over-specification. The observable requirement is that the generated Rust compiles and produces correct behavior. Byte-identical output is a nice property but unnecessarily constrains the designer (e.g., prevents caching tricks, HashMap iteration in Python 3.12+ is deterministic but pinning this as a contract is fragile).

**Why:** The phase plan does not mention determinism. This was added by the requirements doc.

**Consequence:** The designer must guarantee byte-level reproducibility, which adds testing burden (diffing output) and constrains implementation choices (e.g., can't use `set` iteration without sorting, must control all ordering). If this is actually desired for diffability, say so — the justification matters.

**Suggested fix:** Either remove, or weaken to "output should be deterministic to support diffing committed generated files" and specify only the ordering constraints (labels sorted, rules in grammar order).

## requirements-8

**Section:** AC-7, AC-8, AC-9 — `fegen.fltkg` testing (lines 93–98)

**What's wrong:** The requirements say to test against `fegen.fltkg` with 14 rules/classes, and also against a minimal inline grammar. This requires compiling the `fegen.fltkg` Rust code into a loadable extension — but the requirements don't specify how this compilation happens in the test. OQ-fegen-compilation-test acknowledges this but leaves it open. Meanwhile AC-9 says "compiles it (or verifies the generated source text is well-formed)" — hedging on whether compilation is required.

**Why:** If the `fegen.fltkg` generated code must actually compile and be importable from Python (to run the AC-8 smoke test), then it must be wired into `lib.rs` or compiled separately. This is a significant build-system concern that the requirements leave unresolved.

**Consequence:** The designer doesn't know whether to add a second module to `lib.rs`, build a separate extension, or just do source-text validation. The smoke test (AC-8) implies actual compilation and import, but the mechanism is unspecified. This could expand scope significantly.

**Suggested fix:** Decide: either (a) the `fegen.fltkg` generated code is compiled into `fltk._native` (e.g., as a second `mod`) and AC-8 tests import from it, or (b) AC-8 is dropped and only source-text validation is required. Option (a) is recommended since it matches the phase plan's intent.

## requirements-9

**Section:** OQ-empty-label-enum (line 213)

**What's wrong:** This is an open question that should be resolved before design begins, because it affects whether the generator must handle a specific edge case. The exploration notes that Rust enums cannot have zero variants. The requirements doc punts this.

**Why:** Whether zero-label rules exist in practice determines whether the generator needs special handling. If `gsm2tree.py` never produces them, the question is moot and should be closed as "not applicable." If it does, the requirements must specify behavior.

**Consequence:** The designer must investigate this themselves, duplicating exploration work. If zero-label rules do exist and the designer guesses wrong, the generated code won't compile.

**Suggested fix:** Resolve by checking whether `gsm2tree.py` ever produces a rule with no labels. If not, close the OQ with "zero-label rules do not occur in practice; generator may panic on them." If yes, specify which option to use.

## requirements-10

**Section:** "In scope" item 2 (line 22)

**What's wrong:** "Reuse of `CstGenerator` analysis logic — instantiate `CstGenerator` to obtain `rule_models`, `class_name_for_rule_node`, trivia classification. Do not duplicate model-building code." This is a design constraint (how to obtain the model data), not a requirement.

**Why:** The observable requirement is: the generated Rust code must be equivalent to what `gsm2tree.py` would produce for the same grammar. Whether the generator reuses `CstGenerator` internals, subclasses it, or reimplements the analysis is an implementation choice.

**Consequence:** Locks the designer into a specific reuse strategy. If `CstGenerator`'s API turns out to be awkward for Rust generation (e.g., the `py_module` parameter is irrelevant and adds confusion), the designer must still use it.

**Suggested fix:** Reword to: "The Rust generator must produce output consistent with `gsm2tree.py`'s model analysis (same label sets, child types, trivia insertion, class names). How the generator obtains this model data is a design choice."

## requirements-11

**Section:** "Constraints" — "Line length: 120 characters (project convention)" (line 201)

**What's wrong:** Applying the 120-character line-length convention to generated Rust source text is over-specification. The convention (from `pyproject.toml`) applies to hand-written Python code. Generated Rust code will be compiled, not read by humans in normal workflow.

**Consequence:** The generator must implement line-wrapping logic for generated Rust code, adding complexity for no user benefit. The Rust compiler doesn't care about line length.

**Suggested fix:** Remove this constraint for generated Rust output. If readability of generated code is desired for debugging, state that as the goal and let the designer decide the approach (e.g., `rustfmt` post-processing).

The requirements doc correctly uses `fegen.fltkg` (not `fltk.fltkg`), resolving the phase plan's confusion. **Consequence:** Minor — the requirements are correct. But a reader cross-referencing with the phase plan will see a discrepancy.

**Suggested fix:** Add a brief note: "The phase plan references `fltk.fltkg`; the correct grammar is `fegen.fltkg` (from which `fltk_cst.py` was generated)."

## requirements-13

**Section:** Overall — spirit of request

**What's wrong:** The user instruction says "we replace the hand-written phase 2 code with equivalent generated code." The requirements faithfully capture this. However, the requirements also add `fegen.fltkg` testing (AC-7, AC-8) and minimal-grammar testing (AC-9), which go beyond "replace hand-written code." These additions are reasonable — they validate generality — but they expand scope beyond what the user literally said.

**Why:** The phase plan (line 109) does call for testing on at least two grammars: "Also test with at least one non-FLTK grammar from the test suite to validate generality." So the requirements are aligned with the phase plan, even if the user's immediate instruction was narrower.

**Consequence:** Acceptable scope expansion. The user likely approves — generality testing is prudent before Phase 4. No action needed unless the user wants a minimal Phase 3.

## Root causes

### Generated files: single origin, multiple violations

The three `unparsefmt_*.py` files were generated by `fltk/fegen/genparser.py` (a Typer CLI wrapping `gsm2tree.CstGenerator` + `gsm2parser.ParserGenerator`) from the grammar `fltk/unparse/unparsefmt.fltkg`.

**These files are not currently passing lint**, and **the current generator still produces non-compliant output**. Running the generator now produces the same violations:

```
uv run python fltk/fegen/genparser.py unparsefmt.fltkg unparsefmt fltk.unparse.unparsefmt_cst -o /tmp/testgen
```
produces output with single-quote strings, `typing.Optional[...]`, and double-parens — identical to the checked-in files.

#### Root cause A — Single quotes (Q000, Q001): `compiler.py:323`

`fltk/iir/py/compiler.py:323`:
```python
if isinstance(expr, iir.LiteralString | iir.LiteralInt):
    return repr(expr.value)
```
`repr()` of a Python `str` uses single quotes by default (e.g., `repr('foo')` → `"'foo'"`). Every string literal in generated parser code — regex patterns, literal tokens — is emitted with single quotes, violating `Q000`.

#### Root cause B — `typing.Optional[X]` instead of `X | None` (UP045): `gsm2tree.py:128,140,148,154,226`

`fltk/fegen/gsm2tree.py` hardcodes `typing.Optional[...]` in all generated type annotations:
- Line 128: `f"children: list[tuple[typing.Optional[Label], {child_annotation}]]"`
- Line 140: `f"self, child: ..., label: typing.Optional[Label] = None"`
- Line 148: `f"self, children: ..., label: typing.Optional[Label] = None"`
- Line 154: `pygen.function("child", "self", f"tuple[typing.Optional[Label], ..."])`
- Line 226: `f"typing.Optional[{child_annotation_by_labels[label]}]"`

`iir/context.py:87` registers `iir.Maybe` as `typing.Optional`, so IIR→Python compilation also emits `typing.Optional[...]` for all `Maybe`-typed return values in parser methods (UP045 violations in `unparsefmt_parser.py` and `unparsefmt_trivia_parser.py`).

#### Root cause C — Extraneous parentheses (UP034): `gsm2tree.py:185`

`fltk/fegen/gsm2tree.py:185`:
```python
pygen.stmt(f"self.children.extend(({class_name}.Label.{label.upper()}, child) for child in children)")
```
The outer `(...)` around the generator expression is redundant — `extend(...)` already forms the outer parens. Also line 151 for the unlabeled case:
```python
pygen.stmt("self.children.extend((label, child) for child in children)")
```
Wait — line 151 does NOT have the extra parens (it uses `extend((label, child) for ...)` with no outer paren). But what the generator currently emits includes the extra parens, suggesting the checked-in `unparsefmt_cst.py` was generated by a version of `gsm2tree.py` from commit `7914e57` (2026-01-13), which differs from the current `gsm2tree.py` in the repo. **The current `gsm2tree.py` at line 151 emits `extend((label, child) for ...)` without extra parens, but line 185 still uses double parens.** Empirical test above confirms the current generator does produce `extend(((label, child) for ...))` — so line 185 still has UP034.

#### Root cause D — Line length (E501)

`ast.unparse()` produces minimal/compact output with no line wrapping. Long type annotations (e.g., `typing.Optional[fltk.fegen.pyrt.memo.ApplyResult[int, fltk.unparse.unparsefmt_cst.Trivia]]`) easily exceed 120 characters on a single line.

#### Root cause E — Import ordering (I001)

`genparser.py:83-88` defines imports in this order:
```python
imports = [
    pyreg.Module(("collections", "abc")),
    pyreg.Module(("typing",)),
    pyreg.Module(("fltk", "fegen", "pyrt", "errors")),
    pyreg.Module(("fltk", "fegen", "pyrt", "memo")),
    pyreg.Module(("fltk", "fegen", "pyrt", "terminalsrc")),
    cst_module,  # appended at line 88
]
parser_mod = pygen.module(module.import_path for module in imports)
```
`pygen.module()` (`fltk/pygen.py`) emits each as a bare `import X` statement with no blank line between stdlib (`collections.abc`, `typing`) and first-party (`fltk.*`) groups. Isort rule I001 requires a blank line between these groups.

Similarly `gsm2tree.py:96-102` for CST modules:
```python
imports = [
    pyreg.Module(("dataclasses",)),
    pyreg.Module(("enum",)),
    pyreg.Module(("typing",)),
    pyreg.Module(("fltk", "fegen", "pyrt", "terminalsrc")),
]
module = pygen.module(module.import_path for module in imports)
```
No blank line between stdlib (`dataclasses`, `enum`, `typing`) and first-party (`fltk.*`).

#### Root cause F — No newline at end of file (W292)

`ast.unparse()` does not add a trailing newline. All three generated files lack it.

### History: why `fltk_cst.py` passes but `unparsefmt_cst.py` fails

`fltk/fegen/fltk_cst.py` passes all lint rules. It passes because it was **manually reformatted** after generation — specifically in commit `21fc688` ("Fix linter errors in gencode", 2025-07-03) which changed `typing.Optional[Label]` → `Label | None`, double quotes, import grouping, etc. in the hand-edited file.

`fltk/unparse/unparsefmt_cst.py` was originally created in commit `29b4dc1` (2025-07-22) in a clean state (double quotes, `Label | None`). It was then **regenerated** in commit `7914e57` (2026-01-13, "Add preserve_blanks directive and fix formatter comment handling") — that regeneration overwrote the clean version with raw `ast.unparse()` output, losing all manual formatting improvements.

Git diff of commit `7914e57` for `unparsefmt_cst.py` confirms the regression:
```diff
-    children: list[tuple[Label | None, typing.Union["Statement", "Trivia"]]] = ...
+    children: list[tuple[typing.Optional[Label], typing.Union['Statement', 'Trivia']]] = ...
-    def append(self, ..., label: Label | None = None) -> None:
+    def append(self, ..., label: typing.Optional[Label]=None) -> None:
-    def extend(...) -> None:
-        self.children.extend((label, child) for child in children)
+    def extend(...) -> None:
+        self.children.extend(((label, child) for child in children))
```

## What needs to change and where

### Option 1: Fix the generators (permanent)

Fixes the root causes so regeneration never produces broken output:

1. **`fltk/iir/py/compiler.py:323`** — change `repr(expr.value)` to emit double-quoted strings. E.g., `json.dumps(expr.value)` for strings, keep `repr()` for ints. Fixes Q000 in all future-generated parsers.

2. **`fltk/fegen/gsm2tree.py:128,140,148,154,226`** — change `typing.Optional[X]` → `X | None` in all generated type annotation strings. Fixes UP045 in future-generated CST files.

3. **`fltk/iir/context.py:87`** — register `iir.Maybe` as `name="X | None"` pattern, or change `iir_type_to_py_annotation` in `compiler.py` to emit `X | None` instead of `Optional[X]`. Fixes UP045 in future-generated parser files.

4. **`fltk/fegen/gsm2tree.py:185`** (and verify line 151) — remove extra outer parens from `extend((...))`. Fixes UP034.

5. **`fltk/fegen/genparser.py:83-88`** and **`fltk/fegen/gsm2tree.py:96-102`** — insert a blank `ast.parse("")`-derived statement or use `isort`-aware import block assembly to separate stdlib from first-party imports. Alternatively, run `ruff check --fix` on output. Fixes I001.

6. **`genparser.py`** — append `\n` when writing output files. Fixes W292.

7. E501 (line length): `ast.unparse()` produces compact single-line output. Must either post-process with `ruff format` or use `ast.unparse()` + `ruff format` / `black` pipeline on generated output. This is the hardest to fix in the generator.

### Option 2: Post-process generated files with ruff --fix (partial)

2086 of 3112 errors are marked `[*] fixable with --fix`. Running `ruff check --fix` on the three generated files would resolve Q000, UP045, UP034, I001, W292, and some Q001. E501 requires `ruff format`. This is a one-time fix that leaves the generator broken but the checked-in files clean — until next regeneration.

### Per-file ignore (escape hatch, not recommended long-term)

Add `unparsefmt_cst.py`, `unparsefmt_parser.py`, `unparsefmt_trivia_parser.py` to `[tool.ruff.lint.per-file-ignores]` in `pyproject.toml`. Silences errors without fixing generators. Also bypasses the line-length check which would still flag E501 unless E501 is also ignored.

### Hand-written fixes needed regardless of option chosen

- **`fltk/unparse/test_unparser.py`** — 3 pyright errors (lines 1010, 1056, 1099): add `assert fmt_config.trivia_config is not None` before each access, or initialize `fmt_config.trivia_config = TriviaConfig()` explicitly. The PLC0415, Q001, Q000, F401 issues are fixable by moving imports to top-level and using double-quoted strings.

- **`fltk/unparse/resolve_specs.py:308`** — extract constant `MIN_DOCS_FOR_COLLAPSE = 2`.

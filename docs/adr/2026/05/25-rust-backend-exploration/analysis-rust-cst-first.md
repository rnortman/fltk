## Feasibility: Rust+PyO3 CST Nodes as Intermediate Step

All claims anchored to source code. No recommendations.

---

## 1. Exact API Contract of a CST Node

Every CST node class (generated by `gsm2tree.py:109-243`) has this shape:

### Fields
- `span: Span` -- mutable, default `UnknownSpan` (`Span(-1,-1)`). Parser writes twice: `Span(start=pos, end=-1)` at construction, then `result.span = Span(start=result.span.start, end=pos)` at the end of each parse method. Exactly 40 such pairs in `fltk_parser.py`.
- `children: list[tuple[Label|None, ChildUnion]]` -- mutable list, default `[]` via `dataclasses.field(default_factory=list)`.

### Nested Label enum
- `NodeClass.Label` is a nested `enum.Enum` class inside each node.
- Accessed as class attribute: `Grammar.Label.RULE`, `Items.Label.ITEM`, etc.
- Compared with `==` and `in`: `items.children[0][0] in (cst.Items.Label.NO_WS, cst.Items.Label.WS_ALLOWED, ...)` (`fltk2gsm.py:36-40`).

### Generic mutation methods (used by parser)
- `append(child, label=None)` -- `self.children.append((label, child))`
- `extend(children, label=None)` -- `self.children.extend((label, child) for child in children)`
- `child()` -- returns `self.children[0]`, asserts `len == 1`

### Per-label typed methods (5 per label)
For each label FOO:
- `append_foo(child)` -- `self.children.append((ClassName.Label.FOO, child))`
- `extend_foo(children)` -- `self.children.extend((ClassName.Label.FOO, child) for child in children)`
- `children_foo()` -- returns generator `(child for label, child in self.children if label == ClassName.Label.FOO)`
- `child_foo()` -- materializes `children_foo()`, asserts exactly 1
- `maybe_foo()` -- materializes `children_foo()`, asserts 0 or 1, returns Optional

### Dataclass behavior actually used
- `__eq__`: Used on `ApplyResult` (which contains CST nodes) in `test_gsm2parser.py:74`: `assert item_parser(parser, 0) == EXPECT`. Dataclass `__eq__` recursively compares `span` and `children`. Tests depend on this.
- `__repr__`: Used in logging: `LOG.info("Root node children: %s", root_node.children)` (`test_trivia_capture.py:103`).
- `__hash__`: `None` (unhashable). Dataclass with `eq=True` and no `frozen=True` sets `__hash__ = None`. No evidence of hashing CST nodes anywhere.
- **NOT used**: `dataclasses.fields()`, `dataclasses.asdict()`, `dataclasses.astuple()`, `__dataclass_fields__` -- zero occurrences in consumers.

### Direct `children` list access patterns (critical for PyO3)
Parser:
- `result.children.extend(item0.result.children)` -- 11 sites in `fltk_parser.py`. Reads `.children` from one node and extends another's.
- `result.append_foo(child=...)` -- ~30 sites. Uses typed append methods.

Consumers (`fltk2gsm.py`, `bootstrap2gsm.py`):
- `items.children[0][0]` -- index + tuple index (`fltk2gsm.py:36`)
- `items.children[start_idx:]` -- slice (`fltk2gsm.py:51`)
- `children[::2]`, `children[1::2]` -- stride-2 (`fltk2gsm.py:52`)
- `len(children)` -- length (`fltk2gsm.py:62`)
- `label, value = child` -- tuple unpacking

Generated unparser code (emitted by `gsm2unparser.py`):
- `node.children[pos]` -- integer subscript (`gsm2unparser.py:288-289`)
- `child_tuple[0]`, `child_tuple[1]` -- tuple indexing (`gsm2unparser.py:295-316`)
- `len(node.children)` -- length (`gsm2unparser.py:604-618`)
- `isinstance(child_value, SomeType)` -- type dispatch (`gsm2unparser.py:330, 893, 983, 1043`)

### `__class__.__name__` access
Tests use `value.__class__.__name__` for trivia detection (`test_trivia_capture.py:111`, `test_trivia_whitespace_capture.py:41`). This is test code only, not production.

## 2. PyO3 Replicability Assessment

### Easily replicable

| Feature | PyO3 mechanism | Complexity |
|---|---|---|
| `span` field (read/write) | `#[pyo3(get, set)]` on `Span` field, or getter/setter | Trivial |
| `append_foo(child)` | `#[pymethods] fn append_foo(&mut self, child: PyObject)` | Trivial |
| `extend_foo(children)` | `#[pymethods] fn extend_foo(&mut self, children: &Bound<PyAny>)` | Trivial |
| `child_foo()` / `maybe_foo()` | `#[pymethods]` returning `PyObject` or `Option<PyObject>` | Trivial |
| `child()` returning tuple | Return `PyTuple::new(py, [label, child])` | Trivial |
| `__eq__` | `fn __richcmp__` with `CompareOp::Eq`; compare span + children | Moderate (recursive comparison) |
| `__repr__` | `fn __repr__(&self)` | Moderate |
| `isinstance(child, SomeType)` | PyO3 `#[pyclass]` types work with `isinstance` natively | Works as-is |
| `None` return from `maybe_*` | `Option<PyObject>` maps to `None` naturally | Trivial |

### Requires careful design

**Nested `Label` enum as class attribute (`NodeClass.Label.FOO`)**

PyO3 does not support nested `#[pyclass]` definitions. Workarounds:
1. Define `Grammar_Label`, `Rule_Label`, etc. as standalone `#[pyclass]` enums, then attach via `#[classattr] fn Label() -> Py<Grammar_Label>` on the parent class. Consumer code `Grammar.Label.RULE` would work because Python resolves `Grammar.Label` as a class attribute access.
2. The standalone enum class must support `==` comparison and `enum.auto()`-style integer values.

Verified: Labels are compared with `==` (`fltk2gsm.py:42`: `sep_label == cst.Items.Label.WS_REQUIRED`) and with `in` (`fltk2gsm.py:36`: `items.children[0][0] in (cst.Items.Label.NO_WS, ...)`). Both work if `__eq__` and `__hash__` are implemented on the enum.

**`children` as a real Python `list[tuple[...]]`**

The `children` field is accessed as a plain Python list with full sequence protocol: indexing, slicing, striding, `len()`, iteration, `extend()`, `append()`. The tuples inside are accessed by index.

Options:
- **Option A: Store as `Vec<(PyObject, PyObject)>` in Rust, expose via `#[pyo3(get)]` converting to Python list on each access.** Problem: Every access creates a new Python list. Mutations via `append`/`extend` would need to modify the Rust Vec. The list returned by `node.children` would be a snapshot, not a live reference. This breaks `result.children.extend(item0.result.children)` because the parser extends `children` via the returned list object -- but that list is a copy.
- **Option B: Store children in a Python `list` object (held as `PyObject`).** The Rust struct holds a `Py<PyList>`, exposed directly. All list operations delegate to the real Python list. Mutations work naturally. Tuple elements are `PyObject`s. This preserves the exact Python semantics. Cost: each child access crosses the Python-Rust boundary.
- **Option C: Custom `#[pyclass]` implementing `__getitem__`, `__len__`, `__iter__`, `__setitem__`, append, extend.** Must support slicing (including stride) to satisfy `children[::2]`. PyO3's `__getitem__` receives `SliceOrIndex` but implementing full slice semantics (negative indices, strides) is substantial work.

Option B is the only one that preserves semantics without reimplementing the Python list protocol. The performance cost is real but acceptable for an intermediate step.

**Span as both field and child type**

`Span` appears as:
1. A field on every node (`node.span`)
2. A child value in `children` tuples (leaf terminals stored as `Span` directly)

`Span` is `frozen=True, eq=True, slots=True`. A PyO3 `#[pyclass(frozen)]` with `__richcmp__` and `__hash__` replaces this cleanly. The same `Span` PyO3 class works in both roles.

Key: `Span` must be constructable from Python (`Span(-1, -1)`) because `UnknownSpan` is module-level constant, and the parser constructs Spans via `Span(start=pos, end=-1)` and `Span(start=result.span.start, end=pos)`.

## 3. isinstance Dispatch

The generated unparser emits `isinstance` checks in three contexts:

1. **Child type validation** (`gsm2unparser.py:330-331`): `iir.IsInstance(expr=child_var.load(), typ=expected_type)` -- compiles to `isinstance(child, SomeNodeClass)`.
2. **Trivia type check** (`gsm2unparser.py:1043`): `iir.IsInstance(expr=child_value, typ=trivia_type)` -- `isinstance(child, Trivia)`.
3. **Trivia child type check** (`gsm2unparser.py:893`): Checks if trivia children are `LineComment`, `BlockComment`, etc.
4. **Span check** (`gsm2unparser.py:983`): `iir.IsInstance(expr=child_value, typ=span_type)` -- `isinstance(child, Span)`.

Consumer code (`fltk2gsm.py:53`): `isinstance(item, cst.Item)`.

PyO3 `#[pyclass]` types work transparently with Python `isinstance`. Each CST node class must be a distinct `#[pyclass]`. The type object must be importable from the same module path. This is inherently satisfied if Rust CST classes are exposed in a Python module at the same import path as the current generated Python CST module.

**Potential issue**: The generated Python CST module is dynamic -- `plumbing.py:104-112` executes the generated code via `exec()` and creates a `types.ModuleType` at runtime, registering it in `sys.modules`. The generated unparser imports from this dynamic module (`gsm2unparser.py:1544-1547`). For Rust CST nodes, the equivalent module would need to be importable via the same mechanism. This works if the Rust extension exposes a submodule or if the CST classes are injected into the dynamic module.

## 4. The children list Efficiency

As analyzed above, `children` must behave as a real Python `list[tuple[Label|None, ChildUnion]]`.

**Performance implications**: Under Option B (store as `Py<PyList>`), every `append`/`extend`/index/slice call is a Python operation. The Rust backing buys zero speedup for tree traversal. The benefit is only in the overhead of the node object itself (fixed fields, method dispatch). For CST nodes where most time is spent in list operations, the speedup is minimal.

**Can a Rust-native representation work?** Only if all consumers are also ported. Since the question is about Python parser/unparser working with Rust CST nodes, the children list must be Python-native.

## 5. Span Handling

`Span` is used in two distinct ways:

1. **Node field**: `node.span` -- read/write (parser sets it twice). Since `Span` is `frozen=True`, the parser creates new Span objects each time (`result.span = Span(start=..., end=...)`). It never mutates an existing Span. So the node's `span` field is reassigned (not mutated). PyO3: `span` is a `Py<Span>` or `PyObject`, settable.

2. **Child value**: Leaf terminals (literals, regexes) are stored as `Span` objects in `children`. Consumers access `span.start` and `span.end` for text slicing: `terminals[span.start : span.end]` (`fltk2gsm.py:24`).

A PyO3 `Span` with `#[pyclass(frozen)]`, `#[pyo3(get)] start: i64`, `#[pyo3(get)] end: i64`, `__richcmp__`, and `__hash__` satisfies both roles.

`UnknownSpan = Span(-1, -1)` (`terminalsrc.py:15`) -- module-level constant. For Python compatibility, a Rust Span must be constructable as `Span(-1, -1)`.

## 6. Parser Compatibility (Construction Side)

The parser (`fltk_parser.py`) constructs CST nodes via:

1. **Constructor**: `fltk.fegen.fltk_cst.Grammar(span=Span(start=pos, end=-1))` -- keyword argument construction. 40 occurrences.
2. **Typed append**: `result.append_rule(child=one_result.result)` -- keyword argument. ~30 occurrences.
3. **Direct children extend**: `result.children.extend(item0.result.children)` -- 11 occurrences. Reads `.children` from source node, extends target node's `.children`.
4. **Span reassignment**: `result.span = Span(start=result.span.start, end=pos)` -- 40 occurrences.

**Would the parser work unmodified?**

With Option B (children as `Py<PyList>`):
- Constructor: `NodeClass(span=Span(...))` -- works if `#[new]` accepts `span` kwarg and defaults `children` to empty list.
- `append_rule(child=...)` -- works as `#[pymethods]`.
- `result.children.extend(...)` -- works because `result.children` returns a real Python list.
- `result.span = Span(...)` -- works if span has a setter.

**Answer: Yes, the Python parser would work unmodified**, provided:
- Constructor accepts `span` keyword argument with default `UnknownSpan`
- `children` property returns a mutable Python list
- `span` property is readable and writable
- All `append_*/extend_*` methods are defined

## 7. Unparser Compatibility (Read Side)

The generated unparser accesses CST nodes via:

1. **`node.children[pos]`** -- integer subscript on `.children`
2. **`child_tuple[0]`, `child_tuple[1]`** -- tuple indexing
3. **`len(node.children)`** -- length
4. **`isinstance(child, NodeType)`** -- type dispatch
5. **`node.children` field access** -- returns the list

With Option B, all of these work because `children` is a real Python list containing real Python tuples.

The unparser also calls methods on the unparser class itself (`self.unparse_foo(node)`) passing CST nodes as arguments. PyO3 classes are valid arguments to Python functions.

**Answer: Yes, the Python unparser would work unmodified.**

## 8. Migration Path: Mixed Rust/Python CST Nodes

**Can some rules use Rust CST nodes while others use Python?**

Yes, because:
- There is no shared base class. Nodes are structurally typed, not nominally typed (except for `isinstance` checks).
- The `children` list holds heterogeneous Python objects. A Rust-backed `Grammar` node can hold Python `Rule` nodes in its children list, and vice versa.
- `isinstance` checks work across PyO3 and Python classes -- they check the Python type object, not the implementation language.
- The parser constructs nodes by name: `fltk.fegen.fltk_cst.Grammar(...)`. If some classes in the CST module are PyO3 classes and others are Python dataclasses, the parser just imports and instantiates them.

**Constraint**: The module path must be the same. The parser imports from `fltk.fegen.fltk_cst` (or the dynamic equivalent). The Rust CST classes must be importable from the same module.

**Practical approach**: Replace one node class at a time. Start with leaf nodes (`Identifier`, `Literal`, `RawString`) that have only `Span` children (simplest child union). Progress to interior nodes. The parser and unparser work at each step.

## 9. Intermediate Step Quality Assessment

### As a standalone deliverable

**Pros**:
- Rust CST nodes are a real, usable product. Python parser/unparser work unmodified.
- Enables Rust-native CST consumers (analyzers, transformers) to access the tree without Python overhead.
- Establishes the Rust build infrastructure (Cargo, PyO3, maturin/setuptools-rust) needed for all future Rust work.
- Tests provide immediate validation: all existing tests pass with Rust CST nodes.

**Cons**:
- Performance benefit is marginal if the bottleneck is tree traversal (children list is still Python). The speedup comes only from reduced per-node object overhead and potential memory layout improvements.
- With Option B (children as `Py<PyList>`), the Rust struct is essentially a thin wrapper around Python objects. The "Rust backing" is nominal.

### As a stepping stone to Rust parser/unparser

**Good intermediate step because**:
- The Rust CST node structs define the data model that a Rust parser would produce. A Rust parser would populate the same `children: Vec<(Label, CstChild)>` (or Rust-native equivalent) without going through Python.
- When the parser is also Rust, `children` can switch from `Py<PyList>` to `Vec<(Label, CstChild)>` -- a Rust-native representation. The Python-facing `#[pyo3(get)]` on `children` would then materialize the Python list on demand, only when Python code accesses it.
- The `isinstance` dispatch used by the unparser maps to Rust `enum` matching when both CST nodes and unparser are Rust.
- The `Span` type would be shared between Rust parser and Rust CST nodes from day one.

**Technical debt risk**:
- If the final goal is a Rust-native `children` representation, the Option B approach (children as `Py<PyList>`) is throwaway work for the `children` field. The typed `append_*/children_*/child_*/maybe_*` methods would remain, but their implementation would change from delegating to the Python list to iterating a Rust Vec.
- The nested-enum workaround (`Grammar_Label` as standalone class) would persist into the Rust-native version and is not throwaway.
- The `__eq__`/`__repr__` implementations would need updating when children becomes Rust-native.

**Net assessment**: Low technical debt. The primary work products (Rust struct definitions, Span type, Label enums, method signatures, build infrastructure) carry forward. The throwaway part is limited to the `children` field implementation strategy.

## 10. Work Estimate

### Scope: Rust+PyO3 CST nodes with Python parser/unparser unchanged

| Component | Estimated LoC (Rust) | Notes |
|---|---|---|
| `Span` PyO3 class | ~40 | `#[pyclass(frozen)]`, `__new__`, `__richcmp__`, `__hash__`, `__repr__`, `start`/`end` getters |
| `UnknownSpan` module constant | ~5 | Module-level `Span(-1, -1)` |
| Per-node `#[pyclass]` struct | ~15 per node | `span: PyObject`, `children: Py<PyList>` |
| Per-node Label enum class | ~10 per label set | Standalone `#[pyclass]` enum + `#[classattr]` attachment |
| Per-node methods | ~5 per method | `append_*`, `extend_*`, `children_*`, `child_*`, `maybe_*`, `append`, `extend`, `child` |
| Per-node `__eq__`/`__repr__` | ~20 per node | Recursive comparison, string formatting |
| Code generator (Python) | ~300-400 | Modify `gsm2tree.py` or write parallel `gsm2tree_rs.py` to emit Rust source instead of Python AST |
| Build system | ~100 | `Cargo.toml`, `pyproject.toml` changes, maturin config |
| Tests | ~200 | Verify all existing tests pass with Rust CST module swapped in |

For the FLTK grammar (`fltk_cst.py`): 12 node classes, ~35 total labels, ~175 methods.

| Item | Estimate |
|---|---|
| Per-node Rust code (12 nodes) | ~600-800 lines Rust |
| Span + constants | ~50 lines Rust |
| Code generator modifications | ~300-400 lines Python |
| Build/packaging | ~100 lines config |
| Tests/integration | ~200 lines |
| **Total** | **~1,250-1,550 lines** |

This is substantially less than any of the four paths in `synthesis.md` because it deliberately excludes Rust parser runtime, Rust unparser runtime, and Rust code generators for parser/unparser algorithms.

### Key decision: generate vs. hand-write the Rust code

The CST node classes are mechanically generated from the grammar. Two approaches:

1. **Generate Rust source** (parallel to `gsm2tree.py`): Write `gsm2tree_rs.py` (~300-400 lines) that emits `.rs` files instead of Python AST. Each grammar produces a Rust module. Benefits: any grammar automatically gets Rust CST nodes. Cost: writing and maintaining a Rust code generator.

2. **Hand-write for FLTK grammar only**: Write the ~600-800 lines of Rust by hand for the 12 FLTK CST classes. Benefits: simpler, no generator. Cost: doesn't generalize; must hand-update when grammar changes; doesn't help user-defined grammars.

Option 1 is better as an intermediate step because it means any grammar processed by FLTK can opt into Rust CST nodes, and it's the same generator infrastructure needed for future Rust parser generation.

## 11. Open Factual Questions

1. **Performance baseline**: No profiling data exists for FLTK parsing/unparsing. Without knowing whether the bottleneck is node construction, children list manipulation, packrat memoization, or unparse resolve_specs, the performance benefit of Rust CST nodes is unquantifiable.

2. **PyO3 nested class workaround verification**: The `#[classattr]` approach for nested enums needs a proof-of-concept to verify that `Grammar.Label.RULE == Grammar.Label.RULE` and `label in (Grammar.Label.RULE, ...)` work correctly through PyO3.

3. **`typing.cast` in generators**: The `children_foo()` methods use `typing.cast(T, child)` which is a no-op at runtime. If Rust methods return `PyObject`, Python type checkers would need stubs (`.pyi` files) for the generated Rust module.

4. **Dynamic module registration**: `plumbing.py:108-112` creates modules via `types.ModuleType` and registers in `sys.modules`. If Rust CST classes come from a compiled extension module, the integration with this dynamic module system needs validation.

5. **Generator expression return**: `children_foo()` returns a Python generator object `(child for label, child in self.children if ...)`. A Rust implementation would return a PyO3 iterator class. Callers do both `for x in node.children_foo()` and `list(node.children_foo())`. The PyO3 iterator must support both patterns (implement `__iter__` returning self and `__next__`).

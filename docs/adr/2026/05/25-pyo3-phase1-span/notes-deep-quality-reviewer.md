**quality-1** — `fltk/fegen/pyrt/terminalsrc.py:44` — Python `merge()` uses `is not` for source identity, but the Python backend's `with_source` takes a raw `str`. Two different `str` literals with the same content are distinct objects in Python, so `self._source is not other._source` fires and raises even for logically-identical sources (e.g., two spans built from the same string variable are safe; two spans built from separately-interned copies of the same string silently raise). The Rust backend correctly uses `Arc::ptr_eq` because the shared-object contract is enforced at construction time (`SourceText` wrapper = one allocation). The Python backend has no such enforcement — callers can create two `Span.with_source` calls with `str(src)` and get a spurious error.

**Consequence:** The mismatch between the two backends' source-identity semantics is invisible at the abstraction boundary. As soon as parse paths begin constructing source-bearing Python Spans (next phase), merge-across-slices will randomly raise depending on whether the string happens to be interned. This is a latent bug that will propagate into any code that merges two Python Spans with sources, with no indication at the call site that the behaviour differs from the Rust backend.

**Fix:** Either (a) add a `SourceText`-style wrapper to the Python backend too (string interned by identity at construction, same pattern as Rust), or (b) change the Python `merge` check to compare by value (`self._source != other._source`) — value equality for Python strings is always correct because they are immutable, and the performance cost at merge time is negligible for the sizes involved in source text. Option (b) is a one-line change. The design explicitly acknowledges that Python strings are "already immutable and shared," which is true for refcounted sharing, but does not guarantee identity equality for separately-constructed strings. The Rust backend's choice of `Arc::ptr_eq` is correct; the Python backend should use `!=` (value) not `is not` (identity).

**quality-2** — `fltk/fegen/pyrt/terminalsrc.py:59` / `fltk/fegen/pyrt/span.py:8` — `Span.with_source` is a classmethod in both backends but its signature differs in type: Python takes `source: str`, Rust takes `source: SourceText`. The backend-selector module (`span.py`) re-exports whichever `Span` is active, making `Span.with_source` callable from common code — but any caller that uses it must either know which backend is active or provide both arg forms. `SpanProtocol` does not include `with_source` (intentional per design), yet `span.py` is meant to be the "recommended import point for backend-agnostic code" (design §Backend Selection). The asymmetry is structurally invisible — the selector makes `Span` look uniform but the construction API is not.

**Consequence:** Future code written against `from fltk.fegen.pyrt.span import Span` that calls `Span.with_source(start, end, src_str)` works when Rust is absent, silently breaks when Rust is present (wrong arg type). This will be discovered at runtime, not at type-check time, because `# type: ignore[assignment]` on the import suppresses the type information. Every new phase that produces source-bearing spans will have to re-discover and re-work around this asymmetry.

**Fix:** The cleanest resolution for Phase 2 is to expose `SourceText` in the Python backend as a trivial `str`-wrapping class (or just `str` itself via an alias), unifying the constructor signature. A simpler but less clean option: document the asymmetry explicitly in `span.py` with a note that `with_source` is not portable and `SpanProtocol` intentionally omits it, so callers must branch on backend — a `TODO(backend-with-source-signature)` in `TODO.md` at minimum. Currently there is no indication this is a trap.

**quality-3** — `fltk/fegen/pyrt/span.py:5` — `SourceText = None` at module level, then conditionally overwritten from the Rust import. The result is that `SourceText` has type `None` (i.e., `type[None]`) statically when Rust is unavailable, and `type[SourceText]` when available. Code that does `if SourceText is not None: src = SourceText(...)` is forced to use runtime branching rather than static typing. The implementation-log notes this is a "workaround for pyright `Final` reassignment error" — specifically, the design intended `try: ... except ImportError: ...` with Rust first, but pyright rejects reassignment of a `Final`-annotated name. The implemented order (Python default, then conditional Rust override) is correct at runtime but the `None` sentinel creates a stringly-typed-adjacent pattern: `SourceText` is not typed as `type[SourceText] | None`, it is typed as `None`.

**Consequence:** Any caller that wants to create a `SourceText` must check `if SourceText is not None` and will get no type narrowing help from pyright, since the module-level declaration is `SourceText = None` (inferred as `None`, not `type[SourceText] | None`). This is a leaky abstraction — the `None` default leaks the implementation detail that Python backend has no `SourceText`.

**Fix:** Declare `SourceText: type | None = None` (or `Optional[type]`) so the type is accurate and mypy/pyright can narrow. Alternatively, export a no-op Python `SourceText` stub from `terminalsrc.py` that takes a `str` and returns it, making `SourceText` always non-None — this also resolves quality-2's signature asymmetry.

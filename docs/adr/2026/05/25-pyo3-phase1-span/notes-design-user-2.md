# User notes on design, round 2 (verbatim)

OK, the is_ascii thing is badly reasoned; this design claims that ASCII is the common case but every modern language and compiler treats source text as UTF-8 even if it happens ot be ascii, and cannot generally assume that it's ASCII. We will pay the O(N) tax in every case even if a particualr source text happens to be ASCII.

The correct solution is to abandon the guarantee that Python implementation currently provides taht start and end are codepoint indices. There is no *need* for this guarantee to be provided, once you provide users a defined API for retrieving a source span. If we change the interface contract from "these are codepoint indices" to "these are abstract indices and you need to use the provided methods to turn them into source text or compute length instead of slicing an external string yourself" then we have the freedom to make them codepoint indices in Python and byte indices in Rust, and `is_ascii` and the O(N) indexing both go away. In fact, in Rust I'd make them private and force everything through the access methods. That way the future Rust code will be cleaner (and maybe Python will get cleaned up over time). We may need to add some methods like `len()` and `is_empty()`, maybe `merge()` and `intersect()` to handle operations people might otherwise reach for the indexes directly to perform.

## design-1: `requirements_lock.txt` regeneration path is ambiguous and the design's instruction may not produce the pygls entries

Section: "4. Stretch: launch from a Bazel `@fltk//` submodule", step 1 — "Add `args =
["--extra", "lsp"]` ... and regenerate `requirements_lock.txt`."

What's wrong: the design implies the Bazel `lock` target is what produces the committed
lock file, but the committed file's own header says it was generated by a different tool:
`requirements_lock.txt:1-2` reads "autogenerated by uv via ... `uv export --format
requirements-txt --no-editable --output-file requirements_lock.txt`". The Bazel `lock` rule
runs `uv pip compile` (rules_python `lock.bzl`), a different command with different output
format (e.g. it injects a `--custom-compile-command` header, `lock.bzl:98`). The exploration
report itself carries both facts side by side (exploration.md line 13) without reconciling
them; the design inherited the "lock target produces it" half. No Makefile target or doc in
the repo names a canonical regeneration command (checked Makefile, docs/, MODULE.bazel).

Why: two regeneration paths now exist that disagree. An implementer who follows the file's
own header and reruns `uv export` without `--extra lsp` gets a lock file with no pygls —
the `args` addition to the Bazel rule changes nothing about the committed file unless the
file is regenerated *via* `bazel run //:requirements`. Conversely, regenerating via the
Bazel target rewrites the file in `uv pip compile` format, silently changing its provenance
and potentially its resolution relative to `uv.lock`.

Consequence: either `@pypi//pygls` never comes into existence (the new `py_binary` fails to
build — the stretch goal dead on arrival), or the lock file format/provenance churns without
that being a called-out decision. Also note the `uv export` path needs its own `--extra lsp`
flag, which the design never mentions.

Suggested fix: the design should state which command is canonical for regenerating
`requirements_lock.txt` (and update the other path or delete it). If `uv export` remains
canonical, the fix is `uv export --extra lsp ...`, and the `lock` rule's `args` should be
changed to match (or the discrepancy explicitly accepted); if the Bazel `lock` target
becomes canonical, say so and accept the header/format change deliberately.

## design-2: `def name: rule` / `ref ...: rule` use an out-of-legend kind, so the design's own test plan item 2 ("a rule-name def paint") cannot pass against the spec as sketched

Section: "1. Complete the spec triples" — `fegen.fltklsp`: "`rule rule { def name: rule; }`"
and "`rule term { ref rule:identifier: rule; }`"; Test plan item 2: "e.g. for `.fltkg`, a
rule-name def paint, ...".

What's wrong: def-site declaration paint is emitted only when the def kind's first segment
is in the token legend: `lsp_config.py:691-694` (`Paint(...) if def_stmt.kind and
def_stmt.kind[0] in TOKEN_LEGEND else None`). The legend is `features.SEMANTIC_TOKEN_TYPES`
(`features.py:35-52`): keyword, comment, string, number, operator, punctuation, variable,
parameter, property, type, function, enumMember, constant, macro, label, text — no `rule`.
Likewise resolved references only paint for in-legend kinds (`classify.py:287-292`:
"Unresolved references and out-of-legend kinds contribute [nothing]"). So with kind `rule`,
navigation/symbols/rename work (kinds are open vocabulary, `validate_config` docstring), but
rule names at definition sites and every rule reference fall through to the default
`variable` paint.

Why: internal inconsistency — section 1 sketches kind `rule`, while test plan item 2 pins "a
rule-name def paint" for `.fltkg`, which that kind cannot produce. It also undercuts the
"flagship dogfood" highlighting quality: both in-tree precedents paint rule names as a real
legend type (`fltklsp.fltklsp:11` `scope rule_name: type;`; `test_dogfood.py:27`
`def rule_name: type;`).

Consequence: as designed, either test 2 fails, or the implementer silently changes the kind
during implementation — an undesigned decision on the most user-visible surface (what color
grammar-rule names and references get in every downstream `.fltkg` file).

Suggested fix: use an in-legend kind (`type`, matching both precedents) for the def and ref
kinds, or keep kind `rule` for semantic flavor and add explicit `scope name: type;` /
`scope rule:identifier: type;` paints — and align test plan item 2's wording with the
choice.

## design-3: "three grammars and five sidecar specs" — the count is six

Section: "Packaging" — "The three grammars and five sidecar specs live under `fltk/`'s
package tree".

What's wrong: after section 1, each of the three languages has both a `.fltklsp` and a
`.fltkfmt`: 4 new files (fegen.fltklsp, unparsefmt.fltklsp, unparsefmt.fltkfmt,
fltklsp.fltkfmt) + 2 existing (fegen.fltkfmt, fltklsp.fltklsp) = six sidecars, not five.
(The wheel-verification list in the same paragraph is a list of the five files that exist
*today*, which is probably where the number leaked from.)

Consequence: minor — a reader or implementer cross-checking packaging/test coverage against
"five" may conclude one file is intentionally excluded from the wheel. The registry and test
plan item 1 do cover all six, so this is a consistency slip, not a coverage gap.

Suggested fix: say "six sidecar specs", or phrase it as "the grammars and all sidecar specs".

## design-4: test plan item 6 spawns the `fltk-grammar-lsp` console script; the existing e2e harness deliberately uses `python -m` instead

Section: Test plan item 6 — "a pytest-lsp test (same harness as ... `test_server_crossfile.py`)
spawning `fltk-grammar-lsp fltkg`".

What's wrong: the cited harness does not spawn a console script; it spawns
`[sys.executable, "-m", "fltk.lsp.server_cli", ...]` (`test_server_crossfile.py:62-66`).
A console-script binary only exists in the venv after the editable install is refreshed
(`maturin develop` re-run after the `[project.scripts]` addition) and only resolves if the
venv `bin` is on the subprocess PATH.

Consequence: a test that spawns the literal `fltk-grammar-lsp` fails with a confusing
FileNotFoundError in any environment with a stale editable install (the exact papercut the
existing `-m` pattern avoids), and behaves differently under CI vs local runs.

Suggested fix: follow the cited pattern literally — `[sys.executable, "-m",
"fltk.lsp.grammar_cli", "fltkg"]` (which also implies `grammar_cli.py` needs the same
`if __name__ == "__main__": app()` guard `server_cli.py:79-80` has — also required for the
Bazel `py_binary`).

## design-5: "lazy per-language client start ... serializes typical startups" overstates the Bazel lock mitigation

Section: "4. Stretch" — "(a) concurrent `bazel run`s contend for the workspace lock —
mitigated by the lazy per-language client start, which serializes typical startups".

What's wrong: lazy start gates each client on the first *appearance* of a document of that
language, but VS Code session restore reopens all previously open editors at activation
time; a workspace with a `.fltkg` and its sibling `.fltkfmt`/`.fltklsp` open (the normal
state for exactly this feature's audience) starts multiple clients in the same activation
tick, so the `bazel run`s race for the workspace lock on what is arguably the *typical*
startup, not the atypical one. Nothing in the lazy-start design adds actual serialization.

Consequence: docs-level only (the path is already labeled experimental with the
`--script_path` escape hatch), but the README this section drives would tell users the
common failure mode is unlikely when it is in fact the default multi-file-workspace
behavior; bazel's client does queue on the lock, so the practical symptom is slow/timed-out
client startups, not corruption.

Suggested fix: drop the "mitigated by lazy start" claim; document lock contention as
expected whenever more than one fltk language is open and point at `--script_path` (or a
shared wrapper) as the primary answer rather than the escape hatch.

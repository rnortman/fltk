### CLI (`fltk.lsp.server_cli`)

`/home/rnortman/src/fltk/fltk/lsp/server_cli.py`:
- `app = typer.Typer(name="fltk-lsp", ...)`, single `main()` command (lines 36-76).
- Required: `--grammar` (`.fltkg`). Optional: `--lsp` (`.fltklsp`), `--fmt` (`.fltkfmt`), `--rule` (start rule), `--width`/`--indent` (formatting), `--resolver` (`module.path:attr` or `file.py:attr`, cross-file plugin).
- Lazily imports `fltk.lsp.server.create_server` (line 50) so a missing `pygls` prints `"fltk-lsp requires the 'lsp' extra: pip install 'fltk[lsp]'"` and exits 1 (lines 51-53) rather than crashing on import.
- Builds `AnalysisEngine.from_paths(grammar, lsp, start_rule=rule)`, optional `plumbing.parse_format_config_file(fmt)`, optional `load_resolver(resolver)`, then `create_server(engine, formatter_config, renderer_config, resolver=resolver_obj)` and `server.start_io()` (lines 55-76). One process serves exactly one language/grammar (module docstring, lines 1-13): "editors spawn a separate server per language."
- No support currently for a single config file bundling `--grammar`/`--lsp`/`--fmt`/`--resolver` — all four are separate CLI flags; the ADR's D4 (`docs/adr/2026/07/06-fltklsp-lsp-server/README.md` lines 61-64) mentions "or via a workspace config file" as an alternative shape but the implemented CLI only has flags.

### CLI (`fltk.lsp.highlight_cli`)

`/home/rnortman/src/fltk/fltk/lsp/highlight_cli.py`: `fltk-highlight FILE --grammar G [--lsp L] [--rule R]` — standalone ANSI highlighter over stdout, no pygls/server dependency at all (imports only `fltk.lsp.engine.AnalysisEngine` and `fltk.lsp.classify.Token`).

### Server internals

`/home/rnortman/src/fltk/fltk/lsp/server.py` (`create_server`, `_server_version` via `importlib.metadata.version("fltk")`, lines 57-62): pygls `LanguageServer` wiring — capability advertisement, position-encoding negotiation (`_constrain_pygls_encodings`, lines 70+), per-document debounced analysis (`_DEBOUNCE_SECONDS = 0.2`, line 67), diagnostics, stale-token serving, lazy formatting. Requires the `lsp` extra to import (module docstring lines 9-11).

`/home/rnortman/src/fltk/fltk/lsp/resolver.py`: the cross-file resolver plugin contract (**explicitly provisional**, docstring lines 15-17, 61-64: "subject to change until validated against a real downstream language... exactly one in-repo implementation (the `gear` demo)"). Key surface:
- `Resolver` protocol (`file_suffixes: Sequence[str]`, `resolve(doc, host) -> CrossFileResolution`, lines 132-144).
- `ResolverHost` protocol (`document`, `workspace_files`, `root_path`, `uri_to_path`, `path_to_uri`, lines 105-129).
- `load_resolver(spec)` (lines 150-174): parses `module.path:attr` or `path/to/file.py:attr` (default attr `create_resolver`, `_DEFAULT_ATTR`, line 147); `_looks_like_path` (lines 193-203) decides file-vs-module by an existing-file check, `.py` suffix, or path separator (flagged with an open `TODO(resolver-spec-file-recognition)`, lines 196-202, about cwd-dependent recognition).

## 2. The gear demo — end-to-end wiring

Files under `/home/rnortman/src/fltk/examples/gear/` (excluding `node_modules`):
```

```

- **`gear.fltklsp`** (`/home/rnortman/src/fltk/examples/gear/gear.fltklsp`): global `scope string: string;`, `scope number: number;`, `scope ty: type;`, plus `rule` blocks for `import_item`, `shape_def` (with `namespace;`), `field`, `const_def`, `fn_def` (with `namespace;`), `param`, `let_stmt`, `name_ref` (`ref name: *;`), `call`.
- **`gear_resolver.py`** (`/home/rnortman/src/fltk/examples/gear/gear_resolver.py`): reference `Resolver` implementation. `GearResolver.file_suffixes = (".gear",)`; `resolve()` walks `USESTMT`/`IMPORTITEM` CST nodes, maps `a::b` module paths to `<root>/a/b.gear` via `host.root_path()`/`host.path_to_uri()`, calls `host.document(target_uri)`, and redirects both the imported name and its alias to an `ExternalTarget` built from the target's `Symbol` offsets (`symbol_targets`, lines 56-120). Explicitly validates path segments (`part.isidentifier()`, line 83) against traversal.
- **Command line** (README lines 26-32): `uv --project <repo> run --extra lsp fltk-lsp --grammar examples/gear/gear.fltkg --lsp examples/gear/gear.fltklsp --fmt examples/gear/gear.fltkfmt --resolver examples/gear/gear_resolver.py:create_resolver`. README calls `--extra lsp` "load-bearing" since a plain `uv run` only syncs default groups (line 34-36).
- **VS Code client** (`examples/gear/vscode/`):
  - `extension.js`: `repoRoot()` computes `<repo>` via `path.resolve(__dirname, "..", "..", "..")` from the extension's own file location (lines 10-13); `defaultCommand()` builds the exact `uv --project <root> run --extra lsp fltk-lsp --grammar ... --lsp ... --fmt ... --resolver ...:create_resolver` argv (lines 19-39); `serverCommand()` (lines 41-47) lets a `gear.server.command` workspace setting override the array; `activate()` (lines 49-67) starts a `vscode-languageclient` `LanguageClient` over stdio (explicit comment, lines 51-55, that pygls does not accept a `--stdio` flag, so `transport` must be omitted).
  - `package.json`: `"activationEvents": ["onLanguage:gear"]`, `"main": "./extension.js"`, `contributes.languages` registers `.gear`, `contributes.configuration` declares `gear.server.command` as a **machine-scoped** array setting (`"scope": "machine"`, line 44) — comment explains this is deliberate so a cloned repo's `.vscode/settings.json` cannot redirect the launched executable (package.json line 45's markdownDescription).
  - No `.vsix` packaging step is committed; README (lines 69-86) documents `npx @vscode/vsce package` manually and warns that outside the repo the relative-root default breaks and `gear.server.command` must be set to an absolute `uv --project /path/to/fltk ...` argv.
- **Prerequisites called out in README** (lines 40-47): Node/npm (only for the extension's one dependency, `vscode-languageclient`), the Rust toolchain (`rustup`+`cargo`, because `fltk` is a mixed Python/Rust package and `uv run` cannot build it without Rust), and a slow first launch (one-time maturin debug build).
- **Tests**: README references `fltk/lsp/test_gear_demo.py` and `fltk/lsp/test_server_crossfile.py` as the automated coverage for the non-VS-Code half (the language + resolver).
- **Acceptance checklist** (README lines 88-109) is a manual, VS-Code-Extension-Development-Host procedure (F5 launch, open `examples/gear/sample/`, exercise highlighting/formatting/go-to-def/find-refs/live degradation) — not automated.

Detail on the two that exist:
- `fltk/fegen/fegen.fltkfmt`: a real formatting spec (`trivia_preserve: LineComment, BlockComment;`, per-rule `group`/`nest`/`after`/`before` blocks for `rule`, `alternatives`, `items`, `item`, `term`, `block_comment`, `_trivia`). Used by the existing formatter/unparser pipeline, not by any LSP CLI invocation currently.
- `fltk/lsp/fltklsp.fltklsp`: highlights the `.fltklsp` language itself against `fltklsp.fltkg` — global `scope ";", ":": punctuation;` plus `rule` blocks for `rule_config`, `scope_stmt`, `def_stmt`, `ref_stmt`, `namespace_stmt`, `qualifier`, `dotted_name`, `literal`. It carries **only `scope` statements** (highlighting), no `def`/`ref`/`namespace` — confirmed by its own file comment (line 2) and by `fltk/lsp/test_dogfood.py`'s comment (lines 21-24): "a good def/ref/namespace vocabulary for the `.fltklsp` language is its own design question, so semantics dogfooding uses this test-local spec instead" (a separate, not-committed-as-a-file `_SEMANTIC_SPEC` string, lines 25-33, used only inside the test).
- **`fltk/lsp/test_dogfood.py`** is the only current consumer of `fltklsp.fltklsp`: it loads `AnalysisEngine.from_paths(_GRAMMAR_PATH, _SPEC_PATH)` (`_GRAMMAR_PATH`/`_SPEC_PATH` = `fltklsp.fltkg`/`fltklsp.fltklsp` in that directory) purely inside `pytest` assertions on token types (`test_dogfood_spec_highlights_a_sample`, etc.) — it is not wired to `fltk-lsp`/`fltk-highlight` CLI invocation, VS Code, or any packaging.
- No `.fltkg`/`.fltkfmt`/`.fltklsp` triple is complete for any of the three grammars today; none of the three has ever been run through `fltk-lsp` or `fltk-highlight` outside of unit-test-level API calls.

Also present but out of scope per the request: `fltk/lsp/test_data/greet.{fltkg,fltkfmt,fltklsp}` — a minimal fixture trio used by LSP test modules (`test_load_lsp_config.py`, etc.), not one of the three named grammars.

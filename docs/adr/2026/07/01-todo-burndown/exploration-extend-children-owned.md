## `extend_children` implementation — current source

`fltk/fegen/gsm2tree_rs.py:1232-1242` emits, for every generated node struct:

```rust
/// ...
pub fn extend_children(&mut self, other: &Self) {
    self.children.extend(other.children.iter().cloned());
}
```

`children` is `Vec<(Option<LabelEnum>, ChildEnum)>`. `ChildEnum` variants holding
CST nodes hold `Shared<T>` (`Arc<RwLock<T>>`); `Clone` on `Shared<T>` is
`Arc::clone`, an atomic refcount increment. Span-typed children clone `Span`,
which internally holds an `Arc`-backed source text and is also cloned. `iter().cloned()`
does this once per child; when the donor drops immediately after, each Arc is
decremented again. One atomic inc+dec pair per child, as claimed.

## Is the donor always dropped immediately?

Checked every `extend_children(&...)` call site currently generated in
`crates/fegen-rust/src/parser.rs` (12 occurrences: lines 171, 327, 350, 516, 557,
636, 658, 668, 745, 1216, 1230, plus the identical pattern repeated in
`tests/rust_parser_fixture/src/parser.rs` and other fixture crates).

Every occurrence follows one of two shapes, both generated by the same two
Python functions:

- Loop path (`_gen_item_multiple`): `while let Some(one_result) = { .. } { ...; result.extend_children(&one_result.result); }` — `one_result` is the `while let` binding, re-bound and dropped at the top of every iteration.
- Non-loop path (`_gen_append_code`): `if let Some(itemN) = self.parse_xxx(pos) { pos = itemN.pos; result.extend_children(&itemN.result); }` — `itemN` is scoped to the `if let` arm and drops at its closing brace.

In every sampled call site, the bound variable (`one_result` / `itemN`) is used
only for `.pos` (already copied out before the `extend_children` call) and
`.result` (passed by reference to `extend_children`), then never referenced
again in that scope. No call site retains, returns, or re-uses the donor after
the call. `.result` in all these functions has type `cst::NodeType` (a bare,
non-`Shared` struct) — these functions are the non-memoized inline/loop helpers,
not the `apply__`-wrapped top-level rule parsers (which return
`Shared<cst::NodeType>` and never call `extend_children`). Since the packrat
cache (`Cache<T>` in `fltk-parser-core/src/memo.rs`) stores only `Shared<T>`,
and these bare-struct intermediates never enter it, the donor cannot be aliased
by the cache at the call site. This matches the three prior explorations'
conclusion exactly, and I found no exception.

## Does `impl Drop` (added since the 06-11 exploration) newly block the fix?

No — and this was already anticipated by the 06-12 (first triage) exploration.
`fltk/fegen/gsm2tree_rs.py:1111` now emits `impl Drop for {class_name}` for
node structs (iterative-worklist drop, added for the deep-tree-abort fix
referenced in `docs/adr/2026/06/11-rust-cst-debug-depth/design.md`). A type with
`impl Drop` cannot have fields moved out of it while a live binding exists
(`E0509`), but a hypothetical `extend_children_owned(mut other: Self)` would
call `self.children.append(&mut other.children)` — a mutable *borrow* of a
field, not a move-out — and `other` itself is consumed as a whole value by the
function signature (an ordinary move of the whole struct into the function,
which `Drop` does not forbid). No `E0509` conflict. Confirmed by direct
inspection of the current `Drop` impl shape; no partial-field-move pattern is
implicated by the proposed API.

## Is "blocked on gsm2tree_rs.py" a real blocker?

Factually accurate but overstated as an external dependency, matching what the
06-12-triage-2 exploration already flagged: `gsm2tree_rs.py` and
`gsm2parser_rs.py` are both files in this repo, edited together in one commit.
The generated CST API genuinely lacks the method today (confirmed:
`grep -rn extend_children_owned` across `fltk/`, `crates/`, `tests/` returns
zero hits except the descriptive comment mentioning the not-yet-existing name).
So the technical precondition is real (the method must exist before the parser
generator can call it), but it is a same-repo, same-commit coordination
requirement, not a blocker on an external party or a separate release.

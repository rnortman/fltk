# Deep security review — round 2 (M2 fltk-lsp server)

Reviewed: 9719bab7..d9ab841 (HEAD d9ab841). Design: `step2/design.md` (frozen).

## security-1

- **File:line**: `fltk/lsp/server.py:45-59` (`_constrain_pygls_encodings`) and `fltk/lsp/server.py:127-132` (`_encoding`), with `pyproject.toml` pin `pygls>=2,<3`.
- **Issue**: The single-encoding-owner invariant ("whatever pygls advertises, this server can honor") is enforced only by reassigning a pygls *private* module global, `pygls.capabilities._SUPPORTED_ENCODINGS`, whose stock value includes `utf-8`. The pin floats across all pygls 2.x minors, so the private can be renamed/moved in a routine dependency bump; the assignment then silently creates a dead attribute and pygls resumes negotiating from its own set, advertising `utf-8` to any client that prefers it. On the read-back side, `_encoding()` silently coerces every advertised value other than `utf-32` — including `utf-8` — to `PositionEncoding.UTF16`, so the divergence is never detected: the server advertises one encoding and computes positions in another. Verified against the currently installed pygls: the private exists today and the rebinding is effective (`choose_position_encoding` reads the module global), so this is a drift hazard, not a live bug.
- **Trust boundary / data flow**: untrusted workspace document text (any file in an opened repo, attacker-authorable) → `LineIndex` column math in utf-16 units → wire `Range`s/token deltas interpreted by the client in utf-8 units. The mismatch is proportional to non-ASCII content, which the document author controls.
- **Consequence**: Under the drifted-dependency condition, every emitted coordinate is wrong on non-ASCII lines. The damaging case is `textDocument/formatting`: `_format_blocking` returns one whole-document `TextEdit` whose end position comes from `LineIndex.end_position(self._encoding())` (`server.py:325-330`) — in utf-16 units against a client counting utf-8 units, that end falls short of the real end of file, so the client replaces only a prefix and leaves a duplicated tail: silent corruption of the user's file on an explicit format action. The §4.8 verify-reparse guard cannot catch this — it validates the rendered text server-side, before the mis-ranged edit is applied client-side. A hostile repo can maximize the effect by salting files with astral-plane characters; more prosaically it destroys any non-ASCII document. This defeats the design's stated guarantee that formatting "must never destroy" a document.
- **Suggested fix**: Fail closed at both ends. (a) In `_constrain_pygls_encodings`, verify the attribute exists before assignment (`getattr` check) and raise at import/startup if it does not, so a pygls bump that breaks the mechanism is a loud startup failure, not a silent utf-8 advertisement. (b) In `_encoding()`, treat any advertised value outside `{utf-16, utf-32}` as a hard error (raise / log-and-refuse-request) instead of defaulting to UTF16. (c) Optionally tighten the pygls pin to a verified minor range. A regression test asserting the private still exists in the installed pygls would make (a) cheap to keep honest.

## Checked and clear (for the judge's calibration)

- **Code injection via workspace-supplied grammar/spec files**: the engine docstring correctly declares grammar, `.fltklsp`, and document text untrusted, and both exec-codegen paths the server triggers (analysis parser at startup; standard parser + unparser lazily on first format, `server.py:279-282`) are injection-proof for file-sourced input: grammar identifiers are lexically confined to `[_a-z][_a-z0-9]*` (`fegen.fltkg:16`), all grammar/config strings reach generated source only via `repr()` (`iir/py/compiler.py:320-321`), and `.fltklsp` literals go through `ast.literal_eval` (`lsp_config.py:142`). No raw interpolation of untrusted text into exec'd source found.
- **Transport**: stdio only (`start_io`); no TCP/websocket listener, no network-reachable surface added.
- **Secrets**: none in the diff (test fixtures are toy grammars; uv.lock adds hashes only).
- **Deserialization / eval**: `literal_eval` only; no pickle/eval of untrusted data.
- **DoS via runaway parse/ReDoS on untrusted documents** (grammar regexes compiled and run over document text; single analysis worker wedgeable): explicitly dispositioned by the frozen design (§2.3, §5) with `TODO(lsp-analysis-watchdog)` present in `server.py:142` and `TODO.md`; not re-raised.
- **Diagnostics/log content**: parse-error messages (including source-line excerpts) and exception reprs flow only back to the same local client that supplied the document; no cross-boundary disclosure.

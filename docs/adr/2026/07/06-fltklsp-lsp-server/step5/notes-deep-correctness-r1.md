## correctness-1

- **File:line**: `fltk/lsp/resolver.py:229-241` (`_instantiate`), with `_validate` at `resolver.py:244-260`.
- **What's wrong**: `_instantiate` returns `obj` as-is whenever `hasattr(obj, "resolve")` is true тАФ before considering whether `obj` is a class. A `Resolver` *class* (e.g. `--resolver examples/gear/gear_resolver.py:GearResolver`, the most natural spec mistake next to the documented `:create_resolver` factory) has a `resolve` function attribute, so the class object itself is accepted uninstantiated. `_validate` then passes too: `GearResolver.resolve` is callable (an unbound function), and `file_suffixes` is a class attribute, so every check succeeds and the server starts.
- **Why**: at request time the server calls `self._resolver.resolve(doc, host)` (`server.py:_rename_guard_blocking`, `project.py:282`). With `self._resolver` being the class, that is `GearResolver.resolve(doc, host)` тАФ the unbound method receives `doc` as `self` and `host` as `doc`, and is missing its third argument тЖТ `TypeError` on every single call, forever.
- **Consequence**: the exact outcome ┬з4.3 of the design forbids ("a broken resolver is a startup error, never a half-working server"). Startup validation passes; then every definition/references request permanently degrades to same-file (with a per-request error log), and every rename is permanently refused via the fail-closed guard ("could not verify cross-file references") тАФ including renames of purely local `let` bindings. The user sees a server that mostly works but never navigates cross-file and never renames, with no startup-time hint.
- **Suggested fix**: in `_instantiate`, treat classes as factories: check `isinstance(obj, type)` (or `inspect.isclass`) *before* the `hasattr(obj, "resolve")` shortcut and invoke it; alternatively, after instantiation, validate that `resolver.resolve` is a *bound* callable accepting `(doc, host)` (e.g. reject `isinstance(resolver, type)` in `_validate`). Add the class-spec case to `test_resolver_api.py` (currently only "no resolve + not callable" is pinned).

## correctness-2

- **File:line**: `fltk/lsp/project.py:173-175` (`_identity`), `project.py:210-224` (`references` matching), `project.py:146-149` (`_source` open-docs lookup), `project.py:235-244` (`_scan_docs` dedup), `fltk/lsp/server.py:_rename_guard_blocking`; URI producers at `project.py:109-110` (`path_to_uri` = `uris.from_fs_path`) vs client-supplied URIs (`self.workspace.text_documents` keys, request `params.text_document.uri`).
- **What's wrong**: every cross-file identity comparison is a *textual string* comparison between URIs from two different serializers. Resolver-side URIs are produced by pygls `uris.from_fs_path` (via `host.path_to_uri`, e.g. `gear_resolver.py:77`); requesting-document and open-buffer URIs are whatever the client sent. Nothing normalizes either side. The e2e tests construct both sides with `uris.from_fs_path` (`test_server_crossfile.py:44`), so the equality is true by construction there тАФ but real clients serialize differently: VS Code on Windows sends `file:///c%3A/...` (lowercase drive, percent-encoded colon), which never string-equals pygls's `from_fs_path` output for the same file; percent-encoding differences (spaces, non-ASCII) are a same-class hazard on any platform.
- **Why**: three comparisons depend on the equality holding: (a) `_source`'s `self._open_docs.get(uri)` тАФ a resolver-constructed URI for an *open* file misses the snapshot and falls through to disk; (b) `references`' `_canonical_identity(...) != target_id` тАФ an inbound import binding in file B whose redirect target was built with a `from_fs_path` URI never matches a `target_id` built from the client's URI for the same file; (c) `_scan_docs`'s `seen` set тАФ the requesting file can be scanned twice under its two spellings.
- **Consequence**: on any client whose URI serialization differs from pygls's (Windows VS Code being the concrete case тАФ and Windows is explicitly in scope, cf. the drive-letter handling in `resolver.py:_split_spec`):
  - Unsaved edits stop participating in cross-file resolution (disk read instead of open-buffer snapshot), contradicting ┬з4.2's stated behavior;
  - find-references silently drops all cross-file inbound occurrences, and may emit duplicate locations for the requesting file under two URI spellings;
  - worst: the **rename refusal guard fails open** тАФ `_rename_guard_blocking` compares occurrences' URIs against `doc.uri` (client form) while inbound references carry the `from_fs_path` form of *other* files, but the target-identity mismatch in (b) means cross-file inbound references are not even found, the verdict is `"ok"`, and a same-file rename of a symbol other files import proceeds тАФ the exact silent project corruption ┬з2.3/┬з4.6 exist to prevent, on the path documented as fail-*closed*.
- **Suggested fix**: canonicalize URIs at the boundaries тАФ e.g. round-trip every incoming URI through `uris.to_fs_path` тЖТ `uris.from_fs_path` (or compare `pathlib.Path.resolve()`d filesystem paths for identity instead of URI strings) in `ProjectHost`/`ProjectNavigator`, and key the open-docs snapshot by the canonical form. Add a test with a deliberately non-canonical (percent-encoded) client URI.

No other logic findings: the worker-thread snapshot discipline (open-docs map + workspace root captured synchronously on the loop thread after the `_serveable_for` await, no interleaving await before submit), the rename guard's post-await version re-check, the last-good `text` field threading through `_analyze_blocking`/`_store`, the navigator's selection-range-only identity and include_declaration handling, and the gear resolver's one-hop alias/name redirect all trace correctly against the design.

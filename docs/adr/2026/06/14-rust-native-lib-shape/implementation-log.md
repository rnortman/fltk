## Bug fix — cross-repo genrule tool resolution in `fltk_pyo3_cdylib` (commit TBD)

- `rust.bzl:210-222`: in the auto-generate-lib_rs genrule (`name + "_gen_lib"`), changed bare `"//:genparser"` string in `tools` to `Label("//:genparser")`, and changed `$(location //:genparser)` in `cmd` to `$(location @fltk//:genparser)`.
- Root cause: bare string labels passed to genrule `tools` in a Starlark macro resolve at rule-instantiation time in the caller's repo context. When clockwork loaded `@fltk//:rust.bzl` and called `fltk_pyo3_cdylib`, `//:genparser` resolved to `@clockwork//:genparser` (which does not exist). Using `Label("//:genparser")` (which resolves relative to the `.bzl` file's repo, i.e. `@fltk`) fixes the `tools` list; `@fltk//:genparser` as the explicit canonical string in `cmd` ensures `$(location ...)` matches.
- This mirrors the existing pattern at `rust.bzl:282-289` (Label() for rust_shared_library deps) and the `_gen_tool` attr default at line 93.
- Verified: `bazel build //:bootstrap_native` passes in fltk; `bazel build //clockwork/dsl:clockwork_native` and `bazel test //clockwork/dsl:clockwork_rust_roundtrip_test` both pass in clockwork.
- Regression guard note: no in-tree fltk test exercises the `lib_rs == None` auto-generate path from a consumer repo (the bootstrap_native target supplies its own lib_rs). The clockwork build is the de facto regression test for this cross-repo bug; it cannot be replicated purely in-tree without a separate consumer Bazel workspace.

# Requirements review: codegen-rust-lib-boilerplate

Reviewer: requirements-reviewer. Sources: request (verbatim), exploration.md, requirements.md. Did not read code.

## requirements-1 — "fix its shape" intent under-weighted; equivalence constraints may over-constrain

Section: System behavior → "byte-equivalent or behaviorally-equivalent module" (acceptance, line 63-65);
fltk._native special case → "behaviorally equivalent to the current hand-written one" (line 84);
Constraints → "Cross-backend equivalence ... unchanged versus the current hand-written lib.rs" (line 139-140).

What's wrong: The verbatim request says "we can **fix its shape**" — the user is explicitly signalling that the
current hand-written boilerplate's *shape* is not sacred and may be improved/normalized as part of codegenning it.
The requirements doc instead pins the generated output to be equivalent to today's hand-written files
("byte-equivalent or behaviorally-equivalent", "behaviorally equivalent to the current hand-written one",
"unchanged versus the current hand-written lib.rs"). This is a defensible conservative reading, but it elevates
"reproduce exactly what exists" to a hard acceptance criterion when the requester actively invited reshaping.

Why: Request — "there's really no reason for that... it's boilerplate, and **we can fix its shape**." The doc never
acknowledges that the shape is allowed to change; it only ever frames success as reproducing the current files.

Consequence: A designer reading this will treat the existing lib.rs structure (e.g. fltk._native's one-time
Span/SourceText/UnknownSpan registration sitting inline in the module fn, the specific submodule names, the
ordering) as a fixed target to reverse-engineer and replicate, rather than as boilerplate that could be
regularized into a cleaner generated shape. If reshaping would actually simplify the generator (e.g. a uniform
template that fltk._native also fits, vs. a bespoke special-case path), that option is foreclosed by the
equivalence criteria. Risk of building a more complex "exact replica" generator than the user wanted.

Suggested fix: State that behavioral equivalence at the Python-import surface (which submodules/classes are
exposed, UnknownSpan semantics) is required, but the *internal shape* of lib.rs may be normalized/changed where it
reduces boilerplate or simplifies generation — that is explicitly sanctioned by the request. Drop or soften
"byte-equivalent."

## requirements-2 — fltk._native scope is genuine but its cost/benefit is unexamined; possible scope to push back on

Section: In scope (line 20-22) + fltk._native special case (line 71-88) + Open questions → native-special-mechanism.

What's wrong: The exploration (§2, §4, open question 1) repeatedly flags that fltk._native is a *singleton* with
unique, non-reusable responsibilities (Span/SourceText/UnknownSpan, UNKNOWN_SPAN static, multiple custom-named
submodules) and "may warrant a separate specialized codegen path." The requirements doc dutifully puts it in scope
because the request named it, but does not surface the tension: building a generator (or generator mode) whose
*only* user is a single in-tree file may be more machinery than the boilerplate it removes. The doc presents this
as settled in-scope without flagging that a one-off file generated by a one-off code path may not be worth a
generalized generator — and the requirements stage is exactly where that "is this sub-part a good idea" question
should be raised for the requester.

Why: Request — "both fltk._native and the clockwork lib.rs get codegenned" (so it is in scope by the letter).
Exploration §4 — "fltk._native is structurally different from a consumer cdylib; its lib.rs is more complex and may
warrant a separate specialized codegen path." Exploration open question 1 explicitly asks whether to do this at
all via a separate template vs. flags.

Consequence: Designer may build a bespoke generator path used by exactly one file, where a hand-written file +
a comment might have been cheaper, with no decision record acknowledging the trade. The user's "no reason for
[hand-writing] it" applies most strongly to the *replicated consumer* case (clockwork, future downstream); the
singleton case is where the payoff is weakest. Worth a one-line note inviting the requester to confirm fltk._native
codegen is worth the specialized machinery, or to accept it staying hand-written.

Suggested fix: Keep fltk._native in scope (the user asked), but add an explicit note that its value is
lower-confidence than the consumer case and call it out for the requester to confirm, rather than treating it as
equally settled.

## requirements-3 — System behavior pins exact emitted Rust tokens (over-specification into design)

Section: System behavior → Standard consumer generator → "The generated file:" bullets (lines 50-59); fltk._native
special case bullets (lines 81-86).

What's wrong: These bullets enumerate the literal Rust the generator must emit: `use
fltk_cst_core::register_submodule;`, `use pyo3::prelude::*;`, `mod cst;`, `mod parser;`, the exact
`#[pymodule] fn <module-name>(m: &Bound<'_, PyModule>) -> PyResult<()>` signature, and the exact
`register_submodule(m, "cst", cst::register_classes)?;` body. That is the *content of the design* — the specific
token-level output — not an observable requirement. Requirements should constrain observable behavior (the
compiled module exposes `<name>.cst` and `<name>.parser` with the right classes; it imports Span types from
fltk._native rather than re-registering them), and let the designer choose the exact emitted source.

Why: Review rule — "Requirements, not design: flag any tip into ... internal types ... Acceptance criteria are
observable behavior and surfaces, not how-it's-built." The acceptance criteria block (lines 62-69) already states
the right observable form ("exposes the same clockwork_native.cst and clockwork_native.parser submodules with the
same registered classes"); the token-level bullets duplicate this at the wrong altitude.

Consequence: Designer is boxed into emitting exactly these strings even if (per finding requirements-1) a cleaner
shape exists, and the doc/design risk drifting out of sync when the real generated text differs in trivia (import
ordering, glob vs. enumerated imports — note exploration §1 says cst.rs deliberately avoids a pyo3 glob). Pins
trivia that should be the designer's call.

Suggested fix: Replace the literal-source bullets with observable constraints (submodules exposed, Span types not
re-registered, recursion_limit handled per Constraints) and move the sample Rust into a non-normative "illustrative
shape" note if kept at all.

## requirements-4 — recursion_limit constraint correctly identified but left as an unresolved fork inside Constraints

Section: Constraints → "#![recursion_limit] ownership" (lines 141-146).

What's wrong: This is the one genuinely cross-cutting technical hazard from exploration (open question 5), and the
doc states it well — but leaves it as an unresolved either/or ("if the macro keeps injecting it ... if generation
moves to emit lib.rs directly ...") inside a Constraints section. Whether lib.rs is assembled-by-macro or
emitted-directly is a real requirements-affecting fork because it determines whether the Bazel path even changes
shape, yet it is neither an Open question (where the requester would see it as a decision point) nor resolved.

Why: Exploration open question 5 — "this line must be included in the generated file or the macro must still
inject it ... explicitly warns against including it in the consumer file." The doc's Bazel surface (lines 119-123)
assumes "the macro derives the lib.rs," which implies macro-assembly continues, but the Constraint leaves both
forks open — internally inconsistent about which world we are in.

Consequence: Ambiguity about who owns recursion_limit propagates into a Bazel-rule design choice; if unresolved,
the attribute could end up duplicated (compile error) or absent. Low risk of mis-scope but real risk of a
late-discovered design rework.

Suggested fix: Promote to an Open question or resolve it: state the assumed default (macro keeps injecting;
generated lib.rs omits recursion_limit; consistent with the Bazel-surface assumption on lines 119-123).

## requirements-5 — minor: "byte-equivalent" acceptance is likely unachievable and self-undercutting

Section: System behavior acceptance — "produces a byte-equivalent or behaviorally-equivalent module" (line 63-64).

What's wrong: "byte-equivalent ... module" conflates source equivalence with module (compiled/exposed) equivalence,
and offering "byte-equivalent OR behaviorally-equivalent" makes the stronger clause meaningless — any tester will
just satisfy the weaker one. Generated Rust will almost certainly differ in whitespace/import-trivia from the
hand-written file (exploration: generator output needs `make fix`), so byte-equivalence of source is a near-certain
fail and byte-equivalence of a compiled module is not a well-defined notion.

Why: CLAUDE.md / exploration §5 — generated code is reformatted via `make fix`; it is not expected to match
hand-written formatting.

Consequence: A literal-minded implementer could waste effort chasing byte-equivalence, or the criterion gets
silently ignored, eroding trust in the acceptance list.

Suggested fix: Drop "byte-equivalent"; keep behavioral/import-surface equivalence as the single criterion.

## Big picture

Direction is sound and matches the request: removing replicated consumer boilerplate is clearly worthwhile and the
standard-consumer generator is the high-value core. The doc is well-structured, scopes Python-API churn out
correctly, and the Open questions are the right ones. Two substantive concerns: (1) it under-weights the request's
explicit "we can fix its shape" license and over-constrains to reproduce existing files (requirements-1,
requirements-5); (2) it does not surface that the fltk._native singleton is the weakest-payoff part of the scope
and may warrant a flag to the requester (requirements-2). The token-level emission bullets (requirements-3) should
drop to observable-behavior altitude.

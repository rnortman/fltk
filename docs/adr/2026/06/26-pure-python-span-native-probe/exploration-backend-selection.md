# Backend Selection and the Native Probe — Deeper Exploration

## 1. Who imports `span.py` vs `terminalsrc.py`, and for what symbols

### `fltk.fegen.pyrt.span` — runtime (probe-triggering) importers

Every committed generated **parser** file imports `span` unconditionally at module scope:

- `fltk/fegen/fltk_parser.py:7` — `import fltk.fegen.pyrt.span`
- `fltk/fegen/bootstrap_parser.py:7`
- `fltk/fegen/bootstrap_trivia_parser.py:7`
- `fltk/fegen/regex_parser.py:7`
- `fltk/fegen/regex_trivia_parser.py:7`
- `fltk/unparse/toy_parser.py:7`, `toy_trivia_parser.py:7`
- `fltk/unparse/unparsefmt_parser.py:7`, `unparsefmt_trivia_parser.py:7`

These are the only files where importing `span` fires the native probe.  Generated CST files
(`*_cst.py`) and protocol files (`*_cst_protocol.py`) import `span` only under
`if typing.TYPE_CHECKING` (zero runtime cost — see `gsm2tree.py:190-202` for the generation
template).

The `span_protocol.py` also probes `_native` at module scope
(`fltk/fegen/pyrt/span_protocol.py:89-94`) but does so silently, with no `warnings.warn`.

### Symbols from `span` consumed by generated parsers (runtime)

- `fltk.fegen.pyrt.span.SourceText(text=..., filename=...)` — parser `__init__`
  (`fltk_parser.py:16`).
- `fltk.fegen.pyrt.span.Span.with_source(start, end, self._source_text)` — every terminal
  match site (`fltk_parser.py:83`, 93, 115, 124, 225, 237, …, dozens of call sites per parser).

### `fltk.fegen.pyrt.terminalsrc` — runtime importers

Both generated parsers AND generated CST files import `terminalsrc` at module scope (no
native probe; `terminalsrc.py` is pure Python and imports nothing from `_native` or `span`):

- Generated parsers: for `TerminalSource` (passed in as constructor arg).
- Generated CST files: for `TerminalSource`, `Span`, `UnknownSpan` annotations (lazy under
  `from __future__ import annotations`, so the annotation strings are never evaluated at
  runtime).
- Hand-written library code: `plumbing.py`, `errors.py`, `span_protocol.py`, `span.py` (line
  10 — `span.py` imports `terminalsrc` as its Python-backend fallback).

`terminalsrc` itself does not probe `_native` anywhere.

## 2. How backend is actually selected

There is no single "backend selection" flag.  Three orthogonal choices exist:

### (a) Parser logic: Python vs Rust

- **Python parser**: generated by `gsm2parser.ParserGenerator` → Python `ast.Module` exec'd at
  runtime.  Every such parser imports `fltk.fegen.pyrt.span` (triggering the probe).
- **Rust parser**: generated by `gsm2parser_rs.RustParserGenerator` → `.rs` source compiled
  into a cdylib extension (`fltk_parser_core::*`, `fltk_cst_core::*`).  Rust parsers use
  `fltk_cst_core::Span` internally, exposed via PyO3 as `fltk._native.Span`.  No connection
  to `span.py` at all.

The codegen path (Python vs Rust parser) is a build-time decision for downstream consumers.
At runtime inside FLTK, `plumbing.generate_parser` always generates a **Python** parser;
there is no runtime switch between the two.

### (b) CST node classes: Python vs Rust

`plumbing.generate_parser` has a `rust_cst_module: str | None` parameter
(`plumbing.py:212-229`):

- `rust_cst_module=None` (default): `gsm2tree.CstGenerator.gen_py_module()` generates Python
  dataclass CST classes, exec'd via `plumbing.py:248-251`.
- `rust_cst_module="mygrammar.cst"`: the named pre-built Rust extension's CST submodule is
  imported; its public type-attributes replace the generated Python classes
  (`plumbing.py:253-255`).

Regardless of which CST backend is chosen, the **parser logic** is always generated Python
(from `gsm2parser`).

### (c) Span type: `terminalsrc.Span` vs `_native.Span` — controlled by `span.py`'s probe

`span.py:10-18` is the only decision point.  It runs once per process on the first
`import fltk.fegen.pyrt.span` and sets `span.Span`, `span.SourceText`, and `span.UnknownSpan`
to either the `_native` types or the `terminalsrc` types.

This is NOT gated on "are we using the Rust CST?" or "are we in Python-only mode."  It is a
process-wide, unconditional probe that fires whenever any generated parser is imported.

The type registry in `iir/context.py:114-130` registers both `Span` and `SourceText` with
`module=pyreg.Module(("fltk", "fegen", "pyrt", "span"))`:

```python
# context.py:114-130 (condensed)
builtin_types.append(
    pyreg.TypeInfo(
        typ=terminal_span_type,
        module=pyreg.Module(("fltk", "fegen", "pyrt", "span")),
        name="Span",
    )
)
builtin_types.append(
    pyreg.TypeInfo(
        typ=source_text_type,
        module=pyreg.Module(("fltk", "fegen", "pyrt", "span")),
        name="SourceText",
    )
)
```

`iir/py/reg.py:26-29` shows that `TypeInfo.import_name()` returns
`".".join([*module.import_path, name])`, so these resolve to `"fltk.fegen.pyrt.span.Span"` and
`"fltk.fegen.pyrt.span.SourceText"`.

`gsm2parser.py:81` also independently registers `SourceText` with the same `span` module path.

The IIR compiler (`iir/py/compiler.py:329-330`) emits `VarByName(name=...)` as the bare name
string — so `fltk.fegen.pyrt.span.Span` appears verbatim in generated parser code as a dotted
attribute access off the `fltk` package.

Comment in `gsm2parser.py:266-268` makes the intent explicit:

> "On the Python backend this returns a terminalsrc.Span with source; on the Rust backend it
> returns a fltk._native.Span with source.  Both paths accept the resulting object as a node's
> span value."

"Rust backend" in this comment means `_native` was loaded by `span.py`'s probe.

## 3. What `fltk._native` provides and who consumes it

`fltk/_native/__init__.pyi` (the PyO3 stub) documents four public exports:

- `class SourceText` — opaque handle sharing a UTF-8 buffer across all Spans built from it.
  Constructor: `SourceText(text: str, filename: str | None = None)`.
- `class Span` — native `[start, end)` index range.  Key difference from
  `terminalsrc.Span`: `with_source` accepts only `SourceText` (native), not `str`
  (`_native.pyi:50`).  A `terminalsrc.SourceText` passed to `_native.Span.with_source` would
  be rejected.
- `UnknownSpan: Span` — singleton sentinel (`Span(-1, -1)`).
- `class LineColPos` — line/col structure.

### Intended consumers

| Consumer | Access mode | Notes |
|---|---|---|
| `span.py:13` | Runtime re-export | Core selector; native types win if present |
| `span_protocol.py:90-92` | Runtime silent probe | Builds `AnySpan` union |
| Generated `*_cst.py` | `TYPE_CHECKING` only | `gsm2tree.py:196-199` |
| Generated `*_cst_protocol.py` | `TYPE_CHECKING` only | `gsm2tree.py:738-741` |
| Rust-generated `.pyi` stub (`gsm2tree_rs.py:331`) | At stub-eval time (type-checker only) | File is `.pyi`, not imported at runtime |
| Tests (`test_native.py`, `test_phase4_*`, etc.) | Direct import | Parity and binding tests |

### Is the native override load-bearing for any current code path?

Yes.  The Python-generated parser's `__init__` constructs a `SourceText` object and stores it as
`self._source_text`.  Every terminal match then calls `Span.with_source(start, end,
self._source_text)`.  When `_native` is available:

- `self._source_text` is a `_native.SourceText` (sharing the underlying UTF-8 buffer).
- `Span.with_source(...)` returns a `_native.Span` (Rust object, cheaper to allocate).
- The produced CST nodes hold `_native.Span` instances in their `.span` fields.

When `_native` is absent:

- `self._source_text` is a `terminalsrc.SourceText` (Python dataclass).
- `Span.with_source(...)` returns a `terminalsrc.Span` (Python frozen dataclass, carries `_source: str`).
- The produced CST nodes hold `terminalsrc.Span` instances.

The native probe is the mechanism by which the Python-generated parser produces `_native.Span`
objects when `_native` is present.

## 4. Coupling to the native backend

### `terminalsrc.py` itself

`fltk/fegen/pyrt/terminalsrc.py` contains zero imports of `_native` or `span`.  It is a
self-contained pure-Python module.  Importing `terminalsrc` never triggers the native probe.

### Generated CST files

`gsm2tree.py:182-202` shows the pattern:

```python
# from __future__ import annotations  ← lazy annotation evaluation
# if typing.TYPE_CHECKING:
#     import fltk.fegen.pyrt.span
#     import fltk._native
```

Neither `span` nor `_native` is imported at runtime.  Pure-Python installs can import all
generated CST files without any native probe.

### Generated parser files

The generated parser MUST import `fltk.fegen.pyrt.span` at module level, because the parser
code contains `fltk.fegen.pyrt.span.SourceText(...)` and
`fltk.fegen.pyrt.span.Span.with_source(...)` as runtime call sites (not annotations).
These cannot be guarded under `TYPE_CHECKING`.

### Why the parser cannot use `terminalsrc` directly (as currently structured)

- `gsm2parser.py:81` registers `SourceText` to `("fltk","fegen","pyrt","span")`, not
  `("fltk","fegen","pyrt","terminalsrc")`.
- `context.py:120` registers `Span` to `("fltk","fegen","pyrt","span")`.
- The IIR compiler emits the registered module path verbatim into generated code.
- No "pure-Python parser mode" flag is passed to `ParserGenerator` or
  `create_default_context`; both always point to `span`.

To route a pure-Python parser to `terminalsrc` types directly, the type registry registrations
in `context.py` and `gsm2parser.py` would need to be parameterized or replaced with a different
module path.  This is currently architecturally absent: there is a single `create_default_context`
that always registers `span` as the source of `Span` and `SourceText`.

### Interaction between the two probes in `span.py` and `span_protocol.py`

`span_protocol.py:89-94` has its own silent probe.  It imports `span_protocol` is NOT imported
by generated parsers.  It is imported by `pyrt/__init__.py` or other library consumers; its
probe is independent of the parser's.

### Summary of obstacles

1. **No pure-Python-parser codegen path exists**: the type registry always emits `span.Span` /
   `span.SourceText` references in generated parsers.  There is no flag to switch to
   `terminalsrc.Span` / `terminalsrc.SourceText`.

2. **`span.py`'s probe is not conditional on CST backend**: it fires for all parsers equally,
   regardless of whether the Rust CST module is loaded.

3. **`terminalsrc` itself has no `_native` or `span` imports**: no restructuring needed on the
   `terminalsrc` side.

4. **The warning in `span.py` is the only `warnings.warn` call in the codebase**: removing it
   leaves `span.py`'s `try/except` logic intact; the fallback to `terminalsrc` continues to
   work silently (matching the pattern in `span_protocol.py`).

5. **Making the probe conditional** would require knowing, at `span.py` import time, whether
   native is wanted — but `span.py` has no context about which parser is being loaded.  It is
   a module, imported once; subsequent imports are served from `sys.modules`.

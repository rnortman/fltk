# User notes on design — round 2 (user directive, lightly edited for tone)

This doesn't make sense. Right now, we have two *runtime* behaviors that can be active depending on whether _native is available at import time or not: Python span or Rust span. Pyright is perfectly happy with this -- why? Because Pyright gets the Rust import and sees everythign as Rust? If so, that is already a bug and we must not paper over it. Why not change all type annotations to the span protocol, and Pyright never sees anything else? Or, the Python annotations should always resolve to the Python CST/Span and *never* to the Rust one, no matter whether the Rust backend is importable or not? More generally, the Python pipeline shoudl never even attempt to import rust, not even indirectly. Pyright should produce the same results when analyzing Python code whether the Rust backend is importable or not. Send this to a designer. It is critical that the designer must not overwrite the existing design. Overwriting an existing design that's already half implemented is a reliable recipe for derailing the implementation. The new design must be in a new file, written as a *delta* against the original design.

# Design: Two backends only — Python parser ⇒ Python span + Python CST; Rust parser ⇒ Rust span + Rust CST

Status: draft
Base commit: `49e9701e927d1403065f902b99d54acd7c129e41`
Requirements: `./requirements.md`
Exploration: `./exploration.md`, `./exploration-backend-selection.md`
User notes (authoritative): `./notes-design-user.md`

## 1. Root cause / context

### 1.1 The symptom and the user's verdict

The user's spec (`requirements.md`) is unambiguous: a generated **pure-Python parser must
produce pure-Python spans and pure-Python CST**, and a **Rust parser must produce Rust spans
and Rust CST**; the consumer chooses all-Python or all-Rust by *which parser they import*. The
standing statement is emphatic: "It is incorrect for the pure-Python parsers to use Rust-backed
span." The original bug report is a downstream consequence: importing a generated Python parser
in a pure-Python install prints a noisy warning.

Both the warning *and* the silent native-span substitution come from the same mechanism.

### 1.2 The mechanism

`fltk/fegen/pyrt/span.py:12-18` is a process-wide backend *probe*:

```python
from fltk.fegen.pyrt.terminalsrc import SourceText, Span, UnknownSpan
try:
    from fltk._native import SourceText, Span, UnknownSpan   # native wins if importable
except Exception:
    warnings.warn("fltk._native could not be loaded; falling back ...")
```

It runs once per process on first `import fltk.fegen.pyrt.span` and sets `span.Span` /
`span.SourceText` to the native types whenever `fltk._native` happens to be importable —
**regardless of which parser is running**.

The generated **Python parser** is wired to this probe at codegen time:

- `fltk/iir/context.py:114-132` registers the `Span` and `SourceText` IIR types to
  `module=("fltk","fegen","pyrt","span")`. `gsm2parser.py:78-84` re-registers `SourceText` to
  the same module.
- `gsm2parser.py:259-281` (`_make_span_expr`) emits the terminal-match construction call using
  that registry entry's `import_name()`, i.e. `fltk.fegen.pyrt.span.Span.with_source(...)`. The
  `_source_text` field initializer (`gsm2parser.py:105-123`) emits
  `fltk.fegen.pyrt.span.SourceText(text=..., filename=...)`.
- The committed parser top-imports the module at runtime: `fltk/fegen/fltk_parser.py:7`
  (`import fltk.fegen.pyrt.span`), used at `:16` (`SourceText(...)`) and `:83` et al.
  (`Span.with_source(...)`). This runtime import is what fires the probe — and the warning.

Net effect (confirmed in `exploration-backend-selection.md` §2c, §3): a "pure-Python" parser,
imported in *any* process where `fltk._native` is importable, silently produces
`fltk._native.Span` objects in its CST nodes. That is exactly the incorrect behavior. When
`_native` is absent, the same import path fires the warning instead.

The generated **unparser** is wired to the same probe, as a second symptom of the same
mechanism. `gsm2unparser.py` builds its span-guard type via `iir.Type.make(cname="Span")`
(`:87`, used at `:328`, `:1013-1014`, `:1116-1117`); the IIR compiler resolves that type
through the registry (`compiler.py:336-337` → `iir_type_to_py_constructor`) to
`fltk.fegen.pyrt.span.Span`, so the generated unparser emits
`isinstance(child, fltk.fegen.pyrt.span.Span)` — `is fltk._native.Span` under the probe.
Once §2.1 makes the Python parser construct `terminalsrc.Span`, those guards reject the
parser's own spans in any native-present process and unparsing fails (see §2.6, §3).

### 1.3 Two configurations only — the hybrid path is removed

There are exactly **two** valid configurations, and they are chosen by which parser the consumer
imports:

1. **Pure-Python parser ⇒ pure-Python span ⇒ pure-Python CST.** The generated Python parser
   (`gsm2parser`) constructs `terminalsrc.Span` / `terminalsrc.SourceText` and stores them in
   the generated Python dataclass CST nodes (`gsm2tree`). No `fltk._native`, no probe.
2. **Rust parser ⇒ Rust span ⇒ Rust CST.** A compiled PyO3 extension generated by
   `gsm2parser_rs` / `gsm2tree_rs` parses natively, constructs `fltk._native.Span`
   (`fltk_cst_core::Span`) internally, and returns Rust CST nodes. This path never touches
   `span.py` or the Python codegen pipeline at all (`exploration-backend-selection.md` §2a).

A **third, hybrid path exists in the current tree and is being removed**: a *Python* parser
bound to a *Rust* CST module — `plumbing.generate_parser(rust_cst_module=...)` /
`plumbing.parse_grammar(rust_fegen_cst_module=...)` (`plumbing.py:120-183`, `212-297`). Per the
user (authoritative, `notes-design-user.md`): "There is no use case for a python parser producing
rust CST. We should rip that out." It was an intermediate artifact from Rust-backend bring-up.
This design **deletes** it (§2.3) rather than preserving it.

Removing the hybrid path also removes the only reason a codegen-time "native span" selector would
be needed. In the hybrid path the Python parser handed the spans it constructed to
Rust CST node constructors/setters, which call `extract_span`
(`crates/fltk-cst-core/src/cross_cdylib.rs:422-450`); `extract_span` accepts **only** a native
`Span` and raises `TypeError` for a pure-Python `terminalsrc.Span`. With no Python-parser/Rust-CST
combination left, no Python codegen path ever feeds a span into a Rust CST node. The Python parser
therefore constructs `terminalsrc` spans **unconditionally** (§2.1); `extract_span`'s native-only
contract now governs only the all-Rust path, internally to the compiled extension, and is outside
this design's scope.

### 1.4 The other role of `span.py`: a public, intentionally backend-agnostic type name (kept)

`fltk.fegen.pyrt.span` is not only the parser's historical runtime selector; it is also the
**backend-agnostic Span type name** used in the public CST/protocol annotation surface and is a
**tested public module**. This role is what lets a consumer *swap* configuration 1 for
configuration 2 without editing their CST-consuming code — exactly the swap-ability the
requirements call for:

- `test_span_protocol.py:43-54` asserts `span.Span is fltk._native.Span` when native is built,
  else `span.Span is PySpan`. Downstream consumers may import `span.py` to obtain "the active
  backend's span." This probe-as-selector behavior is deliberate and must be preserved.
- The generated CST node `span` field is annotated with the explicit union
  `fltk.fegen.pyrt.terminalsrc.Span | fltk._native.Span` (`gsm2tree.py:269`, protocol
  `:900`, Rust `.pyi` `gsm2tree_rs.py:358`). Span-typed *children* are annotated via the shared
  type registry, which renders `fltk.fegen.pyrt.span.Span` (e.g. committed
  `toy_cst.py:150-170`, `toy_cst_protocol.py:98-106`). **Pyright resolves that bare `span.Span`
  name to the *active backend's single concrete type*** — `fltk._native.Span` when native is
  built, `terminalsrc.Span` otherwise — **not** a union (the `try` branch of the `span.py`
  re-export wins; verified with a pyright probe against the real modules). Cross-backend
  agnosticism for the node `span` *field* is carried by its **explicit** `terminalsrc.Span |
  fltk._native.Span` union annotation (a covariant assignment target), not by the `span.Span`
  name; both the Python CST (config 1) and the Rust CST (config 2) satisfy the protocol because
  every span slot shares that one agnostic name resolving to the one active backend. This
  distinction is load-bearing for the parser (§2.1.1): a constructed `terminalsrc.Span` flows
  *implicitly* into the covariant union field, but **cannot** flow implicitly into the parser's
  *invariant* `ApplyResult[int, span.Span]` return, nor into the `span.Span`-typed CST child
  setters.

This dual role is the crux: **the runtime parser bug must be fixed, and the hybrid path removed,
without disturbing the public, backend-agnostic annotation surface.** CLAUDE.md (and the user's
note) forbid gratuitous annotation churn on generated CST/protocol symbols — they are public API
for out-of-tree consumers, and consumer code must remain agnostic to which backend produced the
CST.

## 2. Proposed approach

Separate the two jobs `span.py` is currently doing for the parser:

1. **Runtime span/SourceText *construction*** inside a generated Python parser → bind it to the
   concrete pure-Python `terminalsrc` backend, never to the process-wide probe. There is no
   selector: the Python parser is always pure-Python.
2. **Type-annotation surface** (parser-internal annotations *and* the public CST/protocol/`.pyi`
   annotations) → leave it referencing the existing backend-agnostic `span` names. No churn.

Because the generated parser's only *runtime* (non-annotation) uses of `span.py` are the two
construction sites (the `with_source(...)` calls and the `_source_text` initializer), retargeting
just those two sites to `terminalsrc` — and making the parser's annotations lazy so they no longer
require a runtime import of `span.py` — fixes the runtime bug while freezing the public surface. Removing
the hybrid plumbing (§2.3) eliminates the codegen-time backend selector the parser would otherwise
need.

One subtlety the construction retarget alone does not cover: the parser's two terminal-consume
helpers return `fltk.fegen.pyrt.memo.ApplyResult[int, span.Span] | None`, and `ApplyResult` is
**invariant** in its result type (`memo.py:69`). Retargeting their construction to build a
`terminalsrc.Span` makes the *constructed* value an `ApplyResult[int, terminalsrc.Span]`, which is
not assignable to the frozen `ApplyResult[int, span.Span]` (=`fltk._native.Span`) return. The
covariant CST `span` field absorbs a `terminalsrc.Span` silently; the invariant `ApplyResult`
return does not. §2.1.1 bridges that one seam with an explicit `typing.cast` to the agnostic span
name — a parser-internal expression that changes no annotation.

The generated **unparser** is the parser's downstream sibling: it consumes the CST the parser
produces, so it too must stop deciding span identity through the probe. §2.6 applies the same
two-config principle — the unparser is a backend-agnostic CST consumer that recognizes *both*
backends' span objects structurally (the dual-backend pattern §2.4 already blesses for the CST
mutators), never the process-wide probe.

### 2.1 The Python parser always constructs `terminalsrc` types (`fltk/fegen/gsm2parser.py`)

Both runtime construction sites are retargeted to the pure-Python `terminalsrc` module,
unconditionally. The type **registry** entries for `Span`/`SourceText` stay registered to
`("fltk","fegen","pyrt","span")` (§2.4), so every *annotation* derived from them is unchanged; the
construction expressions are decoupled from the registry and point at `terminalsrc` directly.

- `_make_span_expr` (`:259-281`): build the class reference from the fixed module-qualified string
  `"fltk.fegen.pyrt.terminalsrc.Span"` rather than from
  `lookup(self.TerminalSpanType).import_name()`. It already constructs an `iir.VarByName` from a
  dotted string and calls `with_source` on it; only the string changes. Update the stale comment at
  `:266-268` ("On the Python backend ... on the Rust backend ...") to state the construction is
  always `terminalsrc`.
- `_source_text` initializer (`:105-137`): this site needs a **structural** change, not a string
  swap. Today it is built as `iir.Construct.make(self.SourceTextType, text=..., filename=...)`
  (`gsm2parser.py:113-123`), and the IIR compiler resolves a `Construct`'s emitted class name
  through the type registry (`compiler.py:312-315` → `iir_type_to_py_constructor` →
  `reg.TypeInfo.import_name(concrete=True)`, `reg.py:26-29`). Because §2.4 leaves the
  `SourceText` registry entry pointed at the `span` module, a `Construct(SourceTextType)` would
  always compile to `fltk.fegen.pyrt.span.SourceText(...)` — reintroducing the runtime `span`
  import — and re-registering the entry is forbidden (it would churn the public annotation surface,
  §1.4/§2.4). Instead, **replace the `Construct` node with a registry-independent, module-qualified
  call expression**, mirroring `_make_span_expr`: build
  `iir.MethodAccess("SourceText", iir.VarByName(name="fltk.fegen.pyrt.terminalsrc")).call(text=..., filename=...)`.
  The `MethodAccess(...).call(**kwargs)` form compiles to
  `fltk.fegen.pyrt.terminalsrc.SourceText(text=..., filename=...)` (the compiler's `MethodCall`
  branch with `_format_args` handling keywords, `compiler.py:305-309,285-289`), so the emitted
  class name no longer flows through the registry.

Both construction surfaces exist and accept keyword construction:
`terminalsrc.SourceText(text, filename=None)` (`terminalsrc.py:22`) and
`terminalsrc.Span.with_source(start, end, source: str | SourceText)` (`terminalsrc.py:212`). The
generated parser always passes `self._source_text` (a `terminalsrc.SourceText`) to
`with_source`, so the produced node `span` value is a pure-Python `terminalsrc.Span`.

### 2.1.1 The two terminal-consume returns bridge construction→annotation with an explicit cast

`consume_literal` / `consume_regex` (`gsm2parser.py:154-200`, `202-249`) are the only sites that
*construct* a span into an **invariant** `ApplyResult[int, span.Span]` return. After §2.1 the
construction is a `terminalsrc.Span`; the return annotation stays the agnostic registry `span.Span`
(§2.4), which pyright pins to `fltk._native.Span` in a native-present `make check` (§1.4). The two
are not the same concrete type and `ApplyResult.ResultType` is invariant, so the bare construction
is rejected (pyright: *"ResultType is invariant, but terminalsrc.Span is not the same as
fltk._native.Span"*) — empirically 2 errors per generated parser, 20 across the committed set: the
§2.5 blocker recorded in the implementation log (increment 11). The covariant CST `span` field is
unaffected; only these two invariant returns fail.

Resolution: wrap each terminal-consume's `ApplyResult` `result=` argument (the
`iir.Construct.make(span_result_type, …)` at `:184-187` and `:231-235`) in an explicit cast to the
agnostic span name. Emit it with IIR the compiler already supports — **no new IIR node** — as
`iir.MethodAccess("cast", iir.VarByName(name="typing")).call(iir.LiteralString(<import_name>),
self._make_span_expr(...))`, which compiles to
`typing.cast("fltk.fegen.pyrt.span.Span", fltk.fegen.pyrt.terminalsrc.Span.with_source(...))`.
`<import_name>` is `python_type_registry.lookup(self.TerminalSpanType).import_name()`, so the cast
target tracks §2.4's frozen registry automatically. `_make_span_expr` itself is left unchanged: its
other callers build the node `span` *field* (the covariant union), which needs no cast.

Why a cast — not honest annotations, variance, or `SpanProtocol`. This is the design's
"decouple construction from annotation" thesis (§2) realized for an *invariant* return: construct
the concrete pure-Python span at runtime, present it under the established agnostic `span.Span` name
statically. A pyright probe against the real `memo.ApplyResult` / `span.Span` / `terminalsrc.Span`
confirms the cast clears the error, and confirms the rejected alternatives (recorded in §5). The
cast is robust across both install shapes (in a pure-Python install `span.Span` *is*
`terminalsrc.Span`, so the cast is an identity; with native present it reconciles the invariant
slot); it needs no runtime `span` import (the type is a quoted forward-ref resolved via the §2.2
`TYPE_CHECKING` import — and ruff's `TC006` *requires* the quoted form for a `TYPE_CHECKING`-only
type; `typing.cast` ignores the string at runtime and returns the `terminalsrc.Span` unchanged, so
§2.1 determinism holds). Generated CST code already emits `typing.cast(...)` (`gsm2tree.py:338`), so
this is an established generated-code pattern. Crucially it changes **no** annotation — parser-
internal or consumer-facing — so the frozen surface (§2.4) is untouched and no consumer-facing
annotation churn is forced.

### 2.2 Generated parsers stop importing `span.py` at runtime; remove the warning

Today the committed parsers have *eager* annotations (no `from __future__ import annotations`,
`fltk_parser.py:1-8`), so `fltk.fegen.pyrt.span.Span` in return annotations is evaluated at
import and forces the runtime `import fltk.fegen.pyrt.span`. Make those annotations lazy and move
the import under `TYPE_CHECKING`, matching the established pattern in generated CST files
(`gsm2tree.py:185-202`):

- **`fltk/fegen/genparser.py`** (`generate_parser`, `:73-118`): prepend
  `from __future__ import annotations`; drop `("fltk","fegen","pyrt","span")` from the runtime
  `imports` list (`:97-105`); emit `import fltk.fegen.pyrt.span` inside an
  `if typing.TYPE_CHECKING:` block (pygen already provides `if_`/`import_`/`module`). The runtime
  `terminalsrc` import stays (already present, now also carries the construction calls).
- **`fltk/fegen/pyrt/span.py`**: remove the `warnings.warn(...)` call and the `import warnings`
  line. **Keep** the `try/except` re-export — `span.py` remains the public, tested backend
  *selector* (`test_span_protocol.py:43-54`) and the type-checking-time agnostic `Span` name
  (§1.4). The except branch now falls back silently, matching `span_protocol.py:89-94`.

Result for a committed (pure-Python) parser: at runtime it imports only `terminalsrc` (plus
`memo`/`errors`/the CST module); it constructs `terminalsrc.Span` / `terminalsrc.SourceText`
deterministically; it never touches `span.py` at runtime, so the probe — and the warning — never
fire from a parser import, in any environment.

### 2.3 Remove the hybrid plumbing (`fltk/plumbing.py`)

Delete the Python-parser/Rust-CST option entirely. This is the user's explicit directive
("rip that out"), not a refactor of convenience.

- `generate_parser` (`:212-297`): remove the `rust_cst_module` parameter and the
  `_load_rust_cst_classes` branch (`:246-255`); always run the Python CST `gen_py_module()` →
  `exec` path. `public` is always the Python CST dataclasses. `parser_globals` already binds
  `terminalsrc` (`:269`) and `Span` (`:266`); no native import is added. The exec'd in-memory
  parser body must still gain `from __future__ import annotations` (prepended): today the exec'd
  parser's eager `span.Span` annotations only resolve because `plumbing.py:18` imports
  `fltk_parser`, which top-imports `span` as a side effect; once §2.2 moves that import under
  `TYPE_CHECKING`, that side effect disappears and eager evaluation would raise `AttributeError`.
  Lazy annotations remove the dependency entirely, leaving only the `terminalsrc` construction
  names to resolve at runtime.
- `parse_grammar` / `parse_grammar_file` (`:120-209`): remove the `rust_fegen_cst_module`
  parameter and the entire Rust-backend branch (`:157-183`). Always use the committed Python
  `fltk_parser` + Python `fltk_cst` path.
- Delete the now-dead support code: `_load_rust_cst_classes` (`:81-117`),
  `RustBackendUnavailableError` (`:70-78`), `_fegen_rust_parser_cache` (`:44`),
  `_fegen_grammar_cache` (`:38`), and `_load_fegen_grammar` (`:47-67`, whose only caller is the
  deleted Rust-fegen branch — confirmed by grep). `_fegen_grammar_cache` is referenced only inside
  `_load_fegen_grammar` (`:53,66,67`), so it is orphaned once that function is gone; ruff's
  unused-binding rules do not flag module-level globals, so `make check` would not catch it — it
  must be deleted explicitly. Remove the now-unused `importlib` import (`plumbing.py:11`; its only
  use is in the deleted `_load_rust_cst_classes`).

`generate_parser` / `parse_grammar` are FLTK library entry points, not generated public symbols;
removing a keyword-only parameter that selects a path the user has declared has no use case does
not touch the generated CST/parser surface that downstream consumers annotate against. This is a
deliberate, user-directed API removal.

### 2.4 What does NOT change (frozen public surface)

- The type **registry** modules for `Span`/`SourceText` stay `("fltk","fegen","pyrt","span")`
  (`context.py:114-132`, `gsm2parser.py:78-84`), so every *annotation* derived from them is
  unchanged. This is why §2.1 retargets construction by replacing the emitting IIR nodes
  (`VarByName` for the span ref; a module-qualified `MethodAccess(...).call(...)` for the
  `SourceText` `Construct`) rather than by editing the registry — the registry continues to drive
  only annotations, to the agnostic `span` name.
- Generated **CST node** files (`*_cst.py`): the `span` field union annotation, span-typed child
  annotations (`fltk.fegen.pyrt.span.Span`), accessor return types, mutator `isinstance` logic
  (`terminalsrc.Span` + lazy `_get_native_span_type()`), `NodeKind`/`Label` equality — all
  unchanged. The Python CST `span` field already defaults to `terminalsrc.UnknownSpan`, so a
  pure-Python CST is now consistently pure-Python end-to-end.
- Generated **protocol** files (`*_cst_protocol.py`) and Rust **`.pyi`** stubs
  (`gsm2tree_rs.py`): unchanged. `gsm2tree_rs.py:330` still emits `import fltk.fegen.pyrt.span`,
  keeping `test_gsm2tree_rs.py:1153` green.
- `span.py` retains its selector semantics (only the warning is removed).
- The two terminal-consume returns gain a `typing.cast` to the agnostic registry name (§2.1.1).
  The cast is a parser-internal *expression*, not an annotation: it renames nothing, changes no
  method signature, and is invisible to consumers. Beyond the §2.1 construction retarget and the
  §2.2 lazy-import move, it is the only addition §2.5 makes to the parser.
- No generated public symbol is renamed; no downstream annotation is forced to change. CST-consumer
  code stays agnostic across both configurations.

The one generated runtime surface that **does** change beyond the parser is the unparser's span
**guard** (§2.6). This is not a frozen surface and not an annotation: the guards were the parser's
probe-bound twin and must change for the same reason the parser's construction sites do. The change
widens an `isinstance` test to accept a superset (it now recognizes both backends' span objects),
renames no symbol, and leaves the unparser's `fltk.fegen.pyrt.span.Span` *annotation* name intact
(now lazy). No downstream annotation churn.

### 2.5 Regeneration

Run `make gencode` then `make fix`, and commit. The committed parser files
(`fltk/fegen/{fltk,bootstrap,bootstrap_trivia,fltk_trivia,regex,regex_trivia}_parser.py`,
`fltk/unparse/{toy,toy_trivia,unparsefmt,unparsefmt_trivia}_parser.py`) gain
`from __future__ import annotations`, move the `span` import under `TYPE_CHECKING`, change
their construction sites from `fltk.fegen.pyrt.span.*` to `fltk.fegen.pyrt.terminalsrc.*`, and
wrap the two terminal-consume `ApplyResult` constructions in
`typing.cast("fltk.fegen.pyrt.span.Span", …)` (§2.1.1). The cast is what makes the regenerated
parsers pass pyright despite the invariant `ApplyResult[int, span.Span]` return; without it
`make check` fails at `typecheck` with 2 errors per file (the blocker in the implementation log,
increment 11). The `make gencode` drift check (`make check`) must stay clean — ruff/format **and**
pyright.

**There are no committed unparser files to regenerate.** `make gencode` generates no unparser
(verified against the `gencode` target: it emits CST/protocol/parser modules and Rust artifacts
only), and `genunparser.py` writes to stdout rather than to a committed module. Every unparser is
generated **in-memory** by `plumbing.generate_unparser` (`plumbing.py:412-424`) at runtime — for
in-tree tests and for out-of-tree consumers alike. The unparser fix (§2.6) is therefore a
**code-generator** change (`gsm2unparser.py`) plus a runtime-support helper
(`fltk/unparse/pyrt.py`); it produces no committed-artifact diff and adds no new `make gencode`
target. `make check` stays clean because the unparse tests that §2.1 breaks (listed in §4) are
restored by §2.6, not because the unparser is untouched.

### 2.6 The generated unparser is a backend-agnostic CST consumer (no probe)

The unparser consumes the CST the parser produces. Per `requirements.md` ("all of the
CST-consuming code remains agnostic to which backend was used"), it must accept **whichever**
span backend the CST actually carries — `terminalsrc.Span` from a Python parser (config 1) or
`fltk._native.Span` from a Rust CST (config 2) — and decide that from the object itself, never
from `span.py`'s process-wide probe. Today it does the opposite: its three span guards resolve
through the registry to the single probe-selected `fltk.fegen.pyrt.span.Span`, so after §2.1 a
native-present process makes the guard `fltk._native.Span` while the Python parser hands it
`terminalsrc.Span`, and the guard rejects every span child. This is the parser's probe-binding
defect (§2.1) reappearing in its downstream sibling; the fix mirrors the dual-backend CST-mutator
pattern §2.4 already blesses (`regex_cst.py:744-750`).

This is the only unparser generator — there is no Rust unparser. A consumer who unparses a Rust
CST (config 2) generates a *Python* unparser over native-span children, so accepting both backends
is required for config-2 unparsing, not merely for the config-1 native-present case.

Two coordinated changes, both in the in-memory generation path (no committed artifact, §2.5):

**(a) Runtime guards → dual-backend, centralized in `fltk/unparse/pyrt.py`.** Add a helper to the
unparser's existing runtime-support module — already imported by every generated unparser
(`gsm2unparser.py:1818`) and already the home of the dual-backend `extract_span_text` /
`count_span_newlines` (which handle both span backends, `pyrt.py:33-58`):

```python
import sys
# Span is the module-top `from fltk.fegen.pyrt.terminalsrc import Span`
def is_span(obj: object) -> bool:
    if isinstance(obj, Span):
        return True
    native = sys.modules.get("fltk._native")
    return native is not None and isinstance(obj, native.Span)
```

This mirrors `gsm2tree.py`'s `_get_native_span_type()` (`:217-219`): it resolves the native span
type lazily via `sys.modules.get("fltk._native")`, so `pyrt.py` keeps its pure-Python
importability (no top-level `fltk._native` import) and recognizes both backends without ever
importing `span.py` or firing the probe. Centralizing in `pyrt.py` keeps the generated unparser
output minimal and consistent with where backend-handling already lives, rather than emitting
`_get_native_span_type()` boilerplate into every generated unparser as the CST generator does
(the CST module must avoid a `fltk.unparse.pyrt` dependency; the unparser already has one).

In `gsm2unparser.py`, retarget the three probe-bound guards to call this helper, building the
call like the existing `fltk.unparse.pyrt.*` calls (`:936-945`, `:1737-1746`),
i.e. `iir.VarByName(name="fltk.unparse.pyrt", ...).method.is_span.call(child)`:

- `_extract_and_validate_nonsequence_child` (`:327-331`): this site guards *all* non-sequence
  children, span and rule-node alike — `expected_type` comes from

  `iir.LogicalNegation(operand=<is_span(child) call>)`; the `Identifier` case keeps
  `iir.IsInstance` against the concrete rule class (unchanged).
- `_count_newlines_in_trivia` (`:1013-1016`): replace `iir.IsInstance(child_value, Span)` with the
  `is_span(child_value)` call (it is the `if_` condition).
- `_gen_trivia_processing` (`:1116-1119`): replace `is_span_check = iir.IsInstance(...)` with the
  `is_span(child_value)` call; it stays the rhs of the `LogicalAnd` with `is_unlabeled`.

The remaining `IsInstance` sites in this file (`:891` trivia-child types, `:1271` `Trivia`) are
not span guards and are unchanged. The `whitespace_span` / `_count_newlines` `span` annotations
(`:925`, `:1127`) are annotations, not runtime guards, and are handled by (b).

**(b) Annotations → lazy, agnostic (consistent with §2.2).** The generated unparser currently has
**no** `from __future__ import annotations` and does **not** import the span module, yet carries
an eager `def _count_newlines(self, span: fltk.fegen.pyrt.span.Span)` annotation (verified by
generating the toy unparser). That annotation resolves today only because *some other* module in
the process has already imported `fltk.fegen.pyrt.span` as a side effect — exactly the fragility
§2.2 removes from the parser. Once §2.5 regenerates the committed parsers to drop their runtime
`span` import, nothing guarantees `span` is imported when an unparser is exec'd, so the eager
annotation would fail (and, until then, evaluating it imports the probe — the thing this design
forbids). Apply the §2.2 treatment: prepend `from __future__ import annotations` as the first
statement of the generated unparser module and add `import fltk.fegen.pyrt.span` under
`if typing.TYPE_CHECKING:`. The agnostic `fltk.fegen.pyrt.span.Span` annotation name is preserved
(frozen surface, §2.4) but becomes a never-evaluated string.

The `from __future__` line and the `TYPE_CHECKING` block are added to the `imports` list returned
by `gsm2unparser.generate_unparser` (`:1815-1824`), so both assembly sites
(`plumbing.generate_unparser`, `genunparser.py`) inherit them; `from __future__` must be
`imports[0]`, ahead of the existing `import typing`. `typing` is already imported (`:1816`), so
the `TYPE_CHECKING` guard needs no new top-level import.

This makes a Python-generated unparser fully probe-free at runtime: it constructs nothing through
`span.py`, evaluates no `span.Span` annotation, and recognizes span children of either backend
structurally — the same agnosticism the CST mutators have.

## 3. Edge cases / failure modes

1. **Pure-Python parser in a process where `_native` IS installed** (the core bug). After the
   change, construction targets `terminalsrc` unconditionally; the produced span is
   `terminalsrc.Span`. No probe runs. Covered by a regression test (§4).
2. **`fltk._native` genuinely absent in a pure-Python install.** The pure-Python parser path
   imports only `terminalsrc` at runtime; nothing imports `span.py`, so no warning and no
   `ImportError`. `_native`-bearing TYPE_CHECKING annotations remain lazy strings and are never
   evaluated at runtime.
3. **Cross-backend node/span equality in parity tests (config 1 vs config 2).**
   `terminalsrc.Span` and `fltk._native.Span` are not `==` even for equal `(start,end)` (noted in
   `test_cst_mutators_parity.py:14`). The parser-parity helpers already compare spans by
   `.start`/`.end`, not `==` (`tests/parser_parity.py` `assert_cst_equal`), so a Python-parser
   CST (`terminalsrc.Span`) vs a Rust-parser CST (`fltk._native.Span`) still matches. No
   regression; this is verified, not assumed.
4. **In-memory exec parser annotations (plumbing).** The exec'd parser body carries `span.Span`
   annotations that today resolve only because importing `fltk_parser` (`plumbing.py:18`) loads
   the `span` submodule as a side effect — a side effect §2.2 removes. §2.3 therefore prepends
   `from __future__ import annotations` to the exec'd module, making those annotations lazy
   strings that are never evaluated; only the `terminalsrc` construction names need to resolve,
   and `terminalsrc` is already bound in `parser_globals`.
5. **Removing `rust_cst_module` / `rust_fegen_cst_module` breaks in-tree callers.** Several tests
   and one consumer-API test pass these keywords (§4). They are removed as part of this change
   per the user directive; no out-of-tree generated-symbol surface is affected.
6. **`make check` (pyright) on regenerated parsers — the invariant-`ApplyResult` gap.** Two
   pressures: (a) `from __future__ import annotations` + a `TYPE_CHECKING` `span` import (the exact
   pattern generated CST files already pass); and (b) the §2.1 construction retarget makes
   `consume_literal`/`consume_regex` build a `terminalsrc.Span` into an **invariant**
   `ApplyResult[int, span.Span]` (=`fltk._native.Span`) return — rejected by pyright (2 errors per
   file). (b) is closed by the §2.1.1 cast, which presents the constructed span under the agnostic
   `span.Span` name statically while leaving the runtime object a `terminalsrc.Span`. A pyright
   probe against the real `memo.ApplyResult` / `span.Span` / `terminalsrc.Span` confirms both the
   failure and the cast fix. The regen drift check enforces a clean `make check`.
7. **Pure-Python parser CST unparsed in a native-present process (the §2.1-exposed gap).** Before
   §2.6, the unparser's probe-bound guards reject `terminalsrc.Span` children whenever
   `fltk._native` is importable, so `unparse_*` returns `None` and `plumbing.unparse_cst` raises
   `ValueError("Unparsing failed")`. After §2.6, `is_span` recognizes `terminalsrc.Span`
   regardless of native presence; unparsing succeeds. Covered by the restored unparse tests and a
   new native-present round-trip regression (§4).
8. **Rust CST (config 2) unparsed by a Python-generated unparser.** A consumer may generate a
   Python unparser over a Rust CST module; its span children are `fltk._native.Span`. `is_span`
   accepts the native type via its lazy `sys.modules.get("fltk._native")` branch, so this path is
   unaffected — the unparser is agnostic to which backend produced the CST, as required.
9. **`fltk._native` absent.** `is_span` short-circuits on `isinstance(obj, terminalsrc.Span)`;
   the native branch resolves `sys.modules.get("fltk._native")` to `None` and returns `False`
   without importing anything. `pyrt.py` stays pure-Python importable.

## 5. Open questions

None requiring user judgment. One interpretive call is recorded as a decision, forced by the code
and by the requirement that consumer code stay backend-agnostic:

- **Agnostic annotation surface kept (alternative rejected).** Making the concrete CST's
  span-typed child annotations *honest per backend* (`terminalsrc.Span` for Python,
  `fltk._native.Span` for Rust) instead of the agnostic `fltk.fegen.pyrt.span.Span` was
  considered and rejected: it churns the public CST/protocol annotation surface (CLAUDE.md, and
  the user's explicit "no forced downstream annotation churn") while delivering no runtime benefit
  — the runtime objects are already correct via §2.1, and consumers remain agnostic (and able to
  swap config 1 ↔ config 2) through the protocol.

- **Unparser span guard: concrete dual-backend `is_span` helper (alternatives rejected).** The
  guard could instead be a *structural* `SpanProtocol`-shaped check (duck-typing on `.start`/`.end`
  or `.kind`), or `_get_native_span_type()` boilerplate emitted into every generated unparser
  (mirroring the CST generator). Rejected: the concrete dual-backend check is the exact pattern
  §2.4 already blesses for the CST mutators (`regex_cst.py:744-750`) — using a second philosophy
  for the unparser would diverge from it and risk accepting non-span objects that happen to be
  protocol-shaped. The per-module `_get_native_span_type()` mirror is what the CST generator does
  *because a CST module must not depend on `fltk.unparse.pyrt`*; the unparser already imports that
  module, so centralizing `is_span` there (next to the dual-backend `extract_span_text` /
  `count_span_newlines`) keeps generated output minimal and backend-handling in one place. This is
  a decision, not an open question — no user judgment is required.

- **Invariant-`ApplyResult` span return: explicit cast, not honest annotations / variance /
  `SpanProtocol` (alternatives rejected, empirically).** The parser's terminal-consume returns are
  *parser-internal*: a consumer calls `apply__parse_<root>`, whose `ApplyResult` is parameterized by
  the **CST node** type, never a span; `consume_literal`/`consume_regex` and the per-item forwarders
  are decomposition helpers no consumer annotates against. But the spans they carry flow into the
  **consumer-facing, frozen** CST span-child setters, annotated `span.Span` (=`fltk._native.Span`).
  A pyright probe against the real modules confirms three rejections: (i) making the parser's span
  annotation honestly `terminalsrc.Span` (the implementation log's recommended option) makes
  `append_<label>(child=…)` reject the span — `terminalsrc.Span` is not assignable to the
  native-typed setter param — so it cascades into the frozen consumer surface and would force
  CST-setter/accessor annotation churn (CLAUDE.md forbids); (ii) making `ApplyResult.ResultType`
  covariant does **not** help — covariance needs `terminalsrc.Span <: fltk._native.Span`, but they
  are disjoint siblings, not sub/supertypes; (iii) `SpanProtocol` as the annotation fails like (i) —
  a protocol supertype is not assignable to the concrete native setter param. Only the §2.1.1 cast
  keeps the consumer-facing annotation surface fully stable, so it is chosen and no escalation is
  required. `SpanProtocol` remains the right tool for *new* backend-agnostic surfaces, but cannot be
  retrofitted onto the frozen concrete `span.Span` setter surface without churn.

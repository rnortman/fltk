## efficiency-1 — `is_span` re-resolves the native span type on every non-span child during unparse

File: `fltk/unparse/pyrt.py:61-80` (`is_span`); call sites generated by
`fltk/unparse/gsm2unparser.py:332` (`_extract_and_validate_nonsequence_child`), `:1035`
(`_count_newlines_in_trivia`, inside the per-child `while` loop), `:1138` (`_gen_trivia_processing`).

Problem: the unparser's three span guards changed from an inline
`isinstance(child, fltk.fegen.pyrt.span.Span)` to a call to the centralized helper
`fltk.unparse.pyrt.is_span(child)`. For any child that is NOT a `terminalsrc.Span` (every rule-node
child, and every span child under config 2), each call runs `sys.modules.get("fltk._native")`
(dict lookup) + `getattr(..., "Span", None)` + a second `isinstance`, and adds Python
function-call + attribute-chain (`fltk.unparse.pyrt.is_span`) indirection that the old inline
`isinstance` did not. The native-type resolution is invariant for the life of the process (None
forever in a pure-Python install; a fixed type once native is loaded) yet is recomputed on every
call. `_count_newlines_in_trivia` invokes it once per child of every trivia node.

Consequence: per-child overhead on the unparse/render hot path. Bites when unparsing large
documents with many trivia/separator children — the repeated `sys.modules.get`+`getattr` and the
call-frame indirection scale with child count. Absolute cost is small (microsecond-scale per child)
and is dominated by Doc construction/rendering, so this is a minor regression, not a scale ceiling.

Direction: optional. Memoize the resolved native `Span` type once it is found (cache only the
non-None resolution so a later native import is still picked up; keep returning the cached type
thereafter), or accept as-is. The current form matches the established lazy
`_get_native_span_type()` pattern in `gsm2tree.py`, so consistency is a legitimate reason to leave
it; flagging only because it is new per-child work added to a render path.

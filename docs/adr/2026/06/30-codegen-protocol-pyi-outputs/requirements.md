# Codegen protocol and .pyi outputs

## Original request

> OK, it's time for a full development cycle. Never mind the isinstance/kind thing -- we need to fix both .pyi generation and protocol generation. Generating a Rust CST should just automatically generate the .pyi and the bazel rules should expose this. We do not need a separate bazel rule. Secondly, I think both Python and Rust generate_parser variants need to be capable of producing a protocol output. It would not be enabled by default for either of them.

## What the user is asking for

Two related changes to FLTK's code generation pipeline, touching the CLI subcommands, the Python and Rust generators, and the Bazel rules that wrap them.

**Change 1 -- Opt-in protocol generation for both backends.** Both the Python code generation path (the `generate` CLI subcommand, exposed via the `generate_parser` Bazel rule in `rules.bzl`) and the Rust code generation path (the `gen-rust-cst` CLI subcommand, exposed via `generate_rust_parser` in `rust.bzl`) should be capable of producing the protocol module (`{base_name}_cst_protocol.py`). This capability should be opt-in -- not enabled by default -- for both backends.

**Change 2 -- .pyi generation for Rust CST, automatic when the protocol is available.** The `.pyi` type stub should be produced whenever the protocol module's dotted Python import path is known. There are exactly two ways that import path becomes known:

1. **The Rust generator is itself producing the protocol** (because opt-in protocol output from Change 1 is enabled). In this case the protocol module's dotted import path is already supplied as part of enabling protocol output, so the `.pyi` reuses it automatically. The caller does not separately pass a `--protocol-module` flag; enabling protocol output on the Rust generator is sufficient to also get the `.pyi`.

2. **The import path is supplied explicitly** as a parameter (the current `--protocol-module` flag on `gen-rust-cst`). This covers the case where the protocol module was generated separately (e.g., by the Python generator or a prior `--protocol-only` invocation) and the caller tells the Rust generator where it lives.

If neither condition is met -- protocol output is not enabled and no import path is supplied -- no `.pyi` is generated.

The existing Bazel rule `generate_rust_parser` (in `rust.bzl`) should expose the `.pyi` in its outputs when it is produced. No new, separate Bazel rule should be introduced for `.pyi` generation.

## What matters in the codebase

### The protocol module

The protocol module is a Python file (e.g., `fltk_cst_protocol.py`) generated from a grammar. It contains `typing.Protocol` classes -- one per grammar rule -- that describe the structural API of a CST node without binding to any concrete implementation. It also contains a `NodeKind` enum, nested `Label` classes, a `Span` protocol, and a `CstModule` protocol. The protocol module is backend-agnostic: it is equally valid as a type contract for the Python-backed CST and the Rust-backed CST.

The protocol module is currently generated by `CstGenerator.gen_protocol_module()` in `gsm2tree.py`. This is the Python CST generator, but the protocol output is structurally independent of the concrete CST -- it only reads the grammar and the rule models. The Rust CST generator (`RustCstGenerator` in `gsm2tree_rs.py`) does not generate protocol modules itself, though it internally constructs a `CstGenerator` instance (stored as `self._py_gen`) and therefore has access to the protocol generation machinery.

### The .pyi type stub

The `.pyi` stub provides type annotations for the compiled Rust extension so that pyright can check code that uses the Rust CST. It is generated by `RustCstGenerator.generate_pyi(protocol_module)` in `gsm2tree_rs.py`. Critically, the `.pyi` stub depends on the protocol module: it contains `import {protocol_module} as _proto` and references protocol types throughout (e.g., `_proto.NodeKind`, `_proto.ClassName.Label`, `_proto.ClassName`). This dependency is by design -- it ensures that the Rust CST's type surface is structurally compatible with the protocol.

The `protocol_module` parameter is a dotted Python import path (e.g., `fltk.fegen.fltk_cst_protocol`) that is interpolated verbatim into the stub's import line. The `.pyi` generator does not generate or read the protocol module; it just references it by import path.

### CLI surface today

The `generate` subcommand (Python path) accepts a grammar file, base name, and CST module name. It always writes the protocol file alongside the CST and parsers -- the protocol output is unconditional, with no flag to suppress it. There is a `--protocol-only` flag that skips CST and parser generation while still emitting the protocol module, but there is no way to generate CST/parsers without also generating the protocol.

The `gen-rust-cst` subcommand (Rust path) accepts a grammar file and output path. It has two optional flags: `--protocol-module` (the import path of an existing protocol module) and `--pyi-output` (custom path for the stub). Without `--protocol-module`, no `.pyi` is generated. With it, the `.pyi` is generated and written alongside (or to the `--pyi-output` path). The `.rs` output is identical regardless of whether `--protocol-module` is provided. Neither `gen-rust-cst` nor `gen-rust-parser` has any ability to generate the protocol module itself.

### Bazel rules today

The Python `generate_parser` rule in `rules.bzl` runs the `generate` CLI subcommand. It declares output files for `_cst.py`, `_parser.py`, and `_trivia_parser.py`, but does not declare `_cst_protocol.py` as an output. The CLI writes the protocol file unconditionally (it lands in the Bazel action directory), but because Bazel does not declare it, Bazel cannot track, cache, or expose it to downstream targets.

The Rust `generate_rust_parser` rule in `rust.bzl` runs two actions: `gen-rust-cst` (producing `cst.rs`) and `gen-rust-parser` (producing `parser.rs`). It declares only those two `.rs` files as outputs. No `--protocol-module` or `--pyi-output` flags are passed, so no `.pyi` is produced. The rule has no protocol-related attributes at all.

### Makefile (non-Bazel) conventions

The Makefile's `gencode` target invokes the generators directly. For Python grammars, it calls `generate` with no protocol-related flags (the protocol is unconditionally written). For Rust CST, it explicitly passes `--protocol-module` and `--pyi-output`. Protocol-only generation is used for the `rust_parser_fixture` test grammar. The overall flow is: run generators, then `make fix` (ruff formatting), then commit.

### Existing tests

Tests in `test_cst_protocol.py` maintain a hardcoded list of all committed protocol module paths (`ALL_PROTOCOL_MODULES`) and assert structural properties (no native imports, no span-selector references). Tests in `test_genparser.py` cover the `--protocol-module` / `--pyi-output` flag interactions, including the rule that `--pyi-output` without `--protocol-module` is rejected. Tests in `test_gsm2tree_rs.py` assert `.pyi` stub structure (imports, header, per-rule stubs). Tests in `test_clean_protocol_consumer_api.py` verify the protocol module's consumer-facing surface.

## Where the request is in tension with the codebase

### Python protocol generation is currently unconditional

The user asks for protocol generation to be opt-in (not enabled by default) for the Python `generate` subcommand. Today, the protocol file is always written, with no way to suppress it. The CLI's help text, docstring, and the `generate_parser` Bazel rule all assume this unconditional behavior. Making protocol opt-in for the Python path is a behavior change: any caller that relies on the protocol file being a side effect of `generate` (including `make gencode` in the Makefile and any out-of-tree Bazel consumers using `generate_parser`) would need to explicitly enable it.

### The .pyi stub structurally depends on the protocol module

The `.pyi` stub references the protocol module by import path throughout. The `.pyi` can only be generated when the protocol module's dotted Python import path is known. The user has clarified that this is the intended behavior: the `.pyi` is produced when the import path is known (either because the Rust generator is co-generating the protocol, or because the path is explicitly supplied), and is not produced otherwise. This resolves the tension -- `.pyi` generation is not unconditional; it is gated on the protocol path being available.

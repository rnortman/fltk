## quality-1

**File:** `fltk/fegen/genparser.py:494`

```python
if protocol_output is not None and protocol_text is not None:
    _write_output_file(protocol_output, protocol_text, "protocol module")
```

**Issue:** The `and protocol_text is not None` guard is logically unreachable. After the preceding `try/except` block (lines 483–491), which exits via `typer.Exit(1)` on any generation error, `protocol_text` is always non-None when `protocol_output is not None` — the branch inside the try at line 487 (`protocol_text = gen.generate_protocol()`) is the only assignment, and any exception exits before reaching line 494. The guard makes the write look defensively optional; in practice it silently skips writing in a state that cannot occur, which is worse than asserting.

Compare with the parallel `init_pyi_text` path seven lines later (line 505): `assert init_pyi_text is not None` — that approach correctly documents the invariant and narrows the type without false optionality.

**Consequence:** Future authors adding more conditional outputs in `gen_rust_cst` will likely copy this `if X is not None and Y is not None:` pattern rather than the `assert` pattern, propagating misleading silent-skip semantics throughout the output-writing section. The pattern suggests `protocol_text` can be None here, which trains incorrect mental models.

**Fix:** Replace `if protocol_output is not None and protocol_text is not None:` with the same idiom used below:

```python
if protocol_output is not None:
    assert protocol_text is not None
    _write_output_file(protocol_output, protocol_text, "protocol module")
```

## quality-2

**File:** `fltk/fegen/gsm2lib_rs.py:56`

```python
"# Generated by `make gencode` (gen-rust-* with --init-pyi-output); do not edit by hand.",
```

**Issue:** `render_stub_package_init` is a grammar-independent utility in `gsm2lib_rs.py`, described in the design as a shared helper for both the in-tree Makefile flow and the Bazel `generate_rust_parser` rule. But it hardcodes a reference to `` `make gencode` `` in the generated marker body. The Bazel rule (`rust.bzl:148–155`) calls the same `gen-rust-cst --init-pyi-output` CLI path, so out-of-tree Bazel consumers who use `generate_rust_parser` will receive a committed `__init__.pyi` that says "Generated by `make gencode`" — factually wrong for anyone not using this project's Makefile.

**Consequence:** Every out-of-tree user of `generate_rust_parser` with `protocol_module` set gets a marker with an inaccurate provenance comment. The helper is billed as a generic utility but the baked-in `` `make gencode` `` reference makes it accurate only for the in-tree Make workflow. As the Bazel rule proliferates to downstream projects, the incorrect provenance propagates into each one.

**Fix:** Remove the `` `make gencode` `` prefix; state provenance in terms of the CLI path, which is accurate for all callers:

```python
"# Generated by gen-rust-* --init-pyi-output; do not edit by hand.",
```

## quality-3

**Files:** `fltk/fegen/genparser.py:253` and `fltk/fegen/gsm2tree_rs.py:450`

```python
# genparser.py (generate --protocol path)
protocol_text = "# ruff: noqa: N802\n" + ast.unparse(protocol_mod)

# gsm2tree_rs.py (RustCstGenerator.generate_protocol)
return "# ruff: noqa: N802\n" + ast.unparse(protocol_mod)
```

**Issue:** The formula for rendering protocol text — ruff noqa prefix + `ast.unparse` — is duplicated verbatim across two files. The design §2.2 called for "a shared `render_protocol_text(grammar) -> str` helper" precisely to prevent drift and named the byte-identity test as "the guardrail regardless of the exact factoring chosen." The implementation chose duplication over extraction.

The byte-identity test (`test_gen_rust_cst_protocol_output_matches_python_protocol`) guards against output drift. It does not guard against editing the formula in one place without the other: if the ruff noqa comment gains additional codes in one file (which the existing comments in `genparser.py:248–252` explain in detail), the other file requires a matching update that no static analysis or test will catch until the byte-identity test runs.

**Consequence:** The two-file maintenance point is a latent drift hazard. The comment explaining *why* only N802 is suppressed (not E501, not F821) lives only in `genparser.py`; `gsm2tree_rs.py` just copies the string without the rationale. A maintainer updating the noqa comment in one file has no automated signal to update the other.

**Fix:** Extract the formula into a module-level helper, as the design anticipated:

```python
# in genparser.py or a shared module
def _render_protocol_text(protocol_mod) -> str:
    """Render protocol module source with file-level ruff suppressions.

    N802: CstModule @property methods have PascalCase names ...
    E501 is NOT added: make fix reformats so no line exceeds the limit ...
    F821 is NOT added: nested Label refs resolve via from __future__ ...
    """
    return "# ruff: noqa: N802\n" + ast.unparse(protocol_mod)
```

`generate_protocol()` in `gsm2tree_rs.py` imports and calls this helper so both paths share a single maintenance point and a single home for the noqa rationale.

## quality-4

**File:** `fltk/fegen/gsm2tree_rs.py:446–449`

```python
protocol_gen = CstGenerator(
    grammar=self.grammar,
    py_module=pyreg.Module(["_protocol"]),  # placeholder: any non-empty value
    context=create_default_context(),
)
```

**Issue:** `generate_protocol()` works around the `py_module.import_path` truthiness gate in `CstGenerator.gen_protocol_module()` (design §1.2's "trap") by providing a placeholder `pyreg.Module(["_protocol"])`. The design documents this as an "implementer hazard" and the docstring explains it thoroughly. However, no TODO is recorded against the underlying dual-use of `py_module` — used simultaneously as the concrete-CST module path for annotation emission and as a truthiness sentinel for the `kind: Literal` discriminant in protocol output.

**Consequence:** Any future Rust generator (or test helper) that calls `gen_protocol_module()` without knowing about this trap will get the degraded `kind: object` form. The placeholder workaround will need to be discovered and repeated each time. Without a TODO in either the TODO system or inline on `gen_protocol_module()` itself, the trap is visible only to readers of `gsm2tree_rs.py`'s docstring, not to readers of `gsm2tree.py` where the gate lives. The workaround-for-existing-bug pattern is in place but unlinked from the underlying code.

**Fix:** Add a `TODO(protocol-module-truthiness-gate)` comment at the truthiness gate in `gsm2tree.py` (the `if rule_name and self.py_module.import_path:` line, currently ~line 891) explaining that callers needing the Literal form from a Builtins-backed context must use a non-empty placeholder, and record a corresponding entry in `TODO.md` so the dual-use is tracked and future callers (and a future clean-up to take an explicit flag) have a single place to find the context.

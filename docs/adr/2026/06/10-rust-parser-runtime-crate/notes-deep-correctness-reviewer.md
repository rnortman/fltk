## Findings

### correctness-1
- **File:line**: `crates/fltk-parser-core/tests/memo_toy.rs:88-107` (rule body `rule_expr`); header claim at lines 1-18.
- **What's wrong**: `rule_expr` is not semantically the same grammar as the Python reference it claims to port. Python's `rule_expr` (test_memo.py:51-83), after a *successful* recursive call, commits: if the `"+" num` continuation fails it `return None` — it does not fall back to the `num` alternative. The Rust body uses plain fall-through: when the recursive call succeeds but `token(p1) != "+"` or `num_at(p1+1)` fails, control falls into "Alternative 2" and returns `Some(Num(...), start_pos+1)`.
- **Why**: Trace the final growth iteration on `"0+1+2+3+4+i"`: cached head result is pos 9; `token(9) == "+"` but `num_at(10)` is `"i"` → fails. Python returns `None`, so `grow_seed` breaks via the `result is None` branch (memo.py:248 left disjunct). Rust returns `Some(ApplyResult{pos: 1, result: Num(0)})`, so `grow_seed` breaks via `new_pos (1) <= entry.final_pos (9)` instead (memo.rs:334 right disjunct). The file header ("Ports of all `fltk/fegen/pyrt/test_memo.py` test cases ... same grammars") and the design (§4.1 "same grammars, same inputs") assert exact-grammar fidelity that the code contradicts. The grammar comment `a := b "+" num | <nothing>` (line 12) is also wrong — an empty alternative would *succeed* consuming nothing; `indirect_a` has no second alternative (it correctly matches Python's `a := b "+" num`).
- **Consequence**: `test_direct` passes only because the fallback's `new_pos` can never exceed the seed's `final_pos` (seed is already `Num` at `start+1`), so the termination guard masks the divergence. The test therefore does not exercise the Python direct-recursion path it claims to replicate (the `!has_result` break is exercised in this port only via `test_indirect`, whose body *is* commit-shaped). If the memoizer's growth loop were later edited in a way that mishandles the failed-final-iteration case for direct recursion specifically, this suite would not catch it while the Python suite would — defeating the test's stated parity purpose. No runtime misbehavior; the divergence is confined to the toy test.
- **Suggested fix**: Make `rule_expr` commit like Python: on `Some` from the recursive call, return `None` when the `"+" num` continuation fails instead of falling through to Alternative 2 (e.g. wrap Alternative 1 so a successful recursion short-circuits the function). Fix the `<nothing>` line in the grammar comment to `a := b "+" num`.

No other findings: memo, terminalsrc, errors runtime logic is a faithful port; all divergences found are the ones the design declares (negative-pos rejection, source-bearing spans, deterministic error ordering, `py_repr_str` non-ASCII domain).

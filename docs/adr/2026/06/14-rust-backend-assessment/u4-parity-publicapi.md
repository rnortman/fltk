# U4 — Cross-Backend Parity & the Public API / Drop-In Story

Scope: how Python/Rust behavioral equivalence is established and tested; the generated
Rust CST public surface vs the Python backend; where they MATCH and where they DIVERGE in
ways that would force a downstream consumer to edit annotations/call sites; the protocol
layer and backend selection. This is the CLAUDE.md crown-jewel concern.

Verdict in one line: **The drop-in story is real and unusually well-defended.** A shared
Protocol module is the single contract both backends conform to; class names, label
namespaces, accessor names, NodeKind, and equality/hash are byte-for-byte identical across
backends by construction; a deep parity corpus runs *non-skipped* in the gate. The residual
divergences are documented, narrow, and mostly intrinsic to Rust ownership (span hand-in,
`children` list identity). The biggest structural risk is not correctness but **maintenance
duplication**: two independent string-emitting generators that must be kept in lockstep, with
"keep them identical" enforced largely by tests rather than by shared code.

---

## 1. The architecture of equivalence: one Protocol, two backends

The whole drop-in story hinges on one design choice: there is a single generated **Protocol
module** that *both* backends must structurally satisfy. It is generated by the **Python**
generator (`gsm2tree.py:gen_protocol_module`, referenced at `gsm2tree.py:188` and
`test_cst_protocol.py:72`), committed as `fltk/fegen/fltk_cst_protocol.py`, and is the only
CST import that consumer code (e.g. `fltk2gsm.py`) is supposed to touch.

The protocol module (`fltk/fegen/fltk_cst_protocol.py`) defines, per grammar rule:
- a `typing.Protocol` class named exactly after the rule (e.g. `Items`, `Grammar`) —
  `fltk_cst_protocol.py:103,231`;
- a nested `Label` class with `ClassVar[object]` members (`fltk_cst_protocol.py:104-106,
  232-236`);
- a `kind: typing.Literal[NodeKind.X]` discriminant (`:107,:238`);
- `span: terminalsrc.Span | fltk._native.Span` (the deliberately *widened* union, `:108`);
- `children: list[tuple[Label | None, <child union>]]` (`:109`);
- the full generic + per-label accessor/mutator surface (`append`, `extend`, `child`,
  `insert`, `remove_at`, `replace_at`, `clear`, and the `append_<lbl>/extend_<lbl>/
  children_<lbl>/child_<lbl>/maybe_<lbl>` quintet, e.g. `:264-302`);
- a module-level `NodeKind` enum (`:33`) and a `CstModule` Protocol exposing one
  `@property` per rule type (`:895-936`).

Because the Protocol is the contract, "Python vs Rust public surface" is really "does each
backend conform to the same Protocol." The conformance is checked by **pyright** against the
`CstModule` Protocol and against per-node Protocols, both in the test suite and at the
`make check` gate.

### Backend selection (how a consumer chooses)

Selection is **not** runtime-polymorphic dispatch; it is a string parameter at
parser-generation time:
- `plumbing.generate_parser(..., rust_cst_module="mygrammar.cst")` — `plumbing.py:212-258`.
  If `rust_cst_module is None` it generates+`exec`s the Python dataclass CST
  (`plumbing.py:247-251`); otherwise it imports the pre-built Rust extension and reads its
  public classes (`_load_rust_cst_classes`, `plumbing.py:81-117`, called at `:255`).
- `plumbing.parse_grammar(..., rust_fegen_cst_module="fegen_rust_cst.cst")` —
  `plumbing.py:120-183` does the same for FLTK's own self-host path.
- The CLI mirrors this in `genparser.py` (`gen-rust-cst` / `gen-rust-parser` commands,
  `genparser.py:265+`).

Consequence for a downstream consumer: switching backends means **changing one import / one
build step**, not editing annotations or call sites — exactly the CLAUDE.md goal. The Rust
backend "hard-errors on failure; never falls back to Python" (`plumbing.py:252-255`,
`RustBackendUnavailableError` at `:70-78`), so a misconfigured Rust backend fails loudly
rather than silently diverging.

## 2. Public surface: where Python and Rust MATCH (by construction)

The match is not aspirational — it is forced by shared naming code and `#[pyclass(name=...)]`
pins.

| Surface element | Python backend | Rust backend | Match mechanism |
|---|---|---|---|
| Node class name | `snake_to_upper_camel(rule)` (`gsm2tree.py:46-47`) | same, delegated via `_py_gen.class_name_for_rule_node` (`gsm2tree_rs.py:180,757`) and pinned by `#[pyclass(name = "{class_name}")]` (`gsm2tree_rs.py:1166`) | **shared function** + name pin. **No `Node` suffix** anywhere. |
| `NodeKind.X` member | `class_name.upper()` (`gsm2tree.py:95-97`) | same, `#[pyo3(name = "{python_name}")]` (`gsm2tree_rs.py:595-599`) | shared rule + pin |
| Label namespace access | nested `ClassName.Label.MEMBER` (Python `enum`) | exposed as `ClassName.Label` via `#[classattr] fn Label(...)` returning the PyO3 enum type object (`gsm2tree_rs.py:1209,1293-1302`) | both reachable as `Cls.Label.MEMBER` |
| Label member name | `label.upper()` (`gsm2tree.py:166`) | `_python_label_name = label.upper()` + `#[pyo3(name=...)]` (`gsm2tree_rs.py:150-152,694`) | shared rule + pin |
| Accessor/mutator method names | `append_<lbl>`, `child_<lbl>`, `maybe_<lbl>`, `insert`, `remove_at`, `replace_at`, `clear`, … | identical method names emitted in `#[pymethods]` | both generated to satisfy the same Protocol |
| `kind` discriminant value | `NodeKind.X` (same enum identity via canonical name) | PyO3 enum with same canonical name | canonical-string equality (below) |

### Equality / hash / ordering — the cross-backend invariant

This is the subtle, load-bearing part. Python's `enum.Enum` Label/NodeKind and Rust's PyO3
enums are **distinct Python types** (`test_cross_backend_label_equality.py:157-161`), so
naive `==` would be `False`. Both generators therefore emit a **canonical-name eq/hash**:

- Python: `_emit_cross_backend_eq_hash` (`gsm2tree.py:99-132`) — `__eq__` is identity →
  same-type member-name → `_fltk_canonical_name` cross-type → `NotImplemented`; `__hash__`
  hashes the canonical string. Same code is hand-mirrored in the committed protocol module
  (`fltk_cst_protocol.py:50-62` for NodeKind, `:86-97` for `_ProtocolLabelMember`).
- Rust: `_emit_rust_cross_backend_eq_hash` (`gsm2tree_rs.py:540+`) plus a `_fltk_canonical_name`
  getter returning the same `"ClassName.Label.MEMBER"` / `"NodeKind.X"` string
  (`gsm2tree_rs.py:715,719-722`).

Result: `py.Items.Label.NO_WS == ext.Items.Label.NO_WS` both directions, hashes agree, sets
collapse, dict round-trips work, and comparison against unrelated objects never raises. This
is exhaustively tested (AC1-AC8) in `test_cross_backend_label_equality.py` and again in
`test_clean_protocol_consumer_api.py::TestCrossBackendEqualityHash`
(`:428-557`). The three-way agreement (protocol / Python-concrete / Rust-concrete) is the
literal mechanism that lets `match child.kind: case cst.Item.kind:` dispatch identically over
trees from either backend (`test_clean_protocol_consumer_api.py:565-723`).

`Span.kind` even returns the **shared Python `SpanKind.SPAN` object by identity** from a Rust
native span (`test_clean_protocol_consumer_api.py:773-778`,
`:696-701`), so `match`-dispatch over span children works regardless of which Span class the
runtime produced.

### Error-message parity

The Rust backend reproduces Python's exact `IndexError`/`TypeError` message text for mutators
— e.g. `"Identifier.remove_at: index 5 out of range (1 children)"`,
`"Identifier.insert: label argument is not a Identifier_Label; got str"` — asserted byte-equal
in `test_cst_mutators_parity.py::TestMessageParity` (`:542-609`) and via regex `match=` in the
parametrized backend tests (`:266-305,:463-474`). Note the message uses the
`Identifier_Label` Python-visible name (see §3).

## 3. Where Python and Rust DIVERGE (and whether it forces consumer edits)

The honest list. Each item is tagged with whether it forces downstream annotation/call-site churn.

1. **Label enum Python-visible name: `ClassName_Label` (Rust) vs nested `ClassName.Label`
   (Python).** The Rust label enum's `#[pyclass(name = "ClassName_Label")]`
   (`gsm2tree_rs.py:657,690`) gives it a module-level Python name `Items_Label`, whereas the
   Python backend's Label is *only* a nested class `Items.Label`. **Does NOT force churn for
   the supported access path:** both backends expose `Cls.Label.MEMBER` (Rust via the
   `#[classattr] Label` shim, `gsm2tree_rs.py:1293-1302`), and the Protocol declares only the
   nested form. The `Items_Label` flat name leaks into the error string
   (`"...is not a Identifier_Label..."`) — a cosmetic, intentional, parity-matched detail
   (both backends produce the *same* string). A consumer who reached for a bare module-level
   `Items_Label` symbol on the Python backend would find it absent, but the protocol never
   advertised that name, so this is out-of-contract.

2. **`children` getter identity: Rust returns a fresh per-call snapshot list; Python returns
   the node's live internal list.** Explicitly documented at `gsm2tree_rs.py:1304-1338`
   (esp. `:1307-1311`): "The returned list is a per-call snapshot; in-place mutation of the
   list is a silent no-op on the tree. The Python backend returns the node's actual internal
   list, so there is a known backend divergence on list-level identity." **Could force a
   behavioral change** for a consumer that mutated `node.children` in place — but the
   sanctioned mutation path (`insert/remove_at/replace_at/clear`) is identical on both
   backends and is what the Protocol exposes. The Protocol types `children` as `list[...]`
   (not a read-only view), so a consumer *could* legally write to it; on Rust that write is
   silently dropped. This is the single most likely "works on Python, silently wrong on Rust"
   trap. (Cross-ref: this is a runtime-semantics divergence, not an annotation one.)

3. **Span hand-in asymmetry.** Python mutators accept both `terminalsrc.Span` and native
   `fltk._native.Span`; Rust mutators accept **only** native `Span` and reject
   `terminalsrc.Span` with `TypeError("unsupported child type")`
   (`test_cst_mutators_parity.py:672-699`, esp. `:695-699`). **Could force a call-site change**
   for a consumer that constructs `terminalsrc.Span` objects and hands them to a Rust node.
   The Protocol's mutator signatures use `fltk.fegen.pyrt.span.Span` (the backend-selecting
   alias, see `span_protocol`/selector in `test_span_protocol.py`), which papers over this at
   the type level but not at runtime. This asymmetry is deliberately **excluded from the
   exact-parity matrix** (`test_cst_mutators_parity.py:18-21,667-671`).

4. **`children_<label>()` return type: stub says `Iterator[T]`, Rust runtime returns a
   `list`.** Documented at `gsm2tree_rs.py:381-384` ("stub/runtime diverge"). Harmless for
   the declared contract (callers iterate), but a consumer doing `next(...)` or relying on
   laziness would see different behavior. Does not force annotation edits.

5. **Span runtime type differs by backend** (`terminalsrc.Span` vs `fltk._native.Span`), which
   is *why* the Protocol `span` annotation was **widened** to the union
   `terminalsrc.Span | fltk._native.Span` (`fltk_cst_protocol.py:108` etc.). The widening is
   **additive** and is specifically tested to NOT break existing `terminalsrc.Span`-annotated
   consumers (`test_cst_protocol.py:493-568`, `test_clean_protocol_consumer_api.py:914-953`).
   **Caveat surfaced honestly in-tree** (`test_cst_protocol.py:592-614`): an *uncast* call
   site passing `node.span` to a `terminalsrc.Span`-typed parameter *does* now require a
   narrowing cast — the backward-compat guarantee in the design (§2.7) explicitly applies to
   the `typing.cast`-using production pattern, not to bare uncast assignments. So the widening
   is "near-drop-in," with a documented small-print exception.

6. **`Span` types are not `==` across backends** even for matching `(start, end)`
   (`test_cst_mutators_parity.py:14-16`). Parity helpers therefore compare `.start`/`.end`
   directly (`parser_parity.py:39-42`, `test_cst_mutators_parity.py:64-67`). A consumer
   comparing whole spans across backends would get `False`. Narrow, rarely relevant.

Net: **no divergence forces wholesale annotation churn**; the CLAUDE.md red line (renaming
public symbols, forcing annotation rewrites) is not crossed. The real-world risks are
runtime-behavioral (#2 in-place `children` mutation, #3 span hand-in) and one type-level
small-print (#5 uncast span call sites).

## 4. How equivalence is *tested* — and whether the tests actually run

### The parity corpus (parser-level structural + error equivalence)

`tests/parser_parity.py` is the shared engine (not itself a test module). It provides:
- `assert_cst_equal` (`:10-42`): recursively compares `kind`, `span.start/end`, child count,
  per-child label equality, and node-vs-span "species" (`hasattr(child, "children")`),
  recursing into node children and comparing span-child `start/end`.
- `assert_error_equiv` (`:119-148`): compares error *position*
  (`py.error_tracker.longest_parse_len` vs `rust.error_position()`) and structurally compares
  the formatted error message (header byte-equal; rule-group order equal; per-rule token sets
  equal, order-insensitive).
- `run_parity_corpus_entry` (`:151-192`): dispatches SUCCESS / PARTIAL(pos) / FAIL and asserts
  both backends agree on outcome, end position, and tree/error.

Two corpora drive it:
- `test_rust_parser_parity_fegen.py` — Python fegen parser vs `fegen_rust_cst.Parser`, over a
  19-entry corpus including the **full fegen grammar self-parse** (`:46-85`), multibyte,
  comments, and multiple failure modes; parametrized over `capture_trivia ∈ {False, True}`
  (`:90-96`). Also contains thorough **comparator self-tests** (`:104-339`) that prove
  `assert_cst_equal`/`assert_error_equiv` actually fail on kind/span/child-count/label/species
  /deep-nesting mismatches — i.e. the parity oracle is not a no-op.
- `test_rust_parser_parity_fixture.py` — generated Python fixture parser vs
  `rust_parser_fixture.Parser`, over a ~45-entry corpus covering left/indirect recursion,
  quantifiers, WS_REQUIRED, union labels, multibyte, paren/grouped sub-expressions, and a rich
  set of control/bidi/invisible-character error-escaping cases (`:53-128`).

### CST-node-level parity (construction & mutation)

`test_cst_mutators_parity.py` parametrizes `insert/remove_at/replace_at/clear` over both
backends (`_BACKENDS`, `:49-53`) and asserts identical trees, identical exception types, and
byte-equal messages. `test_cross_backend_label_equality.py` and
`test_clean_protocol_consumer_api.py` cover the Label/NodeKind eq/hash matrix and the
dual-shape (interleaved walk vs `match/case`) dispatch over both Python- and Rust-produced
trees.

### Self-host GSM equivalence (end-to-end)

`test_clean_protocol_consumer_api.py:353-361`: parse the fegen grammar through BOTH backends
and assert `python_gsm == rust_gsm` — i.e. the entire `Cst2Gsm` pipeline produces an identical
Grammar Semantic Model whether the underlying CST came from Python dataclasses or Rust PyO3
nodes. This is the strongest single equivalence claim.

### Type-level conformance (pyright)

The Protocol surface is checked by pyright in `test_cst_protocol.py` (member access, CstModule
conformance, plain-class stand-in proving the Protocol is not dataclass-specific) and
`test_clean_protocol_consumer_api.py` (Shape 1 / Shape 2 fixtures pyright+ruff clean, no
`cast`/`runtime_checkable`/`type: ignore`). A `.pyi` stub for the Rust extension is generated
by `gsm2tree_rs.py:gen_pyi_stub` (`:300-396`) and references the **protocol module's** types
under alias `_proto` so the stub's nominal identities match what consumers program against
(`:337,341,349,361`). Note the stub deliberately diverges from runtime in two spots
(`children_<label>` Iterator-vs-list `:384`; `NodeKind = _proto.NodeKind` alias `:322`),
both annotated.

### Are these tests gated, or silently skipped?

**They run in the gate.** Every Rust parity test guards with
`pytest.importorskip("fegen_rust_cst" / "rust_parser_fixture")` and the module docstrings warn
"A CI lane where every test here is skipped is a failure signal"
(`test_rust_parser_parity_fegen.py:4`, `test_rust_parser_parity_fixture.py:4`). The gate
satisfies those imports: `make check` → `check-ci` → `check-common`
(`Makefile:39-51,76`) runs the `test` step, and `test` depends on `build-test-fixtures`
(`Makefile:94-97`), which builds `fegen_rust_cst`, `rust_parser_fixture`, and `poc_cst`
(`Makefile:205,210,214`) via `maturin develop` before pytest runs. So under `make check` the
`importorskip` guards are satisfied and the parity corpus genuinely executes — parity is
enforced, not aspirational. (Risk: outside `make`, e.g. a bare `uv run pytest` without
building the fixtures, every parity test silently skips and a green run proves nothing. The
docstring warning is the only guard against that human error.)

## 5. Structural critique (the "did we build it right" lens)

- **Two generators, one contract, kept in sync by tests, not by shared structure.** The
  Python concrete generator (`gsm2tree.py`), the Protocol generator (also in `gsm2tree.py`),
  and the Rust generator (`gsm2tree_rs.py`, which *delegates naming to* `gsm2tree.py` via
  `self._py_gen`, e.g. `:180,:757`) all emit **strings/ASTs**, not a shared IIR. Naming is
  genuinely shared (`naming.snake_to_upper_camel`, `class_name_for_rule_node`), which is the
  thing that most needs to be shared. But the *eq/hash logic, message text, mutator semantics,
  and accessor surface* are independently re-emitted in each generator and held equal by the
  parity suite. Comments throughout `gsm2tree_rs.py` say "mirrors the protocol generator
  exactly" / "shared with NodeKind" — the lockstep is real but is **convention + tests**, not
  a compiler guarantee. This is the central long-term maintenance liability: a change to the
  Python surface that forgets the Rust mirror is caught only if a parity test covers it.

- **The canonical-name equality scheme is clever and necessary, but it is a hand-rolled
  cross-type protocol** (`_fltk_canonical_name` string matching with `NotImplemented`
  fallback) re-implemented in three places (Python concrete, committed protocol, Rust). It is
  well-tested (the AC1-AC8 matrix, including the disjointness of `NodeKind.ITEM` vs
  `Items.Label.ITEM`, `test_cross_backend_label_equality.py:214-234`), but it is exactly the
  kind of subtle invariant that rots silently if someone adds a fourth backend or a new enum
  family without re-deriving the canonical string.

- **The `children` live-list-vs-snapshot divergence (#2) is the one I'd escalate.** It is the
  only divergence where correct-looking consumer code (mutating `node.children` in place)
  works on Python and is a silent no-op on Rust, and the Protocol's `list[...]` typing
  actively invites it. A read-only sequence type in the Protocol would have closed this, at
  the cost of an annotation change — a deliberate tradeoff that was apparently not taken.

- **Span hand-in asymmetry (#3) and the uncast-call-site span caveat (#5)** are the two places
  where the "no call-site edits" promise has fine print. Both are documented in-tree, which is
  the right call, but they mean "near-drop-in" is the honest phrasing, not "drop-in."

---

## 6. Bottom line for the retrospective

On the crown-jewel axis — *does the Rust backend present the same public API so downstream
consumers don't have to rewrite annotations or call sites* — the answer is **yes, to a high
and well-tested degree.** Class names, label namespaces, NodeKind, accessor method names, and
cross-backend equality/hash are identical by construction and verified by a parity corpus that
actually runs in the gate, up to and including a full self-host `python_gsm == rust_gsm`
equivalence. The protocol-as-contract design is the right shape and is the thing that makes
the drop-in claim defensible.

The reservations are: (a) equivalence is maintained by **two parallel string generators kept
in lockstep by tests**, which is a standing maintenance tax and a silent-drift risk for any
surface a test forgot to cover; (b) **three documented runtime/type divergences** (`children`
in-place mutation no-op on Rust, span hand-in asymmetry, uncast span call sites) mean the
honest label is "near-drop-in with documented small print," not "drop-in"; and (c) the parity
gate's correctness depends on the fixtures being *built* — a bare `pytest` invocation skips
the entire parity suite and looks green.

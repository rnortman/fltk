## 1. The crate topology (what gets built, and by which build system)

There are **two disjoint Cargo build universes**, by design:

**Root workspace** (`Cargo.toml:2`): `members = [".", "crates/fltk-cst-core", "crates/fltk-cst-spike", "crates/fltk-parser-core"]`.
Confirmed by `cargo metadata`: the 4 members are `fltk-native`, `fltk-cst-core`, `fltk-cst-spike`, `fltk-parser-core`.
- `fltk-native` (root `[package]`, `src/lib.rs`) — cdylib → Python ext `fltk._native`. `default = ["extension-module"]` → pulls pyo3 (`Cargo.toml:18-19,24`).
- `fltk-cst-core` v0.2.0 (`crates/fltk-cst-core/Cargo.toml`) — rlib, pyo3 **optional**, `default = ["python"]`.
- `fltk-parser-core` v0.2.0 — rlib, **never** links pyo3 (no python feature at all; "pyo3-freedom is structural absence", `crates/fltk-parser-core/Cargo.toml:11-13`). Depends on `fltk-cst-core` with `default-features=false` and `regex-automata`.
- `fltk-cst-spike` — leftover PoC, `default = []` (python OFF by default), has a criterion bench `traverse`. **Still a workspace member** (`Cargo.toml:2`).

**Four standalone crates**, each carrying its own `[workspace]` line to detach from the root workspace, each with its own `Cargo.lock`:
- `crates/fegen-rust/` (`fegen-rust-cst`, cdylib+rlib) — `[workspace]` at line 4; "canonical first-class fegen Rust artifact … Promoted from tests/rust_cst_fegen/" (`crates/fegen-rust/Cargo.toml:1-3`). `cst.rs` is 15.5K LoC generated.
- `tests/rust_cst_fixture/` (`phase4-roundtrip-cst`, cdylib) — `[workspace]` at line 3.
- `tests/rust_parser_fixture/` (`rust-parser-fixture`, rlib+cdylib).
- `tests/rust_poc_cst/` (`poc-cst`, cdylib+rlib).

Confirmed standalone: none of `phase4-roundtrip-cst`, `rust-parser-fixture`, `poc-cst`, `fegen-rust-cst`
appear in the root `Cargo.lock`; only `fltk-cst-spike` does.

**Why detached:** workspace feature unification. If a fixture were a root member, the root's
`fltk-native` (default `extension-module` → pyo3) would re-enable pyo3 across the whole graph,
breaking the python-off feature isolation the no-pyo3 guard depends on. The Makefile comments
state this explicitly (`Makefile:133-134`, `Makefile:152-154`).

### ORPHAN / DEAD WEIGHT (finding)
`tests/rust_cst_fegen/` **still exists and is git-tracked** (`git ls-files`: Cargo.toml, Cargo.lock,
src/cst.rs [645KB], src/lib.rs, src/parser.rs, src/native_parser_tests.rs) even though it was
"Promoted" to `crates/fegen-rust/`. It is referenced by **nothing** except the prose comment in
`crates/fegen-rust/Cargo.toml:3`. It is NOT in any Makefile target — not built, not tested, not
clippy'd, not regenerated by `gencode`, and **not covered by cargo-deny** (the `cargo-deny` target
lists `rust_cst_fixture`, `rust_parser_fixture`, `rust_poc_cst` but NOT `rust_cst_fegen` —
`Makefile:181-186`). It carries a 645KB generated `cst.rs` that can silently rot: no lane regenerates
it, so it is dead, unverified duplicate code in the tree. This is a cleanup/`make check` blind spot.

## 2. The check-target family: `check` vs `check-ci` vs `check-common`

Heavily documented in a banner at `Makefile:7-34`. Structure:

- **`check-common`** (`Makefile:39-51`): the shared base — runs every step EXCEPT cargo-deny, via a
  shell loop over a hardcoded `steps` string:
  `lint format-check typecheck test cargo-check cargo-clippy cargo-test cargo-test-python-features cargo-test-no-python cargo-clippy-no-python check-no-pyo3`.
  Each step runs to a tmpfile; on failure prints `FAILED: <step>` + captured output and exits 1.
- **`check`** (`Makefile:61-70`): LOCAL/PRECOMMIT lane. `check: check-ci` then runs `cargo-deny`.
  The git pre-commit hook is literally `exec make check` (`.git/hooks/pre-commit`).
- **`check-ci`** (`Makefile:76`): `check-ci: check-common` — nothing else. CI calls this.

The **one sanctioned divergence** is `cargo-deny`, present only on `check`. Rationale stated
repeatedly: cargo-deny is **not installed on the GitHub Actions runner** and they chose not to
install it, so supply-chain/advisory enforcement is **local-pre-commit-only** (`Makefile:17-25,53-60`).

**Anti-drift design:** `check` depends on `check-ci` (not `check-common`) so that any step added to
the shared base is structurally inherited by both lanes, and adding a step only to `check` shows up
as an anomaly. The banner declares adding steps directly to `check`/`check-ci` "FORBIDDEN"
(`Makefile:27-32`). This is a thoughtful, well-defended structure.

**CI workflow** (`.github/workflows/ci.yml`): on push-to-main + all PRs; ubuntu-latest; installs uv
(setup-uv pinned by SHA), dtolnay/rust-toolchain stable (pinned, Dependabot-refreshed weekly per
`.github/dependabot.yml`), then `make check-ci`. Single job, single OS, single Python (3.10).

### CI GAPS (findings)
1. **cargo-deny runs nowhere in CI.** Supply-chain/RustSec/license/yanked-crate enforcement depends
   entirely on a developer having the local pre-commit hook installed and not bypassing it
   (`git commit --no-verify` or a fresh clone with no hook). For a "production-ready" judgment this
   is a real hole: an advisory landing in a transitive dep is invisible to CI and to PR review.
2. **Single-platform, single-Python CI.** Only ubuntu-latest / py3.10. No macOS, no Windows, no
   3.11/3.12 (despite pyproject classifiers advertising 3.10–3.12 and PyPy, `pyproject.toml:19-24`).
   abi3-py310 wheels are claimed to work on 3.10+ but nothing in CI exercises 3.11/3.12 at runtime.
3. **No Bazel job in CI.** The entire Bazel packaging surface (rust.bzl, the `bootstrap_native`
   smoke target in BUILD.bazel) is **never built in CI** — `make check-ci` is pure cargo+pytest+ruff.
   The downstream-facing build path has zero automated regression coverage in this repo.
4. **No wheel-build / no publish job.** Nothing builds or smoke-imports a maturin wheel in CI; the
   `verify-pyo3-ext-module` concern (does the produced ext actually avoid linking libpython?) is
   never machine-checked anywhere.

## 3. The feature matrix: python vs --no-default-features, and the no-pyo3 guard

Three cargo lanes encode a 2-axis matrix (python-on / python-off):

- **`cargo-check`** (`Makefile:102-103`): `cargo check -q` — whole root workspace, default features.
  Comment notes per-feature checks are skipped because clippy (a superset) covers them.
- **`cargo-clippy`** (`Makefile:127-131`): `cargo clippy -- -D warnings` on the workspace, **plus**
  three out-of-workspace manifests via `--manifest-path`: fegen-rust, rust_poc_cst,
  rust_parser_fixture (the last with `--features python`). `-D warnings` = warnings are hard errors.
- **`cargo-test`** (`Makefile:105-106`): `cargo test -q` workspace, default features.
- **`cargo-test-python-features`** (`Makefile:117-123`): runs `fltk-cst-core --features python`. This
  is the only step that **links real libpython** (the abi3 ext-module path does not). It resolves a
  uv-managed standalone CPython 3.10 via `env -u VIRTUAL_ENV uv python find --managed-python 3.10`,
  pointing pyo3 at it through `PYO3_PYTHON`. The python-build-standalone interpreter ships the
  unversioned `libpython3.10.so` needed to link; a system venv python does not. Fails with a helpful
  message if no managed 3.10 is found (`Makefile:118-122`). This is genuinely fiddly host-env coupling.
- **`cargo-test-no-python`** (`Makefile:135-141`): the python-OFF lane. Must use `-p` selection (not
  `--no-default-features` on the workspace) because workspace unification would re-enable pyo3 via
  fltk-native (`Makefile:133-134`). Runs: `fltk-cst-core --no-default-features`, `fltk-cst-spike`,
  `fltk-parser-core`, `rust_parser_fixture` (default), `fegen-rust --no-default-features`,
  `rust_poc_cst --no-default-features`.
- **`cargo-clippy-no-python`** (`Makefile:143-150`): clippy `-D warnings` over the same python-off set,
  plus `fltk-cst-spike --features python` (compile-checks the generated output in python-on mode too).

**`check-no-pyo3`** (`Makefile:155-175`): a mechanical guard that asserts pyo3 is **absent** from the
python-off dependency graphs. For each of 6 graphs (cst-spike, cst-core --no-default-features,
parser-core, rust_parser_fixture, fegen-rust --no-default-features, rust_poc_cst --no-default-features)
it runs `cargo tree --edges normal,build`, FIRST asserts a known crate IS present (positive control
to avoid false-pass on a silent `cargo tree` failure), THEN asserts `pyo3` is NOT present. This is a
well-constructed negative test with the positive-control guard. It enforces the core architectural
invariant that pure-Rust consumers never pull pyo3.

**Assessment of the matrix:** thorough and disciplined for the *cargo* side — both feature polarities
are compiled, tested, clippy-linted, and the absence invariant is mechanically verified with a
positive control. The cost is verbosity and per-manifest fan-out (every standalone crate must be
listed by hand in 4 different targets: clippy, clippy-no-python, test-no-python, deny). That hand-
maintained list is the drift surface — and `rust_cst_fegen` already fell off it (see §1 orphan).

## 4. cargo-deny (supply-chain gate)

`deny.toml`: advisories `version=2`, `yanked="deny"` (fails on any unfixed RustSec advisory or yanked
crate). Licenses allow-list = MIT / Apache-2.0 / Apache-2.0-WITH-LLVM-exception / Unicode-3.0,
`confidence-threshold=0.8`; first-party `publish=false` crates exempted via `[licenses.private]
ignore=true`. Bans: `multiple-versions="warn"` (not deny), `wildcards="allow"`. Sources:
unknown-registry/unknown-git both `deny`, only crates.io allowed.

`cargo-deny` target (`Makefile:181-186`) runs deny against **5** manifests sharing the one config:
root, fegen-rust, rust_cst_fixture, rust_parser_fixture, rust_poc_cst. Each standalone crate needs its
own invocation because it has its own Cargo.lock. **`rust_cst_fegen` is missing from this list** (it
is the orphan — see §1). `fltk-cst-spike` is covered transitively via the root manifest.

Gaps already noted: deny runs **only** in the local pre-commit lane, never in CI (§2 gap 1). And
`multiple-versions="warn"` means duplicate-version bloat is tolerated silently.

## 5. How the test fixtures are built (`build-test-fixtures`)

`build-test-fixtures` (`Makefile:94`) is the aggregate, wired as a prerequisite of `test`
(`Makefile:96`) so `make test`/`make check` always rebuild native ext fixtures before pytest — the
comment calls out "no stale-SO risk" (`Makefile:91-93`). It chains 5 `maturin develop` builds:
- `build-native` (`Makefile:193-194`): `uv run --group dev maturin develop` → `fltk._native`.
- `build-test-user-ext` (`198-199`): `cd tests/rust_cst_fixture && maturin develop` → `phase4_roundtrip_cst`.
- `build-fegen-rust-cst` (`205-206`): `cd crates/fegen-rust && maturin develop` → `fegen_rust_cst`.
- `build-rust-parser-fixture` (`210-211`): `cd tests/rust_parser_fixture && maturin develop --features extension-module` → `rust_parser_fixture`.
- `build-poc-cst` (`215-216`): `cd tests/rust_poc_cst && maturin develop` → `poc_cst`.

Each uses `maturin develop` (debug, editable). These produce importable Python modules consumed by
the pytest parity/Tier-2 suites. The standalone fegen-rust crate is thus built the same way a real
downstream maturin consumer would build their grammar crate — it doubles as a build-test fixture and
as the "canonical" dogfood artifact.

`gencode` (`Makefile:247-298`) is the regenerate-everything target: it regenerates all Python CST/
parser/protocol files for 4 grammars, the Rust `src/lib.rs` (via `gen-rust-lib --no-cst
--register-span-types --unknown-span-static`), and Rust cst/parser for poc_cst, rust_cst_fixture,
fegen-rust (cst+parser+pyi stub), rust_parser_fixture (cst+parser), a collision fixture, and `cp`s
poc cst.rs into the spike. Then it normalizes formatting via the ruff check→format→check dance.
`git diff --stat` after `gencode` is the documented cheat-detection for hand-patched generated code
(`Makefile:244-246`). Note: `gencode` does **not** touch `tests/rust_cst_fegen/` — reinforcing that
it is dead (its 645KB cst.rs can never be regression-checked against the generator).

## 6. Bazel packaging for downstream consumers — COMPLETE or STUBBED?

This is a **real, working, end-to-end** Bazel path, not a stub. Evidence:

**MODULE.bazel** (`MODULE.bazel`): module `fltk`; deps `rules_python 1.5.0`, `rules_rust 0.70.0`;
python toolchain 3.10; a pip hub `pypi` from `requirements_lock.txt`; and a crate_universe hub
`fltk_crates` seeded `from_cargo` off the root Cargo.toml + cst-core + parser-core manifests
(`MODULE.bazel:33-50`). So `rules_rust` IS now wired — meaning the headline `bazel-rules-rust` TODO is
substantially DONE for the cdylib path (see §7).

**BUILD.bazel** provides:
- `genparser` py_binary (the codegen CLI, public) and `fltk` py_library.
- `:native` `rust_shared_library` (crate_features `["extension-module","python"]`, deps cst-core +
  `@fltk_crates//:pyo3`) → `:native_so` genrule renames `libfltk_native.so` → `fltk/_native.abi3.so`
  → `:native_py` py_library puts it on the import path (`BUILD.bazel:32-75`). This is the Bazel
  equivalent of `maturin develop` for `fltk._native`, hand-written.
- Smoke targets: `bootstrap_rust_srcs` (generate_rust_parser) + `bootstrap_native` (fltk_pyo3_cdylib
  with `lib_rs` omitted) — exercises the no-lib_rs branch end-to-end in FLTK's own repo,
  "independent of Clockwork" (`BUILD.bazel:108-126`). (Reminder: nothing runs `bazel build` in CI, so
  these smokes only fire when someone runs Bazel locally — §2 gap 3.)

**rust.bzl** — three pieces:
1. `generate_rust_lib` rule (`rust.bzl:58-98`): runs `genparser gen-rust-lib` → `<name>/lib.rs`.
2. `generate_rust_parser` rule (`rust.bzl:151-196`): two `ctx.actions.run` invocations → `<name>/cst.rs`
   and `<name>/parser.rs` with fixed basenames (load-bearing for bare `mod cst;`/`mod parser;`).
3. `fltk_pyo3_cdylib` macro (`rust.bzl:200-391`) — the real downstream entry point. Four steps:
   - **`_assemble_crate` genrule** (`rust.bzl:316-335`): `printf '#![recursion_limit="N"]'` then `cat`
     the consumer/generated lib.rs into a flat gendir, then `cp` every rs_srcs file by basename,
     with `test -f cst.rs`/`parser.rs` guards. The macro **owns** the recursion_limit line (default
     512, sized for "Clockwork's DFL expression chain"; symptom of too-low = E0275 "overflow
     evaluating Shared<X>: Send", `rust.bzl:274-278`). When `lib_rs=None` it auto-generates lib.rs from
     `name` via `generate_rust_lib` (`rust.bzl:288-293`).
   - **`rust_shared_library`** cdylib (`rust.bzl:338-364`): `crate_features = ["extension-module",
     "python"] + crate_features`. Critically, "python" is set **explicitly** because Bazel
     crate_features do NOT forward through Cargo's `extension-module = ["python", ...]` definition
     (`rust.bzl:346-352`) — without it the `#[cfg(feature="python")]` register_classes symbols compile
     out and linking fails. Deps use `Label("//crates/...")` so cross-repo resolution targets the
     FLTK module, not the caller's repo (`rust.bzl:353-362`) — a genuine Bzlmod-correctness subtlety.
   - **abi3 rename** genrule (`rust.bzl:371-376`): `lib<name>.so` → `<name>.abi3.so`.
   - **py_library wrapper** (`rust.bzl:385-391`): puts `<name>.abi3.so` on the path AND carries
     `@fltk//:native_py` as a dep so `import fltk._native` resolves in the sandbox (the generated
     cst.rs resolves the canonical Span via fltk._native at runtime; "invariant #1").

**Real downstream confirmation (Clockwork):** `clockwork/dsl/BUILD.bazel:5-6` loads
`generate_parser`, `fltk_pyo3_cdylib`, `generate_rust_parser` from `@fltk//`. It calls
`generate_rust_parser(name="clockwork_rs_srcs", src="clockwork.fltkg")` then
`fltk_pyo3_cdylib(name="clockwork_native", rs_srcs=":clockwork_rs_srcs")` with lib_rs omitted (the
auto-gen branch), and a `py_test` (`clockwork_rust_roundtrip_test`) depends on `:clockwork_native`.
So the macro is exercised by an actual external consumer, using the exact code path the FLTK
smoke target covers. **Verdict: the Bazel path is complete and used, not stubbed.**

### Caveat — the consumption is not yet a stable release pin
Clockwork's `MODULE.bazel:34-39` consumes FLTK via a **`local_path_override` to
`/home/rnortman/src/fltk`**, explicitly marked TEMPORARY with `TODO(fltk-pin-finalize)`: "MUST be
reverted to the real git pin (bumped to the reviewed FLTK HEAD) before merging." So the downstream
build has never been validated against a committed/published FLTK ref — only against a live local
checkout. There is no PyPI/crates.io release; consumption is **source-only via Bazel git/path pin**.

## 8. How a downstream consumer actually consumes the Rust backend today

Two supported paths, but with very different maturity:

- **maturin (Python/PyPI-style):** The consumer runs `genparser gen-rust-cst` / `gen-rust-parser`
  (and optionally `gen-rust-lib`) to emit cst.rs / parser.rs / lib.rs, writes a Cargo.toml depending
  on path/git `fltk-cst-core` (default-features=false) + `fltk-parser-core`, and builds with
  `maturin develop`/`build`. This is exactly what every in-tree fixture and `crates/fegen-rust` does
  (`Makefile:198-216`). The generated cst.rs depends on `fltk._native` only at runtime for the
  UnknownSpan sentinel (`genparser.py:301-308`). **But FLTK is not published to PyPI or crates.io**
  (root pkg `publish=false`, `Cargo.toml:8-12`; no release/publish job exists), so a maturin consumer
  must pin FLTK's crates by git/path, not by a registry version.
- **Bazel:** `load("@fltk//:rust.bzl", ...)`, call `generate_rust_parser` + `fltk_pyo3_cdylib`. This
  is the path Clockwork actually uses (`clockwork/dsl/BUILD.bazel:68-80`). It works end-to-end and is
  smoke-tested in FLTK's own BUILD.bazel. **But** the only real consumer pins FLTK via a TEMPORARY
  `local_path_override`, never a committed git/version pin (`clockwork/MODULE.bazel:34-39`), and the
  Bazel surface has **no CI coverage in FLTK** (§2 gap 3), and the abi3/no-libpython property is
  still unverified (`verify-pyo3-ext-module`).

**Drop-in-replacement posture (per CLAUDE.md):** the consumer wiring is import-path-level only
(generate cst.rs/parser.rs, point a build at fltk-cst-core/parser-core/native), consistent with the
"update imports, don't rewrite annotations" goal — the build mechanics don't force annotation churn.
The generated `.pyi` stub path (`gen-rust-cst --protocol-module --pyi-output`, `genparser.py:265-362`)
lets pyright verify the compiled ext satisfies `CstModule` without a cast, which supports the
near-drop-in claim on the typing side.

## 9. Bottom-line build/packaging/CI assessment (adversarial)

Strengths (genuinely production-grade engineering):
- The check-target anti-drift structure is rigorous and well-documented (`Makefile:7-76`).
- The python-on/off feature matrix is fully compiled+tested+clippy'd, and `check-no-pyo3` mechanically
  proves the pyo3-absence invariant with a positive control (`Makefile:155-175`).
- The Bazel `fltk_pyo3_cdylib` macro is complete, handles real subtleties (feature non-forwarding,
  cross-repo Label resolution, abi3 rename, recursion_limit, runtime fltk._native dep), and is driven
  by an actual external consumer.

Weaknesses / production-readiness risks:
1. **cargo-deny is CI-invisible** — supply-chain enforcement hinges on a local hook nobody in CI runs.
2. **Bazel surface has zero CI** — the downstream-facing path is only exercised by hand / by Clockwork.
3. **`verify-pyo3-ext-module` is unresolved** — it is unproven that the Bazel cdylib is a true
   no-libpython abi3 extension; a wheel/import smoke test exists nowhere.
4. **Single-platform single-Python CI** vs a 3.10–3.12 + PyPy advertised support matrix.
5. **No release/publish path** — `publish=false`, no PyPI/crates.io; the only real consumer pins via a
   TEMPORARY `local_path_override` (`fltk-pin-finalize`), so no consumption against a committed ref has
   ever been validated.
6. **Hand-maintained per-crate fan-out** (clippy/test/deny each list the standalone crates by hand) is
   a drift surface, and it has already drifted: **`tests/rust_cst_fegen/` is orphaned dead code**
   (git-tracked, 645KB stale generated cst.rs, in no Makefile target, not deny-scanned, not regenerated
   by `gencode`).
7. **`gencode`/regeneration is not in `make check`** — drift between committed generated code and the
   generator is only caught if a human runs `gencode` + `git diff`; CI does not regenerate-and-diff.

# Test review — cst-generated-header

Commit reviewed: 9372b2f

## Coverage

### New code paths

1. `_escape_source_name(s)` — replaces `\` and `` ` `` in source names.
2. `RustCstGenerator.__init__` — new `source_name: str | None = None` parameter stored as `self._source_name`.
3. `RustCstGenerator._gen_header()` — emits `//! Generated by fltk gen-rust-cst from \`{escaped}\`. Do not edit.\n` when `source_name` is set; emits the no-`from` form otherwise.
4. `genparser.gen_rust_cst` — passes `source_name=str(grammar_file)` to `RustCstGenerator`.

## Findings

### test-1

**File:line** `tests/test_gsm2tree_rs.py:222-228` (`TestPreamble.test_preamble_at_start`, `poc_source` fixture)

**What is wrong** The `poc_source` fixture constructs `RustCstGenerator(_make_poc_grammar())` with no `source_name` argument. The test then asserts:

```python
assert poc_source.startswith("//! Generated by fltk gen-rust-cst")
assert ". Do not edit." in poc_source
```

Both branches of `_gen_header` emit the `". Do not edit."` substring and both start with `"//! Generated by fltk gen-rust-cst"`, so these two assertions pass identically whether `source_name` is `None` or set. The `source_name=None` path (no-from clause) is therefore the only path exercised here, and neither assertion distinguishes between the two branches.

No test constructs `RustCstGenerator` with a `source_name` argument and checks that the `from \`...\`` clause appears — i.e., the with-source-name branch of `_gen_header` is untested at the unit level in `test_gsm2tree_rs.py`.

**Consequence** A regression that dropped the `from \`...\`` clause from the with-source-name branch, or wrote the wrong source name into the header, would not be caught by the unit tests. The CLI-level test in `test_genparser.py:64` does cover this end-to-end, but only for the CLI entry point; direct construction via `RustCstGenerator(grammar, source_name=...)` is unverified.

**Fix** Add a unit test in `tests/test_gsm2tree_rs.py` that constructs `RustCstGenerator(_make_minimal_grammar(), source_name="path/to/grammar.fltkg")`, calls `.generate()`, and asserts that the output starts with `"//! Generated by fltk gen-rust-cst from \`path/to/grammar.fltkg\`. Do not edit.\n"`. Optionally also assert the no-source-name path emits `"//! Generated by fltk gen-rust-cst. Do not edit.\n"` precisely (the current `startswith` check does not pin the absence of the `from` clause).

### test-2

**File:line** `tests/test_gsm2tree_rs.py:222-228` (`TestPreamble.test_preamble_at_start`)

**What is wrong** `_escape_source_name` has two distinct escaping rules: backslash → `/`, backtick → `'`. Neither is exercised by any test. The only test that calls `_gen_header` with a non-None source name is the CLI test (`test_genparser.py`), which passes a real filesystem path — on Linux this never contains a backslash or backtick, so neither escape branch executes under CI.

**Consequence** A Windows path `foo\bar.fltkg` would be emitted unescaped in the Rust doc-comment if the backslash replace were accidentally removed; likewise a grammar file path containing a backtick (unusual but valid on Unix) would break the comment syntax without detection.

**Fix** Add two small unit tests (or one parameterised test) for `_escape_source_name` directly, or add parametrised invocations of `RustCstGenerator(grammar, source_name=...)` with a backslash-containing name and a backtick-containing name, asserting the escaped forms appear in the header. Because `_escape_source_name` is a module-level function, it can be imported and tested in one line each.

### test-3

**File:line** `tests/test_gsm2tree_rs.py:222-228`; `tests/test_gsm2tree_rs.py:665` (`test_preamble_in_fegen_source`)

**What is wrong** `test_preamble_in_fegen_source` (line 665) checks many preamble properties for the fegen grammar source but does not check the generated-file header at all. The `fegen_source` fixture also uses `RustCstGenerator(grammar)` with no `source_name`, so there is no coverage of the with-source-name form from the multi-rule fegen path.

**Consequence** Minor: the header omission from `test_preamble_in_fegen_source` means a regression that silently dropped the header from multi-rule grammars would slip past this test. This is mitigated by `test_preamble_at_start` which uses `poc_source`, but it does mean the fegen-grammar-specific preamble test is inconsistent with the others.

**Fix** Add `assert fegen_source.startswith("//! Generated by fltk gen-rust-cst")` to `test_preamble_in_fegen_source`, consistent with how the other preamble tests treat this invariant.

## Quality observations (not findings)

- `test_genparser.py:64` (`test_gen_rust_cst_command_emits_source`) is a genuine, meaningful end-to-end test for the with-source-name path. The assertion `src.startswith("//! Generated by fltk gen-rust-cst from \`")` is concrete and non-vacuous; it pins the presence of the backtick-quoted source name. This is the strongest test of the new feature.
- `test_source_name_in_header` in `test_gsm2parser_rs.py:539` and `test_source_name_none_omits_from_clause` at line 795 provide a useful analogy for what a corresponding `RustCstGenerator` unit test should look like.
- The `". Do not edit."` substring assertion in `test_preamble_at_start` is non-vacuous for what it checks (the phrase appears), but because the same substring appears in both code branches, it provides weaker signal than a startswith check of the full expected line.

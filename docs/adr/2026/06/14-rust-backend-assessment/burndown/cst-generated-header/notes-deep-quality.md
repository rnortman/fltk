# Quality Review — cst-generated-header

Commit reviewed: 9372b2f

## quality-1

**File:** `fltk/fegen/gsm2tree_rs.py:145-151`

The new `_escape_source_name` function is a near-duplicate of the pre-existing
`_rust_str_lit` in `fltk/fegen/gsm2parser_rs.py:31-44`. Both escape a string
for use inside a Rust `//!` doc comment whose payload is backtick-delimited.
The escaping strategies differ: `_escape_source_name` converts backslash to
forward-slash and backtick to single-quote; `_rust_str_lit` applies full Rust
string-literal escaping (double-quote, control characters, DEL). Neither is
obviously wrong for the doc-comment context, but their existence as two
separate functions encoding two different policies for the same conceptual
operation — "make a source path safe to embed in a Rust doc comment" — means
future maintainers must track which one to use, and the policies can diverge
further over time.

**Consequence:** The two generators are already flagged in the ADR
(`emission-ir-decision`) as having duplicated infrastructure. Adding a
second escape function that parallels `_rust_str_lit` extends that
duplication into a third copy of the escape concern (the parser generator
uses `_rust_str_lit` for its header too, at line 248-249). If a third
generator is added, the pattern will propagate again. The backtick-replacement
policy in `_escape_source_name` is also subtly weaker than `_rust_str_lit`:
a source path containing a double-quote would be embedded verbatim, which is
not safe in a Rust string literal context (though `//!` comments are not
string literals, so in practice this is harmless today — but the policy
difference still constitutes a latent trap).

**Fix:** Move the source-name escape logic to a single shared helper (could
live in a small `shared_rs.py` or at the top of `gsm2tree_rs.py` and be
imported by `gsm2parser_rs.py`). The helper should have an explicit policy
comment: "for doc-comment embedding, not for Rust string literals." The
`_rust_str_lit` function remains for Rust string-literal contexts (regex
patterns, string arguments in emitted code) and is clearly distinct from the
doc-comment helper. This also prepares the codebase for the `emission-ir-decision`
refactor by pulling shared state into a shared location rather than inlining it
into both generators.

## quality-2

**File:** `fltk/fegen/gsm2tree_rs.py:473-477` vs `fltk/fegen/gsm2parser_rs.py:245-251`

The two `_gen_header` methods are structurally identical (both accept an
optional source name, escape it, and emit a `//! Generated by fltk
gen-rust-{cst,parser} from \`...\`. Do not edit.` line) but are defined
separately in each class, using different escape functions (see quality-1).
The only intentional differences are: (a) the tool name in the string
(`gen-rust-cst` vs `gen-rust-parser`) and (b) the parser header appends
additional stack-depth advisory lines after the first line.

**Consequence:** Same duplication-propagation dynamic as quality-1. When
the header text needs to change (e.g., the tool is renamed, a copyright
year is added, the `gen-rust-cst` subcommand is renamed), the change must
be made in two places without any mechanical enforcement that they stay in
sync. This has already been flagged as technical debt in the ADR; the
new commit adds more surface area to that debt rather than reducing it.

**Fix:** Extract a shared `_format_generated_header(tool_name: str,
source_name: str | None) -> str` helper (same module-or-shared-module
recommendation as quality-1) that handles the `None`/non-`None` branching
and escaping once. Each generator's `_gen_header` calls it with its own
tool name, then the parser generator appends its extra advisory lines. This
makes the common structure explicit and the single place of truth for the
"generated by" text format.

# Deep correctness review — cst-generated-header

Commit reviewed: 9372b2f (base 205c36b5e63a6ea851cb560ec0357aab475cbc27).
Scope: add a `//! Generated by fltk gen-rust-cst from \`<src>\`. Do not edit.` header to
Rust CST generator output, mirroring the parser generator. Diff = `gsm2tree_rs.py`,
`genparser.py`, two test files, six regenerated fixtures.

## Verdict

No correctness findings.

## What was traced

- Control flow: `_gen_header()` chooses the `from \`<name>\`` branch when `_source_name is
  not None`, else the plain branch. `generate()` prepends the header as `parts[0]` before
  `_preamble()`. `parts` is joined with `"\n"`; the header string ends in `\n`, so the
  emitted file is `//! ...\n` + `\n` + `use ...` — a valid inner-doc-comment line followed
  by a blank line and the first item. Confirmed against committed fixtures.
- Data flow / wiring: `genparser.gen_rust_cst` passes `source_name=str(grammar_file)`,
  matching the parser generator's `gen_rust_parser` call site. No production path
  constructs `RustCstGenerator` for emission without a source name.
- Inner CST generator: `gsm2parser_rs.py:80` constructs `RustCstGenerator(grammar)` with no
  `source_name`. Verified `self._cst` is used only for name/type lookups
  (`class_name_for_rule_node`, `child_enum_name`, `_label_type_info`, `.grammar`) and never
  via `self._cst.generate()`, so the source-name-less inner instance never emits a header
  into `parser.rs`. No wrong/duplicate header leaks.
- Fixture correctness: each of the six regenerated headers names the exact grammar path the
  Makefile `gencode` target passes for that output (fegen.fltkg → crates/fegen-rust;
  poc_grammar → rust_poc_cst and, via `cp`, fltk-cst-spike; phase4_roundtrip →
  rust_cst_fixture; rust_parser_fixture → rust_parser_fixture/cst.rs; collision_fixture →
  collision_cst.rs). All five referenced grammar files exist. No copy-paste path mismatch.
- Escaping: `_escape_source_name` maps `\`→`/` and `` ` ``→`'`, keeping the backtick span
  well-formed for Windows-style paths. Differs from the parser's `_rust_str_lit` (which
  escapes control chars) but is adequate for the actual input domain (filesystem paths).
- Tests: ran `tests/test_gsm2tree_rs.py` + `fltk/fegen/test_genparser.py` — 245 passed.

## Non-finding (latent, out of scope)

`_escape_source_name` does not neutralize newline/control characters; a `source_name`
containing a literal newline would break the single-line `//!` comment. Not reachable from
the only production caller (`str(grammar_file)`); the shipped parser header has a
symmetric practical limitation. Noted, not actionable for this diff.

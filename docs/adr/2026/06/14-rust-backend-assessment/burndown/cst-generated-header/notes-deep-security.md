# Deep security review — cst-generated-header

Commit reviewed: 9372b2f (base 205c36b5e63a6ea851cb560ec0357aab475cbc27)

Scope: adds a `//! Generated by fltk gen-rust-cst from \`{path}\`. Do not edit.`
header to generated Rust CST output; threads `source_name` (= `str(grammar_file)`)
through `RustCstGenerator`; adds `_escape_source_name`; regenerates fixtures.

## security-1 — `_escape_source_name` does not escape newlines (comment break-out) — informational, not a finding

- File: `fltk/fegen/gsm2tree_rs.py:145-151` (`_escape_source_name`), interpolated at
  `gsm2tree_rs.py:474-475` into the `//!` line doc-comment built at the same site.
- Issue: the source name is placed inside a Rust *line* doc-comment (`//!`), which is
  terminated by a newline. `_escape_source_name` rewrites `\` -> `/` and `` ` `` -> `'`
  but leaves `\n` / `\r` untouched. A source name containing a newline would terminate
  the comment early; text after the newline would be emitted as Rust source rather than
  comment content.
- Trust boundary / data flow: `source_name` originates solely from
  `genparser.py:336` as `str(grammar_file)`, where `grammar_file` is the `typer.Argument`
  path passed on the command line by the developer running `gen-rust-cst`
  (`genparser.py:267`). Same flow for the parser generator at `genparser.py:387`. No
  caller feeds this from a network request, uploaded file, or other untrusted source;
  no in-tree or design path runs the generator as a service over externally supplied
  paths. Input and the build live in the same trust domain.
- Consequence: malformed/compile-broken generated Rust that the same developer then
  compiles and inspects. No privilege boundary is crossed — anyone who can set the
  grammar-file path already controls the build and can run arbitrary code via it. A
  filesystem path with a literal newline is also unusual to begin with. No realistic
  attacker/asset; this is a robustness nit, not a code-injection vulnerability.
- Suggested fix (defense-in-depth, optional): also strip/replace `\n` and `\r` in
  `_escape_source_name` (e.g. map to a space), so the comment line is structurally
  guaranteed to stay single-line regardless of input. Mirrors the intent already stated
  in the docstring ("stays well-formed").

## Other vectors checked — clean

- Secrets: none introduced; header text is a fixed string plus a path. No creds/keys.
- Injection (other): the backtick escape already prevents breaking the inline code span;
  the parser-side analogue uses `_rust_str_lit` for a string literal context, which is
  the right encoder for that context.
- Auth/authz, SSRF, deserialization, path traversal, timing, redirects, CSRF: N/A — this
  is offline codegen with no network, no auth surface, no untrusted deserialization, and
  no filesystem operations driven by the header value.
- Regenerated fixtures (`crates/*/src/cst.rs`, `tests/**/src/*.rs`): only add the header
  comment line; no secrets, no behavioral code change.

## Summary

No exploitable security vulnerability. One informational defense-in-depth note
(newline not escaped) against an input that is developer-controlled and in the same
trust domain as the build itself.

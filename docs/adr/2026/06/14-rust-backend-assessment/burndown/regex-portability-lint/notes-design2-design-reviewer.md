# Design review (round 2): `regex-portability-lint` — grammar-based validator

Re-review of the FLTK-grammar-based regex validator design (directive-driven approach).
Base commit `f1e1fef`. Per instructions, the grammar-based approach is taken as given (user
directive is authoritative); this review evaluates the design on its own merits:
feasibility, bootstrapping, fail-closed completeness, trivia handling, test strategy,
groundedness, and requirements coverage.

## Verification summary (claims checked against source)

Nearly every load-bearing factual claim in the design checks out against the code at
`f1e1fef`. Verified accurate:

- Regex execution sites and engines: `terminalsrc.py:177-181` (`re.compile(...).match(...,
  pos=)`) and `crates/fltk-parser-core/src/terminalsrc.rs` + `Cargo.toml` (regex-automata).
- `gsm.Regex.value` is the verbatim raw text between `/.../` (`fltk2gsm.py:168-170`
  `visit_regex` → `_span_text`); escapes like `\/` and `\\` survive as literal two-char
  sequences. The §7 portable test case `([^\/\n\\]|\\.)+` is consistent with this.
- Wiring site: `_gen_consume_term`'s `gsm.Regex` branch (`gsm2parser_rs.py:757-761`) calls
  `_regex_idx`; `_regex_idx` is *also* called for the internal trivia `\s+` at
  `_gen_separator_code` (`gsm2parser_rs.py:577-578`). The design's rationale for checking at
  the term site instead of inside `_regex_idx` is therefore grounded and correct.
- Error path: `gen-rust-parser` wraps `generate()` in `except (ValueError, RuntimeError,
  NotImplementedError)` → `Exit(1)` (`genparser.py:389-391`). A `ValueError` raised in
  `_gen_consume_term` (called during `generate()`) propagates cleanly. Confirmed.
- Committed-pure-Python-parser precedent: `unparsefmt_parser.py` imports only
  `fltk.fegen.pyrt.*` + sibling CST (verified head of file). No Rust at import time.
- `genparser generate` emits exactly `{base}_cst.py`, `{base}_cst_protocol.py`,
  `{base}_parser.py`, `{base}_trivia_parser.py` (`genparser.py:120-252`). The four committed
  artifacts in §5.1/§8 are correct.
- Block-comment lookahead divergence: `fltk.fltkg:76` and `bootstrap.fltkg:21` use
  `/[^*]*(?:\*(?!\/)[^*]*)*/` (lookahead), while `fegen.fltkg:21` uses the lookahead-free
  `(?:[^*]|\*+[^\/\*])*`. The §7 exclusion of `fltk.fltkg`/`bootstrap.fltkg` from the
  completeness test is justified and the distinction is real.
- Parity infra: `tests/parser_parity.py` has `run_parity_corpus_entry`, `assert_cst_equal`,
  `assert_error_equiv`; `tests/test_rust_parser_parity_fixture.py` has `_CORPUS`; fixture
  regexes (`[0-9]+`, `[a-z]+`, `[À-ÿ]+`, `[!@#$]+`) confirmed in `rust_parser_fixture.fltkg`.
- Trivia-injection mechanism (the design's biggest feasibility risk): **verified in the
  actual generated parser.** `fltk_parser.py:117` emits a leading `apply__parse__trivia`
  *because* `grammar := , rule+` has a WS_ALLOWED initial separator; the `rule+` loop
  (`parse_grammar__alt0__item0`, lines 134-138) has **no** inter-iteration trivia; and the
  `.` (NO_WS) alternatives (e.g. `parse_items__...__alts__alt0` for the `.` literal, line
  463-476) emit **no** trivia call. The design's §4.3 hard authoring constraint ("no leading
  separator, only `.` separators throughout") is exactly what makes the injected-but-unused
  `_trivia` rule harmless. This is the design's most important correctness claim and it holds.

The design is unusually well-grounded. The findings below are the residual gaps.

## design-1 — Validator runs the regex grammar's own leaf matchers under Python `re`; the design never verifies these leaf regexes are themselves in `re`'s accepted syntax, and one realistic authoring choice is itself non-portable in a way that bites only on the validator's own parser

**Section:** §3 item 4 ("The validator's own terminals are safe"); §4.1/§4.3.

**What's claimed:** "`regex_subset.fltkg`'s leaf matchers ... are themselves FLTK regexes
that run on Python `re`. These are simple, fixed, hand-audited classes (`[a-z0-9]`, `\\.`,
etc.) ... so the regex-engine-divergence problem does not recurse into the validator."

**What's underspecified / risky:** The validator's own grammar will, at gencode time, be
compiled into `regex_subset_parser.py` by the *same Rust-target-agnostic pipeline*? No — it
is generated by `genparser generate` (Python backend only, §5.1), so its leaf regexes run on
`re` only and never on regex-automata. That part is fine. But the design asserts the leaf
matchers are "safe" without stating the one property that actually matters: **the leaf
regexes must be authorable within the `.fltkg` `raw_string` body grammar**
(`fegen.fltkg:17`: `([^\/\n\\]|\\.)+`). That body forbids a literal `/` (must be `\/`),
forbids a literal newline, and treats `\\` specially. A naive leaf class such as
`/[\/]/` or a class containing a bare `/` cannot be written as-is; it must be escaped. The
design's §4 never calls this out, and §7's escape-handling tests (`[\[:alpha:]]`, `\\p`)
test the *subject* language's escapes, not the *validator grammar's own* `/`-escaping. An
author following this design can write a leaf regex that the `.fltkg` parser silently
mis-tokenizes (the `/` ends the regex terminal early), producing a subtly wrong validator
that still "compiles."

**Consequence:** A leaf-matcher authoring slip is not caught by any test in §7 except
indirectly (a portable in-tree pattern starts failing). Because the regex-subset grammar is
the security boundary, a mis-tokenized leaf rule can silently over-admit (accept a divergent
construct) or under-admit, and the failure mode is a confusing generation error far from its
cause. The design should add an explicit authoring note that every `/` inside the regex-subset
grammar's own regex terminals must be written `\/`, and should add a positive-control test
asserting the regex-subset *grammar itself* round-trips through `genparser generate` and
re-parses the in-tree corpus — i.e., pin that the committed `regex_subset_parser.py` actually
came from a clean `regex_subset.fltkg` (the §6 "drifting from its grammar" bullet names this
risk but defers it entirely to the regen step rather than a committed test).

**Suggested fix:** Add to §4.3 the `\/`-escaping constraint for the validator's own
terminals, parallel to the existing NO_WS-separator constraint. Keep the §7 completeness test
but state it also guards leaf-matcher correctness.

## design-2 — "Stall offset" error reporting is asserted but the generated parser exposes no clean "how far did the start rule get" value; `result.pos` on failure is `None`, not a partial offset

**Section:** §5.2 (`check_regex_portable`, "offset where the subset parser stalled");
§2 ("reject with a clear, located error naming the pattern and the offset where parsing
stalled"); §6 ("Generation error-message quality").

**What's claimed:** "checks the consumed position against `len(pattern)` ... The stall offset
comes from the parser's `error_tracker` / returned `pos`, formatted for the message." And
§5.2's docstring: "Portable iff the parser consumes the entire pattern (result.pos ==
len(pattern))."

**The problem:** These two mechanisms are conflated and only one is real. A generated FLTK
start rule returns `ApplyResult | None`. There are two distinct failure shapes, and the
design treats them as one:

1. **Hard fail** — `apply__parse_<start>(0)` returns `None` (start rule did not match at all,
   e.g. empty pattern or first char unrecognized). There is no `result.pos` to read; the
   design's own §6 bullet "A pattern the subset parser cannot start at all → offset 0" admits
   this, but §5.2's primary "result.pos == len" idiom does not apply to it.
2. **Short parse** — start rule returns `Some` with `pos < len(pattern)` (matched a prefix,
   stopped before end). Here `result.pos` is the prefix length, which is a *usable* stall
   offset.

The "offset where parsing stalled" the design promises for the *located error* is the
furthest-progress position, which lives in `error_tracker.longest_parse_len` (cf.
`gsm2parser_rs.py:391-394` `error_position`, and the Python `format_error_message` path in
`genparser.py:51-56`). But the design says offset comes from "`error_tracker` / returned
`pos`" as if interchangeable — they are not. For a regex like `a(?=x)b`, the start rule may
return `Some(pos=1)` (matched `a`, then alternation/concat dead-ends), while the
furthest-progress error position may be deeper. Reporting `result.pos` vs
`error_tracker.longest_parse_len` gives *different* offsets, and the design never commits to
which, nor verifies the chosen one is meaningful for a recognizer that is designed to
deliberately *not* match the excluded tail.

**Consequence:** The §7 tests assert "returns an issue with a sensible offset" for 13
excluded constructs. Without the design pinning *which* position is reported and confirming
the generated parser actually surfaces it, the implementer can ship a `check_regex_portable`
whose offset is `result.pos` (often pointing at the first character *after* a successfully
matched prefix, which for `[[:alpha:]]` is offset 1 — inside the outer `[`, arguably
unhelpful) and the tests, written to whatever it emits, will pass anyway. The "clear,
located error" requirement (ELI5: "reject them with a clear error message") then degrades to
a technically-present but unhelpful number. This is a real requirement-quality risk, not a
nicety.

**Suggested fix:** §5.2 should commit to a single offset source — recommend
`error_tracker.longest_parse_len` (furthest progress), matching the existing
`format_error_message` machinery the rest of FLTK uses for "where did parsing fail" — and
the design should confirm the generated `regex_subset_parser` exposes it (the Python parser
runtime's `error_tracker` is on the `Parser` instance; check it is populated for a
recognizer with no labels/dispositions, since error tracking keys off rule-application
frames). Drop the "result.pos" wording for the *error* offset; keep `result.pos == len` only
as the accept/reject predicate.

## design-3 — `\b`/`\B` are admitted as portable shorthands, but they are exactly the class of context-sensitive construct that can produce zero-width matches and were flagged as a known under-approximation hole in the codebase; admitting them widens the silent-divergence surface the item exists to close

**Section:** §4.1 ("Admitted shorthands: `\d \D \w \W \s \S \b \B \n \r \t \f \v \0`").

**What's wrong:** `\b`/`\B` are word-boundary assertions whose match depends on the Unicode
word-character table — the *same* `\w` Unicode-DB-version divergence the design itself
quarantines as the "honest limit" for `\d`/`\w`/`\s` (§3, §6, O2). `\b` is defined in terms
of `\w`; if `\w`'s table differs between `re` and regex-automata for non-ASCII, so does `\b`.
The design admits `\d`/`\w`/`\s` *and documents* them as a residual; but it admits `\b`/`\B`
in the same breath with **no** mention that they inherit the identical residual. Worse,
`gsm.py:439-444` (the `validate_no_repeated_nil_items` NOTE, verified) specifically calls out
`\ba*` and `(?=x)` as patterns whose zero-width behavior the static checker cannot see —
i.e. the codebase already treats `\b` as a known soft spot.

**Consequence:** The design's §6 residual list and O2 (the documented permanent boundary)
omit `\b`/`\B`. If a downstream grammar uses `\b` over non-ASCII word characters, it passes
the validator silently and can still diverge cross-backend — the precise outcome
`regex-portability-lint` is chartered to prevent — yet the documentation (§5.4, and the
`document-scope-boundary` hand-off) will not list `\b`/`\B` among the documented-permanent
residuals, because the design never put them there. This is an internal-consistency gap: a
construct is admitted-as-portable in §4.1 but absent from the residual ledger in §6/O2.

**Suggested fix:** Either (a) keep `\b`/`\B` admitted but add them explicitly to the §6/O2
residual list alongside `\d`/`\w`/`\s` (same Unicode-DB caveat), so the documentation hand-off
covers them; or (b) state a decision to admit them as ASCII-portable with the residual noted.
Do not leave them silently admitted with no residual entry.

## design-4 — The completeness/under-admission guard enumerates a hardcoded list of "Rust-target grammars"; this list is itself a drift surface and the design gives no mechanism to keep it in sync with the Makefile

**Section:** §7 ("Whole-tree completeness check"): "the completeness test enumerates the
Rust-target grammars (`fegen.fltkg`, `rust_parser_fixture.fltkg`, `collision_fixture.fltkg`,
`poc_grammar.fltkg`, `phase4_roundtrip.fltkg`)".

**What's wrong on the facts:** `poc_grammar.fltkg` and `phase4_roundtrip.fltkg` are
**CST-only** Rust targets — the Makefile generates `gen-rust-cst` for them but **not**
`gen-rust-parser` (verified in `Makefile` gencode: `tests/rust_poc_cst/src/cst.rs` and
`tests/rust_cst_fixture/src/cst.rs` are CST-only; only `fegen.fltkg`,
`rust_parser_fixture.fltkg`, and `collision_fixture.fltkg` get a generated Rust *parser*).
The portability check fires only in the Rust **parser** generator (§5.3,
`gsm2parser_rs.py`), so it never runs against `poc_grammar`/`phase4_roundtrip` in production.
Including them in the completeness test is harmless (their regexes — `[_a-z][_a-z0-9]*`,
`[0-9]+`, `"[^"]*"` — are portable) but the design's framing that these are the grammars
"that *is* a Rust target" subject to the check is imprecise: two of the five are not parser
targets. More importantly:

**The structural problem:** The list is hand-maintained in a test, duplicating knowledge
that already lives (authoritatively) in the Makefile gencode recipe. The entire
`recommended-actions` corpus repeatedly identifies *exactly this pattern* — hand-maintained
per-target lists that silently fall out of sync — as the project's recurring drift bug
(`gencode-drift-gate`, `remove-dead-duplicate-crate`: "the per-crate fan-out is a drift
surface"). A new downstream-style fixture grammar added to gencode with a non-portable regex
would not be caught by this test until someone remembers to add it to the hardcoded list.

**Consequence:** The completeness guard the design leans on to "pin completeness at scale"
has the same drift weakness as the bugs the assessment is burning down. It will not catch a
future Rust-parser-target grammar that someone forgets to enumerate. Lower severity than
1–3 (the production check at §5.3 is the real gate; this is a test-only completeness aid),
but it is an avoidable instance of the project's named anti-pattern, and the factual
mischaracterization of which fixtures are parser targets should be corrected so the test does
not imply coverage it lacks.

**Suggested fix:** Derive the grammar list from a single source — e.g. glob the grammars the
Makefile feeds to `gen-rust-parser`, or factor the parser-target grammar list into one place
both the Makefile and the test read. At minimum, correct §7 to distinguish CST-only targets
from parser targets, and note the list must be updated when a parser target is added (a
`TODO(slug)` if it stays manual, per CLAUDE.md).

## design-5 — Non-goal "not changing either backend's regex engine" is well-argued, but the requirement's STATUS line explicitly prefers the standardization alternative "if possible"; the design dismisses it via exploration without re-confirming the exploration's key factual claim

**Section:** §9 (Non-goals, second bullet); requirement STATUS
(`recommended-actions-eli5.md` `regex-portability-lint` is silent on this, but
`recommended-actions.md` STATUS per the prompt: "Do this *OR* better if possible: standardize
python and rust on some regex library/standard that has identical feature set and semantics
across languages").

**What's fine:** The design correctly routes to the lint approach and cites
`exploration.md:270-301` for why no off-the-shelf library gives identical Python/Rust
semantics. That exploration section is real and its conclusion ("no current cross-language
regex standard with identical behavior exists as a shipping library") is stated there.

**The gap:** The requirement frames standardization as the *preferred* option ("OR better if
possible"). The design disposes of the preferred option by reference, but the exploration's
own §"Open factual questions" (`exploration.md:320-328`) explicitly flags that it did **not**
determine whether the PyPI `regex` module and regex-automata share a Unicode-DB
source/version — i.e. the exploration left the decisive fact for the "regex module as common
ground" sub-option **unresolved**. The design treats the standardization option as
conclusively closed when the cited source explicitly says one branch of it is unverified.

**Consequence:** This is a groundedness gap on the one place the requirement asked for a
judgment call. It does not change the recommendation (the lint is needed regardless, since
even the larger common subset "is still not perfectly identical" per `exploration.md:294`),
but the design should not present "no shipping library can satisfy" as fully settled when its
own cited exploration lists the deciding fact as an open question. Surface it as a
deliberate, acknowledged deferral (it is arguably already the `differential-property-harness`
/ `document-scope-boundary` territory) rather than a closed finding.

**Suggested fix:** One sentence in §9 acknowledging the `regex`-PyPI-as-common-ground
sub-option remains factually open per `exploration.md:320-328`, and that the lint is the
correct move *irrespective* of how that question resolves (the subset is enforced either
way). This keeps the non-goal honest.

  judgment rather than pre-built.
- Requirements coverage is complete: reject-at-generation (§5.3), clear error (§5.2/§6),
  doc-as-hard-boundary (§5.4 item 1, correctly *not* editing the Accepted ADR), parity-corpus
  expansion (§5.6). The §5.4 decision to leave the Accepted ADR immutable and route permanent-
  boundary prose to `document-scope-boundary` is consistent with CLAUDE.md.
- The §5.6 non-nullable-pattern constraint is correctly grounded in `gsm.py:156-172`
  (`_test_regex_empty`) and `validate_no_repeated_nil_items` (`gsm.py:433-453`); verified.
- Infrastructure-before-features holds: the committed generated parser + wrapper module are
  built and unit-tested (§7) before the generator wiring depends on them.

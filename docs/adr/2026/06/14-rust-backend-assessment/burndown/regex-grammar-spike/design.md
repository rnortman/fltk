# Design: regex-grammar-spike

**Burndown item:** `regex-grammar-spike` (Phase B; supporting the `regex-portability-lint`
work, slug `regex-grammar-spike`).

**Mode:** skip-requirements, implementation spike. The user's spec is authoritative; this
design lays out file layout, codegen wiring, the corpus-test plan, the adversarial-test
suite, and the increment breakdown. It does **not** restate the portability-lint design
(`burndown/regex-portability-lint/design.md`) — it refers to it. Exploration is
`burndown/regex-grammar-spike/exploration.md`. The grammar reference is
`docs/fltk-grammar-reference.md`.

## Scope of this spike (and what it is NOT)

The goal is narrow and stated verbatim: **prove the regex grammar works**. Copy
`regex.fltkg` into the repo, generate a parser + CST from it via the Makefile, build a
**general, reusable tool** that extracts every regex from an arbitrary `.fltkg` grammar
file (path passed as a CLI argument) and runs each through the generated parser + the
accept/reject oracle, run that tool's logic over every regex in the in-tree grammars as a
committed corpus test, and then build an adversarial suite that tries to fool the parser.

**Nothing in this spike is clockwork-specific, and no clockwork path or clockwork data is
committed** (user design-gate directive,
`burndown/regex-grammar-spike/notes-design-user.md`). The same tool extracts and tests
regexes from *any* grammar file. The committed corpus tests run it against in-tree
grammars; the out-of-tree clockwork grammar is exercised ad hoc by a developer who passes
its path on the command line (§3.4) — nothing committed references that path.

This spike is **decoupled from the portability-lint enforcement design**. The lint design
(§5.2–§5.3) wires a `check_regex_portable` wrapper into the Rust generator at the
`gsm.Regex` term site. **This spike does none of that.** It does not touch
`gsm2parser_rs.py`, does not add `regex_portability.py`, does not gate codegen, and does
not change any error path. It only proves that the grammar — as authored — parses the
patterns we expect it to accept and rejects the ones we expect it to reject. The spike is
the empirical evidence that the grammar is fit to become the lint's recognizer; the lint
increment is separate downstream work. (The lint design even names this dependency: it says
the grammar "should be verified programmatically as part of the spike,"
`burndown/regex-grammar-spike/exploration.md:399-407`.)

This boundary matters for naming (below): the lint design calls its grammar
`regex_subset.fltkg`. The spike copies the file the user pointed at, `regex.fltkg`. §1
resolves the name.

## 1. File layout

### 1.1 The grammar file

Copy
`docs/adr/2026/06/14-rust-backend-assessment/burndown/regex-portability-lint/regex.fltkg`
to **`fltk/fegen/regex.fltkg`**, verbatim (no edits in the copy increment — if the grammar
needs a fix, that is a finding the adversarial increment surfaces, applied as its own
change).

Rationale for the location: every committed Python-codegen grammar that ships generated
artifacts lives beside its artifacts — `fltk/fegen/fegen.fltkg`,
`fltk/fegen/bootstrap.fltkg`, `fltk/unparse/toy.fltkg`, `fltk/unparse/unparsefmt.fltkg`
(`Makefile:250-264`). The regex grammar's generated artifacts will live in `fltk/fegen/`
(below), so the grammar belongs there too. This is also exactly where the downstream lint
design places it (`burndown/regex-portability-lint/design.md:181`, §4 heading
`fltk/fegen/regex_subset.fltkg`).

**Name decision: `regex.fltkg` vs `regex_subset.fltkg`.** The user said to copy
`regex.fltkg`; the lint design wants `regex_subset.fltkg`. The spike uses the name
**`regex.fltkg`** (and base name `regex`, CST module `fltk.fegen.regex_cst`) because that is
what the user named and the spike's job is to prove *this* file. The lint design is free to
rename on adoption — a rename of a not-yet-shipped internal grammar file and its generated
`regex_*` modules is cheap (the modules are not yet public API; nothing out-of-tree imports
them) and is the lint increment's decision, not the spike's. The spike should not pre-empt
that naming call. **Open question O1** records the alternative (copy straight to
`regex_subset.fltkg` to save the later rename) for the user.

### 1.2 Generated parser + CST artifacts

Generated by `genparser generate` (the Python pipeline; exploration §2 "generate command",
`burndown/regex-grammar-spike/exploration.md:81-93`). One invocation emits four committed
files into `fltk/fegen/`:

| file | role |
|---|---|
| `fltk/fegen/regex_cst.py` | CST dataclass module |
| `fltk/fegen/regex_cst_protocol.py` | protocol/type-stub module |
| `fltk/fegen/regex_parser.py` | non-trivia parser (the one the tests use) |
| `fltk/fegen/regex_trivia_parser.py` | trivia-preserving parser (unavoidable by-product; harmless, unused) |

These are normalized by `make fix`, per the standard regen → `make fix` → commit flow
(CLAUDE.md "Generated Code and Formatting"). They import only `fltk.fegen.pyrt.*` and the
sibling CST module — no Rust extension — exactly like the committed `unparsefmt_parser.py`
(lint design §3.1).

**What `make check` does and does not enforce.** `make check` enforces only that the
committed generated files are *clean* — they pass lint, format-check, and typecheck — via
`check → check-ci → check-common`, whose steps are `lint format-check typecheck test
cargo-check cargo-clippy cargo-test cargo-test-python-features cargo-test-no-python
cargo-clippy-no-python check-no-pyo3` (`Makefile:39-51`). `make check` does **not**
regenerate code and does **not** diff the committed artifacts against a fresh regen — there
is no drift gate in the check graph. `gencode` is a standalone manual target
(`Makefile:247`) referenced nowhere in `check`; its `git diff --stat` "drift" comment
(`Makefile:245-246`) describes what a *human* sees after manually running `make gencode`,
not an automated CI assertion. So the regen-match is a manual `make gencode` + `git diff`
step the implementer runs before commit. (Adding a real regenerate-and-compare test is the
lint design's deferred "round-trip test" — added scope, not warranted for a spike.)

### 1.3 Language(s): Python only for the spike. Rust out of scope.

**Recommendation: generate Python only.** Rationale, grounded in the codebase:

- The accept/reject oracle the spike needs is "did the parser consume the whole pattern?"
  The Python backend answers this through `fltk.plumbing.parse_text`, which already checks
  `result.pos != len(terminals.terminals)` (`fltk/plumbing.py:319-331`, confirmed). No Rust
  needed to evaluate the grammar's behavior.
- The Python parser is pure Python and importable with no build step beyond the
  already-required `fltk._native` (exploration §4, `:216-218`). A Rust parser for the regex
  grammar would require a whole new Cargo crate under `tests/rust_*_fixture/` and a
  `maturin develop` build wired into `build-test-fixtures` (`Makefile:94`) — significant
  scaffolding (exploration §9, `:413-416`) that proves nothing additional about *the
  grammar*. The grammar's behavior is backend-independent: both backends run the same GSM
  through equivalent generators; what differs between backends is the *regex engine that
  runs each leaf terminal*, not the parse structure of the regex grammar.
- The spike's subject is the **grammar**, not cross-backend parity of the regex grammar's
  own leaf terminals. The leaf terminals here (`[0-9]+`, `[imsU]+`, `[dDwWsS]`, etc.) run on
  Python `re` only — they are the *validator's* terminals and need only be correct under
  Python `re` (lint design §3 item 4). So a Rust build would add cost without adding signal.

**Rust is explicitly out of scope for this spike.** If the downstream lint or a future spike
wants cross-backend confirmation that the regex grammar parses identically on a Rust-built
`regex_parser`, that is a separate item following the `tests/rust_parser_fixture/` pattern
(exploration §3 `:139-157`). Recorded as **Open question O2**.

## 2. Makefile codegen wiring

Add one `generate` invocation to the `gencode` target's Python section
(`Makefile:247-264`), immediately after the `unparsefmt` block, matching the existing shape
exactly:

```makefile
# Python: regex grammar (regex_cst.py, regex_cst_protocol.py, regex_parser.py, regex_trivia_parser.py)
uv run python -m fltk.fegen.genparser generate \
    fltk/fegen/regex.fltkg regex fltk.fegen.regex_cst \
    --output-dir fltk/fegen
```

This slots into the existing `ruff check --fix` → `ruff format` → `ruff check --fix`
normalization tail of `gencode` (`Makefile:289-297`), so the four generated files are
formatted in the same pass as every other generated file. No new target, no new variable —
the spike's grammar is just one more entry in the canonical Python-codegen list, the
smallest possible change that matches the current pattern.

`gencode`'s own contract (`Makefile:241-246`) is that `git diff --stat` after *manually
running it* reveals drift between committed generated files and what the generator produces.
Adding the regex line brings the regex artifacts under that same manual drift-detection step
automatically. This is a manual check (run `make gencode`, inspect `git diff`), **not** a
CI-enforced gate — `gencode` is not in the `make check` graph (§1.2).

**No Rust lines added** (§1.3). The Rust section of `gencode` (`Makefile:265-288`) is
untouched.

## 3. The general extract-and-test tool, and the corpus tests built on it

### 3.1 The general tool and why enumeration must be programmatic

The core capability is **general and reusable**: given an arbitrary `.fltkg` grammar file
path, extract every regex from it and run each through the generated regex parser + the
accept/reject oracle (§3.2). Nothing about it is clockwork-specific — it works on *any*
grammar file (user directive). It is the single source of truth used by both the committed
corpus test (over in-tree grammars) and the developer running it ad hoc against clockwork
(§3.4).

The committed corpus test invokes this tool/logic against the **in-tree** grammars:

1. **`fltk/fegen/fegen.fltkg`** — in-tree, the FLTK meta-grammar.
2. **`fltk/fegen/regex.fltkg`** — the regex grammar itself (self-description: its own leaf
   regexes must be in the portable subset it defines).

The out-of-tree clockwork grammar is **not** a committed corpus and **not** a CI
dependency; it is run ad hoc with its path passed on the command line (§3.4).

**Enumeration must be programmatic, not a hand-transcribed list.** The exploration's §7
hand-lists (`:309-362`) are demonstrably **incomplete and stale**: e.g. it omits clockwork's
`identifier` regex `[_a-zA-Z][_a-zA-Z0-9]*` (`clockwork.fltkg:267`) and its
`nonnegative_integer` regex `[0-9]('?[0-9])*` (`clockwork.fltkg:275`). A hand-list rots the
moment any grammar changes. The tool extracts regexes by walking the GSM, the same field
the generators read.

Enumeration mechanism (exploration §6, `:248-304`, verified): parse a grammar to a
`gsm.Grammar`, then walk every `Rule → Items → Item`, recursing into `Sequence[Items]`
sub-expressions, collecting `item.term.value` for every `term` where
`isinstance(term, gsm.Regex)`. `gsm._for_each_item` already performs the depth-first item
walk including sub-expressions (`gsm.py:291-302`). The tool collects the **distinct** set of
regex bodies per grammar (dedup on the string), since portability is a property of the
pattern, not of where it appears (lint design §6 "Pattern reused across rules").

The tool reads any grammar file via **`parse_grammar_file`** (`fltk.plumbing`, exploration
§4 `:203-205`) — the same entry point for the in-tree corpora and for an ad-hoc clockwork
path. (`gsm.Regex.value` is the raw body with the outer `/.../` already stripped by
`fltk2gsm.visit_regex`, exploration §5–§6, `:222-269` — so the collected strings are exactly
what the regex grammar must parse.)

**Shape of the tool (single source of truth).** The extraction collector
`collect_regexes(grammar)` (§3.1 walk) plus a per-pattern accept/reject classifier (§3.2)
live in one place — a small module importable by the corpus test. A thin CLI entry point
(`python -m ...`, recorded as **O4** for exact packaging) takes a `.fltkg` path argument,
parses it, collects the distinct regexes, runs each through the generated parser, and prints
each pattern with its disposition (and a nonzero exit if any pattern is rejected, so it is
usable as an ad-hoc check). The committed corpus test imports the same collector + classifier
and asserts on the in-tree grammars; it does **not** reimplement the walk. There is exactly
one copy of the enumeration and oracle logic.

### 3.2 The expected-outcome oracle

For each collected pattern, parse it with the generated regex parser and decide accept vs
reject. **Accept** = the start rule matches and consumes the entire pattern:

```python
result = parser.apply__parse_regex(0)
accepted = result is not None and result.pos == len(terminals.terminals)
```

This is precisely `parse_text`'s own success predicate (`plumbing.py:319-331`); the corpus
test may call `parse_text(pr, pattern, "regex")` and read `ParseResult.success` directly
rather than re-implementing it. The start rule is **`regex`** (`regex.fltkg:53`,
`regex := alternation`), confirmed against the grammar.

**The oracle for "should this pattern be accepted or rejected" is the portable subset
definition itself** — i.e. the union of Python `re` and Rust `regex-automata` agreement, as
the `regex.fltkg` header and the lint design §4.1/§4.2 specify. But the spike must not
re-derive that judgement by hand for the corpus; it computes the expected disposition
**empirically** and pins it. Concretely, the corpus test classifies into three buckets and
asserts on each:

- **Bucket A — must be accepted.** A pattern in the portable intersection: `re.compile`
  accepts it on Python **and** `regex_automata` accepts it on Rust. The Python side is
  checked directly with `re.compile`. The Rust side uses **`re.compile` as the authoritative
  "is this even a valid regex" gate and treats Rust-acceptance as a documented assumption**
  for patterns drawn from real working grammars (whose regexes already run on at least one
  backend in production). An optional stronger Rust oracle — a tiny `cargo` helper the test
  shells out to call `regex_automata::meta::Regex::new` — is recorded as **O3**. The spike's
  value is proving the *FLTK regex grammar* accepts the patterns real grammars actually use;
  full Python∩Rust differential evaluation of every pattern is the
  `differential-property-harness` item's job (lint design §9), not the spike's.
- **Bucket B — must be rejected.** Patterns that contain a known non-portable construct.
  None of the real grammars under test is *expected* to contain one (they are working
  grammars); if the parser rejects a corpus pattern, that is a **finding** (either the pattern
  is genuinely
  non-portable — interesting — or the grammar under-admits — a grammar bug). See §3.3.
- **Bucket C — the adversarial set** (§4), which carries hand-authored expected
  dispositions.

For the committed **corpus** specifically, the practical assertion is: **every distinct
regex in the in-tree grammars (`fegen.fltkg`, `regex.fltkg`) must be ACCEPTED by the
generated parser** (parses to end of input). This is the concrete, falsifiable claim that
"the regex grammar works" against real input. When the same tool is run ad hoc against
clockwork (§3.4), the identical accept assertion applies — it is just not committed. Any
corpus pattern that the parser rejects is surfaced as an explicit, named test failure with
the pattern (and, if useful for triage, the furthest-progress offset read from
`parser.error_tracker.longest_parse_len` via the direct `apply__parse_regex(0)` path — this
is *diagnostic only*, never an assertion target; see §4.1 for why `longest_parse_len` is not
sound to assert on), to be triaged as either a grammar gap (fix the grammar) or a genuinely
non-portable pattern in that grammar (document it).

### 3.3 Known risk points to pin explicitly

The exploration flagged several patterns whose acceptance is non-obvious. The **in-tree**
ones (fegen's block-comment regexes) are pinned in the committed corpus test with explaining
comments, so a future grammar edit that breaks one fails loudly. The **clockwork** ones are
documented here as *expected dispositions for the ad-hoc clockwork run* (§3.4) — they are
not committed test pins (no clockwork data is committed), but the developer running the tool
against clockwork should confirm them.

In-tree (committed corpus-test pins):

- **fegen's block-comment regexes.** `fegen.fltkg` itself uses
  `(?:[^*]|\*+[^\/\*])*` (block-comment content) and `\*+\/` (block-comment end), plus
  `[^\n]*`, `[_a-z][_a-z0-9]*`, the `raw_string` body `([^\/\n\\]|\\.)+`, and the literal
  pattern `("([^"\n\\]|\\.)+"|'([^'\n\\]|\\.)+')` (verified by reading `fegen.fltkg:50-55`
  in the grammar reference, and exploration §7 `:309-322`). The `(?:...)` non-capturing
  group, the `\/` escaped slash, and the `\*` escaped star all exercise real corners of
  `regex.fltkg`. **Note:** the *self-escaping* in the grammar reference's reproduction
  (`\/`, `\\`) is the `.fltkg` tokenizer's view; `gsm.Regex.value` holds the *decoded* body
  (one backslash + slash). The collected strings are post-decode, which is exactly what the
  regex grammar parses (lint design §4.3 "The pattern string is the input").

Clockwork (documented expected dispositions for the ad-hoc run, §3.4 — not committed):

- **clockwork's UUID with `\b`.** `[0-9a-fA-F]{8}\b-[0-9a-fA-F]{4}\b-...`
  (`clockwork.fltkg:102`) puts `\b` at top level (outside any class), which `regex.fltkg`
  admits via `escape_body → assertion` (`regex.fltkg:263-274`; exploration §7 `:364-369`).
  Pin accepted.
- **clockwork's ` *` (space-star) and `'?` (apostrophe-question).** The `doc` rule's leading
  ` *` (`clockwork.fltkg:2-3`) is `literal_char(space) . zero_or_more`; the
  `nonnegative_integer` `[0-9]('?[0-9])*` (`clockwork.fltkg:275`) puts an apostrophe inside a
  **capturing** group `('?[0-9])` — a bare `(...)`, parsed by `capturing := %"(" . body . %")"`
  (`regex.fltkg:144`) — whose body has a `'?` optional apostrophe (apostrophe is a
  `literal_char`, not excluded; exploration §7 `:370-374`). Both pinned accepted. (The
  exploration mislabels this group "non-capturing"; it is a capturing group. The grammar
  admits both forms, so the acceptance conclusion is unaffected; the test comment should say
  "capturing group" to be accurate.)
- **clockwork's `[eE]`, `[^\n]*`, `[_a-zA-Z][_a-zA-Z0-9]*`, ` *`, and the string-literal
  pattern.** All ordinary classes/quantifiers; pin accepted.

### 3.4 Exercising the out-of-tree clockwork grammar ad hoc (no committed clockwork data)

`clockwork.fltkg` is an **out-of-tree** grammar that lives only in the developer's own
clockwork checkout (exploration §7 `:343-345`); the spike does not depend on, vendor, or
reference its location. It **must not be a committed artifact or a CI dependency** — CI has
no clockwork checkout, and the user directive is explicit: **do not hard-code a path to
`clockwork.fltkg` in anything committed (including this design doc — it too is committed
under `docs/adr/`), and do not commit clockwork data.** The original snapshot-fixture
approach is therefore **removed**: there is no committed `clockwork_regexes.json`, no
committed clockwork path (literal or otherwise) anywhere in the tree, and no snapshot helper
that bakes the clockwork path into the repo.

Instead, clockwork is exercised exactly like any other grammar, using the **general tool**
(§3.1): a developer who has the clockwork checkout runs the tool with the path passed as a
CLI argument. The general tool reads any `.fltkg` path, extracts its distinct regexes, runs
each through the generated parser, prints each pattern with its disposition, and exits
nonzero if any is rejected. The same tool, the same collector, the same oracle as the
committed in-tree corpus test — only the input path differs.

**Documented command** (run by a developer ad hoc; the path is supplied by the developer at
the command line and is **not** written down anywhere committed — including here):

```
uv run python -m fltk.fegen.regex_corpus <path/to/clockwork.fltkg>
```

The developer substitutes the path to their own clockwork checkout for `<path/to/...>`. The
command is fully grammar-agnostic — the same invocation works for any `.fltkg` file by
swapping the argument. (Exact module/entry-point name is **O4**.) The developer confirms the
§3.3 clockwork expected dispositions (UUID `\b`, ` *`, `'?`) from the printed output. Because
clockwork is run ad hoc, there is **no drift problem to manage** — every run reads live
upstream clockwork directly, so there is nothing to keep fresh and no provenance snapshot to
maintain. (This also retires former open questions about snapshot helper form and freshness;
see §9.)

## 4. Adversarial test suite (the Opus increment)

This is the increment the user assigned to an Opus model. Its purpose: **try to fool the
parser** — find inputs where the grammar accepts something it should reject (over-admission,
the dangerous direction) or rejects something it should accept (over-rejection, the annoying
direction). It is distinct from the corpus tests: the corpus proves the grammar handles
*real* patterns; the adversarial suite proves the grammar handles *hostile* patterns.

### 4.1 Structure and conventions

- **Location:** `tests/test_regex_grammar_adversarial.py` (a new test module under `tests/`,
  the home of the cross-cutting parser tests; exploration §4 `:162-197`).
- **Form:** a table-driven test. Each case is a `(pattern, expected_disposition, rationale)`
  triple, where `expected_disposition ∈ {ACCEPT, REJECT}` and `rationale` is a one-line
  string explaining *why* that is the correct answer, ideally citing which engine(s) accept
  /reject and/or the `regex.fltkg` line that governs it. The rationale string is not just a
  comment — it is asserted to be present (non-empty) so every case is self-documenting, and
  it is printed on failure so a regression is debuggable without re-deriving the intent.
- **Oracle:** same accept predicate as §3.2 (`parse_text(..., "regex").success`). The
  accept/reject boolean is the real signal and is robust. **No `longest_parse_len`
  assertion.** Two facts kill the originally-planned "furthest-progress offset is sane (≥ 0)"
  assertion: (1) `ParseResult` (`fltk/plumbing_types.py:26-32`) exposes only
  `cst/terminals/success/error_message` — it does **not** carry `error_tracker`, so a test
  routing through `parse_text(...).success` *cannot read* `longest_parse_len` at all without
  abandoning that oracle and driving `parser.apply__parse_regex(0)` directly to reach
  `parser.error_tracker`. (2) Even via the direct path, the `≥ 0` lower bound is not
  guaranteed: `ErrorTracker.longest_parse_len` defaults to `-1` and is advanced only by a
  `fail_literal`/`fail_regex` at a position (`errors.py:25-48`). The dominant reject shape in
  this grammar is a *short parse* — the start rule succeeds on a prefix and `parse_text`
  rejects only because `result.pos != len(terminals)` (`plumbing.py:323`) — which records no
  terminal failure, so the tracker can legitimately sit at `-1`. A test asserting `≥ 0` would
  flake. If a future increment wants to pin error-location quality, it must use the direct
  `apply__parse_regex(0)` path and assert only `longest_parse_len <= len(pattern)` (and
  "is an int"), never a positive lower bound. The spike does not assert offsets.
- **Cross-checking the expected disposition — ASYMMETRIC; ACCEPT-only.** The Python `re`
  cross-check (run `re.compile` on the pattern) is sound **only for the ACCEPT direction**: a
  case marked ACCEPT that Python `re` rejects is a mis-spec and is caught. For the REJECT
  direction the cross-check is **effectively useless and must not be relied on**, because the
  suite's REJECT cases are overwhelmingly "Python `re` accepts, Rust diverges/rejects"
  (`[a-z-0]`, `a{`, `{`, `\07` all `re.compile` cleanly on Python — verified). Flagging a
  REJECT mis-spec requires *both* engines to accept, but the spike has no inline Rust oracle
  (O3 deferred), so the Rust half is never evaluated. **The dangerous over-admission
  direction — a REJECT case that is actually portable (both engines accept) but the grammar
  rejects — is therefore unguarded by any automated cross-check.** REJECT-case correctness
  rests entirely on the mandatory `rationale` string + Opus/human judgement, and the rationale
  must state the *Rust-side* basis for the rejection (which engine diverges and how), not just
  the Python side. The cross-check is honest for ACCEPT and explicitly an unverified judgement
  call for REJECT — the design does not pretend otherwise. **If REJECT correctness is judged
  to matter for the spike's value, O3(b) (the tiny `cargo` Rust-acceptance helper) is the only
  thing that makes the REJECT cross-check real, and would need promotion from optional to
  required for this increment (deferred to the user, O3).**

### 4.2 Categories the adversarial suite must cover

The suite is the Opus increment's creative work, but the design fixes the **coverage
obligations** so it is not arbitrary. At minimum it must include cases in every category
below, each chosen to stress a specific grammar decision documented in `regex.fltkg` /
the lint design §4. (The lint design §7's must-accept / must-reject lists are a strong
starting seed; the adversarial suite should *exceed* them with genuinely tricky shapes, not
just copy them.)

Over-admission probes (try to slip a non-portable construct past the grammar):
- **POSIX-class look-alikes:** `[[:alpha:]]`, `[[:^digit:]]`, and the *escaped* near-miss
  `[\[:alpha:]]` (must ACCEPT — it is a literal `[` then `:alpha:` chars then literal `]`;
  lint §4.1, `regex.fltkg:316-322`). The pair pins that the grammar distinguishes the POSIX
  form (reject) from the escaped-bracket form (accept).
- **Set-operation look-alikes:** `[a-z&&[^aeiou]]`, `[\w--_]`, and the interior-dash
  near-miss `[a-z-0]` (must REJECT as a `--` look-alike, `regex.fltkg:172-184`) vs the
  leading/trailing-dash forms `[-a]`, `[a-]`, `[a-z-]` (must ACCEPT).
- **Unicode property:** `\p{L}`, `\P{N}`, `\pL` (reject), vs escaped `\\p` (accept — escaped
  backslash then literal `p`).
- **Lookaround / backref / named group:** `(?=x)`, `(?!x)`, `(?<=x)`, `(?<!x)`, `\1`,
  `(?P<name>x)`, `(?P=name)` (all reject).
- **Divergent escapes:** `\Z` (reject), `\x{41}`, `\u{41}` braced (reject), `\07` octal
  (reject), vs `\xHH`/`\uHHHH`/`\UHHHHHHHH` (accept).
- **Bare-brace divergence:** `a{` (reject — Rust errors, Python literal), `{` (reject), vs
  bounded `a{2}`, `a{2,}`, `a{2,4}` (accept), and the bare *closers* `}`, `a}b`, `]`, `a]b`
  (accept — portable literals).
- **Flag divergence:** `(?x)` verbose (reject), `(?-i)` / `(?i-s:a)` negation (reject), vs
  `(?i)`, `(?ms)`, `(?i:a)`, `(?ms:a)` (accept).
- **In-class escapes:** `[\b]`, `[\B]`, `[\A]`, `[\z]` (reject — divergent/both-reject), vs
  `[\d]`, `[\w\s]`, `[\n-\r]`, `[\x41-\x5a]` (accept).

Over-rejection / boundary probes (try to make the grammar choke on something portable, or
on a degenerate but legal shape):
- **Empty / nilable shapes:** `""` (empty pattern — the lint treats it as portable but
  `regex := alternation` need not match empty; the spike must pin what the *generated parser*
  actually does on `""` and document it — a known corner, `regex.fltkg:51-52`), empty group
  `()` / `(?:)` (accept), empty branches `a|`, `|a`, `a||b` (accept).
- **Adjacency / shadowing under PEG first-match:** patterns where a most-specific-first
  ordering bug would shadow a longer parse — the grammar must try multi-char openers before
  bare ones (`regex.fltkg:106-113, 134-148`). Probes: `(?:ab)` and `(?i:ab)` vs bare `(ab)`
  (the `(?:` / `(?flags:` openers must be tried before bare `(`); `(?i)` standalone inline
  flags vs `(?i:a)` flag group (the `)` vs `:` after the flags disambiguates — both must
  accept and not shadow each other); `\d` shorthand vs a bare `d` literal after a backslash
  (the escape path must prefer the shorthand). A shadowing bug shows as a short parse
  (`result.pos < len`), caught by the accept predicate.
- **Deeply / awkwardly nested:** long alternation chains `a|b|c|...|z`, deep nesting
  `(((((a)))))`, and `((((a|b)c)d)e)` to stress left-recursive seed-growing on
  `alternation`/`concatenation` (`regex.fltkg:55-77`; left-recursion mechanics, grammar
  reference §9.1). (Stacked quantifiers like `a+*?` are covered under the
  quantifier-on-quantifier probe below.)
- **Whitespace-significant patterns:** a leading literal space `" a"`, a literal tab,
  `" *"` (clockwork's actual pattern) — must ACCEPT and must NOT silently strip the leading
  whitespace (this is the §4.3 lint constraint: the regex grammar must use NO_WS separators
  and no leading separator so the injected `_trivia` is never invoked; the adversarial suite
  is where that property is hammered). A failure here means the generated parser is eating
  significant whitespace — a serious grammar bug.
- **Pathological dashes and carets in classes:** `[a^b]` (non-leading caret literal —
  accept), `[a^]` (accept), `[^^]` (accept — `[` then optional-negation `^` consumed, then
  `class_char` matches a literal `^`, since `class_char := /[^\\\]\[\-\n]/` does not exclude
  caret, `regex.fltkg:186,223`), `[-]`, `[+\-]` (accept).
- **Quantifier-on-quantifier and lazy markers:** `a*?`, `a+?`, `a??`, `a{2,4}?` (lazy
  markers — ACCEPT: `quantifier` is `(+|*|?|bound) . lazy:"?"?`, `regex.fltkg:82-83`) vs the
  stacked `a**`, `a*+`, `a+*` (REJECT: `repetition` allows exactly one `quantifier?`,
  `regex.fltkg:80`, and a second `*`/`+` is not the lazy `?`, so the parser matches `a*` and
  stalls on the trailing glyph → short parse → reject). This pair pins that a single lazy
  `?` is admitted but a second full quantifier is not.

UTF-8 / non-ASCII probes (REQUIRED — see §4.3 for why). The regex grammar is parsed as
Unicode and indexed by codepoint at every layer (§4.3); non-ASCII content is legal and
structurally reachable in literals and regex bodies, so the adversarial suite **must**
include the following. These cases stress the codepoint-vs-byte position machinery
(`terminalsrc.py:173,178`; Rust `cp_to_byte`, exploration §2) and the Unicode-aware regex
engines on both backends (exploration §3):
- **Multi-byte literal characters:** a pattern containing 2-byte (`é`, `ü`), 3-byte (`中`,
  `→`), and 4-byte/astral-plane (`𝄞` U+1D11E, emoji) codepoints as ordinary `literal_char`s
  (e.g. `café`, `αβγ`, `中文`, `a𝄞b`) — must ACCEPT. The grammar's `literal_char` /
  `class_char` classes (`regex.fltkg:186,223`) exclude only ASCII metacharacters, so any
  non-ASCII codepoint is an ordinary character on both engines (exploration §4).
- **Multi-byte characters at non-zero offsets:** patterns with multi-byte characters
  *before* the construct under test (e.g. `é+`, `中*`, `café\d`, `αβγ{2}`, `a𝄞b?`) — must
  ACCEPT and the parse must consume to end-of-input. This is the case most likely to expose
  an off-by-one in codepoint↔byte conversion: the Python oracle counts `result.pos` in
  codepoints (`len(terminals)`), and a Rust `cp_to_byte` indexing bug would surface as a
  wrong end position. The accept predicate (`result.pos == len(terminals)`, §3.2) catches a
  position miscount as a short parse → reject, so this directly guards the §4.3 hazard.
- **Multi-byte characters inside classes:** `[é]`, `[中-中]`, `[a-zé]`, `[αβγ]` — must
  ACCEPT (`class_char` admits non-ASCII; the in-class range form must still parse with a
  non-ASCII endpoint). Pins that the class-body machinery and any range handling is
  codepoint-correct, not byte-correct.
- **Unicode-shorthand on non-ASCII (engine-table divergence — the key risk):** patterns
  using `\w`, `\d`, `\s`, `\b` whose *behavior* depends on Unicode category tables (e.g.
  `\w+` intended to match `naïve` / `αβγ`, `\s` against U+00A0 NO-BREAK SPACE / U+2028 LINE
  SEPARATOR). The exploration flags item 7 — Unicode category tables — as **the single most
  likely Python `re` vs Rust `regex-automata` divergence point** (exploration §5, §6 item
  7): both engines run full-Unicode mode, but their category tables may classify a given
  codepoint differently across versions. The *grammar* admits these patterns (they are
  ordinary `\w`/`\s`/`\b` escapes, `regex.fltkg` escape/assertion rules); the adversarial
  case asserts the grammar ACCEPTS them, and the `rationale` string must call out that
  cross-engine *match-set* parity for these shorthands is a documented divergence risk owned
  by the differential harness (lint §9), not something the grammar-acceptance oracle can
  prove. (The grammar's job is to admit the pattern; whether the two engines then agree on
  which codepoints it matches is a separate, out-of-scope question that the case must name so
  it is not silently assumed.)
- **Combining marks / normalization (NFC vs NFD):** `é` precomposed (U+00E9, 1 codepoint)
  vs decomposed `é` (U+0065 U+0301, 2 codepoints) as literal content. Both must ACCEPT as
  *grammar input* (each is just a sequence of `literal_char`s), but the case must document —
  in the `rationale` and a comment — that neither backend normalizes (exploration §5,
  "Literal comparison"): a literal authored NFD will not *match* NFC parser-input at runtime,
  and crucially **both backends agree on this non-match** (no divergence). The probe pins
  that the regex *grammar* treats both byte-sequences as valid patterns; the no-normalization
  behavior is a documented, cross-backend-consistent property, not a bug to fix here.
- **Astral-plane / supplementary codepoints as a 1-codepoint span:** a pattern like `𝄞`
  (U+1D11E) or an emoji, confirming the parser treats it as **one** codepoint (Python `str`
  len 1, Rust `char` 1 scalar value, 4 UTF-8 bytes) — must ACCEPT, and a single-char quantifier
  applied to it (`𝄞?`, `𝄞+`) must ACCEPT and consume to end. This is the strongest
  exerciser of the 4-byte path in `cp_to_byte` and Python's `pos=` codepoint counting.
- **Bidirectional / RTL text in literals:** a pattern containing RTL or bidi codepoints
  (e.g. Hebrew/Arabic letters) as literal content — must ACCEPT, and (where the case also
  reads `span.text()`) the extracted slice must be the raw codepoint range, not visually
  reordered (exploration §6 item 6). No parser logic is direction-sensitive; this pins that.

These UTF-8 cases carry the same `(pattern, expected, rationale)` shape and ACCEPT-direction
Python `re` cross-check as every other case (§4.1). The `re.compile` cross-check is
especially valuable here: it confirms the non-ASCII pattern is a *valid Python regex*, so an
ACCEPT mis-pin is caught. The cross-engine *match-set* parity questions some of them raise
(Unicode tables, normalization) are out of the grammar-acceptance oracle's scope; §4.3's
scope boundary states why and how each `rationale` must flag them.

### 4.3 Why the UTF-8 / non-ASCII cases are required (not optional)

FLTK parses both grammar files and parser input as **Unicode**, and indexes everything by
**codepoint**, at every layer. This is not a guess — it is verified in source:

- **Grammar files are read as Unicode strings.** All grammar-read sites use `Path.open()`
  with **no `encoding=` argument** (`fltk/plumbing.py:56,206`; `fltk/fegen/genparser.py:42`),
  which decodes as the locale preferred encoding — UTF-8 on the Linux/macOS dev+CI hosts. The
  decoded text is a Python `str` (codepoints), so non-ASCII content in a `.fltkg` is legal.
  (The one caveat — non-UTF-8 Windows locales — is orthogonal to this spike; it does not
  change the codepoint model on the platforms the suite runs on, and is not the adversarial
  suite's concern.)
- **Only literals and regex bodies can carry non-ASCII; rule/label names cannot.** The
  identifier rule is ASCII-only (`fegen.fltkg:16`, `identifier := /[_a-z][_a-z0-9]*/`), but
  literal and raw-string bodies use negated classes (`fegen.fltkg:17-18`,
  `[^\/\n\\]` / `[^"\n\\]`) that admit any non-ASCII codepoint under both regex engines
  (exploration §4). A regex body in a grammar — which is exactly what `regex.fltkg` must
  parse — can therefore contain `é`, `中`, `𝄞`, etc.
- **Positions are codepoints on both backends.** Python `TerminalSource` stores a `str`,
  uses `len()` (codepoints) and indexes char-by-char (`terminalsrc.py:165,173`), and passes a
  codepoint `pos` to `re.match(..., pos=pos)` (`terminalsrc.py:178`; `re` defaults to flag 32
  = `re.UNICODE`, verified). The Rust runtime declares the same contract — `Span` start/end
  are *codepoint* indices (`span.rs` doc) — and maintains a `cp_to_byte` table to convert
  (exploration §2, verified in `terminalsrc.rs`). A span `(0, 1)` is one codepoint on both
  sides regardless of byte width.

Because the position unit is the codepoint and the input is multi-byte-bearing Unicode, the
class of bugs that *only* shows up on non-ASCII input — codepoint↔byte off-by-ones, byte-vs-
codepoint span lengths, Unicode category-table divergence between Python `re` and Rust
`regex-automata` — is invisible to an ASCII-only suite. The current in-tree corpus grammars
are all pure ASCII (exploration §4), so the corpus tests (§3) provide **zero** coverage of
this hazard class. The adversarial suite is the only place it is probed, which is why the
§4.2 UTF-8 category is a hard coverage obligation, not a nice-to-have. The user directive is
explicit on this point: grammars are parsed as UTF-8, therefore the adversarial suite must
include UTF-8 characters.

**Scope boundary the cases must respect.** This spike's oracle proves the *grammar accepts/
rejects* a pattern (a single-backend, Python-side parse, §1.3). It does **not** prove the two
runtime regex engines agree on which codepoints a given `\w`/`\s` then *matches* at parse
time — that cross-engine match-set parity (the genuine Unicode-table divergence risk,
exploration §5–§6 item 7) belongs to the `differential-property-harness` (lint §9). The
UTF-8 adversarial cases therefore assert grammar acceptance and *name* the divergence risk in
their `rationale`; they do not (and cannot, given §1.3's no-inline-Rust-oracle scope) assert
cross-engine match equality. Keeping that boundary explicit prevents the suite from silently
over-claiming.

### 4.4 What "fool the parser" findings look like

When the adversarial suite finds a real discrepancy between the grammar's behavior and the
portable-subset intent, the increment's deliverable is **two things**: (1) a committed
adversarial test case that pins the *correct* expected disposition (so the bug is captured as
a red test), and (2) a finding note in the spike's exploration/notes describing the
discrepancy. The increment **may** also fix the grammar (`fltk/fegen/regex.fltkg` + regen)
if the fix is small and clearly correct; a larger or judgement-dependent grammar change is
left as a documented finding for the user / the downstream lint increment to adopt. The spike
proves the grammar works *or* documents precisely where it does not — both are valid spike
outcomes, and the second is arguably the more valuable one.

## 6. Edge cases / failure modes

- **Empty pattern `""`.** `regex := alternation` (`regex.fltkg:53`) and the header says the
  empty pattern is "handled by the caller," so the start rule "need not match empty"
  (`regex.fltkg:51-52`). The spike does not have the lint's `check_regex_portable` caller, so
  it must **pin what the bare generated parser does on `""`** rather than assume. Likely the
  start rule succeeds via the empty-branch path (`branch:concatenation?`, `regex.fltkg:70`)
  and `result.pos == 0 == len("")`, i.e. ACCEPT — but the adversarial suite must assert the
  observed behavior so it is documented, not assumed. If it does *not* accept `""`, that is a
  finding (the spike runs the parser standalone, unlike the lint which special-cases empty).
- **Whitespace-significant patterns silently stripped.** The single most important failure
  mode for *this* grammar: if `regex.fltkg` accidentally carries a WS-bearing separator or a
  leading separator anywhere, the auto-injected `_trivia` rule (`[\s]+`,
  `gsm.add_trivia_rule_to_grammar`, grammar reference §7) would run and eat literal spaces in
  the pattern. The grammar's header (`regex.fltkg:26-30`) asserts it uses NO_WS throughout to
  prevent this. The corpus has a live witness (clockwork's ` *`) and the adversarial suite
  hammers leading/interior whitespace (§4.2). A regression here is a silent mis-parse, so it
  is pinned in two suites.
- **Stale corpus enumeration.** Guarded by §3.1 (programmatic walk, never a hand-list) for
  the in-tree corpora. Clockwork is run ad hoc against the live file via the general tool
  (§3.4), so there is no committed snapshot to go stale — and no committed clockwork path or
  data at all (user directive).
- **Generated parser drifting from `regex.fltkg`.** Caught only by the *manual* `make gencode`
  → `git diff` step (`Makefile:241-246`), which the implementer runs before commit. This is
  **not** automated: `gencode` is not in the `make check` graph, and no test in the suite
  regenerates and diffs (§1.2). `make check` enforces only that the committed artifacts are
  clean (lint/format/typecheck), not that they are up-to-date with the generator. A real
  regenerate-and-compare gate (the lint design's deferred round-trip test) is added scope and
  not warranted for a spike — but the implementer must not assume `make check` will catch a
  stale or hand-edited `regex_parser.py`. It will not.
- **A corpus pattern is genuinely non-portable.** Possible in principle (the grammars under
  test are not guaranteed all-portable). The corpus test surfaces it as a named failure with the
  pattern (and, diagnostically, the §3.2 furthest-progress offset); triage decides grammar-gap
  vs document-as-non-portable (§3.2). This is a desired outcome of the spike (it would be a
  real finding about a real grammar), not a test infrastructure bug.
- **PEG first-match shadowing.** A subtle grammar bug where a shorter alternative shadows a
  longer valid one yields a *short parse* (accepts a prefix, `result.pos < len`). The accept
  predicate (`result.pos == len`) catches this as a rejection, and the adversarial
  shadowing-probe category (§4.2) targets it directly.

Broken error path: `_check_child_type_for_mutators` emits the message `"{ClassName}.{method}: unsupported child type {type}"` (method name included), while the Rust backend's `extract_from_pyobject` emits `"{ClassName}: unsupported child type {type}"` (no method name). The design §2.2 pins the text as Rust's existing form (no method name); the Python generator adds `.{method}` that Rust lacks.
Why: Python's validation helper receives `method: str` and injects it; Rust's `extract_from_pyobject` is a standalone enum method with no method-name context.
Consequence: Any caller matching or logging the child-type `TypeError` message for cross-backend diagnostics sees different strings. The parity tests (`test_bad_child_type_insert`, `test_bad_child_type_replace_at`) only assert `TypeError` type, not message text — the divergence is undetected and will persist silently until a message-parity test is added. On-call cannot write a single log pattern that matches both backends.
What must change: Either (a) the Python generator drops the method name from the child-type message to match Rust's existing format, or (b) `extract_from_pyobject` is made method-aware (or a wrapping layer adds method context) so both backends produce an identical string. Option (a) is the minimal fix consistent with §2.2. Add an exact-message parity test analogous to `test_bad_label_insert_message_parity`.

errhandling-2
File:line: tests/test_cst_mutators_parity.py:475-494 (three `test_non_index_*` tests); fltk/fegen/gsm2tree_rs.py:1228 (`index.call_method0(pyo3::intern!(py, "__index__"))?`)
Broken error path: Rust calls `index.__index__()` via PyO3's `call_method0`, which raises `AttributeError` when the object has no `__index__` method (e.g. a plain string). Python's `operator.index()` raises `TypeError` for the same input. Design §2.2 says "type pinned" (both backends raise the same exception type) but the implementation produces `AttributeError` on Rust vs `TypeError` on Python.
Why: PyO3's `call_method0` on a missing method raises `AttributeError`; `operator.index` maps all non-int-like inputs to `TypeError` per CPython semantics.
Consequence: Code catching `TypeError` for bad-index inputs on the Python backend will fail to catch `AttributeError` on the Rust backend. The tests explicitly accept both (`pytest.raises((TypeError, AttributeError))`), meaning the type divergence is known but unfixed and unresolved against the design. On-call will see different exception types for the same user mistake depending on backend.
What must change: After calling `index.__index__()`, Rust should check whether the PyErr is `AttributeError` and convert it to `TypeError` with a backend-specific message (matching Python's contract). Or document the divergence in design §2.2 by changing "type pinned" to "type backend-specific for non-indexable inputs" and update the test assertions to make the intent explicit.

errhandling-3
File:line: fltk/fegen/gsm2tree.py:535-543 (Python `remove_at`/`replace_at` template); fltk/fegen/gsm2tree_rs.py:1308-1317 (Rust `insert` template); fltk/fegen/fltk_cst.py:118-120 (generated Python `insert`)
Broken error path: Python validates index first (`operator.index`), then label, then child. Rust validates child first (`extract_from_pyobject`), then label (`_label_from_pyobject_match`), then index (`__index__`). Design §3 says "child/label TypeError wins on Rust (extraction first); Python implementation checks in the same order so the surfaced error matches" — this is incorrect: Python checks index before child/label, so when both index and child are invalid, Python raises an index-related `TypeError` while Rust raises a child-type `TypeError`.
Why: The Python generator emits `operator.index` as the first statement to mirror `list.insert`'s early index coercion; Rust validates extractable arguments before touching the Python integer protocol.
Consequence: When a caller passes both a non-integer index and a wrong-type child, the exception type and message differ between backends. The multi-bad case is not tested. Code relying on "TypeError from bad child" will get a different error on the Python backend. On-call diagnosing a caller bug sees a different error depending on which backend is in use.
What must change: Pick one canonical order and implement it on both backends. The least-surprise choice (matching design §3's stated intent) is child → label → index on both. Alternatively, acknowledge in §3 that multi-bad-arg error priority is backend-specific and document which backend gives which error; add a test that explicitly asserts the per-backend behavior.

errhandling-4
File:line: fltk/fegen/gsm2tree_rs.py:1338-1348 (Rust `remove_at` template, generated at e.g. src/cst_fegen.rs:771-781)
Broken error path: `remove_at` removes the child from the `Vec` under the write lock, then drops the lock and calls `child.to_pyobject(py)?`. If `to_pyobject` fails (e.g. OOM in `Py::new` inside `registry::get_or_insert_with`), the `?` propagates a `PyErr` to the caller, but the child has already been removed from the parent's children list and its `Arc` is dropped if no other reference exists. The caller receives an error but the tree is mutated.
Why: Rust cannot "un-remove" after the write lock is released without a backup copy. The design's lock-discipline (§2.3) correctly puts Python work outside the lock but does not address the partial-mutation case.
Consequence: On OOM (extremely rare), `remove_at` leaves the tree in the post-removal state while returning an error — the semantic contract of "atomic: either returns (label, child) or does nothing" is violated. The removed child is permanently lost from the Python-visible tree. No log or structured error accompanies this; the caller's `PyErr` is the only signal, and it will look like an allocation failure, not a mutation side-effect. Diagnosing an unexpected `MemoryError` will not reveal that the tree was silently mutated.
What must change: Before removing from the `Vec`, call `to_pyobject` on a preview of the child (clone the `Arc` / copy the `Span`) while still under a read lock (or outside any lock) to verify wrap-out succeeds; only then acquire the write lock and remove. Alternatively, buffer the removal: remove from `Vec`, attempt wrap-out, and if it fails, re-insert at the same index before propagating the error. The second approach is simpler but requires a re-insert under a second write lock. For spans (value type), the data is already cloned at remove time, so the issue applies primarily to node-typed children. Add a note in the method's doc comment documenting the atomicity contract.

## `rust-cst-child-node-identity` — Keep deferred

**Problem:** With the Rust backend, reading the same child twice through any accessor returns two distinct Python objects (`is` fails, `==` works). The Python backend gives stable identity. This is a real cross-backend behavioral difference visible to out-of-tree consumers (e.g. code that dedupes nodes with `is` during a tree walk silently breaks).

**Why it matters:** Cross-backend equivalence is a stated goal. But: node `__hash__` already raises (so set/dict-key uses fail loudly), `==` is structural and correct, and the instability is a deliberate consequence of the native-ownership design (`Box<ChildNode>` in the native Vec) that enables GIL-free tree manipulation.

**What the work looks like:** A per-node `Py` cache indexed by position at the accessor layer. Exploration confirmed feasibility but surfaced an unmentioned complication: every mutating method (`append_*`, `extend_*`, `extend_children`) must invalidate/grow the cache, and the `children` list object itself is also fresh per call and would need separate treatment. Non-trivial, ongoing maintenance surface in generated code.

**Case for skipping:** No known consumer requires identity stability; the failure mode for `is`-dependent code is at least deterministic; the cache adds real complexity to every generated accessor.

**Recommendation & decision:** Keep the TODO as-is. The deferral rationale is sound and now verified. Implementing a cache speculatively, overnight, without a consumer demanding it would be complexity without a customer. This is the one item where product-owner input would genuinely change the cost/benefit (how much do you weigh `is`-stability for downstream consumers?), and the reasonable default is the status quo the TODO already records.

**Case for skipping:** The one-liner works today and the committed file matches.

**Recommendation & decision:** Do. Verified-safe, removes a nonstandard regen path, adds a drift guard.

## `preamble-helpers-into-cst-core` — Do

**Problem:** Five cross-cdylib helper items (2 statics + 3 functions) are emitted byte-for-byte identically (MD5-verified) into all three independently committed generated Rust files. A bug fix to any of them requires regenerating every consumer's generated code — including out-of-tree consumers we can't reach.

**Why it matters:** These helpers contain `unsafe` code whose soundness rests on a shared-rlib invariant. If a soundness bug is ever found, propagation-by-regeneration is the worst possible fix path for a library with out-of-tree consumers. Centralizing in `fltk-cst-core` means a fix propagates at rebuild time, no regeneration.

**What the work looks like:** Move the three functions as `pub fn` into `fltk-cst-core` (statics stay `pub(crate)` behind the accessors — exploration confirmed per-cdylib static copies are correct and already-precedented in the crate via `SPAN_KIND_SPAN_CACHE`). Shrink `_preamble()` to `use` lines. Regenerate all three files, `make fix`, tests.

**Case for skipping:** None found. Exploration confirmed no blockers, no behavior change.

**Recommendation & decision:** Do, **before** `span-source-as-py-crosscdylib` (whose new helper should land in `fltk-cst-core` directly rather than be added to the preamble and then moved).

## `span-source-as-py-crosscdylib` — Do (reframed; needs design)

**Problem:** Every span-returning accessor read on a Rust-backend node copies the **entire source text twice** (verified: `arc.text.clone()` + `text.to_owned()`), i.e. O(source length) per node read. A downstream consumer walking a CST of a large file pays quadratic-ish cost. An O(1) Arc-clone API (`source_as_py`) exists but can't be used cross-cdylib because pyo3 type extraction in `fltk._native.Span.with_source` rejects `SourceText` objects registered by another cdylib.

**Why it matters:** This is the single biggest performance footgun for the primary use case — out-of-tree consumers reading spans off large parses. The parse path is clean; the regression is exclusively at the read boundary.

**TODO inaccuracy found:** The proposed fix ("add `extract_source_text` to the generated preamble") is insufficient on its own. Exploration traced the data flow: the blocker is on the **receiving** side — `fltk._native.Span.with_source` does pyo3 type-checked extraction and will reject a cross-cdylib `SourceText` no matter what the caller does. The fix needs a new entry point on the `fltk._native` side (e.g. a private classmethod taking `&PyAny` + `downcast_unchecked` under the same shared-rlib invariant `extract_span` already uses), plus generated-code changes to call it.

**Case for skipping:** Correctness is unaffected; only performance. But the cost is real and the consumers are external.

**Recommendation & decision:** Do, with a design stage (the fix spans `fltk-cst-core` public surface + generated-code emission + the unsafe invariant). Sequenced after `preamble-helpers-into-cst-core`.

## `dependabot-branch-pin-gap` — Do (re-pin to `v1`)

**Problem:** The `dtolnay/rust-toolchain` CI action is SHA-pinned against the `stable` *branch*; Dependabot only refreshes SHA pins that reference semver tags, so this pin silently rots.

**What the work looks like:** Upstream has a `v1` tag (verified live). Re-pin to the `v1` SHA with a `# v1` comment; Dependabot then tracks it. The known behavioral difference between the branches (`toolchain` input required on `v1`) is already guarded — `ci.yml` passes `with: toolchain: stable` explicitly. CI validates the change immediately.

**Case for skipping:** Residual uncertainty whether `master`/`v1` and `stable` differ in other ways, and whether Dependabot proposes refreshes for a sole `v1` tag in practice. Both risks are small and loud (CI fails or Dependabot stays quiet — no worse than today).

**Recommendation & decision:** Do. One-line change that converts a permanent manual chore into automation. Worst case equals the status quo.

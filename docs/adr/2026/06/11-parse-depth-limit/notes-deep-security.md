# Security review — parse depth limit

Style: concise, precise, complete, unambiguous. No padding. Audience: smart LLM/human.

Reviewed: `ef315be..d442f56`. Scope: depth-limit guard in `memo.rs`, generator changes in `gsm2parser_rs.py`, regenerated fixtures, new tests.

Overall: the change is a DoS mitigation and the core mechanism is sound. Guard precedes cache writes; increment/decrement are balanced on every non-panic path; sticky flag neutralizes cache pollution per instance; Python bindings check the flag after every `apply__parse_X` and before inspecting the result, so the truncated-`Some` shape cannot leak through the binding layer; `RecursionError` matches the Python backend's failure type; the mixed-version hazard is dispositioned via the 0.2.0 semver bump and lockstep docs. The attacker-supplied value here is input text only; `max_depth` is operator-controlled. No injection, secret, or authz surface in the diff. Two residual findings.

## security-1

- **File:line**: `fltk/fegen/gsm2parser_rs.py:258-263` (generated header template); `crates/fltk-parser-core/src/memo.rs:64-70` (`DEFAULT_MAX_DEPTH` doc); generated copies in `tests/rust_parser_fixture/src/parser.rs`, `tests/rust_cst_fegen/src/parser.rs`.
- **Issue**: The generated header states unconditionally that exceeding `max_depth` fails the parse "instead of overflowing the native stack." The limit counts rule applications, not stack bytes. Native frames per application level are grammar-dependent (the ~5-7 frames/level figure is for the fixture-shaped grammars; a grammar rule with many nested sub-expression helpers consumes more frames per level), and stack size is deployment-dependent: Rust spawned threads default to 2 MiB (vs the 8 MiB main thread the §6 margin math and T6 assume). T6 empirically pins only fixture-grammar × debug-build × CPython-main-thread.
- **Trust boundary / data flow**: untrusted input text → generated `Parser` → `apply` recursion; native frames accumulate per counted level until either the depth guard or the OS guard page fires, whichever comes first.
- **Consequence**: an attacker feeding deeply nested input to an out-of-tree consumer that runs the parser on a small-stack thread (default `std::thread::spawn`, async-runtime worker threads, embedder-controlled stacks) or against a fat-rule grammar can still overflow the native stack below `DEFAULT_MAX_DEPTH` — process abort (SIGSEGV), i.e. exactly the DoS this change claims to remove, while the documentation tells the consumer they are protected. The false assurance is the hazard: pre-change docs told callers to limit nesting upstream or size their thread stacks; the new docs drop that advice.
- **Suggested fix**: qualify the generated header and `DEFAULT_MAX_DEPTH` docs: the limit is in rule applications; native frames per application vary by grammar and build; the default is sized for an ~8 MiB stack — callers on smaller thread stacks or with deep per-rule structure must lower `max_depth` (roughly proportionally) or size the stack. One sentence in the template covers all generated parsers.

## security-2

- **File:line**: `fltk/fegen/gsm2parser_rs.py:387-400` (generated `set_max_depth`/`depth_exceeded` accessors); contract stated only in doc comments (`memo.rs` `apply` docs, generated header).
- **Issue**: For Rust-native callers, the flag-outranks-result contract is documentation-only. A depth-rejected parse can return `Some` with a truncated/wrong tree (grow_seed stall; optional-item skip — design §2, pinned by T4). Nothing in the generated Rust API forces the `depth_exceeded()` check; the natural call shape `if let Some(r) = parser.apply__parse_x(0)` silently accepts the wrong parse. The design explicitly dispositions this (docs enforce; bindings enforce for Python) — recorded here as residual risk, not a missed case.
- **Trust boundary / data flow**: untrusted input text → `apply` depth rejection inside an in-flight frame → truncated `Some` returned to a Rust consumer that checks only `Some`/`None` → consumer logic operates on a CST that is not the parse the grammar defines.
- **Consequence**: attacker crafts input whose prefix parses shallow and whose tail exceeds the limit (e.g. `valid_stmt + deep_nesting`), yielding a `Some` whose span omits the tail. A consumer using the parsed prefix for validation/sanitization while another component consumes the full input gets a parser-differential / validation bypass; even standalone, the consumer silently acts on a wrong tree on attacker-chosen inputs. Conditions: Rust-native (non-Python-binding) consumer that skips the documented flag check — a plausible default, since the pre-limit API had no post-parse check to make.
- **Suggested fix** (future hardening, beyond this change's scope): make misuse structurally harder — e.g. a generated checked entry point returning `Result<Option<ApplyResult<T>>, DepthExceeded>` alongside the raw `apply__parse_X`, or a `debug_assert!(!depth_exceeded)` escape hatch is not possible without breaking the contract, so the checked wrapper is the clean additive option. At minimum keep the discard-on-flag sentence on every generated `apply__parse_X` doc, not only the header.

No other findings.

## Findings

### design-1

**Section**: Test plan, "Bindings (pytest, `tests/test_rust_parser_bindings.py`), using the new `nest`/`nest_sum` rules (§5)"; same file named in §8 table row "binding-level depth tests (T5-T7)".

**What's wrong**: `tests/test_rust_parser_bindings.py` tests `fegen_rust_cst.Parser` — the parser generated from `fegen.fltkg` (`importorskip("fegen_rust_cst")`, lines 13-16; every test constructs `fegen_rust_cst.Parser`). The `nest`/`nest_sum` rules are added to `rust_parser_fixture.fltkg` (§5) and therefore exist only in the `rust_parser_fixture` module. T5-T7 as specified cannot run against the module that file tests; `fegen_rust_cst` has no `apply__parse_nest`.

**Why**: `tests/test_rust_parser_bindings.py:13-16` (fegen module guard) vs. design §5 (rules added to `fltk/fegen/test_data/rust_parser_fixture.fltkg`) and `tests/test_rust_parser_parity_fixture.py:13-16` (the file whose skip-guard targets `rust_parser_fixture`).

**Consequence**: implementer following the design literally writes T5/T6 against `fegen_rust_cst.Parser` (AttributeError — no nest rules), or imports `rust_parser_fixture` into a file whose `importorskip` guards only `fegen_rust_cst` — when one extension is built and the other isn't, tests error instead of skipping. Either way the test plan's file assignment forces mid-implementation rework.

**Suggested fix**: put T5-T7 in `tests/test_rust_parser_parity_fixture.py` or a new `tests/test_rust_parser_fixture_bindings.py` with its own `importorskip("rust_parser_fixture")`. T7 (existing suites unchanged) is unaffected. T6 only needs *a* depth-proportional rule; if the design prefers keeping it in `test_rust_parser_bindings.py`, it must instead use a deeply-nested fegen-grammar input (nested sub-expressions) — but then say so.

### design-2

**Section**: §1, "Signature unchanged — previously generated parser files keep compiling against the new core (additive struct fields only)."

**What's wrong**: compile compatibility for old-generated-parser + new-core is claimed as a benefit, but the *behavioral* semantics of that combination are never addressed. An already-generated parser (out-of-tree consumer who `cargo update`s `fltk-parser-core` without regenerating, e.g. via a published version bump — in-tree fixtures use path deps, but the design's own argument is about exactly the not-regenerated case) gets the new default limit with none of the new surfacing: its bindings have no flag check, its `Parser` has no `depth_exceeded()`/`set_max_depth`, and its header docs still describe the abort behavior. Two regressions follow for inputs nested past 1000 but under the native-overflow threshold (which is far above 1000 at ~5-7 frames/level on an 8 MiB stack — such inputs previously parsed *successfully*): (a) parses that used to succeed now return `None`, indistinguishable from no-match and uncontrollable (no setter); (b) worse, the §2 truncated-`Some` shapes (grow_seed stall, optional-item skip) return a *wrong successful parse* to code that has no way to know a flag exists — silent corruption, strictly worse than the old behavior for those inputs (old behavior: correct parse). CLAUDE.md requires evaluating breakage from the out-of-tree consumer's perspective; the design evaluates only compilability.

**Why**: design §1 (quoted claim) + §2 (truncated-`Some` mechanics, contract enforced only by *newly generated* bindings/docs) + `tests/rust_parser_fixture/Cargo.toml` (path-dep pattern showing core is a separate crate consumers version independently) + CLAUDE.md "Generated Output is Public API for Out-of-Tree Consumers".

**Consequence**: a downstream consumer upgrading `fltk-parser-core` without regenerating their parser can silently get truncated wrong parse results (or unexplainable failures) on deep inputs that previously parsed correctly, with no API to detect or configure it. If unaddressed in the design, this ships as an incidental side effect rather than a deliberate, called-out decision.

**Suggested fix**: add an explicit disposition to the design: declare the core change semver-breaking (bump `fltk-parser-core` minor/major and state that generated parsers and the core crate must be upgraded in lockstep), and/or note it in the core crate docs/changelog. No code-shape change required; this is a documentation/versioning decision the design must own explicitly.

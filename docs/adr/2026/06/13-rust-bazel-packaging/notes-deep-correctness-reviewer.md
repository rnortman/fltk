## correctness-1 — Test reads `result.value`; the parser result has no `value` attribute

File: `clockwork/dsl/clockwork_rust_roundtrip_test.py` (`test_clockwork_native_parses_module`,
the line `module_node = result.value`).

What's wrong: the Python-visible parse-result class is generated by
`fltk/fegen/gsm2parser_rs.py:850-861`:

```
#[pyclass(frozen, name = "ApplyResult")]
pub struct PyApplyResult { pos: i64, result: Py<PyAny> }
#[pymethods] impl PyApplyResult {
    #[getter] fn pos(&self) -> i64 { self.pos }
    #[getter] fn result(&self, py) -> Py<PyAny> { self.result.clone_ref(py) }
}
```

Only two getters exist: `pos` and `result`. There is **no** `value` getter.
Every other consumer in the tree reaches the CST root through `.result`
(`tests/test_phase4_fegen_rust_backend.py:261`, `tests/parser_parity.py:175`,
`tests/test_rust_parser_parity_fegen.py`). `PyApplyResult` is `frozen` with no
`__getattr__`, so `.value` is not an alias.

Why / consequence: `result.value` raises `AttributeError: 'ApplyResult' object
has no attribute 'value'`. The test fails (or, worse, if it ever ran in a
context that swallowed the error, it would never reach the span assertions it is
meant to exercise). Since AC #4 is "parse one string and read node/label/span
without error," this test cannot pass as written. The earlier asserts
(`result is not None`, `result.pos == len(src)`) use real attributes and pass;
the failure is specifically at the `.value` read.

Fix: `module_node = result.result`.

## correctness-2 — `register_classes` is `#[cfg(feature = "python")]`-gated, but the cdylib never sets `feature = "python"` → unresolved symbol at compile

Files: `rust.bzl:219-234` (`rust_shared_library` with
`crate_features = ["extension-module"] + crate_features`);
`clockwork/dsl/clockwork_native_lib.rs` (and the consumer template embedded in
`rust.bzl`'s docstring), which call `cst::register_classes` /
`parser::register_classes` **unconditionally**.

What's wrong: the generated `cst.rs` and `parser.rs` gate all PyO3 machinery —
including `pub use python_bindings::register_classes` — behind
`#[cfg(feature = "python")]`:
- parser: `gsm2parser_rs.py:956-957` (`#[cfg(feature = "python")] pub use
  python_bindings::register_classes;`).
- cst: `gsm2tree_rs.py:382-392` emits the same `#[cfg(feature = "python")]`
  gating around the register fn / pyclass blocks.

In Cargo, `feature = "python"` is reached because the manifest forwards
`extension-module = ["python", "pyo3/extension-module"]`
(`tests/rust_parser_fixture/Cargo.toml`, root `Cargo.toml:19`). **A Bazel
`rust_shared_library` has no manifest**, so `crate_features = ["extension-module"]`
sets only the raw cfg `feature="extension-module"`. There is no Cargo-style
feature-forwarding step that turns it into `feature="python"`. The design §3.1
explicitly assumes this forwarding happens ("`["extension-module"]` → forwarding
to `pyo3/extension-module` + `fltk-cst-core/python`"), but Bazel `crate_features`
do not forward.

Why / consequence: with `feature="python"` unset, the generated
`register_classes` symbols and the `#[pyclass]` wrappers are compiled out of
`cst.rs`/`parser.rs`. `clockwork_native_lib.rs`'s unconditional
`register_submodule(m, "cst", cst::register_classes)` then fails to compile:
`error[E0425]: cannot find function 'register_classes' in module 'cst'`. The
cdylib target (`clockwork_native_cdylib`) does not build — directly defeating
test-plan items §5.2/§5.4 ("the cdylib builds"). Same applies to FLTK's own
smoke target if/when it is wired to `fltk_pyo3_cdylib`.

Fix: the macro must set `crate_features = ["extension-module", "python"] +
crate_features` (the `python` cfg is what the generated code checks), and the
fltk-cst-core rlib it links already sets `crate_features = ["python"]`
(`crates/fltk-cst-core/BUILD.bazel:6`), so they agree. Note the fixture lib.rs
in-tree wraps the calls in `#[cfg(feature = "python")]`, but the
consumer-authored `clockwork_native_lib.rs` does not — so even matching the
fixture pattern would not save it without the `python` cfg being set.

## correctness-3 — pyo3 `extension-module` feature not demonstrably enabled on the shared `@fltk_crates//:pyo3` (needs verification)

Files: `BUILD.bazel:34-45` (`:native`), `rust.bzl:227-232`, `MODULE.bazel:25-35`
(`crate.from_cargo`, no `crate.annotation`).

What's wrong / uncertainty: `pyo3/extension-module` is a feature of the **pyo3
crate**, not of the consumer. In crate_universe (`from_cargo`), enabling it
requires either a manifest that unconditionally activates it or a
`crate.annotation(crate = "pyo3", crate_features = ["extension-module"])`. None
is present. Whether it ends up enabled depends on whether `from_cargo` honors
`fltk-native`'s `default = ["extension-module"]` (root `Cargo.toml:18-19`,
which forwards to `pyo3/extension-module`) when computing the resolved feature
set for the shared `@fltk_crates//:pyo3` repo. This is a crate_universe
resolution nuance I could not confirm from the tree.

Why / consequence: if `extension-module` is **not** active on
`@fltk_crates//:pyo3`, both `:native` and every `fltk_pyo3_cdylib` link
libpython and fail at link/import time — the exact failure mode the design's own
§4 ("Missing extension-module feature → linker pulls libpython") warns about. If
it **is** active globally, note the opposite hazard: pyo3 docs warn against
building non-extension contexts (e.g. anything that embeds the interpreter)
against a pyo3 with `extension-module` on; not relevant for these cdylib-only
targets, but worth a deliberate confirmation rather than relying on transitive
default-feature resolution. Flagging for the implementation spike to verify
empirically (a clean `bazel build //:native` link is the check).

Fix (if it proves unset): add
`crate.annotation(crate = "pyo3", crate_features = ["extension-module"])` to the
`crate.from_cargo` block, or otherwise pin pyo3's feature set explicitly so it is
not an emergent property of default-feature resolution.

## correctness-4 (minor) — trailing `\n` and full-consume assertion

File: `clockwork/dsl/clockwork_rust_roundtrip_test.py`, `src = "cpu_domain main;\n"`
plus `assert result.pos == len(src)`.

`module := … (use* . entity+)`, `entity` includes `cpu_domain`,
`cpu_domain := doc? . "cpu_domain" : identifier , (cpu_domain_options | ";") , ;`
(clockwork.fltkg:1,12,221). `cpu_domain main;` matches. Whether the trailing
`\n` is consumed so `pos == len(src)` (len includes the newline) depends on
trailing-trivia handling of the final `,` separator. This is a fail-loud
assertion (not a silent wrong answer), so it is a lower-severity correctness
risk than -1/-2, but if the newline is not consumed the test fails on `pos`
even after -1 is fixed. Worth confirming against actual parser behavior;
dropping the trailing `\n` (or asserting `result.pos > 0`) removes the
dependency.

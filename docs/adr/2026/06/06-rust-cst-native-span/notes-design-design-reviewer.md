# Design review: rust-cst-native-span

Adversarial fact-check of `design.md` against requirements, exploration, and source @ base 6fd32e7.
Concise; consequence-focused. Audience: smart LLM/human.

## design-1 — `children.extend`/getter-mutation is exercised in-repo; "no in-repo evidence" is false (load-bearing)

Section: §2.3 ("`children` Python getter rebuilds a list on demand"), §3 edge case "children getter rebuilds a list each call", §5 open question **children-getter-identity** ("no in-repo evidence of such a consumer … Resolve as Won't-Do").

What's wrong: The design rebuilds the `children` list on every getter call and treats mutation-via-getter as having no in-repo consumer. But the in-tree generated parsers and the parser generator mutate children through the getter:
- `fltk/fegen/fltk_parser.py:114,224,280,300` and `bootstrap_parser.py:94,198,252,277,282,304,374,436`: `result.children.extend(item0.result.children)` — `.extend()` called on the list returned by the `children` getter, and `.children` iterated as a plain list.
- Generator source: `fltk/fegen/gsm2parser.py:497,715` emit exactly this (`...fld.children.method.extend.call(...children.move())`).

These emitted parsers run against the **Rust backend** CST nodes when `parse_grammar(rust_fegen_cst_module=...)` is selected (`tests/test_phase4_fegen_rust_backend.py` AC8 path runs the real parse over Rust nodes). Under a rebuild-on-each-call getter, `result.children.extend(...)` mutates a throwaway list; the appended children are silently dropped.

Consequence: Rust-backend parsing produces nodes with empty/incomplete children. AC8 (`fltk2gsm` under Rust backend) and the §4 parse-path test would fail. The §5 "Won't-Do" resolution is built on a false premise and would ship a silently-broken parse path. The design must either keep the current append-via-getter mutation semantics (return a live/identity-stable list, or special-case `extend`/`append` on the getter result) or change the parser generator (`gsm2parser.py`) to route through the node's `extend`/`append` methods — and that generator change must be in scope, not dismissed.

## design-2 — §2.6: `text_or_raise()` does NOT "work under both backends"; current Python-backend child spans are sourceless

Section: §2.6 ("replace `self.terminals[span.start:span.end]` with `span.text_or_raise()` … works under both backends").

What's wrong: Two compounding errors.
1. The values read in `fltk2gsm.py:25-26,146-147,150-151` are not `node.span`; `child_name()`/`child_value()` return the stored **child** span (`terminalsrc.Span`), per protocol `fltk_cst_protocol.py:143` (`def child_name(self) -> ...terminalsrc.Span`). So this is the **children** migration surface, not the span-field surface. The design frames it as a `node.span` / Option-B issue; mechanism is misattributed.
2. Current Python-backend child spans are **sourceless**: `terminalsrc.Span._source` defaults `None` (`pyrt/terminalsrc.py:38`) and generated parsers construct `Span(start=..., end=...)` with no source (`fltk_parser.py:108,117,124,...`). `text_or_raise()` on a sourceless span raises `ValueError` ("has no source", `span.rs:153-159` / `terminalsrc.py:55-56`). That is exactly why `fltk2gsm` uses `self.terminals[start:end]` instead.

Consequence: Replacing `terminals[start:end]` with `text_or_raise()` breaks `fltk2gsm` under the **Python** backend immediately (every Identifier/Literal/Regex visit raises). Under the Rust backend it only works after the parse-path stage attaches source to child spans — so the migration is **not** independent of the parse-path/children stage as §2.5/§4 staging implies. The §4 item-7 acceptance ("`text_or_raise()` migration … passes under Rust backend") and the requirements §"In-tree consumer migration" cross-backend acceptance are unmet as written. Fix requires sourced child spans (parse-path stage) as a prerequisite, and a cross-backend-safe read that the Python backend also satisfies.

## design-3 — `cst_generated.rs` / `cst_fegen.rs` are hand-written, not generator output; §2.8 regen will not reproduce them

Section: §1 line 14 ("`gsm2tree_rs.py:289-294` emits every node struct"), §2.8 ("regenerate and commit the four in-repo generated files: `src/cst_generated.rs`, `src/cst_fegen.rs`, …").

What's wrong: The design (and requirements §"In scope") asserts all four files are generated by `gsm2tree_rs.py` and fixed by regeneration. The exploration explicitly contradicts this: `exploration.md:10` (`src/cst_generated.rs` "PoC grammar — **hand-written**") and `:15` (`src/cst_fegen.rs` "fegen grammar — **hand-written**, 5081 lines"). Only `tests/rust_cst_fixture/src/cst.rs` and `tests/rust_cst_fegen/src/cst.rs` are generated. No build/regen script invokes `RustCstGenerator` to emit `src/cst_generated.rs` or `src/cst_fegen.rs` (grep over `*.py` finds `RustCstGenerator` use only in `genparser.py` for downstream crates and in tests).

Consequence: "Edit the generator, regenerate the four files" will not touch `src/cst_generated.rs`/`src/cst_fegen.rs` — those must be hand-edited to mirror the generator change, or first converted to generated artifacts. If the implementer follows §2.8 literally, the two hand-written files keep `span: PyObject`/`Py<PyList>`, failing the requirements §"Native node state" audit acceptance (which names these files) and the no-`PyObject` grep gate (§4 item-2). This is unverified-claim-taken-as-true; the design inherited the requirements' incorrect premise and should call out the divergence and the manual edit burden. (Note: requirements §"In scope" line 30 also asserts "zero hand-written CST node structs" — directly false per exploration; the design propagates it.)

## design-4 — children-pure-rust-variant: design leaves the core requirement acceptance unsatisfiable under its own chosen representation

Section: §2.3 note + §5 **children-pure-rust-variant**; against requirements §"Native node state" 2nd-bullet acceptance (lines 86-90): pure-Rust test, "no `Python::with_gil`, no Python init," constructs a node tree with child nodes and spans, walks, compares.

What's wrong: The design's chosen child representation is `Py<ChildNode>` (§2.3), then concedes (§2.3 note, §5) that `Py<T>` requires a `Python` token to construct, so a fully interpreter-free constructed subtree is impossible — and defers to "user judgment" whether the acceptance is met by typed `Py<ChildNode>`. This is an unresolved contradiction between the chosen design and a stated **acceptance criterion** (the requirement is not an open question; the open question `children-container-shape` covers shape, not whether the acceptance holds). §4 test-1 is marked "*conditional on resolving children-pure-rust-variant*" — i.e. the design cannot currently write the acceptance test it owes.

Consequence: As written, the primary requirement (CST usable "no GIL, no live Python interpreter — for construction, traversal, field access, equality, and cloning", requirements §Goals) is not met for child nodes. Either the representation must change to a native variant (e.g. `Box<ChildNode>` / native node enum — the larger change both docs flag), or the requirement/acceptance must be formally renegotiated with the user. Punting to "user judgment during implementation" leaves the central deliverable undefined. This is legitimately the hardest open question, but it is a gap in the design's ability to satisfy an accepted acceptance criterion, not merely a style choice.

## design-5 — build system: `fltk-native` rlib carries `extension-module` feature by default; downstream link is not validated

Section: §2.1 (`crate-type = ["cdylib","rlib"]`; downstream `fltk-native = { path = "../.." }`) and §5 **build-cdylib-rlib-cycle**.

What's wrong (verification): `Cargo.toml` confirms `crate-type = ["cdylib"]`, `default = ["extension-module"]`, `extension-module = ["pyo3/extension-module"]`. Adding `rlib` is correct as far as it goes, but a downstream crate depending on `fltk-native` with `default-features` on will pull `pyo3/extension-module` transitively into a crate that also declares its own `pyo3` extension-module — the exact double-`extension-module`/runtime-symbol hazard §5 raises. The design names the risk but provides `fltk-native = { path = "../.." }` with no `default-features = false`, so the manifest snippet as written is the failure mode, not a mitigation.

Consequence: Downstream fixture crates may fail to link or mis-link the pyo3 runtime. The build-system change (requirements §"Build-system changes", a hard requirement) is presented as "validate-then-decide" but the one concrete manifest line given is likely wrong. At minimum the dependency must be `default-features = false` (and the rlib path must not force `extension-module`), and the split-crate fallback (`fltk-cst-core`) should be costed before committing, since it is a substantially larger restructure that the requirements' build-system bullet depends on.

## design-6 — §2.4 equality recursion mechanism is asserted but not grounded; current `__eq__` uses Python `.eq()`, and the native path needs a definition

Section: §2.4 ("recursing into the child's native `__eq__`/value equality, **not** Python `.eq()` on stored objects").

What's wrong: Current `_eq_method` (`gsm2tree_rs.py:535-550`, verified) compares via `self.children.bind(py).eq(...)` and `self.span.bind(py).eq(...)` — Python equality, GIL-bound. The design requires native recursion over `Py<ChildNode>` variants "by recursing into the child's native `__eq__`/value equality." But for a `Py<ChildNode>` you still need either a GIL token to borrow the inner Rust struct and call a native `eq`, or you fall back to Python `.eq()`. The design does not specify how native equality is invoked on a `Py<ChildNode>` without re-entering the Python equality machinery it forbids. The requirement (requirements §"Node and span equality" acceptance: equality "without … entering Python equality code for stored node state") is therefore not demonstrably met by the described mechanism.

Consequence: Either equality on child nodes silently routes back through Python `.eq()` (failing the acceptance), or it requires the GIL (failing the pure-Rust no-interpreter goal for the equality path). This is the same unresolved tension as design-4, surfacing on the equality surface specifically; the acceptance criterion for native equality cannot be claimed met until the child representation is pinned to something with a native borrow path.

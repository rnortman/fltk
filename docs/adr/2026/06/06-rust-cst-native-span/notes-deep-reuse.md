**File:line:** `crates/fltk-cst-core/src/span.rs:140`

The generated code never calls it — the span-getter and child-to-pyobject paths in `gsm2tree_rs.py` switched to `source_full_text_str()` + `get_source_text_type(py)?.call1(...)` (the cross-cdylib workaround described in the implementation log). **Existing alternative:** `Span::text_str` (added in the same diff, `span.rs:157`) and `Span::source_full_text_str` (`span.rs:176`) are the helpers that are actually used. `source_as_py` duplicates a subset of `source_full_text_str`'s intent (extracting the Arc to build a Python object) via a different strategy that does not work cross-cdylib — the reason the generated code was changed away from it.

**Consequence:** `source_as_py` is public API on a foundational crate. Future maintainers will assume it is used, attempt to use it in generated code, and re-encounter the cross-cdylib PyO3 type-registration mismatch that this diff explicitly solved. The stale comment in `gsm2tree_rs.py:392` reinforces the confusion.

## reuse-2 — `Span::text_str` duplicates the existing `Span::text` Python method body

**File:line:** `crates/fltk-cst-core/src/span.rs:157–169`

**What's duplicated:** `text_str` (new, Rust-only) reimplements the same boundary-checking slice logic as `fn text` (`span.rs:221–236`, the existing `#[pymethods]` method). Both check `start > end`, `end > text.len()`, and both UTF-8 char boundaries, then return `Option<String>` with identical semantics. The only difference is `text` additionally rejects negative indices before the `usize` cast, while `text_str` uses `usize::try_from(...).ok()?` which already handles negatives.

**Existing function:** `fn text(&self) -> Option<String>` at `crates/fltk-cst-core/src/span.rs:221`. It is in a `#[pymethods]` block, which only prevents calling it from Rust via the pyo3 trait dispatch — the method body itself is accessible by adding a `pub` plain-impl delegation or by extracting the shared logic into a private helper.

**Consequence:** Two independent implementations of text-slicing in the same file. A future change to UTF-8 handling, boundary semantics, or negative-index treatment applied to `text` will silently diverge from `text_str`, producing different results for identical inputs depending on which path is taken. Since `text_str` is called from generated Rust code and `text` is called from Python, the divergence can manifest as cross-language behavioral differences that are hard to diagnose.

## reuse-3 — `gencode` Make target re-invokes `gen-rust-cst` via `$(MAKE)` for 3 of 4 Rust files but inline-constructs the PoC grammar for the 4th

**File:line:** `Makefile:83–88`

**What's duplicated:** `src/cst_generated.rs` is regenerated by an inline `python -c` one-liner that directly imports `_make_poc_grammar` from `tests/test_gsm2tree_rs.py` and `RustCstGenerator` from `fltk.fegen.gsm2tree_rs`. The other three Rust files (`cst_fegen.rs`, `tests/rust_cst_fixture/src/cst.rs`, `tests/rust_cst_fegen/src/cst.rs`) go through the existing `gen-rust-cst` make target which wraps `python -m fltk.fegen.genparser gen-rust-cst`. The `gen-rust-cst` path writes through `genparser.py`'s `gen_rust_cst` command, which adds the `# ruff: noqa` header and handles file I/O consistently.

**Existing entry point:** `make gen-rust-cst` / `fltk.fegen.genparser gen-rust-cst` at `Makefile:54–55` and `fltk/fegen/genparser.py:264–282`. The TODO comment `# TODO(gencode-poc-fltkg)` acknowledges the gap.

**Consequence:** The PoC grammar `cst_generated.rs` is regenerated without the `genparser.py` header-emission and file-write logic, so any future change to the file preamble (e.g., adding a lint-suppression header, a generation timestamp, or a "do not edit" warning) will require updating both `genparser.py` and the inline one-liner in the Makefile separately. The `TODO(gencode-poc-fltkg)` already tracks this but the divergence grows as the preamble evolves.

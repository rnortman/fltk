## errhandling-1

**File:** `crates/fltk-cst-core/src/span.rs:217`

**Path:** `Span::text_or_raise` — `self.source.as_ref().unwrap()`

**Why:** At line 214 the function returns `Err` if `self.source.is_none()`, so by line 217 `source` is
guaranteed `Some`. The `unwrap()` is therefore a true invariant, not a reachable panic on bad input.
However: the invariant is local and fragile — if a future edit reorders the guards (e.g. adds an early
`Ok(...)` path above), the `unwrap` becomes reachable without a compile error. The invariant is also not
commented at the call site; a reader has to mentally re-trace the guards to confirm safety.

**Consequence:** Currently unreachable, so no runtime consequence today. Future maintenance risk: a
refactor that adds a branch above this point silently removes the safety guarantee, producing a panic
on a Python call with no diagnostic other than a generic rust panic traceback.

**What must change:** Replace `unwrap()` with an explicit `unreachable!()` or an `expect()` with a
message stating the invariant, e.g.:
```rust
let inner = self.source.as_ref()
    .expect("invariant: source is Some — checked by is_none guard above");
```
Alternatively restructure using `if let Some(inner) = &self.source { ... }` to make the guard and the
use structurally adjacent.

## errhandling-2

**Files:** `src/cst_fegen.rs`, `src/cst_generated.rs`, `tests/rust_cst_fegen/src/cst.rs`,
`tests/rust_cst_fixture/src/cst.rs` — generated and hand-written `_label_from_pyobject_match` code
(emitted by `gsm2tree_rs.py::_label_from_pyobject_match`).

**Path:** `append` and `extend` methods — label conversion when a label is provided but fails extraction:

```rust
Some(lbl) => {
    if let Ok(native_lbl) = lbl.bind(py).extract::<Grammar_Label>() {
        Some(native_lbl)
    } else {
        None  // ← silent discard
    }
}
```

**Why:** If the Python caller passes a label value that is the wrong type (e.g., a label enum from a
different grammar rule, or a stale Python-side label object), the extraction silently fails and the child
is stored with label `None`. The child is appended without error; subsequent `child_{label}` /
`children_{label}` / `maybe_{label}` queries will miss it entirely.

**Consequence:** A mis-labelled `append` call — plausible during parser development or cross-grammar
reuse — produces a child that is invisible to all label-based accessors and invisible to `__repr__`
(which only shows count). No exception is raised, no log is emitted. On-call cannot distinguish
"bug: wrong label passed" from "no child was appended" without a debugger. The invariant assumed by
`child_{label}` (count == 1) silently breaks, producing a misleading "Expected one X child but have 0"
error later, far from the actual `append` call.

**What must change:** The `else` branch in `_label_from_pyobject_match` should raise a `PyTypeError`
instead of silently discarding:
```rust
} else {
    return Err(PyTypeError::new_err(format!(
        "{class_name}.append: label argument is not a {enum_name}"
    )));
}
```
Same fix applies to the `extend` methods and to both the hand-written
`cst_fegen.rs`/`cst_generated.rs` files and the generator template.

## errhandling-3

**Files:** `src/cst_fegen.rs`, `src/cst_generated.rs`, `tests/rust_cst_fegen/src/cst.rs`,
`tests/rust_cst_fixture/src/cst.rs` — `span` getter (generated by `_span_getter_setter`).

**Path:** `span` getter: `span_cls.bind(py).call1((self.span.start(), self.span.end()))` — creates a
**sourceless** `Span(start, end)` regardless of whether the stored span has a source.

**Why:** `extract_span` (called on `set_span` and `new`) correctly preserves the source reference
into the native `Span`. But the getter reconstructs the Python span using only `(start, end)`, calling
`Span.__new__` which maps to `Span::py_new` — the sourceless constructor. So a span stored with a
source comes back to Python without one: `node.span.has_source()` is `False` even if the node was
constructed with a source-bearing span.

Similarly, `IdentifierChild::to_pyobject` (and all other `XChild::to_pyobject` `Span` variants) do:
```rust
span_type.call1((s.start(), s.end()))
```
dropping the source.

**Consequence:** Any downstream consumer that relies on `node.span.text()` or `node.span.has_source()`
after round-tripping through a Rust node will silently get `None` / `False`. The source is not lost in
the native store — only in the Python-visible projection. A consumer who stores a source-bearing span
into a node and reads it back cannot diagnose why `text()` returns `None`; `repr(node)` only shows
`Span(start=..., end=...)` and `has_source()` is `False`, giving no indication the source was present
at store time.

**What must change:** The span getter (and `XChild::to_pyobject` for the `Span` variant) must use
`Span.with_source` when a source is present. Since the getter holds a `Span` with a potential
`Option<Arc<SourceInner>>`, it must either:
- Call `fltk._native.Span.with_source(start, end, SourceText(...))` when `self.span.source.is_some()`,
  reconstructing a `SourceText` Python object from the stored `Arc<SourceInner>` — which requires
  `SourceInner` to be exposed or `SourceText` to be constructable from an `Arc`, or
- Directly allocate a `Span` Rust-side (via `Py::new`) with the source intact and return it, bypassing
  the Python constructor.

The `SourceText` / `Span` types in `fltk-cst-core` need to support this round-trip path. This is
likely a design gap that needs to be addressed as part of §2.6/§2.7 source propagation work, but the
current silent source-drop is not marked as a known gap and will confuse consumers.

## errhandling-4

**File:** `fltk/fegen/gsm2tree_rs.py:220–221` — `_child_enum_block`, degenerate case.

**Path:** Generated code for a rule with no known child types (no Span, no node refs):
```python
if not has_span and not child_classes:
    lines.append("        let _ = (py, span_type);")
```
The generated `extract_from_pyobject` always reaches the final `Err(PyTypeError)` for this case —
correct. But the generator silently accepts this degenerate grammar structure and emits code that
cannot store any child. Any `append` or `extend` call on such a node will always fail with a type
error, with no warning at code-generation time that the grammar model is degenerate.

**Consequence:** If a grammar has a rule that accidentally has no typed children (e.g. due to a
`_child_variants_for_rule` model lookup gap), the generator produces a node type where every
`append`/`extend` always raises. No diagnostic is emitted by the generator; the author discovers
this at runtime.

**What must change:** `_child_enum_block` should raise `RuntimeError` or emit a compile-time
`compile_error!()` macro call when `not has_span and not child_classes`, or at minimum log a warning
at generation time identifying the rule name.

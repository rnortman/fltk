design-1:
- Disposition: Fixed
- Action: §2.3 (children getter) and §3 (children-getter edge case) and §5 (children-getter-identity
  open question). Verified real and load-bearing: `fltk/fegen/fltk_parser.py:114,224,280,300,312,317,…`
  and `bootstrap_parser.py:94,198,252,…` emit `result.children.extend(item.result.children)` —
  `.extend()` mutates the object returned by the `children` getter; `gsm2parser.py:497,715` is the
  emitter (`...fld.children.method.extend.call(...children.move())`). A rebuild-on-each-call getter
  makes this mutate a throwaway list, silently dropping children. Design rewritten: the `children`
  getter no longer claimed safe to mutate; the parser generator (`gsm2parser.py`) is brought IN SCOPE
  to route child-merging through the node's own `extend`/`append` methods (which mutate the native
  `Vec`) rather than `.children.extend(...)` on the getter result. The §5 "Won't-Do" resolution
  premised on "no in-repo consumer" is removed; the consumer is in-repo (the generated parsers).
- Severity assessment: Without this, Rust-backend parsing yields nodes with empty/incomplete children;
  the parse-path test and fltk2gsm-under-Rust acceptance fail. Silently-broken parse path shipped.

design-2:
- Disposition: Fixed
- Action: §2.6 (in-tree consumer migration). Verified both sub-claims. (1) The reads in
  `fltk2gsm.py:26,147,151` are on child spans, not `node.span`: `visit_identifier` reads
  `identifier.child_name()`, `visit_literal`/`visit_regex` read `child_value()`; protocol types these
  as `fltk.fegen.pyrt.terminalsrc.Span` (`fltk_cst_protocol.py:532,562,592`) — so this is the
  CHILDREN-migration surface, not the span-field / Option-B surface. (2) Current Python-backend child
  spans are sourceless: `terminalsrc.Span._source` defaults `None` (`terminalsrc.py:38`); generated
  parsers build `Span(start=…, end=…)` with no source (`fltk_parser.py:108,117,124,…`).
  `text_or_raise()` on a sourceless span raises `ValueError` (`terminalsrc.py:55-56`, `span.rs:153-159`)
  — which is precisely why fltk2gsm uses `self.terminals[start:end]`. Design corrected: (a) the
  migration is reframed as a CHILDREN-surface concern, not `node.span`; (b) the §2.5/§4 staging claim
  that the fltk2gsm migration is independent of the parse-path stage is removed — sourced child spans
  (parse-path stage) are now stated as a PREREQUISITE; (c) the cross-backend-safe read is fixed to one
  that the Python backend also satisfies today (continue using `self.terminals[start:end]` via a
  backend-agnostic extent accessor, OR gate the `text()`-based read on the parse-path stage attaching
  source to child spans under BOTH backends — and the Python-backend child-span sourcing change is
  called out as required for the `text()` route, since Out-of-scope forbids changing the Python
  backend's `terminalsrc.Span` type but not what source the parser attaches). Note: Option B
  (`native-span-start-end`) means the Rust child Span has no `.start`/`.end`, so a `terminals[start:end]`
  read is itself unavailable under the Rust backend — the design now states the read must go through
  `.text()`/`.text_or_raise()` on a SOURCED span under the Rust backend, which forces the parse-path
  prerequisite. §4 item-7 acceptance updated to depend on the parse-path stage.
- Severity assessment: As written, the migration breaks fltk2gsm under the Python backend immediately
  (every Identifier/Literal/Regex visit raises ValueError) and cannot work under the Rust backend until
  child spans carry source. Both the in-tree-migration and cross-backend acceptances are unmet.

design-3:
- Disposition: Fixed (premise rejected; residual correction applied)
- Action: §1 line 14, §2.8. The reviewer's premise — `src/cst_generated.rs` and `src/cst_fegen.rs` are
  hand-written — is FALSE. Targeted investigation (per task note) and my verification establish they are
  GENERATED by `RustCstGenerator` (`gsm2tree_rs.py`) and checked in: both files carry the generator's
  exact preamble (`UNKNOWN_SPAN_CACHE` GILOnceCell comment + imports, byte-identical to generator
  output); `tests/test_gsm2tree_rs.py:24` states the PoC grammar "used to generate src/cst_generated.rs";
  `Makefile:53-55` provides `gen-rust-cst GRAMMAR=… RS_OUT=…` which invokes
  `fltk.fegen.genparser gen-rust-cst` → `RustCstGenerator`. So "edit generator, regenerate" is the
  correct mechanism for ALL FOUR files; the requirements §"In scope" "zero hand-written CST node structs"
  claim stands. The reviewer's Consequence (the two files keep `PyObject` after regen) does not follow.
  RESIDUAL (valid, applied): I found the on-disk files have drifted from the CURRENT generator output —
  generator emits `other_kind` in `__eq__` (`gsm2tree_rs.py:157`) and the label-enum `#[allow(non_camel_case_types)]`
  (`:240`); the committed `src/cst_generated.rs` shows `other_label` and differing line-wrap. This means
  the committed files were produced by an earlier generator/formatter and are NOT byte-reproducible from
  HEAD's generator. §2.8 amended: the regen step must (a) use `make gen-rust-cst` with the PoC grammar →
  `RS_OUT=src/cst_generated.rs` and the fegen grammar → `RS_OUT=src/cst_fegen.rs` (the exact grammars are
  in `tests/test_gsm2tree_rs.py` fixtures / `fltk/fegen/fegen.fltkg`), then `make fix`/`cargo fmt`, then
  commit; (b) the post-regen audit (§4 item-2 grep gate) is the correctness check, NOT byte-equality to
  the prior committed files (which encode pre-existing drift). The "two hand-written files must be
  hand-edited" remediation the reviewer proposes is rejected as based on the false premise.
- Severity assessment: If accepted as written, the design would gain a spurious "hand-edit two large
  files" task and the requirements' generated-artifact framing would be wrongly overturned. The real
  residual (generator/disk drift) is minor and is handled by regenerating from grammars, not from disk.

design-4:
- Disposition: Fixed
- Action: §2.3 note, §5 children-pure-rust-variant. Verified the contradiction: requirements §"Native
  node state" 2nd bullet (lines 86-90) is an ACCEPTANCE CRITERION (pure-Rust construct/walk/compare a
  node-with-child-nodes tree, no `Python::with_gil`, no Python init), and `Py<ChildNode>` storage cannot
  satisfy it because `Py<T>` construction requires a `Python` token. The open question
  `children-container-shape` covers shape, not whether the acceptance holds — so this is a design gap,
  not a sanctioned open question. Design changed: the child representation is no longer left to
  "user judgment during implementation." The design now COMMITS to a native child-node variant
  (`Box<ChildNode>` / native per-rule node enum) as the required representation to meet the accepted
  pure-Rust criterion, with `Py<…>` wrapping applied only at the Python binding boundary (getters), not
  in stored state. The Python-identity-stability concern that motivated `Py<ChildNode>` is moved to a
  scoped, called-out consequence (see TODO below) rather than a reason to violate the acceptance. The
  larger-change cost (every accessor's boundary translation) is stated explicitly as the real,
  in-scope cost of the requirement. §4 test-1 is no longer "conditional"; it is the acceptance test the
  design owes and can now write against the native variant.
- Severity assessment: As written, the primary requirement (pure-Rust CST, no interpreter, for
  construction/traversal/equality/cloning) is unmet for child nodes — the central deliverable is
  undefined. Pinning the native variant makes the acceptance satisfiable.
- TODO(rust-cst-child-node-identity): Native `Box<ChildNode>` storage means a child node accessed twice
  via the Python getter may not be the same `PyObject` identity as before (previously `Py<PyList>`
  preserved identity). Track whether any consumer relies on Python child-object identity; if so, layer a
  per-node `Py` cache at the binding boundary. Marked TODO because it is a real, concrete follow-on whose
  "done" is checkable (identity test across getter calls), but it does not block the native-state
  acceptance.

design-5:
- Disposition: Fixed
- Action: §2.1 (build system) and §5 build-cdylib-rlib-cycle. Verified against `Cargo.toml`:
  `crate-type = ["cdylib"]`, `default = ["extension-module"]`, `extension-module = ["pyo3/extension-module"]`.
  Adding `rlib` is necessary but the design's downstream snippet `fltk-native = { path = "../.." }` with
  default features ON would pull `pyo3/extension-module` transitively into a crate that ALSO declares its
  own pyo3 extension-module — the exact double-extension-module/runtime hazard §5 names, so the snippet
  as written IS the failure mode. Design corrected: (a) the downstream dependency is
  `fltk-native = { path = "../..", default-features = false }`; (b) the `extension-module` feature must
  NOT be in `fltk-native`'s default set if the rlib is to be linked by another pyo3 crate — the design
  now either moves `extension-module` out of `default` (opt-in for the cdylib build) or splits a
  non-pyo3-`extension-module` core; (c) the split-crate fallback (`fltk-cst-core` rlib holding `Span` +
  child enums, pyo3 bindings layered in the extension crate) is costed up front as the safe structure
  rather than "validate-then-decide", because §2.4/§2.3's native-equality + native-child requirements
  (post design-4) push the native types toward a pyo3-free core anyway.
- Severity assessment: The one concrete manifest line given would likely fail to link or mis-link the
  pyo3 runtime in downstream crates, blocking the build-system requirement (a hard requirement).

design-6:
- Disposition: Fixed
- Action: §2.4 (native equality). Verified: current `_eq_method` (`gsm2tree_rs.py:535-550`) compares via
  `self.children.bind(py).eq(...)` / `self.span.bind(py).eq(...)` — Python `.eq()`, GIL-bound. The design
  asserted native recursion "into the child's native `__eq__`/value equality" but did not define how that
  is invoked on a `Py<ChildNode>` without a GIL token or Python `.eq()`. Resolved by the same change as
  design-4: with stored children as a native `Box<ChildNode>` enum, equality recurses structurally in
  pure Rust (`PartialEq` on the node enum: native `Span` variant compares per `span.rs`; child-node
  variants recurse into derived/native field equality), never borrowing a `Py<T>` and never entering
  Python `.eq()`. §2.4 now references the native enum's `PartialEq` as the equality mechanism and states
  explicitly that no stored-state comparison enters Python equality code — satisfying requirements
  §"Node and span equality" acceptance.
- Severity assessment: Without the native-variant change, child equality either silently routes through
  Python `.eq()` (fails the acceptance) or requires the GIL (fails the pure-Rust goal) — the native-
  equality acceptance is unclaimable until child storage is native.

---

Note on scope coupling: design-1, design-4, and design-6 share a root — stored children must be a native,
GIL-free, structurally-comparable enum (not `Py<ChildNode>`, not the getter-mutation pattern). The design
was reworked around that single decision; the §5 open questions reduce to (a) the build-crate structure
(design-5: split-crate `fltk-cst-core` recommended) and (b) staging order (parse-path before fltk2gsm
migration, per design-2). The genuine remaining user-judgment item is delivery staging, not the child
representation (now decided).

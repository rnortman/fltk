## gsm2tree_rs.py — Rust-exposed Python API changes from commit 4c8f0ad

### Design table row: `span` — "PyObject get/set (`_node_block`, l.295-296) — opaque"

**STALE.** After commit 4c8f0ad, the Rust struct holds `span: Span` (native `fltk_cst_core::Span`), not `PyObject`. The Python getter `fn span(&self, py: Python<'_>) -> PyResult<PyObject>` (generated by `_span_getter_setter`, `gsm2tree_rs.py:584-612`) now returns a Python `fltk._native.Span` object constructed on-the-fly via `get_span_type(py)`. The runtime span returned to Python is now the **canonical `fltk._native.Span`** type, always — not an opaque PyObject whose type was unknown.

The setter (`set_span`, `gsm2tree_rs.py:607-611`) extracts a native Span via `extract_span(py, value)`, accepting either the locally-registered `Span` or a cross-cdylib `fltk._native.Span`.

### Design table row: `children` — "Py<PyList> get (l.297-298)"

**Partially stale — Python surface unchanged, internal representation changed.** The Python getter `fn children(&self, py: Python<'_>) -> PyResult<Py<PyList>>` still returns `Py<PyList>` (generated by `_children_getter`, `gsm2tree_rs.py:636-655`). But internally the struct field is now `children: Vec<(Option<Label>, EnumChild)>` — native Rust types — and the getter rebuilds the list per call by calling `child.to_pyobject(py, &span_type)?`. The **Python-visible return type** (`Py<PyList>`) is unchanged, so the design's table entry for `children` is still accurate as a description of the *Python surface*.

### Design table row: `span` type for `.pyi` — "typed as `terminalsrc.Span`"

**Stale.** The committed `CstModule` protocol (`fltk_cst_protocol.py:89,120,...`) now annotates `span` as `fltk.fegen.pyrt.terminalsrc.Span | fltk._native.Span` (union), not just `terminalsrc.Span`. The design (`design.md:92`, `design.md:170`) says "Type the stub `span` as `terminalsrc.Span` to match the protocol" — but the protocol now uses the union type. The design's instruction to use only `terminalsrc.Span` for the `.pyi` `span` field would produce a narrower annotation than the committed protocol, which may or may not cause a pyright conformance failure. The design's §3 edge-case reasoning about the Rust span opacity is also stale: the runtime span getter now returns an `fltk._native.Span` object, so `.start`/`.end` are accessible on the returned span (not "no `.start`/`.end`" as stated in `design.md:168`).

### Design table row: `Span.start`/`.end` — "not present (Rust span is opaque PyObject)"

**STALE.** The span getter now constructs `fltk._native.Span(start, end)` or `fltk._native.Span.with_source(start, end, src)` via `span_cls.call1(...)`. The returned Python object is a `fltk._native.Span` with `.start` and `.end` properties. The design's constraint "Do not invent `.start`/`.end` on the Rust span" (`design.md:170`) was motivated by the opacity of the old PyObject span; that rationale no longer applies.

### `extend_children` — new method not in design

**New surface not covered by design.** `_generic_extend_children` (`gsm2tree_rs.py:725-740`) emits `fn extend_children(&mut self, other: PyRef<'_, ClassName>) -> PyResult<()>` for each node. The committed `CstModule` protocol (`fltk_cst_protocol.py:96,129,...`) includes `def extend_children(self, other: <NodeType>) -> None`. The design does not mention `extend_children` in its `.pyi` structure (§2.1). The `.pyi` emitter must also cover this method or the conformance check will fail.

### NodeKind Python names

`_node_kind_python_name` (`gsm2tree_rs.py:224-226`) returns `class_name.upper()`. The design (`design.md:85`) says "one member per rule (`_node_kind_python_name`) typed so the per-node `kind` annotation `Literal[NodeKind.X]` resolves." The `_node_kind_canonical_name` (`gsm2tree_rs.py:228-230`) returns `f"NodeKind.{class_name.upper()}"`. These are consistent with the design, no change.

### `_rule_info` — cited as gsm2tree_rs.py:69-95

Current: `gsm2tree_rs.py:69-95`. Confirmed present and unchanged in structure. Design's reliance on it is still valid.

### New preamble content (`_preamble` output)

`_preamble` (`gsm2tree_rs.py:136-214`) now emits `extract_span`, `get_span_type`, `get_source_text_type`, `FLTK_NATIVE_SPAN_TYPE`, `FLTK_NATIVE_SOURCE_TEXT_TYPE`. These are implementation-internal to the `.rs`; they do not affect the `.pyi` structure except that the `span` getter now returns a canonical `fltk._native.Span`, which affects the correct annotation for `span` in the `.pyi`.

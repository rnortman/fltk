## TODO(rust-cst-child-node-identity)

### TODO.md entry (lines 11–13)

> Native child ownership (`Box<ChildNode>` in the native Vec) means a child returned twice through a Python getter/accessor wraps a fresh `Py<ConcreteNode>` per call; the same child read twice is not the same Python object (identity differs). Tests in `tests/test_phase4_rust_fixture.py` that formerly used `is` for child identity were relaxed to `==` (value equality). If a consumer requires stable Python child-object identity, add a per-node boundary cache (e.g. `Py` cache indexed by position) at the generated accessor layer. Deferred: no in-tree consumer currently requires identity stability. Location: `fltk/fegen/gsm2tree_rs.py` (accessor methods in `_per_label_methods`); see also `tests/test_phase4_rust_fixture.py:242,276,291,350,371`.

### Code occurrences

All five are in `tests/test_phase4_rust_fixture.py`:

- Line 242: `test_ac6_list_protocol_index` — `assert tup[1] == child`
- Line 276: `test_ac6_list_protocol_negative_index` — `assert last[1] == b`
- Line 291: `test_ac7_tuple_items` — `assert value == ident`
- Line 350: `test_ac11_iterator_methods` — `assert node.child_key() == ident`
- Line 371: `test_ac12_generic_child` — `assert tup[1] == ident`

Also pinned at `tests/test_rust_cst_poc.py:47`: `test_children_rebuilt_each_call` — explicitly asserts list-level (not element-level) identity is not preserved per-call.

### Mechanism

Node struct layout (generated by `gsm2tree_rs.py:543–546`, emitted for every rule):

```rust
pub struct NodeName {
    span: Span,
    children: Vec<(Option<NodeName_Label>, NodeNameChild)>,
}
```

Child enum for node-typed children (`gsm2tree_rs.py:453`):

```rust
NodeName(Box<NodeName>),
```

Conversion to Python (`gsm2tree_rs.py:488`):

```rust
Self::NodeName(n) => Py::new(py, (**n).clone()).map(|p| p.into_any()),
```

Every call to `to_pyobject` on a node-typed child calls `(**n).clone()` then `Py::new(...)`. This allocates a fresh `PyCell<T>` on the Python heap. Python `is` is therefore `False` between two reads of the same positional child, even though `==` (`PartialEq`) is `True`.

This pattern is called from:
- `_children_getter` (`gsm2tree_rs.py:682–700`): rebuilds the entire list on every `node.children` call; the list object itself is also fresh each time
- `_generic_child` (`gsm2tree_rs.py:787–805`): called by `node.child()`
- `_per_label_methods` (`gsm2tree_rs.py:807–909`): `children_<lbl>`, `child_<lbl>`, `maybe_<lbl>` all call `child.to_pyobject(py)?` or `child.to_pyobject(py)` inside a Vec iteration

Span children are also reconstructed per-call via `span_to_pyobject` (`cross_cdylib.rs:113`), but `Span` is `#[pyclass(frozen, eq, hash)]` so `is` on spans is practically irrelevant.

The Python backend (`gsm2tree.py`) stores children in a plain `list` on a `@dataclass`; accessing `node.children[0][1]` twice returns the same Python object. Python backend has stable identity; Rust backend does not.

### Why deferred

The TODO was deferred because no in-tree consumer requires `is`-identity on children. The immediate fix asked: calling tests were relaxed from `is` to `==`.

### Proposed fix (from TODO.md)

A per-node boundary cache — `Vec<Option<PyObject>>` (or `HashMap<usize, PyObject>`) — stored on each node struct, lazily populated. Feasibility complications not mentioned in the TODO entry (identified in `docs/adr/2026/06/10-todo-burndown/exploration-rust-cst-child-node-identity.md`):

1. **Cache invalidation on mutation.** `append`, `extend`, `extend_children`, `append_<lbl>`, `extend_<lbl>` all mutate `self.children`. Any positional cache is invalidated by appends (position N is stable only if nothing is ever inserted before N; appends are the only current mutation). The cache must grow in step with the Vec.
2. **`Py<T>` inside a `#[pyclass]` struct.** `Py<PyObject>` drops its refcount on `Drop` of the Rust struct; pyo3 handles this (queuing the decrement when not holding the GIL). Not a blocker, standard pyo3 usage.
3. **List-level identity separately.** The `children` getter rebuilds a `PyList` per call (`gsm2tree_rs.py:682–700`). Stabilizing per-element identity does not stabilize the list object.

A more structural alternative (mentioned in `design.md §4.3` and `§7 Q1`): store children as `Py<Child>` when `python` feature is on. This would give true stable identity but changes the ownership model, affecting the pure-Rust API and `extend_children` semantics.

### Connection to the idiomatic-Rust-CST design

`design.md §7 Q1` asks whether snapshot-on-read (mutations to a Python-read child don't propagate to the parent; `is`-identity unstable) is acceptable, or whether shared mutable identity is needed. The user's provisional answer (design.md USER A1): "We are going to try to pull shared identity/mutability into this design."

`design.md §4.3` explicitly defers shared identity and mutation as a structural redesign requiring its own requirements + ADR pass. The design names the two candidate approaches: store children as `Py<Child>` or use `Arc<RwLock<...>>` throughout — both change ownership, perf, and the pure-Rust API shape.

`design.md §8 (Consequences)` states: "`TODO(rust-cst-child-node-identity)` ... remain[s] open and [is] now explicitly load-bearing for Phase 3 scope decisions."

Phase 3 in the design is the "Mixed Rust/Python story (R3): document + decide" phase. Pulling this TODO into scope of the idiomatic-CST project means choosing the ownership model for children before Phase 2's native accessor surface is finalised — because the return type of per-label native accessors (`pub fn child_<lbl>(&self) -> Result<&T, CstError>`) assumes `Box<T>` ownership and borrows through it. If children switch to `Py<Child>`, the native accessor returns something different.

## TODO(crosscdylib-abi-sentinel)

### TODO.md entry (lines 15–17)

> `extract_span` in `crates/fltk-cst-core/src/cross_cdylib.rs` uses `downcast_unchecked` after an `isinstance` check that can pass under version skew (consumer pins revision A, installed `fltk._native` wheel built from revision B with different `Span` layout) — turning a packaging error into in-process memory corruption rather than a clean error. Fix: unify the ABI sentinel mechanism across both the `Span` path (`get_span_type`, `extract_span`) and the `SourceText` path (`extract_source_text`, `FLTK_CST_CORE_ABI` classattr on `SourceText`). The `FLTK_CST_CORE_ABI` constant and `SourceText._fltk_cst_core_abi` classattr (introduced alongside `extract_source_text`) are the seed mechanism for the `SourceText` path, but the current derivation (`CARGO_PKG_VERSION` alone) does NOT cover pyo3-resolution skew — two builds of the same `fltk-cst-core` version with different pyo3 resolutions produce identical markers over different `PyClassObject<SourceText>` layouts (→ UB, not TypeError). The strengthened derivation must fold in the pyo3 version and/or a layout hash, and should be applied to both paths. Location: `crates/fltk-cst-core/src/cross_cdylib.rs` (`get_span_type`, `extract_span`, `extract_source_text`, `FLTK_CST_CORE_ABI`).

### Current state of each unsafe site

**`extract_span` (line 147–176):** Fast path is `obj.extract::<Span>()` — safe, same cdylib. Slow path at line 156: `obj.is_instance(&native_span_type)?` using the cached `fltk._native.Span` type object. If that check passes, line 169: `unsafe { obj.downcast_unchecked::<Span>() }`. The `SAFETY` comment (lines 157–165) documents the invariant ("both cdylibs MUST link the same fltk-cst-core rlib") and names the vulnerability: `is_instance` can succeed under version skew when `Span` layout differs, producing UB. There is currently **no ABI string check** on the Span path. The full slow-path gate is `isinstance` only.

**`extract_source_text` (line 50–93):** Fast path `obj.downcast::<SourceText>()` — safe. Slow path at line 59: reads `obj.get_type().getattr("_fltk_cst_core_abi")`, extracts as `&str`, compares with `FLTK_CST_CORE_ABI`. If equal, line 68: `unsafe { obj.downcast_unchecked::<SourceText>() }`. Has an ABI string check; does NOT have `isinstance`. The `FLTK_CST_CORE_ABI` value is `concat!("fltk-cst-core/", env!("CARGO_PKG_VERSION"))` = `"fltk-cst-core/0.1.0"` (Cargo.toml line 3: `version = "0.1.0"`).

**`span_to_pyobject` (line 113–137):** Calls `get_span_type(py)` every invocation (no caching of the fast-path bool or the `_with_source_unchecked` method reference). The second TODO at line 114 is a performance concern, not a safety one.

**`get_span_type` (line 179–193):** Caches `fltk._native.Span` type object in `FLTK_NATIVE_SPAN_TYPE: GILOnceCell<Py<PyType>>`. No ABI check at init time. The TODO at line 166 proposes moving the sentinel check into this `GILOnceCell` init, so it fires once per process (fail fast, clean error, not repeated-UB-on-every-call).

### The pyo3-resolution-skew gap

`crates/fltk-cst-core/Cargo.toml:15`:

```toml
pyo3 = { version = "0.23", features = ["abi3-py310"], optional = true }
```

`tests/rust_cst_fixture/Cargo.toml` and all real out-of-tree consumer crates are standalone Cargo workspaces that resolve `pyo3 = { version = "0.23" }` independently. Two builds with different patch versions of pyo3 (e.g. 0.23.3 vs 0.23.4) could produce different `PyClassObject<Span>` layouts or different `PyClassObject<SourceText>` layouts while both reporting `FLTK_CST_CORE_ABI = "fltk-cst-core/0.1.0"`. The `isinstance`-only gate on the `Span` path and the version-only ABI string on the `SourceText` path both admit this gap.

Proposed strengthening (from TODO.md): fold in the pyo3 version (e.g. `concat!("fltk-cst-core/", env!("CARGO_PKG_VERSION"), "/pyo3/", env!("CARGO_PKG_VERSION" for pyo3))`) and/or a `size_of::<PyClassObject<T>>()` layout hash. The design doc (`span-source-as-py-crosscdylib/design.md §3`, `§2.2` note) explicitly delegated strengthening to this TODO.

### Two distinct sub-problems in this one TODO

1. **Safety gap on the `Span` path** — `extract_span`'s slow path uses only `isinstance`; add an ABI string check (same mechanism as `extract_source_text`, but adapted for `Span` which has no classattr yet). Proposed location: check in `get_span_type`'s `GILOnceCell` init so it fires once.

2. **Derivation weakness on both paths** — `FLTK_CST_CORE_ABI = "fltk-cst-core/0.1.0"` does not distinguish pyo3 minor versions. Strengthening requires either: (a) embedding the pyo3 version string at compile time (`CARGO_PKG_VERSION` of the pyo3 dependency), or (b) a `size_of::<PyClassObject<T>>()` layout check at startup.

3. **Performance concern in `span_to_pyobject`** — `get_span_type` is called on every span accessor; the fast-path bool (is this cdylib `fltk._native`?) and the bound `_with_source_unchecked` method reference could be cached in a `GILOnceCell`. This is correctness-neutral but listed alongside the safety items.

### Connection to the idiomatic-Rust-CST design and node identity

`design.md §4.3` states: "Cross-cdylib node passing (nodes created by extension module A consumed by extension module B's Rust). Requires per-node-type ABI-marker machinery analogous to `extract_span`/`crosscdylib-abi-sentinel`, which is itself still being hardened."

`design.md §7 Q2` asks whether "passing CST objects through the barrier" ever means between two different extension modules. The user's answer (design.md USER A2): "There is never a need for cross-cdylib passing." This scopes out the need for node-type ABI markers, so cross-cdylib node passing is off the table.

**The remaining connection:** `crosscdylib-abi-sentinel` directly underlies `span_to_pyobject` and `extract_span`, which are called by every generated span getter and setter. These span operations are part of the idiomatic-CST API:

- `span_to_pyobject` is called by `_span_getter_setter` (generator `gsm2tree_rs.py:645–658`) and by the `Span` arm of `to_pyobject` in every child enum (`gsm2tree_rs.py:484–486`)
- `extract_span` is called by `_new_method` (`gsm2tree_rs.py:628–643`), `_span_getter_setter` (setter, line 651–657), `_generic_append` (`gsm2tree_rs.py:702–715`), and every `append_<lbl>` and `extend_<lbl>` in `_per_label_methods` (lines 814–840)

If `crosscdylib-abi-sentinel` is resolved (ABI check added to `extract_span`'s slow path), a call with a mismatched-version `Span` from a consumer cdylib now raises `TypeError` with a diagnostic message instead of silently corrupting memory. Every generated node method that receives a span argument benefits.

The performance sub-problem (caching in `GILOnceCell`) directly affects the hot path of the idiomatic native API: the Phase 1–2 span getter/setter are called on every node read from Python. Currently `get_span_type` resolves the cached type object on each call — one `GILOnceCell::get_or_try_init` call, cheap but non-zero.

## Constraints and risks relevant to the designer

### rust-cst-child-node-identity

- The `Box<T>` ownership model was deliberately chosen to enable GIL-free `extend_children` bulk-copy (generator docstring `gsm2tree_rs.py:770–785`). Switching to `Py<Child>` enables Python identity stability but removes GIL-free tree manipulation for the node-typed children (only `Span` children could remain GIL-free with no stored Python object).
- The `Arc<RwLock<...>>` approach keeps GIL-free manipulation but introduces interior mutability across the Python/Rust boundary and requires a custom pyo3 integration (not a `#[pyclass]` struct in the normal sense).
- Phase 2's native accessor signatures (`pub fn child_<lbl>(&self) -> Result<&T, CstError>`) return `&T` by borrowing through `Box<T>`. If children switch to `Py<Child>`, the return type must change — `Py<Child>` does not deref to a borrowable `T` without the GIL; the native-only path would need a different API shape.
- The `extend_children` native method proposed in `design.md §4.2 item 3` (`pub fn extend_children(&mut self, other: &Self)`) clones children from `other`'s Vec. Under `Py<Child>` ownership, this would clone Python refcounts (requiring the GIL) rather than native `Box<T>` values — fundamentally different cost and semantics.
- Cache-based fixes (per-node `Vec<Option<PyObject>>`) avoid restructuring ownership but require careful invalidation on `append`/`extend`/`extend_children` and don't solve list-level identity (`node.children` returning the same list object across calls).

### crosscdylib-abi-sentinel

- The safety gap on the `Span` path (`extract_span` line 169 `downcast_unchecked`) is currently ungated by any ABI string. Adding a `_fltk_cst_core_abi` classattr to `Span` (parallel to `SourceText`) and checking it in `get_span_type`'s init would unify the mechanism.
- `FLTK_CST_CORE_ABI = "fltk-cst-core/0.1.0"` is effectively a constant: `fltk-cst-core` has never been bumped past 0.1.0 (`Cargo.toml:3`). The version-only derivation will not distinguish any two builds until the version is bumped — meaning the "clean error instead of UB" guarantee is only as strong as the version discipline.
- Adding `size_of::<PyClassObject<T>>()` as a layout component to the ABI string requires Rust `const` evaluation of pyo3 internals. `PyClassObject` is a pyo3 internal type; its layout is not guaranteed stable across pyo3 minor versions, which is exactly the threat model. Whether `size_of::` is accessible as a `const` for `PyClassObject<Span>` at build time requires verification.
- The performance sub-problem (caching in `GILOnceCell`) is orthogonal to correctness and can be addressed independently; it does not require resolving the derivation question.
- Since cross-cdylib node passing is off the table (USER A2), the `crosscdylib-abi-sentinel` work does not need to extend to generated node types. It is scoped to `Span` and `SourceText` in `fltk-cst-core`.

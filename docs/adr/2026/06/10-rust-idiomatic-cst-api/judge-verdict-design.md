## Findings walk

### design-1 — Fixed
Claim: §4's "make gencode covers all five outputs" was false — `tests/rust_cst_fegen/src/cst.rs` is
regenerated by no target; consequence: stale fegen Rust CST lets the §6 item 3 cross-backend gates pass
against old clone-semantics code.
Evidence: design.md §4 opening now reads "covers only **four** of the five outputs ... regenerated by no
target (only built, `build-fegen-rust-cst` Makefile:89–90) and its crate is excluded from the root
workspace", and commits Phase 1 to extending `gencode` with
`gen-rust-cst GRAMMAR=fltk/fegen/fegen.fltkg RS_OUT=tests/rust_cst_fegen/src/cst.rs`, naming the
stale-extension/green-gates failure mode. Independently spot-checked against the Makefile: `gencode` emits
cst_generated.rs, cst_fegen.rs, rust_cst_fixture cst.rs, and the spike `cp` — rust_cst_fegen absent;
`build-fegen-rust-cst` only runs maturin develop; the parameterized `gen-rust-cst GRAMMAR= RS_OUT=` target
exists, so the proposed extension is mechanically real.
Assessment: false claim corrected, regen gap closed with a concrete committed step. Accept.

### design-2 — Fixed
Claim: `from_shared(s) -> Self` was a registry-bypassing handle-producing path the §4.0 invariant
enumeration omitted; consequence: `is`-stability (the R3 definition, §1) breaks in exactly the mixed
scenario R3 serves, while §6 item 4 stays green testing only registry-routed paths.
Evidence: §4.0 handle spec has no `from_shared`; defines
`to_py_canonical(py: Python<'_>, s: &Shared<Identifier>) -> PyResult<Py<PyIdentifier>>` as the
registry-routed wrap-out bridge, with an explicit paragraph on why no bare constructor may exist
(registration needs the post-`Py::new` heap object; a bare path mints unregistered aliases). Invariant
paragraph now reads "no handle-producing path bypasses the registry (no bare `from_shared` exists)". §4.4
uses `to_py_canonical` outbound / `.shared()` inbound; §4.5 routes the parser hand-off through
`to_py_canonical`. §6 item 4 adds the both-directions test (native wrap of an already-Python-read child →
`is` the existing handle; two `to_py_canonical` calls on one `Shared` → `is`).
Assessment: adopts the reviewer's suggested shape, closes the enumeration gap, pins it with a test. Accept.

### design-3 — Fixed
Claim: deep `PartialEq` holds read guards across recursion; `x == x` (CPython calls `__eq__` on identical
operands; the registry makes repeated reads the same object) and same-position DAG compares read-lock the
same std `RwLock` twice on one thread — documented may-deadlock under a queued writer; "excluded by
construction" was recorded false.
Evidence: §4.0 `Shared` spec specifies "`PartialEq` = `ptr_eq` short-circuit, then deep", comment marking
the short-circuit "load-bearing, not an optimization" and citing both same-lock cases plus std `RwLock`
semantics. §4.0 lock-discipline paragraph qualified (deep `PartialEq` holds guards but calls no Python;
same-lock re-entry handled by the short-circuit; see §5). §5 rewritten: deep `PartialEq` named as the one
generated path that cannot follow snapshot-then-drop; residual same-lock re-entry under `eq` requires a
true cycle, already documented out of contract. §6 item 1 adds native `x == x` / same-position DAG
no-deadlock tests; item 4 adds Python-side `x == x` returns `True` without hang.
Assessment: hazard eliminated by spec, false claim corrected, both sides test-pinned. Accept.

### design-4 — Fixed
Claim: §6 item 6's fake-span ABI-mismatch test was not executable against the init-time `GILOnceCell` gate
(a fake object dies at plain `isinstance`, cross_cdylib.rs:156; the cell is not resettable in the shared
pytest process); consequence: the TDD-first test passes pre-feature or never reaches the new code, shipping
Phase 0's headline fail-loud behavior unverified.
Evidence: §6 item 6 rewritten with per-path mechanisms. Span path: subprocess test — fresh interpreter,
`fltk._native.Span._fltk_cst_core_abi` patched/removed via import shim before the first span boundary
crossing, crossing driven through the `rust_cst_fixture` consumer cdylib, asserting a `TypeError` naming
both ABI strings, with an unpatched control subprocess (precedent: commit 3217a14). SourceText path:
in-process fake-marker shape retained (per-object gate, cross_cdylib.rs:59–77). The item records why
in-process Span-path testing is impossible, and layout-probe tests inherit the same per-path mechanisms.
Assessment: test plan now matches the §4.1 mechanism per path; isolation mechanism concretely specified.
Accept.

### design-5 — Fixed
Claim (minor, two parts): (a) Phase-1 snippets used the Phase-2 name `IdentifierLabel`, risking the rename
being pulled forward and bloating the "mechanical" Phase-1 diff; (b) §5 wrongly listed the Rust `__repr__`
as a cycle-recursion hazard headed into an immutable ADR.
Evidence: (a) §4.0 snippet reads `Vec<(Option<Identifier_Label>, IdentifierChild)>` with inline note
"Identifier_Label until the Phase 2 rename (§4.3 item 5)"; the §4.2 plain-impl snippet spells
`Identifier_Label` with the same deferral footnote. (b) §5 "Reference cycles" lists
`__eq__`/`PartialEq`/the new `Debug` only, with the parenthetical that the Rust `__repr__` prints span +
child count (gsm2tree_rs.py:933–943, moved unchanged) and only the Python backend's dataclass repr
recurses.
Assessment: anachronism and overstated hazard both corrected in place. Accept.

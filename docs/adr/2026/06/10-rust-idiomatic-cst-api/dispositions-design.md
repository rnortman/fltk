design-1:
- Disposition: Fixed
- Action: §4 opening paragraph rewritten — now states `make gencode` covers only **four** of the five
  outputs, that `tests/rust_cst_fegen/src/cst.rs` is regenerated by no target (built only,
  `build-fegen-rust-cst` Makefile:89–90) and its crate is workspace-excluded so staleness is invisible to
  root `cargo check`, and commits Phase 1 to extending `gencode` with a
  `gen-rust-cst GRAMMAR=fltk/fegen/fegen.fltkg RS_OUT=tests/rust_cst_fegen/src/cst.rs` step, naming the
  stale-extension/green-gates failure mode the extension prevents.
- Severity assessment: Verified against Makefile:103–141 — the "covers all five" claim was false. Without
  the fix, an implementer following the doc's regen path leaves the fegen Rust CST stale, and a previously
  built `fegen_rust_cst` .so could keep serving old clone-semantics nodes to the AC8/cross-backend pytest
  path, letting the §6 item 3 R2/R3 gates pass against code the change never touched.

design-2:
- Disposition: Fixed
- Action: §4.0 handle-pyclass spec — `from_shared(s) -> Self` removed, replaced with registry-routed,
  GIL-bound `to_py_canonical(py, s: &Shared<T>) -> PyResult<Py<PyT>>` (reviewer's suggested shape); new
  paragraph states why no bare constructor may exist (registration needs the Python heap object, which
  exists only post-`Py::new`; a bare path mints unregistered aliases and breaks `is`-stability in exactly
  the R3 mixed scenario). Invariant-soundness paragraph updated ("no handle-producing path bypasses the
  registry"). §4.4 bridge and §4.5 parser-hand-off references updated to `to_py_canonical`. §6 item 4 gains
  the both-directions test (native wrap of an already-Python-read child → `is` the existing handle; two
  `to_py_canonical` calls on one `Shared` → `is`).
- Severity assessment: As previously specified, the documented §4.4 mixed-app pattern itself violated the
  single-canonical-handle invariant — `is`-stability (the R3 definition, §1) broke precisely in the mixed
  Rust/Python scenario R3 exists to serve, while §6 item 4 stayed green because it tested only
  registry-routed paths.

design-3:
- Disposition: Fixed
- Action: §4.0 `Shared` spec — `PartialEq` now specified as `ptr_eq` short-circuit then deep compare, with
  the comment marking the short-circuit load-bearing, not an optimization (CPython calls `__eq__` for
  identical operands; the registry makes repeated reads the same object; DAG same-position case; std
  `RwLock` documented same-thread read-read may-deadlock under a queued writer). §4.0 lock-discipline
  paragraph qualified (deep `PartialEq` holds guards across recursion but calls no Python; same-lock
  re-entry handled by the short-circuit). §5 "Lock recursion / re-entrancy" rewritten: deep `PartialEq`
  named as the one generated path that cannot follow snapshot-then-drop; residual same-lock re-entry under
  `eq` requires a true cycle, already documented out of contract. §6 item 1 gains the native `x == x` / DAG
  same-position no-deadlock test; §6 item 4 gains Python-side `x == x`.
- Severity assessment: Without the short-circuit, a plain `==` on a self-identical operand or a DAG-sharing
  tree read-locks the same std `RwLock` twice on one thread — a potential hang in exactly the multithreaded
  mixed apps R3 targets — and the ADR recorded a false "deadlocks excluded by construction" claim.

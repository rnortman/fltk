## Other findings walk

### design-1 — Fixed (special-attention: filename must reach Rust-parser-produced spans)
Claim: on the Rust backend, parser-produced spans never receive a filename — the generated pyo3 parser ctor takes a raw `str` and builds the `SourceText` itself, so the pyo3 `SourceText(text, filename=None)` ctor (the prior draft's named "Rust caller entry point") is **not on the parse path**. Consequence: the headline feature ("a parser-produced span can tell you its file") is unmet on the Rust backend, and the §4.1/§4.5 cross-backend filename equivalence tests are unimplementable as a real parse (they silently degrade to the hand-built-`SourceText` path).

Source verification of the finding's premise:
- `crates/fegen-rust/src/parser.rs:57-59` — `Parser::new(text: &str, capture_trivia)` → `Self::from_source_text(SourceText::from_str(text), …)`. Confirmed: no filename path.
- `crates/fegen-rust/src/parser.rs:1399-1407` — `PyParser::new(text: &str, capture_trivia, max_depth)`, builds `Parser::new(text, …)`. Confirmed: raw string, ctor builds the `SourceText` internally.
- `parser.rs:1649-1651` — `register_classes` adds only `PyApplyResult`/`PyParser`; no `Parser.from_source_text` pyo3 entry. Confirmed.
- clockwork `parser_backend.py:145` — `self._rparser_mod.Parser(source)` (bare string). Confirmed.
The finding is correct and is the highest-severity item: the feature would ship half-working on the exact backend clockwork is porting *to*.

Disposition (Fixed) — revised threading path verified as **real and implementable**:
- The design's §2.8 "Rust *parser* path" subsection (design.md:428-467) names the emitter as `fltk/fegen/gsm2parser_rs.py` (distinct from `gsm2parser.py`, which emits the Python parser). I confirmed the emitter and the exact cited lines:
  - `gsm2parser_rs.py:380-381` emits `pub fn new(text: &str, capture_trivia: bool)` → `Self::from_source_text(SourceText::from_str(text), …)` — i.e. it emits verbatim the `parser.rs:57-59` I verified above.
  - `gsm2parser_rs.py:384-385` emits `from_source_text`.
  - `gsm2parser_rs.py:944-947` emits the pyo3 `#[new] PyParser::new(text, capture_trivia=false, max_depth=None)` — verbatim `parser.rs:1399-1407`.
- The proposed change (add `filename: Option<&str>` to the emitted `Parser::new` → `SourceText::from_str(text, filename)`; add `#[pyo3(signature = (text, filename=None, …))]` to `PyParser::new` and forward it) is mechanically sound: it makes `Parser(source, filename="f.clk")` a **real parser entry point**, mirroring the Python `TerminalSource(text, filename=...)` path. The `parser.rs` header "Generated by fltk gen-rust-parser … Do not edit" confirms regen is the right vehicle.
- §2.9 (design.md:517-532) rewritten to assert filename equivalence via **actual parses** on both backends; §4.1 (686-691) names the parse path as the load-bearing case and as the regression guard for the generator change; §4.5 (751-756) parses with a filename on both sides; §2.12/§4.6/§5/§7 corrected from "Python parsers only" to "BOTH parser backends" (617-621, 764-773, 800-806).

Assessment: the revised threading path is real, the cited emitters are accurate, and the cross-backend filename feature is now implementable on the Rust backend. Fix addresses the consequence at the named mechanism. Accept.

### design-3 — Fixed
Claim: the two legacy `pos_to_line_col` impls do not share a low-end domain, and the new `start < 0 → None` guard makes `Span.line_col()` diverge from the legacy method on both backends — undocumented, will surface during TDD as a false "bug."
Source verification: `terminalsrc.py:183-205` — Python `pos_to_line_col` guards only `pos > len` (line 184); no negative guard; any `pos < 0` flows into `bisect.bisect_left`. `terminalsrc.rs:182` — Rust rejects `pos < -1`, treats `pos == -1` as in-domain. Divergence is real and accurately described.
Action verified: §2.6.2 (design.md:323-341) states the guard deliberately diverges on both backends and that the legacy low-end domains themselves differ; §4.3 (711-715) adds a deliberate Python negative-position divergence test mirroring the Rust one; §4.4 (729-738) scopes the drift anchor to non-negative positions with the reason stated.
Assessment: the stated mechanism now matches source on both backends; divergence pinned by deliberate tests. Accept.

### design-5 — Fixed
Claim: a `line_span: Span` getter on a frozen pyclass must return an owned (cloned) `Span` (a bare field-borrow won't compile), and `pyclass(eq)` requires `PartialEq` with a noted value-equal/source-ignoring semantics.
Source verification: `span.rs:156` `#[derive(Clone)]` on `Span`; `span.rs:155` `pyclass(frozen, eq, hash, from_py_object)`; `span.rs:176-180` `PartialEq` ignores source (compares only `start`/`end`). Clone is an O(1) `Arc` bump. All hold.
Action verified: §2.6 step 1 "Getter and equality specifics" (design.md:285-298) specifies the clone-getter `#[getter] fn line_span(&self) -> Span { self.line_span.clone() }` and documents the value-equal/source-ignoring equality, matching the Python `@dataclass(eq=True)` `LineColPos` (both ignore source).
Assessment: prevents a wrong first implementation; equality semantics correctly grounded. Accept.

### design-9 — Fixed
Claim: the Python regen and the Rust `SourceText::new` filename param must land **atomically** — the generated Python parser's `_source_text` line resolves, under the Rust backend, to the Rust pyo3 `SourceText`, so a Python regen merged ahead of the Rust ctor param raises `TypeError: unexpected keyword argument 'filename'`.
Source verification: `gsm2parser.py:105-118` is the `_source_text` construction (`SourceText(text=terminalsrc.terminals)`); generated `fltk_parser.py:16` is `fltk.fegen.pyrt.span.SourceText(text=terminalsrc.terminals)` — the backend selector; `span.py:10-13` re-exports the Rust `SourceText` over the Python one when `fltk._native` imports. The coupling argument is sound and grounded.
Action verified: §2.8 "Atomicity" subsection (design.md:505-515) and §5 bullet (807-810) state the two changes are not independently mergeable and must land together.
Assessment: latent integration-order trap correctly flagged with an accurate mechanism. Accept.

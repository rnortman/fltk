design-1:
- Disposition: Fixed
- Action: §2.8 reworked. Added a dedicated **"Rust *parser* path — filename must reach the real parse"**
  subsection with a generator-change table (`gsm2parser_rs.py:380-381` `Parser::new`,
  `gsm2parser_rs.py:944-947` `PyParser::new`), making the **pyo3 `Parser(source, filename=...)`**
  constructor the real Rust parser entry point (mirroring the Python `TerminalSource` path). §2.9
  rewritten to assert filename equivalence via *actual parses* on both backends. §4.1 filename test now
  names the parse path as the load-bearing case and as the regression guard for the generator change.
  §4.5 cross-backend formatter case parses with a filename on both sides. §2.12, §4.6, §5, §7 corrected
  from "Python parsers only" to "BOTH parser backends." Verified against source: `parser.rs:57-59`
  (`Parser::new` → `SourceText::from_str(text)`) and `parser.rs:1399-1407` (`PyParser::new(text: &str,
  …)`) confirm the filename never enters the parse path through the pyo3 `SourceText` ctor; `parser.rs`
  header "Generated by fltk gen-rust-parser … Do not edit" and `gsm2parser_rs.py:380-385,944-947` confirm
  the emitter is the Rust-parser generator; clockwork `parser_backend.py:145` (`self._rparser_mod.Parser(source)`)
  confirms the consumer calls a bare-string parser ctor.
- Severity assessment: Highest. As written, the headline feature ("a parser-produced span can tell you
  its file") was unmet on the Rust backend — the backend clockwork is porting *to* — and the §4.1/§4.5
  cross-backend filename tests were unimplementable as a real parse. Left unfixed, the feature ships
  half-working and the equivalence tests silently degrade to only exercising the hand-built `SourceText`
  path.

design-2:
- Disposition: Fixed
- Action: §4.5 "Cross-backend" bullet updated — the cross-backend formatter-equivalence case now parses
  on both backends with a filename set (so both headers render `In <file>:` instead of one degrading to

design-3:
- Disposition: Fixed
- Action: §2.6.2 gained a subsection stating that `Span.line_col()`'s `start < 0 → None` guard
  deliberately diverges from legacy `pos_to_line_col` on **both** backends, and that the two legacy
  methods' low-end domains themselves differ (Rust rejects `pos < -1`; Python has no negative guard and
  silently bisects any `pos < 0`). §4.3 adds a deliberate Python negative-position divergence test
  mirroring the Rust one; §4.4 drift anchor scoped to non-negative positions with the reason stated.
  Verified against source: `terminalsrc.py:184` (Python `pos_to_line_col` guards only `pos > len`, no
  negative guard) and `terminalsrc.rs:182` (Rust rejects `pos < -1`) confirm the divergence.
- Severity assessment: Low-medium. Not a code bug, but an unstated cross-method inconsistency that would
  surface during TDD if the implementer wrote a negative-start drift-anchor case and mistook the genuine
  divergence for an implementation error.

design-4:
- Disposition: Fixed
- Action: §2.6 step 1 registration path expanded from three steps to four — inserted the
  `src/lib.rs:6` (`use span::{SourceText, Span};`) import edit that binds the name `LineColPos` for the

design-5:
- Disposition: Fixed
- Action: §2.6 step 1 gained a "Getter and equality specifics" subsection: the `line_span` getter
  returns an **owned (cloned) `Span`** (O(1) `Arc` bump), shown as
  `#[getter] fn line_span(&self) -> Span { self.line_span.clone() }`, not a field-borrow; and the
  intended `pyclass(eq)` equality is documented as value-equal/source-ignoring, matching the Python
  `@dataclass(eq=True)` `LineColPos` because both ignore source. Verified against source: `span.rs:156`
  `#[derive(Clone)]` on `Span`, `span.rs:155` `pyclass(frozen, eq, hash, from_py_object)`, and
  `span.rs:176-180` `PartialEq` ignoring source.
- Severity assessment: Low. A field-borrow getter on a frozen pyclass won't compile; specifying the clone
  prevents a wrong first implementation. The equality note prevents surprise in the cross-backend
  `LineColPos` equality assertions.

design-9:
- Disposition: Fixed
- Action: Added an "Atomicity" subsection to §2.8 and a bullet to §5: the Python regen and the Rust
  `SourceText::new` `filename` param must land together, because the generated Python parser's
  `_source_text` line uses the backend selector `fltk.fegen.pyrt.span.SourceText`, which resolves to the
  **Rust** pyo3 `SourceText` under the Rust backend — a Python regen merged ahead of the Rust ctor param
  raises `TypeError: unexpected keyword argument 'filename'` at runtime. Verified against source:
  `span.py:10-13` re-exports the Rust `SourceText` over the Python one when `fltk._native` imports;
  `gsm2parser.py:105-118` is the `_source_text` construction site.
- Severity assessment: Medium-high. A partial/interleaved merge (Python regen before the Rust ctor param)
  breaks **every** run of a generated Python parser under the Rust span backend — a latent
  integration-order trap the design did not flag.

# Exploration: Clean Protocol-Only Consumer API

**Gating criterion (user-stated, verbatim):** "A gating criteria is that user code (and fltk2gsm.py is a model for user code) must be *clean*. No double-importing, no TYPE_CHECKING hacks, no `cast`, no `noqa` or other pyright/ruff suppressions forced by the cst. This needs to carry through from requirements to design."

---

## 1. Protocol Module: How it is Generated and What it Contains

### Generator

`fltk/fegen/gsm2tree.py` contains both the concrete CST generator and the protocol generator in a single `CstGenerator` class. The entry point for the protocol module is:

- `CstGenerator.gen_protocol_module()` — `gsm2tree.py:434`
- `CstGenerator._protocol_class_for_model()` — `gsm2tree.py:465`
- `CstGenerator._cst_module_protocol()` — `gsm2tree.py:561`

### What Protocol Classes Contain

Each rule's Protocol class (`_protocol_class_for_model`, `gsm2tree.py:465`) contains:

- A nested `class Label:` (plain class, **not** `enum.Enum`) with one `ClassVar[object]` annotation per label name — `gsm2tree.py:476-483`
- `kind: typing.Literal[NodeKind.X]` — `gsm2tree.py:488-489`
- `span`, `children` — `gsm2tree.py:493-505`
- `append`, `extend`, `child` and per-label `append_<label>`, `extend_<label>`, `children_<label>`, `child_<label>`, `maybe_<label>` — `gsm2tree.py:509-558`

### Why `Label` Members Are Value-Less

The protocol Label is a **plain class** (`pygen.klass(name="Label")` with no bases, `gsm2tree.py:476`) whose members are bare `ClassVar[object]` annotations (`gsm2tree.py:483`):

```python
class Label:
    RULE: typing.ClassVar[object]
```

The comment at `gsm2tree.py:478-483` explains: `ClassVar[Label]` causes pyright to flag `reportUndefinedVariable` inside the nested class body (self-referential annotation), so `ClassVar[object]` is used instead. The result is that Protocol Label members carry **no runtime value at all** — they are pure annotations with no assignment.

### NodeKind Import in Protocol Module

`gen_protocol_module()` emits a `TYPE_CHECKING`-guarded import of `NodeKind` from the concrete module (`gsm2tree.py:444-454`):

```python
if typing.TYPE_CHECKING:
    from fltk.fegen.fltk_cst import NodeKind
```

This means `NodeKind` is available for the `Literal[NodeKind.X]` annotation **at type-check time only**. At runtime, the protocol module has no `NodeKind` import. This is safe because `from __future__ import annotations` (emitted at `gsm2tree.py:438`) makes all annotations lazy strings.

### Generated Protocol Module (fltk_cst_protocol.py) — Top-Level Shape

File: `fltk/fegen/fltk_cst_protocol.py`

- Line 1: `# ruff: noqa: N802` — file-level suppression (already present, not consumer-induced)
- Lines 8-9: `if typing.TYPE_CHECKING: from fltk.fegen.fltk_cst import NodeKind`
- Lines 12-638: one Protocol class per grammar rule, then `CstModule`
- Each nested `Label` class: plain class, members are `ClassVar[object]` with **no value** at runtime — `fltk_cst_protocol.py:14`, `39`, `77`, etc.

## 2. Concrete Module `fltk_cst.py`: Enums, NodeKind, Canonical-Name Equality

### NodeKind Enum

`fltk_cst.py:8-52`. Module-level `class NodeKind(enum.Enum)` with one member per grammar rule (e.g., `GRAMMAR = enum.auto()`). Members are assigned `_fltk_canonical_name` as plain string attributes post-class:

```python
NodeKind.GRAMMAR._fltk_canonical_name = "NodeKind.GRAMMAR"  # fltk_cst.py:39
```

Generator: `CstGenerator._node_kind_enum()` — `gsm2tree.py:134` and `_emit_node_kind_canonical_name_assignments()` — `gsm2tree.py:145`.

### Per-Node Label Enums

Each node class (`Grammar`, `Rule`, etc.) contains a nested `class Label(enum.Enum)` — e.g., `fltk_cst.py:57-73`. Members assigned `_fltk_canonical_name` post-class:

```python
Grammar.Label.RULE._fltk_canonical_name = "Grammar.Label.RULE"  # fltk_cst.py:114
```

Generator: `CstGenerator.py_class_for_model()` — `gsm2tree.py:192`, calls `_emit_cross_backend_eq_hash()` — `gsm2tree.py:100` and `_emit_label_canonical_name_assignments()` — `gsm2tree.py:158`.

### Cross-Backend `__eq__`/`__hash__` Mechanism

Both `NodeKind` and each `ClassName.Label` enum carry the same `__eq__`/`__hash__` pattern generated by `_emit_cross_backend_eq_hash()` — `gsm2tree.py:100-132`:

1. **Same-object:** `if other is self: return True`
2. **Same-type:** `if type(other) is type(self): return self.name == other.name`
3. **Cross-type (canonical-name bridge):** `cn = getattr(other, '_fltk_canonical_name', None); if cn is not None: return self._fltk_canonical_name == cn`
4. **Foreign:** `return NotImplemented`

`__hash__` returns `hash(self._fltk_canonical_name)` — `gsm2tree.py:131`.

**Canonical name format:**
- `NodeKind`: `"NodeKind.<UPPER>"` — `gsm2tree.py:155`
- Label: `"<ClassName>.Label.<UPPER>"` — `gsm2tree.py:167`

A **third enum set** (e.g., enums in the protocol module) would compare-equal to both backends **if and only if** it also sets `_fltk_canonical_name` to the exact same string and exposes it as an attribute (either via `@property` or plain attribute). The `getattr` lookup is unconditional on type.

## 3. `fltk2gsm.py`: Current Cleanliness Violations

File: `fltk/fegen/fltk2gsm.py`

### The Dual-Import Pattern

```python
from fltk.fegen import fltk_cst as cst          # fltk2gsm.py:8 — runtime import
if TYPE_CHECKING:
    from fltk.fegen import fltk_cst_protocol as cst  # fltk2gsm.py:12 — shadows at type-check time
```

- Runtime binding: `fltk_cst` (concrete module, has enum values).
- Type-checking binding: `fltk_cst_protocol` (Protocol module, has type-correct annotations).
- The `TYPE_CHECKING` guard and the name-shadowing (`as cst` in both) are the core cleanliness violation: a consumer is **forced** to maintain two imports of the same logical name, one for types and one for runtime values.

### `typing.cast` Usages

Two `typing.cast` calls forced by `children` being `list[tuple[Optional[Label], child_union]]` and needing to narrow the child type after label-based filtering:

- `fltk2gsm.py:63`: `item = typing.cast("cst.Item", item)`
- `fltk2gsm.py:75`: `item = typing.cast("cst.Item", item)`

Both occur inside `visit_items()` where the code iterates `items.children` interleaved pairs and needs to assert the child type after confirming its label.

### `# noqa: S101` Suppressions

Multiple `assert` statements use `# noqa: S101` to suppress ruff's S101 (assert-in-non-test) rule:
- `fltk2gsm.py:62`, `fltk2gsm.py:70`, `fltk2gsm.py:74`, `fltk2gsm.py:78`

These are ruff suppressions required by the coding style (assert-in-non-test bodies), but they are not directly caused by the CST surface. They would exist regardless.

### Summary of Violations Caused by the CST Surface

| Line | Violation | Cause |
|------|-----------|-------|
| 8 | `from fltk.fegen import fltk_cst as cst` (runtime) | Enum values not in protocol |
| 11-12 | `if TYPE_CHECKING: from ... fltk_cst_protocol as cst` | Dual-import to get Protocol types |
| 63 | `typing.cast("cst.Item", item)` | children type is Union, narrowing requires cast |
| 75 | `typing.cast("cst.Item", item)` | Same reason |

The `typing.cast` calls are partially caused by the CST surface: `children` is `list[tuple[Optional[Label], Union[...]]]` and after filtering by label, pyright cannot narrow without a cast. If the protocol module had real enum values, the `children_item()` method would be used instead of the manual loop — eliminating the cast need.

## 4. Rust Backend: Enum and NodeKind Emission

File: `fltk/fegen/gsm2tree_rs.py`

### NodeKind (Rust)

`RustCstGenerator._node_kind_block()` — `gsm2tree_rs.py:181`. Emits a `#[pyclass(frozen)] pub enum NodeKind` with one variant per rule, Python-visible name = `class_name.upper()` — `gsm2tree_rs.py:194-197`. Canonical name = `"NodeKind.<UPPER>"` — `gsm2tree_rs.py:148`. Both `__repr__` and `_fltk_canonical_name` getter return this canonical string — `gsm2tree_rs.py:206-216`.

Cross-backend `__eq__`/`__hash__` are in `_emit_rust_cross_backend_eq_hash()` — `gsm2tree_rs.py:151-179`. The Rust `__eq__` does:
1. Same-type fast path via `extract::<NodeKind>()` — `gsm2tree_rs.py:162-163`
2. Canonical-name bridge via `getattr(other, "_fltk_canonical_name")` — `gsm2tree_rs.py:165-171`
3. Returns `py.NotImplemented()` otherwise

`__hash__` uses `hash(PyString::new(py, self.__repr__()))` — `gsm2tree_rs.py:175-179` — matching the Python backend's `hash(self._fltk_canonical_name)`.

### Label Enum (Rust)

`RustCstGenerator._label_enum_block()` — `gsm2tree_rs.py:228`. Emits `{ClassName}_Label` (e.g., `Items_Label`) as a `#[pyclass(frozen)] pub enum`. Python-visible name = `"{ClassName}_Label"`. Each variant's canonical name = `"{ClassName}.Label.{PYTHON_NAME}"` — `gsm2tree_rs.py:265`.

Label enum is exposed as a classattr `Label` on each node class via `_label_classattr()` — `gsm2tree_rs.py:360-369`: returns `{ClassName}_Label::type_object(py)` i.e. the enum **class** itself as an object.

### Key Asymmetry: Rust Label Type Name

The Rust Label enum's Python-visible class name is `"{ClassName}_Label"` (e.g., `Items_Label`), not `"{ClassName}.Label"`. It is accessed as `Items.Label` via the `#[classattr]` property, but `type(Items.Label.ITEM).__name__ == "Items_Label"`. This is relevant if protocol enums need `isinstance` checks.

### Cross-Backend Compatibility of a Third Enum

A third enum in the protocol module would be compatible with both backends for equality/hashing **iff** it exposes `_fltk_canonical_name` matching the canonical string format. The `getattr` in both Python (`gsm2tree.py:122`) and Rust (`gsm2tree_rs.py:165`) implementations is duck-typed: it does not care about the object's type, only the attribute value.

## 5. Constraints and Risks

### Constraint: `typing.runtime_checkable` Not Used

Current Protocol classes are **not** decorated with `@typing.runtime_checkable`. This is deliberate: runtime Protocol checking is expensive and was not needed. Adding enum values to the protocol module does not require adding `runtime_checkable`.

### Risk: Enum Identity Across Modules

If the protocol module defines its own `Label` enum (a third set), comparisons using `is` (identity) rather than `==` (equality via canonical-name bridge) will fail between protocol-module enum members and concrete-module enum members. The existing code in `fltk2gsm.py` uses `==` throughout (e.g., `fltk2gsm.py:46`, `51`, `65`, `70`), so `==` is the contract. The canonical-name bridge handles cross-module `==`. **But**: any downstream code using `is` or `match`/`case` with enum members would break if they receive protocol-module enum instances that are a different type.

### Risk: `Literal[NodeKind.X]` Annotation in Protocol

Protocol classes declare `kind: typing.Literal[NodeKind.X]`. `NodeKind` is imported only under `TYPE_CHECKING`. If protocol-module enum members were runtime-importable, the `NodeKind` import would also need to be runtime. Today it is lazy-string under `from __future__ import annotations`, so no runtime cost. If the protocol module gains a runtime `NodeKind` (or its own `NodeKind` equivalent), the import dependency changes.

### Risk: Public-API Breaking Change

The generated `fltk_cst_protocol.py` is **public API for out-of-tree consumers** (CLAUDE.md). Any change to the protocol module's Label members from `ClassVar[object]` (annotation-only, no value) to an enum or assigned value is a **structural change** to the generated public API. Existing downstream code that references `SomeNode.Label.MEMBER` may rely on the current type (`object`) in ways that break if the type becomes `LabelEnum`. Additionally, if the protocol module previously had no runtime `NodeKind`, adding one is an additive but potentially surprising dependency change.

### Risk: `# ruff: noqa: N802` on Protocol Module

`fltk_cst_protocol.py:1` has `# ruff: noqa: N802` (function names should be lowercase) — this is a **file-level suppression already in the generated output**, not consumer-induced. It is there because Protocol method names like `Grammar()` (the `@property` in `CstModule`) are capitalized. Any change to the protocol module must keep this or eliminate the need for it.

### Risk: T2b Test Confirms Cast Is Required Today

`test_cst_protocol.py:359-376` (test `test_boundary_probe_documents_label_mismatch`) explicitly asserts that assigning `fltk_cst` (concrete module) directly to a `cstp.CstModule`-typed variable produces pyright errors — because the nested `Label(enum.Enum)` in the concrete module does not structurally match the plain `class Label` in the Protocol. This test would need to be revisited if the protocol's Label becomes an enum or if the concrete Label changes to match the protocol's plain class.

### Constraint: Children-Loop Cast Pattern in `fltk2gsm.py`

`fltk2gsm.py:61-76` manually iterates `items.children` in interleaved (item, separator) pairs. This pattern bypasses `children_item()` because it needs to process separator labels too. Even if enums existed in the protocol module, the `typing.cast` calls at lines 63 and 75 would remain unless the `children` type annotation became more precise (e.g., via overloaded return types for the per-label methods). The per-label `children_item()` method on the Protocol already returns `typing.Iterator[Item]` with no cast needed — the cast is a result of using `.children` directly rather than `children_item()`.

### Open Factual Question: Why Does `visit_items` Use `.children` Directly?

`fltk2gsm.py:61-76` iterates `items.children` directly rather than calling `children_item()`, `children_no_ws()`, etc. This is because the code needs to process **interleaved** item/separator pairs in order, preserving ordering between items and their following separator labels. The per-label accessor methods would lose ordering. This means the `typing.cast` at lines 63 and 75 is inherent to the algorithm, not an artifact of missing enum values — it would persist even with protocol-module enums unless the `children` type annotation were narrowed.

### Open Factual Question: Does `fltk2gsm.py` Require Label Enum Values at All?

The `fltk2gsm.py` dual-import exists primarily so that `cst.Items.Label.NO_WS` etc. resolve as runtime values for comparisons (lines 46, 51, 65, 70). If those Label enum members existed in `fltk_cst_protocol.py` with real values (not just `ClassVar[object]` annotations), `fltk2gsm.py` could drop the runtime `fltk_cst` import and rely solely on `fltk_cst_protocol`. The `NodeKind` issue is secondary — `NodeKind` is not directly referenced in `fltk2gsm.py`; only Label members are.

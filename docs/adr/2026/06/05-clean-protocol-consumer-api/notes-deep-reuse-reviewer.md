**What's duplicated:** `_emit_protocol_label_member_class` emits `_ProtocolLabelMember.__eq__`/`__hash__` by calling `ast.parse` on a hardcoded source string. `_emit_cross_backend_eq_hash` (`gsm2tree.py:100–132`) already exists to generate exactly this bridge shape as AST nodes, and is called for both `NodeKind` and `Label` enums in `gen_py_module`.

**Existing utility:** `CstGenerator._emit_cross_backend_eq_hash` — `fltk/fegen/gsm2tree.py:100`.

**Consequence:** The two implementations of the cross-backend `__eq__`/`__hash__` bridge can drift independently. Today there is already a divergence: `_emit_cross_backend_eq_hash` generates `type(other) is type(self): return self.name == other.name` (enum `.name` comparison), while `_emit_protocol_label_member_class` generates `type(other) is type(self): return self._fltk_canonical_name == other._fltk_canonical_name` (canonical-name comparison). The divergence is currently intentional (non-enum vs enum class), but the string-parse path has no structural coupling to the canonical generator, so future edits to one (e.g. adding an `__ne__`, changing the `getattr` sentinel, updating the same-type fast path) will not automatically propagate to the other.

## reuse-2

**File:line:** `fltk/fegen/pyrt/terminalsrc.py:15–23` (`SpanKind.__eq__`)

**What's duplicated:** `SpanKind.__eq__` hand-implements the cross-backend bridge. The same bridge shape is generated by `_emit_cross_backend_eq_hash` for every `NodeKind` and `Label` enum in both concrete and protocol modules. The hand-written `SpanKind` version uses `isinstance(other, SpanKind)` for the same-type fast path while every generated version uses `type(other) is type(self)`. These are not equivalent: `isinstance` matches subclasses; `type(...) is type(...)` does not.

**Existing utility:** `CstGenerator._emit_cross_backend_eq_hash` — `fltk/fegen/gsm2tree.py:100`.

**Consequence:** The `isinstance` vs `type(…) is type(…)` asymmetry is a latent behavioral divergence. If a subclass of `SpanKind` is ever introduced (unlikely but not impossible), `isinstance` would match it while `type(…) is type(…)` would not, breaking the symmetry AC 7 requires. More practically: when the bridge is updated for any reason (new edge case, stricter `NotImplemented` propagation, etc.), `SpanKind` will be missed because it is hand-written outside the generator's scope, and there is no test that exercises the reflected operand-order equality specifically for `SpanKind` vs a foreign bridge object (the AC 7 tests cover `NodeKind` and `Label` members, not `SpanKind`).

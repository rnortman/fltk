We have (I hope) completed Rust CST codegen. It is time to design and plan Rust parser codegen. There should be some ADR docs from a while back about the parser; the explorer needs to find these, but they are not necessarily controlling. Many things have changed since we wrote those. Overall goal: fltk generates both Python parsers (as it already does, unchanged) and Rust parsers. The Rust parsers can be used directly by Rust code *or* can be used from python. The CST already supports both Rust and Python users with the same codegen; for parsing it is acceptable to generate *different* parser classes for Rust and Python, because nobody ever needs to pass the parser itself across a boundary. Parsers generate CST; the CST is the thing that crosses boundaries. Just like with CST, a pure-Rust consumer should not pay a price for python; the python bindings should only be linked in with a python consumer. The pattern used by CST to support this is fine, or separate classes is also fine. Designer chooses.

## 2. Python Parser Generation Path

### 2.1 Entry Point

`fltk/fegen/genparser.py` is the CLI and orchestration layer. The `generate` command (line 119) produces three files per grammar:

- `<base>_cst.py` — via `gsm2tree.CstGenerator.gen_py_module()`
- `<base>_parser.py` — via `generate_parser(..., preserve_trivia=False)`
- `<base>_trivia_parser.py` — via `generate_parser(..., preserve_trivia=True)`

`generate_parser` (genparser.py:72) instantiates `gsm2tree.CstGenerator` (for type info) and `gsm2parser.ParserGenerator` (which builds the IIR in its `__init__`), then calls `compiler.compile_class(pgen.parser_class, context)` (via `fltk/iir/py/compiler.py:77`) to produce `ast.ClassDef`, wraps in an `ast.Module` with imports, and writes `ast.unparse(parser_mod)` to file.

The `gen-rust-cst` command (genparser.py:264) calls `gsm2tree_rs.RustCstGenerator(grammar).generate()` — a separate, Rust-specific generator that emits `.rs` source directly as a string. There is no analogous `gen-rust-parser` command yet.

### 2.2 `gsm2parser.py` — Structure

**File:** `fltk/fegen/gsm2parser.py` (832 lines as of current HEAD)

`ParserGenerator.__init__` (line 27) does all generation work: builds an `iir.ClassType` named `"Parser"`, populates it with fields and methods via the IIR proxy-builder API. The IIR class is later compiled to Python by `py/compiler.py`.

**Key `CompilerContext` flag:** `context.capture_trivia: bool` (context.py:47). When `True`, trivia nodes are appended to CST node children in separator-handling code. The two parser variants (`_parser.py` vs `_trivia_parser.py`) differ only in this flag — genparser.py:85.

**Parser fields registered on the IIR class:**

- `packrat: Packrat[int, int]` — gsm2parser.py:54-65
- `terminalsrc: TerminalSource` — gsm2parser.py:67-74
- `_source_text: SourceText` — gsm2parser.py:85; initialized from `terminalsrc.terminals` in the constructor init list — gsm2parser.py:107-131
- `error_tracker: ErrorTracker[int]` — gsm2parser.py:92-97
- `rule_names: Sequence[str]` — gsm2parser.py:99-103
- `_cache__parse_<rule>: MutableMapping[int, MemoEntry[int, int, <CstNodeType>]]` — one per rule, generated by `_gen_parser_callable` — gsm2parser.py:464-472

**Constructor:** single parameter `terminalsrc: TerminalSource` (owning). Sets `_source_text = SourceText(text=terminalsrc.terminals)` — gsm2parser.py:119-131.

**The `_make_span_expr` method** (gsm2parser.py:254-276): emits `Span.with_source(start, end, self._source_text)` rather than bare `Span(start, end)`. This is the portable form that works on both the Python and Rust Span backends. The import path for `Span` is resolved from the type registry (`fltk.fegen.pyrt.span.Span`) — context.py:113-123.

**Method call chain:**
```
ParserGenerator.__init__
  → _make_parser_info (pre-register, one pass per rule)
  → gen_alternatives_parser (per rule, second pass)
    → gen_alternative_parser (per alternative)
      → gen_item_parser (per item)
          → gen_item_parser_single_or_optional  OR  gen_item_parser_multiple
            → _gen_consume_term_expr → ConsumeTermInfo(expr, result_type, inline_to_parent)
      → _gen_separator_handling (per separator, per item)
```

**`_gen_consume_term_expr`** (gsm2parser.py:303-375): dispatches on `gsm.Term` type:
- `gsm.Identifier` → `self.apply__<rule_name>(pos=pos)` — calls memoised rule parser
- `gsm.Literal` → `self.consume_literal(pos=pos, literal="...")`
- `gsm.Regex` → `self.consume_regex(pos=pos, regex="...")`
- `Sequence[Items]` (sub-expression) → recursively creates `gen_alternatives_parser`, `inline_to_parent=True`
- Anything else (e.g. `gsm.Invocation`) → `raise NotImplementedError` — line 374-375

**`_gen_separator_handling`** (gsm2parser.py:606-683): three cases:
- `NO_WS` → no-op
- trivia rule context: uses raw `consume_regex(r"\s+")` to avoid recursion — lines 641-659
- non-trivia context: calls `self.apply__parse__trivia(pos=pos)` — lines 661-683
- If `WS_REQUIRED` and trivia fails: return `Failure` — line 682-683

**Naming conventions for generated methods:**
- `parse_<rule>` — actual parse body
- `apply__parse_<rule>` — memoisation wrapper
- `parse_<rule>__alt<N>` — per-alternative parser (not memoised, called directly)
- `parse_<rule>__alt<N>__item<M>` — per-item parser (not memoised)
- `_cache__parse_<rule>` — `MutableMapping` field
- `apply__parse_<rule>__alt<N>__item<M>` — if `memoize=True`; currently only top-level rules are memoised

**`_gen_parser_callable`** (gsm2parser.py:407-473): creates an `iir.Method` (the parse body) plus optionally a second memoizer `iir.Method` and a `_cache__` field. The memoizer body is:
```python
return self.packrat.apply(
    rule_callable=self.<name>.bind(),
    rule_id=<int>,
    rule_cache=self._cache__<name>,
    pos=pos,
)
```

### 2.3 Generated Parser Shape

Concrete example from `fltk/fegen/fltk_parser.py`:

```python
class Parser:
    def __init__(self, *, terminalsrc: TerminalSource):
        self.terminalsrc = terminalsrc
        self._source_text = SourceText(text=terminalsrc.terminals)
        self.packrat: Packrat[int, int] = Packrat()
        self.error_tracker: ErrorTracker[int] = ErrorTracker()
        self.rule_names: Sequence[str] = [...]
        self._cache__parse_grammar: MutableMapping[int, MemoEntry[int, int, Grammar]] = {}
        # ... one cache per rule

    def parse_grammar(self, pos: int) -> ApplyResult[int, Grammar] | None:
        if alt0 := self.parse_grammar__alt0(pos=pos):
            return alt0
        return None

    def apply__parse_grammar(self, pos: int) -> ApplyResult[int, Grammar] | None:
        return self.packrat.apply(
            rule_callable=self.parse_grammar,
            rule_id=0,
            rule_cache=self._cache__parse_grammar,
            pos=pos,
        )

    def parse_grammar__alt0(self, pos: int) -> ApplyResult[int, Grammar] | None:
        result: Grammar = Grammar(span=Span.with_source(pos, -1, self._source_text))
        if initial_ws := self.apply__parse__trivia(pos=pos):
            pos = initial_ws.pos
        if item0 := self.parse_grammar__alt0__item0(pos=pos):
            pos = item0.pos
            result.extend_children(item0.result)   # inline_to_parent case
        else:
            return None
        result.span = Span.with_source(_span_start, pos, self._source_text)
        return ApplyResult(pos=pos, result=result)
```

### 2.4 Parser Runtime

**`fltk/fegen/pyrt/memo.py`** (257 lines):
- `ApplyResult(pos, result)` — frozen dataclass; return type of all parse methods
- `MemoEntry(result, final_pos)` — cache value; `result` may be `Poison`, actual result, or `None` (failure)
- `Poison(recursion_info)` — placed in cache at recursion entry to detect left-recursion
- `RecursionInfo(rule_id, involved, eval_set)` — tracks active left-recursive cycle
- `Packrat[RuleId, PosType]` — core packrat engine; `invocation_stack: list[RuleId]`; `apply(rule_callable, rule_id, rule_cache, pos)` — seed-growing Warth/Douglass/Millstein algorithm (memo.py:82-156)

**`fltk/fegen/pyrt/terminalsrc.py`** (206 lines as of current HEAD):
- `TerminalSource(terminals: str)` — `consume_literal(pos, literal) -> Span | None`; `consume_regex(pos, regex) -> Span | None`; `pos_to_line_col(pos) -> LineColPos`
- `Span(start, end)` — frozen dataclass; fields `start: int`, `end: int`, `_source: str | None`; `with_source(start, end, source)` classmethod
- `UnknownSpan: Final = Span(-1, -1)` — sentinel
- `SourceText(_text: str)` — immutable wrapper; `SourceText(text)` constructor

**`fltk/fegen/pyrt/span.py`** (21 lines) — backend selector:
```python
from fltk.fegen.pyrt.terminalsrc import SourceText, Span, UnknownSpan
try:
    from fltk._native import SourceText, Span, UnknownSpan
except Exception:
    warnings.warn(...)
```
Generated parsers import `Span` and `SourceText` from `fltk.fegen.pyrt.span`, so they pick up the Rust `Span` when `fltk._native` is available — genparser.py:101, context.py:117.

**`fltk/fegen/pyrt/errors.py`** (71 lines):
- `ErrorTracker[RuleId]` — tracks farthest-right parse failure; `fail_literal(pos, rule_id, literal)`, `fail_regex(pos, rule_id, regex)`
- `format_error_message(tracker, terminals, rule_name_lookup) -> str` — human-readable error with line/col

### 2.5 How Parsers Build CST Nodes

The parser calls exactly these operations on CST nodes (Python backend):

1. `NodeClass(span=Span.with_source(start, -1, self._source_text))` — construct with placeholder end span
2. `result.span = Span.with_source(span_start, pos, self._source_text)` — set final span on completion
3. `result.append_<label>(child=...)` — add labeled child (typed, per-label)
4. `result.append(child=..., label=None)` — add unlabeled child (trivia case)
5. `result.extend_children(other)` — copy all children from another node (inline_to_parent case)
6. `result.extend_<label>(children=...)` — not directly used by parser; used by `children.extend(...)` in inline cases

Rust CST data struct API (as of Phase 2, `src/cst_generated.rs`):

1. `NodeClass::new(span: Span) -> Self` — construct with span
2. `node.set_span(span: Span)` — replace span
3. `node.append_<label>(child)` — typed labeled append (e.g., `append_name(span: Span)` or `append_item(child: impl Into<Shared<Identifier>>)`)
4. `node.push_child(label: Option<NodeLabel>, child: NodeChild)` — untyped append (covers unlabeled/trivia case)
5. `node.extend_children(other: &Self)` — Arc-clone all children from other into self

The Rust data struct constructors require a `Span` from `fltk_cst_core::Span` (not `fltk._native.Span` — those are the same type, but the constructor is GIL-free). A Rust parser builds `Span` directly: `Span::new_with_source(start, end, &source_text)` or `Span::new_sourceless(start, end)`.

## 3. Just-Completed Rust CST Codegen

### 3.1 Generator

**`fltk/fegen/gsm2tree_rs.py`** — `RustCstGenerator` class. Generates a complete `.rs` file as a string (no IIR; direct string emission). Invoked via `genparser gen-rust-cst` CLI command.

`generate() -> str` (line 222 approximately): calls `_preamble()`, `_node_kind_block()`, per-rule `_label_enum_block()` / `_child_enum_block()` / `_node_block()`, `_register_classes_fn()`, `_cst_module_fn()`.

Also `generate_pyi(protocol_module: str) -> str` — emits a `.pyi` stub for pyright.

### 3.2 Handle/Data Split (Phase 1 + 2)

Each generated node type consists of two Rust items:

**Data struct** (`pub struct Identifier`, always compiled):
- `span: Span` (private field)
- `children: Vec<(Option<IdentifierLabel>, IdentifierChild)>` (private field)
- Native API (no `#[cfg]` gate): `new(span) -> Self`, `kind() -> NodeKind`, `span() -> &Span`, `set_span(span)`, `children() -> &[...]`, `push_child(label, child)`, `child() -> Result<..., CstError>`, `extend_children(&other)`, per-label quintet (`children_<lbl>`, `child_<lbl>`, `maybe_<lbl>`, `append_<lbl>`, `extend_<lbl>`)

**Handle pyclass** (`#[cfg(feature = "python")] #[pyclass(frozen, weakref)] pub struct PyIdentifier`):
- Single field: `inner: Shared<Identifier>` (private)
- `shared() -> &Shared<Identifier>` — Rust-side access to the inner `Shared`
- `to_py_canonical(py, s: &Shared<Identifier>) -> PyResult<Py<PyIdentifier>>` — registry lookup or mint
- `#[pymethods]` block: `new(py, span)`, `span` getter/setter, `kind`, `Label` classattr, `children` getter, `append`, `extend`, `extend_children`, `child`, per-label quintet, `__eq__`, `__hash__`, `__repr__`

The split was introduced to eliminate the `_native` suffix collision: pymethod block uses natural Python names (`fn span`, `fn new`, `fn children`); data struct can also use those names without collision because they are on different structs.

The split also enables `Shared<T>` ownership.

### 3.3 `Shared<T>`

**`crates/fltk-cst-core/src/shared.rs`**:
```rust
pub struct Shared<T>(Arc<RwLock<T>>);
```
- `new(value: T)`, `read() -> RwLockReadGuard`, `write() -> RwLockWriteGuard` (both poison-transparent)
- `ptr_eq(&self, other: &Self) -> bool` — pointer equality, used as short-circuit in `PartialEq`
- `arc_ptr() -> usize` — used as registry key
- `Clone` = shallow (Arc clone); `PartialEq` = ptr_eq short-circuit then deep value comparison; `From<T>` = `Shared::new(value)`

Node-typed child variants: `ItemsChild::Identifier(Shared<Identifier>)` rather than `Box<Identifier>`. Arc clone on every child-ref, not deep copy. This matches Python's reference semantics.

### 3.5 Cross-cdylib ABI Concerns

**`crates/fltk-cst-core/src/cross_cdylib.rs`** (python-only, entirely `#[cfg(feature = "python")]`):

- `get_span_type(py)` — imports `fltk._native.Span`, caches in `GILOnceCell`, verifies ABI string + layout probe against locally-compiled values before caching
- `extract_span(py, obj)` — fast path: `obj.extract::<Span>()`; slow path: `isinstance(fltk._native.Span)` + `downcast_unchecked` (with ABI sentinel check). The ABI check compares `_fltk_cst_core_abi` classattrs (string `"fltk-cst-core/<version>"`) and layout probes (`size_of::<PyClassObject<Span>>()`).
- `span_to_pyobject(py, span)` — fast path: type_object identity match (same cdylib); slow path: calls `Span._with_source_unchecked(start, end, source)` on the canonical type

ABI sentinel constants: `FLTK_CST_CORE_ABI: &str = "fltk-cst-core/0.1.0"` (baked at compile time). The layout probe (`size_of::<PyClassObject<T>>()`) catches pyo3-resolution skew.

`Span` has `_fltk_cst_core_abi` and `_fltk_cst_core_abi_layout` classattrs (span.rs:375-393); so does `SourceText` (span.rs:97-122). Generated node types do **not** carry ABI markers — cross-cdylib node passing is permanently out of scope (design.md §3).

### 3.6 Canonical Wrapper Registry

**`crates/fltk-cst-core/src/registry.rs`** (python-only):

A `WeakValueDictionary`-based registry mapping `usize` arc-pointer → `PyObject`. Ensures the same `Shared<T>` allocation always returns the same Python handle object (identity stability).

- `get_or_insert_with(py, addr, factory)` — look up or mint and register
- `force_register(py, addr, obj)` — register a newly-minted handle as canonical (used in `PyNodeName::new`)
- `register_if_absent(py, addr, obj)` — register on hand-in from Python (used in `extract_from_pyobject`)

Handles are weak references; GC may evict them. On next Python read, a new handle is minted pointing to the same `Shared<T>`.

### 3.7 CST Construction Interface for a Rust Parser

A Rust parser generating into the current Rust CST data structs would use:

```rust
use fltk_cst_core::{Span, SourceText};
use crate::cst::{Grammar, GrammarChild, GrammarLabel, Rule, ...};

// TerminalSource equivalent (Rust-side — needs implementing)
// let source = SourceText::from_str(input);

// Node construction
let mut node = Grammar::new(Span::new_with_source(pos, end, &source));

// Typed append
node.append_rule(rule_shared);           // impl Into<Shared<Rule>>
node.append_name(name_span);             // Span

// Untyped append (trivia, unlabeled)
node.push_child(None, GrammarChild::Trivia(trivia_shared));

// Inline-to-parent (extend_children replaces append for sub-expression results)
node.extend_children(&sub_result);

// Set final span
node.set_span(Span::new_with_source(span_start, pos, &source));
```

No GIL required. The data struct methods are all `pub`, GIL-free, and take no `py: Python` parameter.

### 5.3 `CompilerContext` for Parser Generation

`fltk/iir/context.py:42-47`:

## 6. Current State: What Exists for Rust Parser Codegen

**Nothing exists yet.** There is no `gsm2parser_rs.py`, no Rust parser runtime, no `gen-rust-parser` CLI command. The immediate groundwork is the Rust CST, which is complete.

The Rust CST data struct API (`new`, `set_span`, `push_child`, `extend_children`, `append_<lbl>`, etc.) is the exact interface a Rust parser would call when building a CST. This API is GIL-free, pure-Rust, and intentionally designed with the parser use case in mind (design.md §2, R4: "R4 — parser-ready construction API, improved: `new`/`push_child`/`extend_children` on the data struct, GIL-free").

The Python parser runtime (`memo.py`, `terminalsrc.py`, `errors.py`) has no Rust equivalent. A Rust parser needs a Rust `TerminalSource` (consuming `&str`, returning `Span`), a Rust `Packrat` implementing the seed-growing algorithm, and a Rust `ErrorTracker`. None exist.

## 7. Open Factual Questions

1. **Python parser wrapping a Rust parser**: the task description says "can be used from python". The CST already supports this via handle pyclass + `#[pymethods]`. The question is what the Python-visible parser API looks like — does `PyParser.parse(text) -> PyGrammar` suffice, or does it need to match the Python-generated `Parser` interface precisely (e.g., `Parser(terminalsrc=TerminalSource(text))`, `apply__parse_grammar(0)`, `.error_tracker`, `.rule_names`)?

2. **Trivia parser variant**: The Python side generates two parser classes (`_parser.py` and `_trivia_parser.py`). Does the Rust side need the same pair, or is one variant sufficient?

3. **How a generated Rust parser crate is compiled**: The CST fixture crates (`tests/rust_cst_fixture/`, `tests/rust_cst_fegen/`) are standalone out-of-workspace cdylibs built by Makefile. A Rust parser used from Python needs the same cdylib treatment. If the parser crate is compiled into `fltk._native` (in-tree for fltk's own grammar), the plumbing exists. If it is an out-of-tree consumer extension, it needs the same manifest pattern as the fixture crates plus a Rust parser runtime dependency.

4. **Rust parser runtime as crate or in-crate**: The Rust packrat runtime could live in `fltk-cst-core` (requiring a version bump and re-export), in a new `fltk-parser-core` crate, or generated inline per-grammar. The CST precedent (`fltk-cst-core`) argues for a shared runtime crate.

5. **Regex handling**: Python uses `re.compile(regex).match(terminals, pos=pos)`. The Rust equivalent requires a regex crate (`regex` crate, pre-compiled `Regex` instances). Regex compilation at call time vs. pre-compiled statics is a structural difference. The Python generator already notes `# TODO pre-compile regexes` (gsm2parser.py:341).

6. **Position type**: Python uses `int` (codepoint index into `str`). Rust `Span` fields are `i64` (codepoints, matching Python). A Rust parser would track position as `i64` or `usize`. The CST boundary (`Span::new_with_source(start: i64, end: i64, ...)`) uses `i64`.

## Background

`src/cst_generated.rs` (the PoC Rust CST) is regenerated by an inline Python one-liner at `Makefile:99-103` that imports `_make_poc_grammar` from `tests/test_gsm2tree_rs.py` and calls `RustCstGenerator(...).generate()` directly. Every other generated Rust CST goes through the declarative path: a `.fltkg` grammar file fed to `fltk.fegen.genparser gen-rust-cst` (see `Makefile:105-108` for `src/cst_fegen.rs` and `Makefile:110` for the fixture crate).

Validation (see `exploration.md` in this dir) established ground truth:

- The PoC grammar is fully expressible in `.fltkg`. The following two-rule file parses to a GSM field-for-field identical to `_make_poc_grammar()` and produces a **byte-identical** `src/cst_generated.rs` through `gen-rust-cst` (diff verified against the committed file):

```
identifier := name:$/[_a-z][_a-z0-9]*/ ;
items := no_ws:$"." . ws_allowed:$"," , ws_required:$":" : item:$identifier ;
```

- `_parse_grammar_raw` (`fltk/fegen/genparser.py:254-261`) deliberately skips trivia processing because `RustCstGenerator.__init__` applies it internally (`fltk/fegen/gsm2tree_rs.py:40`) — both paths reach the generator with the same pre-trivia GSM.
- `_make_poc_grammar()` (`tests/test_gsm2tree_rs.py:29-96`) is used by 8+ generator unit tests (lines 136, 485, 491, 498, 505, 749) and must stay.
- The original TODO claimed the one-liner misses preamble changes; that is false (both paths share `_preamble()`). The real value here is a single declarative source of truth and regen-path uniformity.

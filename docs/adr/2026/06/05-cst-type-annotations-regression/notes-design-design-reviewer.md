## Findings

### design-1 — `py_annotation_for_model_types` reuse glosses over a required name remapping
Section "New generated artifact" (design.md:29): "Child/label value types reference *other Protocols* … reusing the existing `py_annotation_for_model_types` machinery (`gsm2tree.py:85-93`) retargeted to emit Protocol names."

What's wrong: as written, `py_annotation_for_model_types` (`gsm2tree.py:85-93`) emits the *concrete* class names produced by `class_name_for_rule_node` (`Rule`, `Items`, `Alternatives`, …) and `Span` for literal/regex children — it has no knowledge of a `<Node>Node` suffix. The design's protocol classes are named `RuleNode`, `ItemsNode`, etc. (design.md:25). "Retargeted" hides a non-trivial transform: every child/label union member name must be rewritten to its `…Node` protocol name, while leaving non-node members (`terminalsrc.Span`) un-suffixed. The method builds names via `iir_type_to_py_annotation` over `iir.Type` objects registered with the concrete name (`gsm2tree.py:76` registers `name=class_name_for_rule_node(...)`); there is no node-vs-Span discriminator at the string level.

Consequence: implementer either (a) under-budgets this as a trivial reuse and produces a protocol module whose member annotations name `Rule`/`Items` (undefined in the protocol module → pyright `reportUndefinedVariable`, failing B5), or (b) must build a parallel annotation path. Either way the "reuse machinery" framing understates the work and risks a generated module that does not typecheck. Span (`terminalsrc.Span`) and node names need different treatment.

Fix: state explicitly that protocol generation needs a node-name→protocol-name mapping (suffix `Node`) applied only to rule-node types, with `Span`/`terminalsrc.*` passed through unchanged; either a new annotation method or a parameter on the existing one. Note that `Span` literal/regex children (`self.Span.key`, `gsm2tree.py:253`) stay `Span`, not `SpanNode`.

### design-2 — `Label` enum equality across backends is asserted, not shown; it is the load-bearing risk for B1's Rust acceptance and is under-analyzed
Section "fltk2gsm.py changes" (design.md:44) and "edge cases": design types `self.cst.Items.Label.NO_WS` etc. via a "nested `Label` enum surface … typed as the concrete enum is structurally used: equality against `self.cst.<Node>.Label.<NAME>`" (design.md:28).

What's wrong: `visit_items`/`visit_disposition`/`visit_quantifier` do `label == self.cst.Items.Label.NO_WS`, `if label == self.cst.Disposition.Label.INCLUDE`, etc. (`fltk2gsm.py:44-50,63-68,114-130`), where `label` comes off `items.children`/`disposition.child()` typed as `Label | None` from the *node's own* Label. For this to typecheck non-trivially (and for B4's "deliberately-wrong access flagged"), the protocol's per-node `Label` member and the `children` tuple's `Label` element must be the *same* static type. A `Protocol`-nested `class Label` is a real concern: pyright treats a Protocol member class as a nominal type, and two backends' `Label` enums are distinct nominal types — but the comparison is `==`, which pyright permits across types without error (always allowed, never flagged). So the equality "typechecks" trivially but provides *no* safety: a typo'd label constant would still be flagged only if it is an undefined attribute, not if it is a wrong-but-existing enum member. The design does not spell out how `Label` is represented in the Protocol (enum subclass? bare class with class-var members? `int`?), yet B4 acceptance ("a deliberately-wrong access … is flagged") leans on it.

Consequence: the `Label` surface is the single most intricate part of the structural-Protocol approach and the design leaves its representation unspecified ("typed as the concrete enum is structurally used" is not a concrete prescription). Implementer may discover late that (a) a Protocol cannot cleanly declare a nested enum that both a Python `enum.Enum` subclass and a PyO3 class satisfy structurally, and/or (b) the resulting type gives weaker checking than B4 implies. This drives whether B1's "no implicit Any / attribute access resolves" holds for the label-heavy methods, which are the bulk of `visit_items`.

Fix: specify the exact static form of the per-node `Label` in the Protocol (e.g. a nested `class Label:` with `NO_WS: <SomeType>` class attributes, and the element type of `children`'s tuple). State what wrong-label safety B4 actually buys here (likely: attribute-presence only, not value-correctness) so acceptance is not overclaimed.

### design-3 — Module-object-satisfies-`CstModule` is the primary mechanism and is left empirically undetermined; B5/B6 hinge on it
Section "DI boundary" (design.md:53-56) and "Edge cases" (design.md:59): whether pyright accepts the `fltk_cst` *module* assigned to a `cst: CstModule` Protocol parameter is "resolved during implementation."

What's wrong: this is acknowledged as "Primary risk" (design.md:59), but it is not a peripheral edge case — it is the load-bearing mechanism for the *entire* chosen approach. If module-to-Protocol structural matching does not hold, the fallback is "a single documented cast at the default binding" (design.md:55). But that cast is `cast("CstModule", _default_cst)` — casting a concretely-typed module to the Protocol. If pyright would have *rejected* the structural match (because the module genuinely does not satisfy the Protocol — e.g. the `Label`/`type[...]` representation mismatch from design-2), the cast silently masks a real incompatibility, undermining B4's "stub accuracy" guarantee for the default Python backend (the one backend B4 makes *mandatory*). A cast that hides a true structural mismatch is exactly the `cast(Any)` escape B5 forbids in spirit, dressed as the permitted single boundary cast.

Consequence: the design's confidence that "either way `visit_*` bodies are fully typed" (design.md:59) is only valid if the Protocol is *actually* structurally satisfied by the module; the boundary cast does not make the bodies safe, it makes pyright *stop checking* the boundary. If the real reason a cast is needed is a Protocol/backend mismatch, B4 (Python-mandatory) silently fails. The design should not treat "module matches Protocol" and "use a cast" as interchangeable success paths — only the first satisfies B4.

Fix: require, as a gating step before declaring the approach viable, an empirical pyright check that the `fltk_cst` module structurally satisfies `CstModule` *without* a cast; permit the boundary cast only for the `ModuleType`→Protocol narrowing at the Rust injection site (`plumbing.py:171`, where the value is genuinely `ModuleType`/`Any`), not to paper over a Python-backend structural mismatch. Tie this to Test plan item 2 explicitly as a precondition, not a post-hoc resolution.

### design-4 — `isinstance(item, self.cst.Item)` with a Protocol-derived `type[...]` is a known pyright failure mode; design's mitigation may itself violate B5
Section "Edge cases" (design.md:60): `isinstance(item, self.cst.Item)` (`fltk2gsm.py:61,72`) with `self.cst.Item` typed `type[ItemNode]`. Design notes Protocols aren't valid `isinstance` second args unless `runtime_checkable`, and proposes fallback "type that one member `type[Any]` locally" or "guard with the single boundary cast."

What's wrong: design.md:32 explicitly chooses `Protocol` (not `runtime_checkable`). pyright flags `isinstance(x, P)` where `P` is a non-`@runtime_checkable` Protocol as an error (`reportGeneralTypeIssues`) — and even runtime-checkable protocols only support `isinstance` for non-data-member protocols. `self.cst.Item` typed `type[ItemNode]` where `ItemNode` is a Protocol is therefore a likely hard pyright error at `fltk2gsm.py:61,72`, directly contradicting B5 ("zero new errors") for the very file B1 must fix. The proposed escape ("type that one member `type[Any]` locally") degrades `self.cst.Item` to `type[Any]`, which makes the `isinstance`-narrowed `item` become `Any` — losing the typing B1 demands on exactly the `Item` path, and arguably is the `cast(Any)`-equivalent B5 forbids.

Consequence: two of the methods in the B1 coverage list (`visit_items` uses `isinstance(item, self.cst.Item)`) cannot be both error-free under pyright *and* non-`Any` for `item` under the pure-Protocol mechanism without an unresolved tension. The design lists this as an edge case "verified by Test plan" but does not resolve it; it could force either a `# type: ignore` (B5 violation) or `Any`-degradation (B1 violation) precisely where B1 and B5 intersect. This is more than an edge case — it is a possible dead-end for mechanism (a) on the `Item` path.

Fix: resolve before implementation. Options: make the node protocols `@runtime_checkable` (cost: enables `isinstance` but Protocols with methods may still be rejected as data; verify), or restructure the `isinstance(item, self.cst.Item)` check to a non-isinstance form, or accept a single documented `type: ignore` here and justify it under `di-boundary-escape`. Pick one and state it; do not defer to "Test plan decides," because the outcome determines whether mechanism (a) satisfies B1+B5 on this path at all.

### design-5 — Test plan items 2/3/4 assume a pyright-invoking test harness whose mechanism is undefined; infra-before-feature gap
Section "Test plan" (design.md:72-74): items 2, 3, 4 require running `pyright --outputjson` on fixtures and asserting zero/expected errors, including a negative case.

What's wrong: the repo has no existing pyright-as-a-test harness (verified: tests are `pytest`; `uv run pyright` is a separate `make check` gate, not a pytest assertion). The design proposes "a test that shells `pyright --outputjson` on the fixture" (design.md:72) and "an xfail→pass marker or a focused subprocess assertion" (design.md:73) as if these are drop-in, but specifies no location for the harness, no handling for pyright-not-installed-in-test-env, and no baseline strategy (pyright reports errors repo-wide; a subprocess run scoped to one fixture file still needs config isolation). The negative-test requirement (B4: "deliberately-wrong access is flagged") fundamentally needs a pyright-returns-nonzero assertion, which is new infrastructure.

Consequence: B4's two mandatory checks (present-members + wrong-access-flagged) for the Python backend depend entirely on this unspecified harness. If it is harder than assumed (config isolation, CI pyright availability, output-parsing brittleness), B4 acceptance slips or is faked with a weaker check. The design budgets it as a one-line test but it is the verification backbone for B4/B5 — infrastructure that should be specified before the feature tests rely on it.

Fix: specify the pyright-invocation harness concretely (helper that runs `pyright --outputjson <file>` with a scoped/explicit config, parses diagnostics, asserts on count/rule for a target file; how it skips/fails when pyright is absent). Confirm CI runs pyright (it does, via `make check`) so the harness can rely on it.

### design-6 — `from __future__ import annotations` interaction with runtime `cstp` reference is correct, but one runtime use of `cst` types is unguarded
Section "fltk2gsm.py changes" (design.md:43): add `from __future__ import annotations` so `TYPE_CHECKING`-only `cstp` names "never evaluate at runtime."

What's wrong (scope-completeness): `from __future__ import annotations` defers *annotation* evaluation only. It does not affect non-annotation runtime references. `fltk2gsm.py` uses `self.cst.Item` as a runtime value in `isinstance(item, self.cst.Item)` (`fltk2gsm.py:61,72`) and `self.cst.Items.Label.*`, `self.cst.Disposition.Label.*`, `self.cst.Quantifier.Label.*` as runtime values (`fltk2gsm.py:44-50,63-68,114-130`). These go through `self.cst` (the runtime `ModuleType`), not through `cstp`, so they are fine at runtime — but the design should state that the *only* place `cstp` names appear is in annotations (parameter annotations + the `cst: cstp.CstModule` param), never in runtime position, which is what makes the `TYPE_CHECKING`-only import safe. Constraint (requirements line 97) demands no `NameError` under runtime injection; the design's reasoning is correct but relies on an unstated invariant (no `cstp.*` in runtime position). Worth making explicit so an implementer doesn't, e.g., write `isinstance(item, cstp.ItemNode)`.

Consequence: low — design is correct as intended; risk is only that an implementer reading design.md:44 (which mentions `isinstance(item, self.cst.Item)` right after introducing `cstp`) could conflate `self.cst.Item` with a `cstp` reference and introduce a runtime `NameError`. Minor clarity gap, not a defect.

Fix: add one sentence: all `cstp.*` names appear only in annotation position; all runtime CST access stays on `self.cst` (the injected `ModuleType`).

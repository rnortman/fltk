# Requirements: Make CST-node type annotations available and pyright-checkable

Concise. Precise. Complete. Unambiguous. No padding. Audience: smart LLM/human.

## Goals

Make it *possible* to write correct, statically-checkable type annotations on every piece of generated or hand-written code that accepts or returns a CST node — regardless of whether the CST is the Python dataclass backend (`fltk_cst` etc.) or the Rust/PyO3-backed CST module. "Possible" means: an annotation can be written that names the CST node type, and pyright resolves it to a type with the correct attribute/method surface, without `Any`-degradation and without runtime cost.

## In scope

- Restoring/enabling CST-node-typed parameter and return annotations on `fltk/fegen/fltk2gsm.py` `visit_*` methods (the intentionally-removed annotations from commit `214dbe1`), in a form that remains valid under dependency-injected CST backends (`cst: ModuleType` injection).
- Whatever mechanism is required so that a `ModuleType`-injected or Rust-backed CST is statically typeable from the perspective of consumer code (exploration suggests type stubs `.pyi` and/or a `Protocol`; design decides).
- Ensuring the code generators (`gsm2tree`, `gsm2parser`/`genparser`, `fltk2gsm`, `gsm2tree_rs`) can emit, or are accompanied by, the artifacts needed for the above to typecheck.
- Pyright (at the repo's *current* configured strictness) passing cleanly over: the generated Python CST/parser modules, `fltk2gsm.py` with annotations restored, and any new stub/protocol artifacts.

## Out of scope

- Tightening pyright configuration (`reportMissingParameterType`, basic→strict, etc.). That is the explicitly-deferred LATER effort. Acceptance here is measured against the strictness configured *today*.
- Authoring the ADR recording this decision.
- The style/formatting regression in `unparsefmt_*` generated files (single quotes, `typing.Optional[X]` vs `X | None`, parens, line length, trailing newline) — that is exploration Regression 2, a separate `make check` / ruff concern, not a *type-correctness* concern. Out of scope unless design finds the type work cannot proceed without touching it.
- Runtime behavior of parsing/CST construction. No change to parse output, CST shape, or runtime semantics.
- Changing the DI design (`cst: ModuleType` injection) itself; the typing solution must work *with* it.

## System behavior

Inputs → outputs are about the *generator toolchain and the typecheck gate*, not parser runtime.

### B1 — fltk2gsm annotations restored

Every `visit_*` method in `fltk/fegen/fltk2gsm.py` that consumes a CST node has a static type annotation naming that CST node type on its primary parameter (exact spelling per chosen mechanism — e.g. a bare imported `Grammar`, or an attribute-on-Protocol like `<CstType>.Grammar`; B1 does not prejudge this — see open question `mechanism`), matching the pre-`214dbe1` coverage list:
`visit_grammar`, `visit_rule`, `visit_identifier`, `visit_alternatives`, `visit_items`, `visit_item`, `visit_term`, `visit_disposition`, `visit_quantifier`, `visit_literal`, `visit_regex`.

Acceptance:
- Each listed method's primary CST parameter has a non-`Any` annotation.
- Pyright resolves attribute/method accesses on that parameter inside the method body against the annotated type (no `reportAttributeAccessIssue`, no implicit `Any`).
- The annotation form remains valid when `Cst2Gsm` is constructed with a non-default (e.g. Rust-backed) CST module — i.e. the static type used for annotation and the injected runtime module are compatible per the typing mechanism design chooses.

### B2 — CST node types are statically nameable

For both the Python dataclass CST backend and the Rust/PyO3-backed CST module, there exists a statically-resolvable name (importable type, stub, or protocol member) for each generated CST node class such that consumer code can annotate parameters/returns/variables with it.

Acceptance:
- A consumer module can import or reference the CST node type and use it in an annotation; pyright resolves it to a class with the node's documented attribute/method surface (label accessors, `children`, `span`, append/extend where applicable).
- For the Rust-backed CST, *to the extent a static surface for it is produced this cycle* (see B3a), the surface pyright sees matches the actual PyO3 runtime surface (stub accuracy — see B4). If Rust stub generation is deferred per B3a, the shared static type must still be *defined* such that the Rust backend can satisfy it.

### B3 — Generators emit or provide the typing artifacts

The generator toolchain produces (or is accompanied by checked-in, regenerable) the artifacts required for B1/B2. If the chosen mechanism is `.pyi` stubs for the Rust extension and/or a `Protocol` describing the CST module, those artifacts are generated by the same toolchain that generates the CST (so they stay in sync with the grammar), not hand-maintained out of band.

Acceptance:
- Regenerating from a grammar (via the documented generator command) (re)produces the typing artifacts alongside the CST/parser modules.
- No manual *type-correctness* editing is required to make the generated CST + its consumers typecheck under current pyright strictness. (The generator emitting code that does not yet pass the formatter/linter is normal and expected: `make fix` after regeneration is the established convention that resolves style/formatting — including annotation-style — output. Running `make fix` post-regen does not count as "manual editing" for this acceptance, and is not the type-of-editing this criterion forbids.)

### B3a — Rust backend status (precondition, confirmed)

A compiled Rust/PyO3 CST extension (`fltk._native`, generated from GSM via `gsm2tree_rs.py` into `src/cst_generated.rs`) exists in-tree and **is** injected into `Cst2Gsm` on the opt-in path (`plumbing.parse_grammar(..., rust_fegen_cst_module=...)` → `Cst2Gsm(..., cst=pr.cst_module)`). The Rust path is caller-opt-in, not the default. **No stub/protocol generator for the Rust (or Python) CST surface exists today** — the Rust CST module is loaded dynamically by dotted name and is fully `Any` statically.

Consequence for scope: B4 Rust-surface verification is achievable in principle (a runtime surface exists) but requires building the stub/protocol generator this cycle. Design decides how thin: the minimum is a shared static type both backends *satisfy* with B4 verified against the Python backend; full Rust `.pyi` emission may be a thin or deferred increment (see open question `rust-stub-source`). Do not over-build a Rust `.pyi` + compile-and-import test harness unless design judges it necessary for B1's Rust-injection acceptance.

### B4 — Stub/protocol accuracy

Any generated stub or protocol accurately reflects the runtime surface of the backend it describes: every attribute/method a consumer is allowed to call is present with a correct (non-`Any` where a real type exists) signature; no phantom members that don't exist at runtime.

Acceptance:
- For a representative grammar, every label accessor / child-access method / span field present at runtime on a CST node is present in the static surface with a matching type.
- A deliberately-wrong access (calling a method that doesn't exist, or passing a wrong-typed child) is flagged by pyright.
- These two checks are mandatory for the Python dataclass backend (the backend `fltk2gsm` runs by default). For the Rust backend they are mandatory only if Rust stub generation is in scope this cycle per B3a; otherwise they are deferred to the increment that adds Rust stub generation.

### B5 — Gate passes at current strictness

Running the repo's pyright check (`uv run pyright`) over the affected files passes with zero new errors attributable to CST typing, at today's configured strictness.

Acceptance:
- `uv run pyright` exits clean on `fltk2gsm.py`, the generated CST/parser modules in scope, and any new stub/protocol artifacts.
- No `# type: ignore` / `cast(Any, ...)` is introduced to suppress CST-typing errors that the chosen mechanism should resolve. (If a narrowly-scoped, documented escape hatch is unavoidable for the DI boundary, it must be justified — see Open questions `di-boundary-escape`.)

### B6 — Backend-agnostic annotations preserve swappability

Code above the backend-selection level can annotate CST-node parameters/returns/variables with a single static type that does not name a concrete backend (neither the Python dataclass classes nor the Rust/PyO3 classes specifically). Switching which backend is injected at the DI boundary does not require editing any annotation in such code.

Acceptance:
- There exists a backend-agnostic static CST-node type (Protocol, alias, or equivalent) usable in annotations on `Cst2Gsm` and any future consumer that holds the CST module by injection rather than by concrete import.
- `fltk2gsm.py`'s `visit_*` annotations (B1) are written in this backend-agnostic form — they do not name `fltk_cst.Grammar`-the-concrete-class in a way that would break if the Rust backend were injected instead.
- Pyright accepts the same annotated `Cst2Gsm` source whether the default Python backend or the Rust backend is the one the static type is checked against, to the extent both backends are statically described this cycle (Rust per B3a/B4 scoping). The Python backend must satisfy the agnostic type unconditionally; the Rust backend must satisfy it once its static surface is produced (B3a).
- A backend-specific access (a member that exists on one backend but not the agnostic type) used in backend-agnostic code is flagged by pyright — the agnostic type is the intersection consumers above the selection level may rely on.

## User-visible surface

Audience = developers of this repo and downstream grammar authors who run the generators.

- **Generator CLI**: if new outputs (e.g. `.pyi`) are produced, the `genparser generate` command (and `gen-rust-cst` for the Rust path) produce them; any new flags/output paths are documented. Default behavior should produce the typing artifacts without requiring extra flags, unless design justifies opt-in.
- **Generated artifacts**: new files (stubs/protocols) appear in the same directory convention as existing gencode (`*_cst.py`, etc.). They follow the existing "no DO-NOT-EDIT header" convention unless design adds one.
- **No new runtime config, env vars, or error messages** are required by this cycle.

## Constraints

- **No runtime cost / no runtime import cycles**: typing artifacts must be import-time-cheap or guarded (e.g. `TYPE_CHECKING`), and must not alter parser/CST runtime behavior. CST-type annotations may rely on PEP 563 deferred evaluation (`from __future__ import annotations`) or string-form annotations; runtime evaluation of the CST-type annotations is not required and must not be forced — so a `TYPE_CHECKING`-only Protocol/stub import used in an annotation must not raise `NameError` under the runtime `cst: ModuleType` injection.
- **DI compatibility**: the solution must coexist with `Cst2Gsm.__init__(..., cst: ModuleType = _default_cst)` accepting either Python or Rust CST backends. Annotations must not force a single concrete backend at runtime.
- **Dual-backend typeability** (weaker than a single shared type): `fltk2gsm.py`'s `visit_*` annotations must typecheck for whichever backend is injected at the DI boundary. A single shared consumer-facing static type (Protocol/alias both backends satisfy) is the *preferred* design, since `Cst2Gsm` is one class annotated once; but it is not a hard requirement — a structural Protocol that the Python dataclass CST satisfies, with the Rust backend reconciled at the DI boundary per `di-boundary-escape`, also qualifies. Do not invest in unifying Python-dataclass and PyO3 surfaces into one nominal hierarchy unless design judges it the cleanest path.

- **Backend-agnostic annotations above the selection level** (see B6): code that lives *above* the point where a concrete CST backend is selected (e.g. `Cst2Gsm` and any consumer that holds a `cst: ModuleType` or otherwise has not committed to Python vs Rust) must be able to write type annotations that name CST node types *without* naming a backend-specific concrete type. The chosen static type for CST nodes must be one that *both* the Python dataclass backend and the Rust/PyO3 backend satisfy (structurally or nominally), so that swapping the injected backend requires no annotation changes in backend-agnostic code. This is the typing-level guarantee that preserves the runtime DI swappability the `cst: ModuleType` design already provides.
- **Sync with grammar**: typing artifacts are derived from the grammar/GSM, so a grammar change that changes node shape regenerates correct typing artifacts.
- **Current-strictness only**: do not depend on, nor pre-enable, stricter pyright settings.
- **Backward compat**: existing generated/committed CST modules that are currently clean (`fltk_cst.py`, `bootstrap_*`, etc.) must continue to typecheck (pyright). Regeneration is permitted to emit not-yet-formatted output; the established convention is to run `make fix` after regenerating, which restores formatter/linter compliance (including annotation style). "Must not reintroduce errors" here means *type* errors that survive `make fix` — not transient formatter/ruff-style violations that `make fix` resolves. Regenerating a currently-clean module and then running `make fix` is an acceptable path and is not a backward-compat violation.

## Open questions

- **mechanism** — Exploration (Regression 1, lines 66, 160) states the blocker is that the injected `cst: ModuleType` has no stub/protocol, so `cst.Grammar` etc. are not statically reachable. Two plausible mechanisms:
  - (a) A generated `Protocol` describing the CST module's class surface, used as the static type of the `cst` parameter, so `visit_*` can annotate against `self.cst.Grammar`-equivalents.
  - (b) Generated `.pyi` stubs for the (Rust) CST module plus a shared static type/alias that both backends satisfy, annotated directly.
  Design decides. Both must satisfy B1–B5. Which does the user prefer, or is the choice left to design? Proposed default: leave to design; require only that B1–B5 hold.
- **di-boundary-escape** — Is a single, narrowly-scoped, documented escape hatch at the DI boundary (the point where a `ModuleType` is reconciled with a static CST type) acceptable, or must the boundary be fully statically typed with zero `cast`/`ignore`? Proposed default: a single documented boundary cast is acceptable if unavoidable; zero elsewhere.
- **rust-stub-source** — For the Rust/PyO3 backend, is the static surface to be generated from the same GSM that drives `gsm2tree_rs.py` (preferred, stays in sync), or extracted from the compiled extension? Exploration confirms `gsm2tree_rs.py` generates the `.rs` from GSM; it does not mention an existing stub generator. Proposed default: generate from GSM alongside the `.rs`.
- **scope-of-regen** — Should currently-clean committed CST modules (`fltk_cst.py`, `unparsefmt_*`) be regenerated this cycle to gain/refresh typing artifacts, or only the toolchain + `fltk2gsm.py` changed and regeneration deferred? Per the user ruling, regenerating a module and then running `make fix` is an acceptable, low-cost path (the formatter/linter resolves style output), so regeneration is not blocked by style concerns. The remaining question is purely whether to do the regeneration *this cycle* or defer it as a mechanical follow-up. Proposed default: change toolchain + restore `fltk2gsm.py` annotations + add typing artifacts for the CST backend(s) `fltk2gsm` depends on, regenerating + `make fix`-ing the affected module(s) as needed; broader mass regeneration of unrelated modules may be deferred.

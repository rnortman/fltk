# Design: CST-node type annotations available and pyright-checkable

Concise. Precise. Complete. Unambiguous. No padding. Audience: smart LLM/human.

Requirements: `./requirements.md` (B1–B6, constraints, open questions). Exploration: `./exploration.md`. This doc does not restate them; it refers.

## Root cause / context

Two facts from exploration + code combine into the gap requirements close:

1. **Annotations dropped at the DI boundary.** Commit `214dbe1` added `Cst2Gsm.__init__(self, terminals, cst: ModuleType = _default_cst)` (`fltk/fegen/fltk2gsm.py:10`) and rewrote every CST class reference to dispatch through `self.cst` (`self.cst.Items.Label.*`, `self.cst.Item`, etc. — e.g. lines 44–48, 61, 114–119, 125–130). Because `cst` is a runtime `ModuleType`, `cst.Grammar` is not a statically-resolvable name, so the `visit_*` parameter annotations (`grammar: cst.Grammar`, …) were removed (`fltk2gsm.py:14,18,24,28,31,78,100,112,123,134,138`). The injected module has **no static surface** — neither a stub nor a Protocol describes it.

2. **The gate does not catch the loss.** pyright runs at *basic* mode (`pyproject.toml:48-53` sets only `pythonVersion`/`include`/`exclude`; no `typeCheckingMode`, so the default is `basic`). `reportMissingParameterType` is a strict-only check, so the bare `grammar` parameters silently infer `Any` and the gate stays green. Consequence: today nothing *forces* a CST annotation, and nothing *enables* one either — inside `visit_grammar`, `grammar.children_rule()` is `Any`-typed, so a typo or wrong accessor is not caught. This is the "make it possible" gap, not a red gate. (Tightening pyright to flag it is the deferred LATER effort, requirements "Out of scope".)

The Python dataclass backend already has a fully-typed concrete surface (`fltk/fegen/fltk_cst.py`: each node is a `@dataclass` with `child_*`/`children_*`/`maybe_*`/`append*`/`extend*`/`span`/`children`). The Rust backend (`gsm2tree_rs.py` → `src/cst_generated.rs`, loaded by dotted name in `plumbing._load_rust_cst_classes`, `plumbing.py:79-108`) exposes the *same* Python-visible API by construction but is `Any` statically (dynamic `importlib` load, `dict[str, object]`). Both backends are generated from one GSM via the shared `CstGenerator` model (`gsm2tree.py:33`, reused by `RustCstGenerator.__init__`, `gsm2tree_rs.py:45`), so a single structural description fits both — confirming requirements' "preferred single shared type" is achievable.

## Proposed approach

Mechanism (open question `mechanism`): **(a) a generated `Protocol` describing the CST module surface**, used as the static type of the `cst` parameter and as the backend-agnostic node type. Rationale over (b) `.pyi` stubs: the runtime DI value is a *module object* (`ModuleType`), not an instance of a nominally-imported class; a Protocol typing the module's class-valued attributes is the natural fit, is backend-agnostic by construction (structural — both Python dataclasses and PyO3 classes satisfy it without a shared base, satisfying B6 and the dual-backend constraint), carries no runtime import or cost under `TYPE_CHECKING`, and is emitted from the same GSM that drives both generators (`rust-stub-source` default: generate-from-GSM). `.pyi` stubs for the Rust extension become an optional later increment (B3a permits deferral); they are *not* needed for B1's Rust-injection acceptance because the shared Protocol is the static type `Cst2Gsm` is checked against regardless of which backend is injected.

**Empirically why the Rust `.pyi` is deferrable (pyright 1.1.402, verified).** The Rust CST module is always held as `types.ModuleType` (`ParserResult.cst_module: types.ModuleType` in `plumbing_types.py`; loaded by `importlib.import_module` in `plumbing._load_rust_cst_classes`, `plumbing.py:79-108`, `240-246`). Pyright never sees the Rust classes at any site. At the Rust injection site (`plumbing.py:171`, `Cst2Gsm(..., cst=pr.cst_module)`): bare `ModuleType` is **rejected** against `cst: CstModule` (`reportArgumentType`, verified); the documented narrowing cast `cast("CstModule", pr.cst_module)` makes it clean. The cast yields `CstModule` — **not** `Any` (`reveal_type` verified) — so it does not poison downstream checking. Every consumer downstream of the cast (`visit_*`, any future injection-holding code) is then checked against the Protocol: member access resolves fully typed (`cst.Grammar : type[GrammarNode]`, verified) and a deliberately-wrong access **is flagged** (`reportAttributeAccessIssue`, verified) — *identically* whether the Python or Rust module is injected, because both present as `CstModule` past the cast. This is exactly B1's Rust-injection acceptance (requirements line 36) and B6 swappability (line 84, "to the extent both backends are statically described this cycle") satisfied for the Rust path with the Protocol alone. The *single* thing a Rust `.pyi` would add is B4-Rust **accuracy** — letting pyright verify the real PyO3 surface genuinely satisfies `CstModule` (that the cast does not mask a missing Rust member). No in-tree consumer holds Rust nodes by a concrete type, so the `.pyi` is never on the annotation-authoring/checking path; it serves only that accuracy guarantee, which B3a/B4 (lines 54-67) explicitly make conditional and defer.

### New generated artifact: a CST Protocol module

Add a generator (new method(s) on `CstGenerator` in `fltk/fegen/gsm2tree.py`, emitted as a sibling module) producing, per grammar, a `*_cst_protocol.py` (naming parallels `*_cst.py`; see Resolved below). It contains:

- **One `Protocol` class per CST node** (`Grammar`, `Rule`, … — Protocol names exactly match concrete CST node class names; module separation provides disambiguation). Each declares the node's public surface as `...`-bodied members:
  - `span: Span` (read), `children: list[tuple[Label | None, <union>]]`,
  - `child_<label>(self) -> <T>`, `children_<label>(self) -> Iterator[<T>]`, `maybe_<label>(self) -> <T> | None`, `child(self) -> tuple[...]`, and the `append*`/`extend*` family,
  - a nested `Label` surface where the node has labels — see **Label representation** below.
  - Child/label value types reference *other Protocols* (e.g. `Rule`, `Item`) and `Span`. Protocol node names are identical to concrete class names (`Rule`, `Items`, …); `terminalsrc.Span`/`terminalsrc.*` and library types are unaffected. **Annotation-name remapping is required and is not a free reuse of `py_annotation_for_model_types` (`gsm2tree.py:85-93`).** That method emits the *concrete* class names from `class_name_for_rule_node` and passes `terminalsrc.Span` through; there is no node-vs-Span discriminator at the string level. Protocol generation uses a separate `protocol_annotation_for_model_types` method that applies quoting to rule-name references (forward refs within the protocol module) while leaving library types unquoted. Implement as either a new annotation method or a `suffix`/`rename` parameter threaded through `iir_type_for_model_type`; the rename keys off whether the model type is a rule name (str → `iir_type_for_rule`) vs a registered library type (`Span`). Under-budgeting this as "retarget the existing machinery" yields a module whose member annotations name undefined identifiers → `reportUndefinedVariable`, failing B5.
- **One `CstModule` Protocol** typing the module surface: a **read-only** member per node (declared via `@property def Grammar(self) -> type[GrammarNode]: ...`), plus any module-level names `Cst2Gsm` reads (`Items`, `Item`, `Disposition`, `Quantifier`, …). **Read-only (property), not a plain `Grammar: type[GrammarNode]` attribute** — empirically required: pyright treats a mutable attribute as invariant, so a plain `type[Grammar]`→`type[GrammarNode]` match is *rejected* for invariance; a `@property`-returning member is covariant and the concrete module's class attribute satisfies it (verified with pyright 1.1.402). This is the type used for the `cst` parameter.

Members are typed structurally; `Protocol` (not `runtime_checkable`) — no runtime use, pure static surface.

### Label representation (the intricate part — was under-specified)

Empirically (pyright 1.1.402), a nested class inside a Protocol is matched **nominally**, not structurally: `type[Items.Label]` (concrete `enum.Enum` subclass) is *not* assignable to a Protocol's nested `Label` class regardless of how the protocol's `Label` members are typed (`object`, `ClassVar[Label]`, empty enum — all rejected: "`type[Items.Label]` is not assignable to type[ItemsNode.Label]"). Consequences, which the design now states honestly rather than hand-waving:

- The per-node `Label` is declared as a nested `class Label:` with `NO_WS: ClassVar[Label]` (etc.) members typed as the nested `Label` itself. This makes `self.cst.<Node>.Label.<NAME>` **resolve** (attribute present) and `label == self.cst.<Node>.Label.<NAME>` typecheck. A *non-existent* label (`.WRONG`) is flagged (verified: "Cannot access attribute WRONG"). That is the safety B4 actually buys on the label path: **attribute-presence only**.
- It does **not** buy wrong-but-existing-value safety: `==` across distinct nominal enum types is never flagged by pyright (verified). B4 acceptance ("deliberately-wrong access flagged") is satisfied for *missing* members, **not** for a valid-but-semantically-wrong label constant. Do not overclaim.
- Because nested-class matching is nominal, the presence of any `Label` on a node means the concrete module does **not** structurally satisfy `CstModule` at the default-arg site without help. This is the dominant driver of the boundary-cast decision (see DI boundary) — not a peripheral edge case. Every node in the B1 list except `Identifier`/`Literal`/`RawString` carries a `Label`.

### `fltk2gsm.py` changes (B1, B6)

- Import the Protocol module under `TYPE_CHECKING` only (constraint: no runtime import cycle/cost; the `cst: ModuleType` runtime value is unaffected):
  ```python
  if TYPE_CHECKING:
      from fltk.fegen import fltk_cst_protocol as cstp
  ```
- Annotate `__init__`'s `cst` parameter as `cstp.CstModule` instead of bare `ModuleType` (keep `ModuleType` import only if still needed; `CstModule` is the static type). The default binding needs a documented cast (nested-`Label` nominal mismatch — see DI boundary): `cst: cstp.CstModule = cast("cstp.CstModule", _default_cst)`.
- Restore each `visit_*` primary-parameter annotation in backend-agnostic Protocol form, per the B1 coverage list: `visit_grammar(self, grammar: cst.Grammar)`, `visit_rule(self, rule: cst.Rule)`, `visit_identifier(... cst.Identifier)`, `visit_alternatives(... cst.Alternatives)`, `visit_items(... cst.Items)`, `visit_item(... cst.Item)`, `visit_term(... cst.Term)`, `visit_disposition(... cst.Disposition)`, `visit_quantifier(... cst.Quantifier)`, `visit_literal(... cst.Literal)`, `visit_regex(... cst.RawString)`. (Protocol module imported as `cst`, bare names now match concrete CST class names — disambiguation is by module qualifier, not suffix.)
- Add `from __future__ import annotations` (absent today, `fltk2gsm.py:1-6`) so the `TYPE_CHECKING`-only names in annotations never evaluate at runtime — satisfies the constraint that a `TYPE_CHECKING`-only Protocol used in an annotation must not `NameError` under runtime injection. **Invariant the implementer must hold:** `cstp.*` names appear *only* in annotation position (`__init__`'s `cst` param + each `visit_*` primary param). All runtime CST access stays on `self.cst` (the injected `ModuleType`) — `isinstance(item, self.cst.Item)`, `self.cst.Items.Label.*`, etc. Never write `cstp.ItemNode` in runtime position (e.g. as an `isinstance` arg); that would `NameError` under runtime injection because `cstp` is `TYPE_CHECKING`-only. `from __future__ import annotations` defers annotation evaluation only; it does not protect runtime references.
- `self.cst` is `CstModule`-typed, so `self.cst.Items.Label.NO_WS`, `isinstance(item, self.cst.Item)`, `self.cst.Disposition.Label.*` resolve statically. **`isinstance(item, self.cst.Item)` is fine** (verified, pyright 1.1.402): the second arg is a *value* of type `type[ItemNode]`, which pyright accepts as a runtime class object and narrows `item` to `ItemNode` — no error, no `Any`-degradation. The `runtime_checkable`-required restriction applies only when a Protocol *name* is the literal second arg (`isinstance(item, ItemNode)` → error); the design never does that. So the B1×B5 intersection on the `Item` path holds cleanly with no `type: ignore` and no `Any`. (Previously flagged as a possible dead-end; empirically it is not.)

### Generator wiring (B3, user-visible surface)

- `genparser generate` emits `{base_name}_cst_protocol.py` alongside `{base_name}_cst.py` (new write in `genparser.generate`, mirroring the `shared_cst` write at `genparser.py:170-186`). Default-on, no new flag (requirements: "Default behavior should produce the typing artifacts without requiring extra flags"). Same directory convention, no DO-NOT-EDIT header.
- Regenerating the fegen grammar's own artifacts produces `fltk/fegen/fltk_cst_protocol.py` next to `fltk_cst.py`.
- The Rust path (`gen-rust-cst`): no new emission required this cycle. The Rust CST module is loaded dynamically and is statically `Any`/`ModuleType` at the injection site (`plumbing.py:171`), so it crosses into `CstModule` via the documented boundary cast there (see DI boundary); no structural match is asserted for it this cycle. `.pyi` emission that would give it a checkable static surface is deferred (B3a/`rust-stub-source` — TODO below).
- Emission reuses `ast.unparse` like the existing CST/parser writes; the output is expected to need `make fix` for style (single quotes, `Optional` vs `| None`), which is the established post-regen convention (requirements B3a acceptance, Backward-compat note) — not counted as manual editing.

### DI boundary (open question `di-boundary-escape`)

The default `_default_cst = fltk_cst` module is assigned to a `cst: CstModule` parameter. pyright *does* check a module object against a Protocol-typed parameter structurally (verified: it reports per-attribute incompatibilities, not a blanket "module is not a Protocol"). Two sub-problems, with empirical answers:

1. **Read-only members** — solved by declaring each node member as a `@property` returning `type[<Node>Node]` (covariant), not a mutable attribute (invariant, rejected). See "New generated artifact" above. With property members and a node carrying *no* `Label`, the module satisfies the Protocol with zero casts (verified).
2. **Nested `Label` nominal mismatch** — **not** solvable structurally (see Label representation). Any node with a `Label` makes `type[<ConcreteNode>]` non-assignable to `type[<Node>Node]`, so the `fltk_cst` module does **not** structurally satisfy `CstModule` for the label-bearing nodes. A cast *is* required at the default binding.

**Gating precondition.** Run an empirical pyright check (Test plan item 2b) before committing to the cast: assert which node members structurally match without it. Expected given the nested-`Label` finding: a boundary cast is needed. The cast is permitted under `di-boundary-escape` **only** because the underlying mismatch is the *known, benign* nested-enum nominal-identity limitation — the concrete `Label` enums genuinely carry the same members the Protocol declares; pyright simply refuses to match nested classes structurally. Document it as "narrowing past pyright's nominal nested-class limitation," **not** a blanket `cast(Any)`. The hazard a boundary cast otherwise carries: it could mask a *real* surface gap (a genuinely-missing member) and silently void B4. Guard against that with Test plan item 2a, which — without the cast — accesses node members through a `CstModule`-typed binding and asserts every promised member resolves on the concrete module.

Casts therefore: **one** at the `__init__` default binding (`cst: CstModule = cast("CstModule", _default_cst)`), and **one** at the Rust injection site (`plumbing.py:171`, `cst=pr.cst_module` where `pr.cst_module` is genuinely `ModuleType`/`Any`). Both documented; **none inside `visit_*`**.

## Edge cases / failure modes

- **Module-object ↔ Protocol structural match.** pyright matches the module structurally (per-attribute). Property members fix invariance; nested `Label` forces the documented boundary cast (see DI boundary). `visit_*` bodies are fully typed regardless — the cast is only at the binding, and the separate member-access fixture (Test plan item 2) guards against the cast hiding a real gap.
- **`isinstance(x, self.cst.Item)` — resolved, not a risk.** `self.cst.Item` is a *value* of type `type[ItemNode]`. Verified (pyright 1.1.402): `isinstance(item, self.cst.Item)` produces no error and narrows `item` to `ItemNode`. The `runtime_checkable` restriction applies only to a literal Protocol-name second arg (`isinstance(item, ItemNode)`), which the design never writes. No `type: ignore`, no `type[Any]` degradation needed on the `Item` path.
- **Forward references among Protocols.** Nodes reference each other cyclically (`Grammar`→`Rule`→`Alternatives`→…→`Term`→`Alternatives`). `from __future__ import annotations` in the generated protocol module makes all member annotations strings; no ordering constraint. Matches how `fltk_cst.py` already uses quoted forward refs.
- **Trivia / `Span` children.** Several nodes' child unions include `Trivia` and `terminalsrc.Span` (e.g. `Items`, `fltk_cst.py:181`). The Protocol must include `TriviaNode` and import `Span` from `terminalsrc`. The Rust backend inserts `(None, Trivia(...))` entries (`fltk2gsm.py:36-40`); `children` typed `list[tuple[Label | None, <union incl. TriviaNode>]]` covers both backends. The `label is not None` filter (`fltk2gsm.py:40`) narrows correctly under the `Label | None` element type.
- **Backend-specific member used in agnostic code (B6 acceptance).** Because the Protocol is the *intersection* surface (only what `CstGenerator`'s model emits for both), a member existing on only one backend is absent from the Protocol and pyright flags its use in `fltk2gsm` — the desired behavior. No action needed beyond emitting from the shared model.
- **Grammar drift.** Protocol is GSM-derived and regenerated with the CST, so a node-shape change regenerates a matching Protocol (constraint "Sync with grammar"). Stale hand-edits are not a concern (no manual maintenance).
- **`make fix` reordering/style.** Generated protocol module will have `typing.Optional`/single quotes from `ast.unparse`; `make fix` normalizes. No *type* change results, so B4/B5 hold post-fix.

## Test plan

After this cycle the following tests exist (location: `fltk/fegen/test_genparser.py` exists; add a typing-focused test module, e.g. `fltk/fegen/test_cst_protocol.py`, plus a pyright-gate assertion).

**Pyright test harness (specify before the feature tests depend on it).** The repo has no pyright-as-pytest harness today (pyright runs only as a `make check` gate). Add a helper, e.g. `run_pyright(file_path, *, extra_config=None) -> list[Diagnostic]`, that:
- shells `uv run pyright --outputjson <file>` and parses `generalDiagnostics` (severity, rule, line) from JSON;
- runs against an explicit, scoped fixture file (or a tiny fixture package under `test_data/`) so repo-wide errors don't contaminate the result; `--outputjson` already reports per-file, so filter diagnostics to the target path;
- `pytest.skip(...)` when pyright is not importable/spawnable (so the suite still runs in a pyright-less env), but CI *does* run pyright (via `make check`), so the harness is exercised there;
- exposes two assertions: "target file has zero CST-typing diagnostics" and "target file has a diagnostic of rule R at line L" (for negative cases).
This harness is the verification backbone for B4/B5; budget it as real infrastructure, not a one-liner.

Tests:

1. **Protocol generation (unit).** For a representative grammar (the fegen grammar, and/or the existing fixture `fltk/fegen/test_data/phase4_roundtrip.fltkg`), `CstGenerator` protocol emission produces a module with one Protocol per node, each carrying `span`, `children`, `child_<label>`, `children_<label>`, `maybe_<label>`, `child`, `append*`, `extend*`, and `Label` where applicable. Assert member presence against the node model (B4 "no phantom members, all real members present").
2. **Default backend satisfies Protocol (static, B4 Python mandatory).** Two distinct fixtures, because the nested-`Label` finding splits "module assignable to `CstModule`" from "members resolve":
   - **(2a) Member-access fixture (the real B4 guarantee).** A fixture that holds a `CstModule`-typed binding and *accesses* node members through it (`m.Grammar`, `g.child_name()`, `m.Items.Label.NO_WS`, etc.) — asserted zero errors via the harness. This catches a genuinely-missing member that the boundary cast would otherwise mask. Includes the **negative** case: a deliberately-wrong access (`grammar.no_such_method()`, or `.WRONG` label) asserted to *produce* a pyright diagnostic (B4 "deliberately-wrong access is flagged"). Note explicitly: a wrong-but-existing label compared via `==` is *not* flagged (nominal-enum limitation) — assert this as a known boundary so the test does not falsely promise it.
   - **(2b) Boundary-assignability probe.** Run the harness on a fixture that binds the bare `fltk_cst` module to `CstModule` *without* a cast, and record which members match. This is the empirical precondition gate for the DI boundary (confirms the expected nested-`Label` cast necessity, and that nothing *else* fails the match). The boundary cast is justified only against the diagnostics this probe produces.
3. **`fltk2gsm.py` typechecks with restored annotations (B1, B5).** `uv run pyright` over `fltk2gsm.py` + the generated `fltk_cst_protocol.py` exits clean; assert no `reportAttributeAccessIssue`/implicit-`Any` on the annotated params. (Gate-level: part of the repo's existing `uv run pyright` run; add an xfail→pass marker or a focused subprocess assertion.)
4. **Backend-agnostic / swap (B6).** Static fixture annotating a `Cst2Gsm`-like consumer holding `cst: CstModule`; assert it typechecks identically whether the Python concrete module or a Rust-surface stand-in is the bound value. Since Rust `.pyi` is deferred, the Rust arm is satisfied by "the Protocol is defined such that the Rust backend can satisfy it" (B2/B6 conditional clause) — tested via a hand-written minimal stand-in module that structurally matches `CstModule`, asserting the Protocol is not Python-dataclass-specific.
5. **Runtime unaffected.** Existing `parse_grammar` / Rust-path tests (`test_plumbing.py`) continue to pass unchanged — annotations are `TYPE_CHECKING`-only; no runtime import of the protocol module. Assert `fltk_cst_protocol` is not imported at runtime by `fltk2gsm` (e.g. it is absent from `sys.modules` after `import fltk.fegen.fltk2gsm` without `TYPE_CHECKING`).
6. **No new gate regressions.** `uv run pyright` clean over the full in-scope set; `make fix` then `make check` clean on the regenerated `fltk_cst_protocol.py` (style normalized).

## Resolved (were ambiguous; decided here, not deferred)

- **mechanism**: Protocol (a), not `.pyi` (b) — rationale in Proposed approach.
- **di-boundary-escape**: two documented boundary casts — one at `__init__`'s default binding (needed: nested-`Label` nominal mismatch, empirically confirmed), one at the Rust injection site (`plumbing.py:171`, genuine `ModuleType`). None inside `visit_*`. The default-binding cast is justified against the (2b) probe diagnostics so it cannot mask a real surface gap; (2a) independently verifies member resolution.
- **Protocol naming**: bare names matching concrete classes (e.g. `Rule`, `Grammar`) + `CstModule`, in `*_cst_protocol.py`. Protocol and concrete classes live in separate modules and are always module-qualified in annotations (`cstp.Rule`, `cst.Grammar`, etc.), so bare names do not collide. The prior `*Node` suffix justification ("avoids same-namespace collision") was retracted as factually incorrect — the generator never co-locates Protocol and concrete classes. See `protocol-namespace-investigation.md` for the full analysis.

## 2. What is `UnknownSpan` and what does skew actually mean?

`src/lib.rs:24–33` — at `fltk._native` module init, `UnknownSpan` is created as a singleton `Span(-1, -1)` instance:

```rust
let unknown_span_obj = Py::new(m.py(), Span { start: -1, end: -1, source: None })?.into_any();
m.add("UnknownSpan", unknown_span_obj.clone_ref(m.py()))?;
```

`src/span.rs:65–69` — `Span` struct layout:

```rust
pub struct Span {
    pub(crate) start: i64,
    pub(crate) end: i64,
    pub(crate) source: Option<Arc<SourceInner>>,
}
```

The user extension's `UNKNOWN_SPAN_CACHE` holds a `PyObject` reference to the sentinel. It does not link against `fltk._native` at the Rust level — there is no `extern crate fltk_native` in any user extension Cargo.toml. All three Cargo.toml files (`Cargo.toml:15`, `tests/rust_cst_fixture/Cargo.toml:15`, `tests/rust_cst_fegen/Cargo.toml:15`) depend only on `pyo3 = { version = "0.23", features = ["abi3-py310"] }`.

**The user extension stores `UnknownSpan` as an opaque `PyObject`.** There is no Rust-level type assertion that the object is a `Span`, no field access at the Rust level. The sentinel is used solely as a Python object to assign to `self.span: PyObject` when no span is passed to `__new__`.

## 3. What is the actual skew risk?

The TODO's concern is: if `Span`/`UnknownSpan` fields change between FLTK versions, a user extension compiled against an old FLTK might misbehave.

**Reality:** the user extension does not link `fltk._native` at compile time, only at runtime via `py.import("fltk._native")`. There is nothing to recompile against. The extension fetches `UnknownSpan` from whatever `fltk._native` is currently installed. So "built against an older FLTK" is not the correct framing; the extension always gets the currently-installed `UnknownSpan`.

**The concrete risk that remains:**

1. **`Span` struct layout** is invisible to the user extension at the Rust level (the extension stores spans as `PyObject`). Node equality (`_eq_method`, `gsm2tree_rs.py:544–558`) calls Python's `self.span.bind(py).eq(other_node.span.bind(py))`, delegating to `Span.__eq__`. If `Span.__eq__` semantics change, behavior changes — but this is a normal Python semantic break, not a silent ABI skew.

2. **`UnknownSpan` identity/equality** — if `fltk._native.UnknownSpan` is replaced by a different type in a future FLTK version, the cached `PyObject` would hold a stale reference for the process lifetime (GILOnceCell is initialized once). But since the extension imports it from the live `fltk._native` at first use, skew requires running a user extension against a different FLTK version than was present when the cache was first populated within the same process — an unusual scenario (hot-swapping modules).

3. **No Rust-level linkage means no Rust ABI breakage.** The pyo3 stable ABI (`abi3-py310`) insulates the extension from CPython version changes, not from FLTK semantic changes.

**Summary:** the risk the TODO describes ("user extension built against an older FLTK could misbehave silently") is real only at the *semantic* level (Python API of `Span`/`UnknownSpan` changes), not at the binary/ABI level. There is no binary skew possible because there is no Rust-level linkage.

## 4. No ABI-version check exists

No code in `gsm2tree_rs.py`, `src/lib.rs`, or `src/span.rs` emits or checks any version token. `fltk._native` exposes no `__version__`, `_abi_version`, or equivalent attribute (verified by searching all `fltk/` Python sources — no `__version__` or `FLTK_VERSION` constants exist in the package). The `pyproject.toml` version (`0.1.1`) is not exposed via any runtime attribute.

## 6. Actionability verdict

The TODO is **real but misframed**:

- The UNKNOWN_SPAN_CACHE sentinel fetch in `_new_method` exists exactly as described (fact verified).
- The "built against an older FLTK" framing is inaccurate: there is no compile-time linkage; the extension always fetches the sentinel from the currently-installed `fltk._native`.
- The genuine risk is a Python semantic break (type, attributes, or equality semantics of `Span`/`UnknownSpan` change without incrementing a detectable version token), not a binary ABI skew.
- An ABI-version check at `UNKNOWN_SPAN_CACHE` init would need `fltk._native` to expose a version attribute; none currently exists.
- The TODO marker is missing from `gsm2tree_rs.py:_new_method` (present only in docs).

The TODO is a valid future-revisit: low urgency because there is no binary skew vector today, and `Span`'s Python surface (`start`/`end` are not exposed as attributes per `span.rs:56` — "intentionally not exposed as Python attributes") is minimal. If `Span`'s Python-visible API ever changes, a version attribute on `fltk._native` and a check in the generated sentinel-fetch would be the implementation site.

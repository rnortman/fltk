## design-1: generated header names the grammar file, but the generator API cannot know it

Section ┬¦2.2 item 1: "`//!` doc: \"Generated by fltk gen-rust-parser from `<grammar filename>`. ...\""

┬¦2.1 fixes the constructor as `RustParserGenerator(grammar: gsm.Grammar, cst_mod_path: str = "super::cst")`
and `generate(self) -> str`. The grammar filename exists only in the CLI (┬¦2.5); no parameter carries it
into the generator. Precedent cuts the other way: `RustCstGenerator.__init__(grammar)` takes no filename
and its preamble (gsm2tree_rs.py:261-279) emits none.

**Consequence**: the header cannot be emitted as specified. The implementer must either silently drop the
filename (divergence from the design text, and ┬¦4.1's structural text assertions become ambiguous about
what to pin) or invent an unplanned constructor parameter mid-implementation.

Suggested fix: add an explicit optional `source_name: str | None` constructor parameter that the CLI
passes (and that unit tests can pass directly), or change the header spec to omit the filename.

## design-2: fixed import block contradicts conditional regex-table emission under `-D warnings`

Section ┬¦2.2 item 1 (the `use` block, presented as fixed) vs ┬¦2.2 item 2: "The regex table, `regex_at`,
and `consume_regex` are emitted only when the grammar has at least one regex ... unconditional emission
of unused private items fails the `-D warnings` clippy lanes via `dead_code`."

When the regex table is omitted, `use std::sync::OnceLock;` and `use fltk_parser_core::regex::Regex;`
are unused imports ŌĆö `unused_imports` fails the same `-D warnings` lanes (┬¦2.7) the conditional emission
exists to satisfy. The zero-regex case is reachable: the default `_trivia` rule (which contains `[\s]+`)
is only injected when the grammar lacks its own (gsm.py:380-401), and the `\s+` separator pattern enters
the table only via separators *inside trivia rules* (gsm2parser.py:641-650) ŌĆö so a grammar with a custom
literal-only `_trivia` rule and no regex terms produces an empty table.

**Consequence**: for such a grammar the generated file fails the design's own clippy gate; the generator
violates its stated invariant ("the generated `.rs` must be clippy-clean under both lanes", ┬¦2.7).

Suggested fix: emit the `OnceLock` and `regex::Regex` imports under the same condition as the regex table
(one sentence in ┬¦2.2 item 1). Same treatment is already implied for `consume_literal`; no other import
in the block is conditional (all others are used unconditionally).

## design-3: out-of-range/negative-`pos` claim and tests are wrong for nullable rules

Section ┬¦3 bullet: "**Out-of-range / negative `pos` at `apply__parse_<rule>`**: ... returns `None`/fails
the parse"; ┬¦4.3: "out-of-range and negative `pos` return `None`".

A rule that can match empty (an alternative whose items are all `?`/`*`, or a `*`-quantified sole item ŌĆö
the loop fails immediately, `pos == span_start`, and with `min == ZERO` no progress check is emitted,
gsm2parser.py:581-585) returns `Some(ApplyResult)` with an empty span at *any* `pos`, including `-1` and
`len+1`: the inner `consume_*`/`apply__*` calls fail (terminalsrc.rs:109-125 bounds checks), but failure
of optional items is not failure of the alternative. This matches Python's behavior, so it is not a
parity bug ŌĆö but the design's blanket "returns `None`" is false as stated, and ┬¦2.6 deliberately puts
`?`/`*` items in the fixture grammar, so the ┬¦4.3 test as written passes or fails depending on which
rule the implementer happens to pick.

**Consequence**: a test implementing ┬¦4.3 literally against a nullable rule fails (or worse, the
implementer "fixes" the generator to reject out-of-range `pos`, diverging from Python); the edge-case
bullet documents a contract the generated code does not have.

Suggested fix: restate as "never panics, never indexes out of bounds; rules that cannot match empty
return `None`" and scope the ┬¦4.3 assertions to a non-nullable rule (or additionally assert the
empty-match-at-out-of-range result for a nullable rule, pinning the Python-equivalent behavior).

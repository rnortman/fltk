# Security review — Phase 2 Rust parser generator

Style note: concise, precise, complete, unambiguous. Audience: smart LLM/human. No padding.

Reviewed: `git diff 490bccf..b95f772` (HEAD b95f772).

Checked and clean (no findings filed):

- **Generated-code injection via grammar content**: rule names and item labels are interpolated into Rust identifiers (`parse_<rule>`, `append_<label>`, cache fields). Covered: `RustParserGenerator.__init__` composes `RustCstGenerator`, whose constructor validates every rule name and label against `^[_a-z][_a-z0-9]*$` before any emission (fltk/fegen/gsm2tree_rs.py:57-81); the parser generator emits nothing before that runs (fltk/fegen/gsm2parser_rs.py:70).
- **String-literal injection**: literals, regex patterns, rule names, and `source_name` all pass through `_rust_str_lit` (fltk/fegen/gsm2parser_rs.py:21-34), which escapes `\`, `"`, controls < 0x20, and DEL. `\` and `"` are the only chars that can terminate/escape a Rust string literal; newlines (the only comment-breaker for the `//!` header line) are escaped. No breakout path found.
- **`--cst-mod-path` injection**: validated against `^[A-Za-z_][A-Za-z0-9_]*(::[A-Za-z_][A-Za-z0-9_]*)*$` before interpolation into the `use` line (fltk/fegen/genparser.py:348, 364-366). Python `$` admits one trailing `\n`, but `\n` is plain whitespace inside a Rust `use` item — no consequence.
- **Generated regex table indexing**: `consume_regex(pos, idx)` indices are generator-emitted constants always in range of `REGEX_PATTERNS`; not input-derived.
- **ReDoS on the Rust side**: grammar regexes compile with the `regex` crate (linear-time guarantee; Cargo.lock pins regex 1.12.4, no known advisories). The new fegen `block_comment` patterns (`(?:[^*]|\*+[^\/\*])*`, `\*+\/`) also re-enter the Python backtracking engine via regenerated fltk_parser.py; the star group has no required suffix inside the same regex (it can always succeed early, possibly empty), so backtracking into earlier iterations never triggers — no catastrophic case found.
- **Dependencies**: new fixture Cargo.lock entries are regex/memchr/aho-corasick/pyo3 at current versions; nothing with known vulns noticed.

## security-1 — Unbounded recursion on untrusted input aborts the process (stack overflow), a harder failure than the Python backend's catchable RecursionError

- **File:line**: generated parsers, e.g. tests/rust_parser_fixture/src/parser.rs (`parse_expr`/`parse_atom`/`parse_paren_expr` chain), tests/rust_cst_fegen/src/parser.rs; emission sites fltk/fegen/gsm2parser_rs.py:383-418, 637-704.
- **Issue**: the generated parser is recursive-descent with no depth limit. Recursion depth is proportional to input nesting depth (e.g. `((((…))))` through `paren_expr -> expr -> atom -> paren_expr`). Packrat memoization caps left recursion but not plain nested recursion on first descent.
- **Trust boundary / data flow**: the parse input (`Parser::new(text, ...)` → `apply__parse_<rule>(0)`) is the primary untrusted input of any downstream consumer — source files from arbitrary users. Nesting depth flows directly to native stack depth.
- **Consequence**: an attacker submitting a deeply nested document to any service or tool built on a generated Rust parser exhausts the thread stack. In Rust this is a guaranteed process abort (SIGSEGV/`abort`), not an unwindable panic — it cannot be caught with `catch_unwind`, takes down co-tenant work in the same process, and is strictly worse than the Python backend's catchable `RecursionError`. Cheap input (a few hundred KB of `(`) suffices on default 8 MB stacks; far less on small thread stacks (Tokio default 2 MB). This is input-dependent process death, which sits uneasily next to the controlling design's no-input-dependent-panic contract (§4) even though stack exhaustion is technically not a panic.
- **Suggested fix**: emit a depth counter in the generated `Parser` (incremented in `apply__*` or rule bodies) with a configurable limit returning a parse failure (and a distinguishable error) when exceeded; alternatively document a stack-size requirement and the abort behavior prominently in the generated header so downstream consumers parsing untrusted input know to isolate the parser. If parity with Python is the blocker, note that Python's effective limit (recursion limit ~1000) is itself a de facto depth cap the Rust backend currently lacks.

## security-2 — Input-triggered infinite loop in `*`/`+` item parsers when the repeated term matches empty

- **File:line**: fltk/fegen/gsm2parser_rs.py:605-624 (`_gen_item_multiple` `while let` loop, no per-iteration progress check); design acknowledges at docs/adr/2026/06/10-rust-parser-generator/design.md §3 ("Empty-match inner loop divergence").
- **Issue**: the emitted loop `while let Some(one_result) = { … } { pos = one_result.pos; … }` has no in-loop progress guard. For a grammar where the repeated term can match empty at a fixed position (e.g. an inner alternative whose items are all `?`/`*`, or a regex that matches empty), the loop never advances and never exits. The `+` check (`pos == span_start`) runs only after the loop.
- **Trust boundary / data flow**: whether the inner term matches empty at a given position is input-dependent — the attacker chooses input that lands the parser at a position where the nullable term matches empty inside the loop. Memoized inner rules return the same empty hit from cache every iteration, so memoization does not break the loop.
- **Consequence**: for affected grammars, a crafted input hangs the parser in a 100%-CPU infinite loop — an unbounded denial of service in any downstream consumer (no timeout, no allocation growth to trip a limiter). The design deliberately mirrors the Python backend's identical behavior for cross-backend parity (recorded decision, design §3, fltk_parser.py:134 reference); filing because the consequence is attacker-triggerable and the marginal fix (break when `one_result.pos == pos`) is small and could be applied to both backends in lockstep rather than perpetuated.
- **Suggested fix**: emit `if one_result.pos == pos { break; }` (or refuse at generation time to emit `*`/`+` over a provably nullable term). If parity forbids a Rust-only fix, track a cross-backend fix as a TODO(slug) so the hazard is at least recorded outside one design doc.

# Design: `fegen-cst-rs-single-source`

Style note (applies to this doc): concise, precise, complete, unambiguous. No padding. Audience: smart LLM/human.

This is a design doc, not a spec. It refers to `request.md` (the spec) and `exploration.md` rather than restating them. Scope: a small drift-elimination refactor of file layout/build wiring. No production-logic change.

## Root cause / context

Two generated files are byte-for-byte identical (same md5 `0a30124f...`, 5080 lines each; confirmed in `exploration.md`) but committed as independent copies:

- `src/cst_fegen.rs` — generated by `make gen-rust-cst` (`Makefile:54-55`, command `python -m fltk.fegen.genparser gen-rust-cst $(GRAMMAR) $(RS_OUT)`); loaded via `mod cst_fegen;` (`src/lib.rs:5`) and registered into `fltk._native.fegen_cst` (`src/lib.rs:44`).
- `tests/rust_cst_fegen/src/cst.rs` — loaded via `mod cst;` (`tests/rust_cst_fegen/src/lib.rs:11`); compiled into the independent `fegen_rust_cst` cdylib (`Makefile:49-50`); consumed by `tests/test_phase4_fegen_rust_backend.py`, `tests/test_clean_protocol_consumer_api.py`, `tests/test_cross_backend_label_equality.py` via `importorskip("fegen_rust_cst")`.

Only the first has a regeneration command. No Makefile target writes the second (verified: `Makefile` has no target touching `tests/rust_cst_fegen/src/cst.rs`). After a grammar change, regenerating `src/cst_fegen.rs` leaves `cst.rs` stale; the test crate then compiles outdated generated code and the cross-backend tests validate the wrong thing — silently. This is the drift foot-gun the TODO `fegen-cst-rs-single-source` (`TODO.md:39-41`) targets.

## Proposed approach

Make `src/cst_fegen.rs` the single source of truth. The test crate includes it via the Rust `include!` macro rather than holding a second copy.

Changes (no production code; build/layout only):

1. **Replace the contents of `tests/rust_cst_fegen/src/cst.rs`** (currently 5080 lines, identical copy) with a single line:
   ```rust
   include!("../../../src/cst_fegen.rs");
   ```
   `include!` resolves relative to the file containing the macro, so the path is relative to `tests/rust_cst_fegen/src/cst.rs`. Confirmed correct: `realpath --relative-to=tests/rust_cst_fegen/src src/cst_fegen.rs` → `../../../src/cst_fegen.rs`.

2. **Leave `tests/rust_cst_fegen/src/lib.rs` unchanged.** It keeps `mod cst;` (`:11`) and `cst::register_classes(m)?;` (`:15`). `mod cst;` loads the one-line `cst.rs`, whose `include!` pastes the canonical source into module `cst`'s scope. This is the standard `include!`-generated-file pattern.

3. **`src/cst_fegen.rs` is untouched** — still generated by `gen-rust-cst`, still committed, still subject to `make fix` formatting. It remains the sole regeneration target.

No new files, no Cargo workspace, no symlink, no build.rs, no copy step, no `.gitignore` change. `cst.rs` stays committed (now a one-liner). After this change, editing or regenerating `src/cst_fegen.rs` necessarily changes the test crate's compiled output because there is no second copy.

### Why this option over the alternatives

`request.md` already selects `include!`; `exploration.md` confirms feasibility of all three candidates. Rationale, grounded in exploration:

- **Symlink** — works under Cargo/maturin/git-on-Linux, but breaks on Windows with `core.symlinks=false` (git materializes the link as a text file containing the target path, which Cargo then tries to compile). `include!` has no platform dependency.
- **Makefile copy step** — either leaves a committed copy that can still drift, or `.gitignore`s `cst.rs` so the crate won't build from a clean checkout without a prior `make` step. `include!` keeps a clean-checkout build working with no extra step.
- **`include!`** — single committed source, plain-checkout build, platform-independent. Selected.

### Scope-of-edit note on the `include!` target

`src/cst_fegen.rs` opens with top-level `use pyo3::...` items (`src/cst_fegen.rs:1-5`). Under `include!` these become `use` items at the scope of module `cst` (because `mod cst;` + `include!` injects the text into module `cst`). That is valid Rust — `use` is legal at module scope. The file is otherwise plain Rust with no `include!`-incompatible constructs (e.g. no `#![...]` inner attributes that must be crate-root; confirmed the file starts with `use`, not an inner attribute). Compilation is the verification gate (see Test plan).

## Edge cases / failure modes

- **Inner crate-level attributes (`#![...]`).** If `src/cst_fegen.rs` ever began with an inner attribute, `include!` into a non-root module would fail to compile. Today it begins with `use` (verified `:1`). The generator (`gsm2tree_rs.py`) is not in scope to change; if a future generator change adds inner attributes, the test crate build breaks loudly (not silently) — which is acceptable and far better than the current silent-drift failure mode.
- **Path fragility.** The relative path `../../../src/cst_fegen.rs` hard-codes the crate's location two directories under repo root plus `src/`. If `tests/rust_cst_fegen/` is relocated, the path must be updated; failure is a compile-time "file not found", loud and immediate.
- **`use` collision / unused-import warnings in the test crate.** `lib.rs` already does `use pyo3::prelude::*;` (`:9`); the included module gets its own `use pyo3::prelude::*;`. These live in different module scopes (`lib` root vs. `cst`), so no conflict. `cargo clippy -- -D warnings` (`Makefile:30-31`) runs against the root crate (`Cargo.toml`), not the test crate, so test-crate import warnings, if any, do not fail `make check`. No action needed; noted for completeness.
- **Formatting of the one-liner.** `cst.rs` becomes hand-written (not generator output), but it is Rust, outside ruff's scope, and `cargo fmt` is not part of `make check` (`Makefile:6`). The one-liner is trivially well-formed; nothing in the gate formats it.
- **TODO bookkeeping.** `request.md` and the exploration both assume a `TODO(fegen-cst-rs-single-source)` comment "at the top of `tests/rust_cst_fegen/src/cst.rs`". Verified: no such comment exists — `cst.rs` is byte-identical to `cst_fegen.rs` and contains no TODO marker (`grep` found the slug only in `TODO.md:39`). So the code-comment removal step is a no-op; only the `TODO.md` entry (`:39-41`) is removed. The implementer must not hunt for a nonexistent comment.

## Test plan

No new automated tests; this is a build-wiring refactor whose contract is "the test crate compiles the canonical source." After the change, the verification surface is:

1. **Build the test crate:** `cd tests/rust_cst_fegen && uv run --group dev maturin develop` succeeds. This is the compile-time proof that `include!` works (the load-bearing unknown from `request.md` / `exploration.md`).
2. **Existing consumer tests pass:** `uv run pytest tests/test_phase4_fegen_rust_backend.py tests/test_clean_protocol_consumer_api.py tests/test_cross_backend_label_equality.py`. These exercise the `fegen_rust_cst` extension and are the regression net proving the included source produces identical runtime behavior to the prior copy.
3. **Equivalence check (one-time, manual):** `git diff --no-index src/cst_fegen.rs <cpp-expanded include>` conceptually equal — i.e. the included text is the canonical file. Trivially true by construction (the include names that exact path); this is a sanity assertion, not a standing test.
4. **Full gate unaffected:** `make check` still passes. The root-crate `cargo check`/`clippy`/`test` targets do not touch the test crate, so they are unaffected; the Python `test` target now exercises the include-built extension via (2).

Drift-elimination is structurally guaranteed, not test-enforced: with one source file there is nothing to diverge.

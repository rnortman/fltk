# Design: `fegen-cst-rs-single-source`

Style note (applies to this doc): concise, precise, complete, unambiguous. No padding. Audience: smart LLM/human.

This is a design doc, not a spec. It refers to `request.md` (the spec) and `exploration.md` rather than restating them. Scope: a small drift-elimination refactor of file layout/build wiring. No production-logic change.

## Root cause / context

Two generated files are byte-for-byte identical (md5 `4bff0dbe...`, 6857 lines each, verified at HEAD `af6e6f3`; `exploration.md`'s figures — md5 `0a30124f...`, 5080 lines — predate commit `4c8f0ad`, which grew both copies identically) but committed as independent copies:

- `src/cst_fegen.rs` — generated by `make gen-rust-cst` (`Makefile:69-70`, command `python -m fltk.fegen.genparser gen-rust-cst $(GRAMMAR) $(RS_OUT)`); loaded via `mod cst_fegen;` (`src/lib.rs:4`) and registered into `fltk._native.fegen_cst` (`src/lib.rs:34`).
- `tests/rust_cst_fegen/src/cst.rs` — loaded via `mod cst;` (`tests/rust_cst_fegen/src/lib.rs:15`); compiled into the independent `fegen_rust_cst` cdylib (`Makefile:64-65`); consumed by `tests/test_phase4_fegen_rust_backend.py` (`:29`) and `tests/test_cross_backend_label_equality.py` (`:24`) via `pytest.importorskip("fegen_rust_cst")`, and by `tests/test_clean_protocol_consumer_api.py` via a try/except import plus `pytest.mark.skipif` (`:46-55`) — equivalent gating, different mechanism.

Both copies now have regeneration commands: commit `4c8f0ad` added `make gencode` (`Makefile:80-118`), which regenerates `src/cst_fegen.rs` (`Makefile:105`) and then `tests/rust_cst_fegen/src/cst.rs` (`Makefile:108-109`, the latter step carrying a `TODO(fegen-cst-rs-single-source)` comment marking it a duplicate). This mitigates but does not eliminate the foot-gun: the two files are still independent committed copies, `make gen-rust-cst` invoked on one leaves the other stale, and drift only stays invisible until someone runs `gencode` and diffs. After a partial regeneration the test crate compiles outdated generated code and the cross-backend tests validate the wrong thing — silently. This is the drift foot-gun the TODO `fegen-cst-rs-single-source` (the `## fegen-cst-rs-single-source` entry, `TODO.md:19-21`) targets.

Workspace note (post-`4c8f0ad` correction to `request.md`/`exploration.md`): the root `Cargo.toml:1-3` now declares `[workspace] members = [".", "crates/fltk-cst-core"]`. The test crate stays excluded via an empty `[workspace]` opt-out table in `tests/rust_cst_fegen/Cargo.toml:3` (the fixture crate likewise). Read the request's "NOT a Cargo workspace" constraint as: do not add the test crates to any workspace, and **preserve the empty `[workspace]` opt-out** — removing that seemingly pointless table makes cargo error on a non-member crate nested inside a workspace directory.

## Proposed approach

Make `src/cst_fegen.rs` the single source of truth. The test crate includes it via the Rust `include!` macro rather than holding a second copy.

Changes (no production code; build/layout only):

1. **Replace the contents of `tests/rust_cst_fegen/src/cst.rs`** (currently 6857 lines, identical copy) with a single line:
   ```rust
   include!("../../../src/cst_fegen.rs");
   ```
   `include!` resolves relative to the file containing the macro, so the path is relative to `tests/rust_cst_fegen/src/cst.rs`. Confirmed correct: `realpath --relative-to=tests/rust_cst_fegen/src src/cst_fegen.rs` → `../../../src/cst_fegen.rs`.

2. **Leave `tests/rust_cst_fegen/src/lib.rs` unchanged.** It keeps `mod cst;` (`:15`) and `cst::register_classes(m)?;` (`:21`). `mod cst;` loads the one-line `cst.rs`, whose `include!` pastes the canonical source into module `cst`'s scope. This is the standard `include!`-generated-file pattern.

3. **`src/cst_fegen.rs` is untouched** — still generated by `gen-rust-cst`, still committed, still subject to `make fix` formatting. It remains the sole regeneration target.

4. **Remove the second regeneration step from `make gencode`** (`Makefile:108-109`): delete both the comment line carrying `TODO(fegen-cst-rs-single-source)` and the `$(MAKE) gen-rust-cst ... RS_OUT=tests/rust_cst_fegen/src/cst.rs` command. After this change `cst.rs` is a hand-written one-liner, not generator output; regenerating it would clobber the `include!`. No replacement step is needed — `gencode`'s `src/cst_fegen.rs` step (`Makefile:105`) now covers the test crate by construction.

No new files, no workspace changes (the test crate's empty `[workspace]` opt-out stays — see workspace note above), no symlink, no build.rs, no copy step, no `.gitignore` change. `cst.rs` stays committed (now a one-liner). After this change, editing or regenerating `src/cst_fegen.rs` necessarily changes the test crate's compiled output because there is no second copy.

### Why this option over the alternatives

`request.md` already selects `include!`; `exploration.md` confirms feasibility of all three candidates. Rationale, grounded in exploration:

- **Symlink** — works under Cargo/maturin/git-on-Linux, but breaks on Windows with `core.symlinks=false` (git materializes the link as a text file containing the target path, which Cargo then tries to compile). `include!` has no platform dependency.
- **Makefile copy step** — either leaves a committed copy that can still drift, or `.gitignore`s `cst.rs` so the crate won't build from a clean checkout without a prior `make` step. `include!` keeps a clean-checkout build working with no extra step.
- **`include!`** — single committed source, plain-checkout build, platform-independent. Selected.

### Scope-of-edit note on the `include!` target

`src/cst_fegen.rs` opens with top-level `use` items (`src/cst_fegen.rs:1-6`: `use fltk_cst_core::Span;` then `use pyo3::...`). Under `include!` these become `use` items at the scope of module `cst` (because `mod cst;` + `include!` injects the text into module `cst`). That is valid Rust — `use` is legal at module scope. The `fltk_cst_core` dependency the included source needs is already declared in the test crate (`tests/rust_cst_fegen/Cargo.toml:20`, added by `4c8f0ad`), so the `use fltk_cst_core::Span;` resolves. The file is otherwise plain Rust with no `include!`-incompatible constructs (e.g. no `#![...]` inner attributes that must be crate-root; confirmed the file starts with `use`, not an inner attribute). Compilation is the verification gate (see Test plan).

## Edge cases / failure modes

- **Inner crate-level attributes (`#![...]`).** If `src/cst_fegen.rs` ever began with an inner attribute, `include!` into a non-root module would fail to compile. Today it begins with `use` (verified `:1` at HEAD `af6e6f3`, after `4c8f0ad`'s generator rewrite). The generator (`gsm2tree_rs.py`) is not in scope to change; if a future generator change adds inner attributes, the test crate build breaks loudly (not silently) — which is acceptable and far better than the current silent-drift failure mode.
- **Path fragility.** The relative path `../../../src/cst_fegen.rs` hard-codes the crate's location two directories under repo root plus `src/`. If `tests/rust_cst_fegen/` is relocated, the path must be updated; failure is a compile-time "file not found", loud and immediate.
- **`use` collision / unused-import warnings in the test crate.** `tests/rust_cst_fegen/src/lib.rs` already does `use pyo3::prelude::*;` (`:13`) and `use fltk_cst_core::{SourceText, Span};` (`:12`); the included module gets its own `use pyo3::prelude::*;` and `use fltk_cst_core::Span;`. These live in different module scopes (`lib` root vs. `cst`), so no conflict. `cargo clippy -q -- -D warnings` (`Makefile:45-46`) runs against the root workspace (`fltk-native` + `crates/fltk-cst-core`), not the test crate (excluded via its empty `[workspace]` opt-out), so test-crate import warnings, if any, do not fail `make check`. No action needed; noted for completeness.
- **Formatting of the one-liner.** `cst.rs` becomes hand-written (not generator output), but it is Rust, outside ruff's scope, and `cargo fmt` is not part of `make check` (`Makefile:8-21`). The one-liner is trivially well-formed; nothing in the gate formats it.
- **Stale `gencode` step would clobber the fix.** If change 4 (remove `Makefile:108-109`) were skipped, the next `make gencode` would overwrite the one-line `cst.rs` with a full regenerated copy, reintroducing the duplicate (visible only as a 6800-line diff at review time). Change 4 is therefore mandatory, not cleanup; test-plan step 5 verifies it.
- **TODO bookkeeping.** The `TODO(fegen-cst-rs-single-source)` code comment lives in the Makefile (`Makefile:108`, on the `gencode` step's comment line — placed there by `4c8f0ad`; it is not in `cst.rs` itself, contrary to `request.md`'s "top of cst.rs" assumption). It is deleted as part of change 4. The `TODO.md` entry (the `## fegen-cst-rs-single-source` heading + body, `TODO.md:19-21` — not `:23-25`, which is the unrelated live `rust-cst-pyi` entry) is removed in the same commit. After both, `grep -r 'fegen-cst-rs-single-source'` must return nothing.

## Test plan

No new automated tests; this is a build-wiring refactor whose contract is "the test crate compiles the canonical source." After the change, the verification surface is:

1. **Build the test crate:** `cd tests/rust_cst_fegen && uv run --group dev maturin develop` succeeds. This is the compile-time proof that `include!` works (the load-bearing unknown from `request.md` / `exploration.md`).
2. **Existing consumer tests pass — and actually run:** with the extension built (step 1, equivalently `make build-fegen-rust-cst`), `uv run pytest tests/test_phase4_fegen_rust_backend.py tests/test_clean_protocol_consumer_api.py tests/test_cross_backend_label_equality.py`. All three skip when the extension is unbuilt (two via `importorskip`, one via `skipif`), so confirm the pytest summary shows passes, not skips. These exercise the `fegen_rust_cst` extension and are the regression net proving the included source produces identical runtime behavior to the prior copy.
3. **Equivalence check (one-time, manual):** the text the test crate compiles is the canonical file — true by construction, since `include!` pastes the named file verbatim. A sanity assertion, not a standing test.
4. **Full gate unaffected:** `make check` still passes. The root-workspace `cargo check`/`clippy`/`test` targets do not touch the test crate (workspace opt-out, see Root cause), so they are unaffected; the Python `test` target now exercises the include-built extension via (2).
5. **`gencode` no longer touches `cst.rs`:** run `make gencode`; `git status` must show `tests/rust_cst_fegen/src/cst.rs` unmodified (still the one-liner). Verifies change 4 landed and the regeneration path cannot reintroduce the duplicate.

Drift-elimination is structurally guaranteed, not test-enforced: with one source file there is nothing to diverge.

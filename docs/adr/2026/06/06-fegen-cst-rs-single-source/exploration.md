## Feasibility of proposed fixes

### Symlink

- **Cargo**: Cargo follows symlinks transparently; a symlink at `tests/rust_cst_fegen/src/cst.rs` pointing to `../../../src/cst_fegen.rs` would work at compile time.
- **Git**: Git on Linux tracks symlinks as symlinks (storing the link target as a blob). This works correctly in typical Linux/macOS environments but breaks on Windows with `core.symlinks=false`. The relative path from `tests/rust_cst_fegen/src/cst.rs` to `src/cst_fegen.rs` is `../../../src/cst_fegen.rs`.
- **Bazel**: `BUILD.bazel` has no rules for `tests/rust_cst_fegen`; Bazel is documented as not currently supporting the Rust extension (`TODO(bazel-rules-rust)`). No Bazel blocker.
- **maturin**: Both crates are built independently via `maturin develop`; maturin just invokes cargo, which follows symlinks. No blocker.

### Makefile copy step

- Add a target that copies `src/cst_fegen.rs` to `tests/rust_cst_fegen/src/cst.rs` before (or as part of) `build-fegen-rust-cst`. The copy would still be a committed file unless `.gitignore` excludes it, in which case the test crate wouldn't build from a clean checkout without running the copy step first.
- Alternatively, keep `cst.rs` committed but add a `check` step that diffs the two and fails if they diverge (a guard, not a fix).

### Rust `include!` macro

- `include!` in Rust resolves paths relative to the file containing the macro, not the crate root. The relative path from `tests/rust_cst_fegen/src/cst.rs` to `src/cst_fegen.rs` is `../../../src/cst_fegen.rs`.
- `cst.rs` could be replaced with a one-liner: `include!("../../../src/cst_fegen.rs");`
- **Blocker**: `include!` pastes the file content verbatim into the calling module's scope. `src/cst_fegen.rs` begins with `use pyo3::...` imports at the top level — these would become inner `use` items inside the module, which is valid Rust. The content is otherwise pure Rust (no macro-incompatible constructs), so this approach is syntactically feasible.
- `tests/rust_cst_fegen/src/lib.rs:11` currently has `mod cst;` which loads `cst.rs` as a module. With `include!`, `cst.rs` would still be loaded via `mod cst;` and would include the source of `cst_fegen.rs` — this is the standard `include!` pattern for generated files and works.
- No Cargo workspace required; `include!` operates purely at the file level.
